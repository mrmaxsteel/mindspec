# ADR-0011: AgentMind as Unified OTLP Collector

- **Date**: 2026-02-16
- **Status**: Accepted
- **Domain(s)**: viz, bench, observability
- **Deciders**: Max
- **Supersedes**: ADR-0010
- **Superseded-by**: n/a

## Context

ADR-0009 established AgentMind as a real-time 3D visualization tool that accepts OTLP/HTTP on port 4318. ADR-0010 then introduced a separate headless recording collector on port 4319 to automatically capture per-spec telemetry. The bench system added a third collector on port 4320 for benchmark sessions.

This resulted in three independent OTLP/HTTP receivers:

| Component | Port | Purpose |
|-----------|------|---------|
| AgentMind | 4318 | Live visualization |
| Recording | 4319 | Per-spec NDJSON capture |
| Bench | 4320 | Benchmark NDJSON capture |

All three parse identical OTLP payloads using the same extraction functions (`bench.ExtractLogEvents`, `bench.ExtractMetricEvents`). The recording and bench collectors are literally the same `bench.Collector` struct on different ports. AgentMind's `LiveReceiver` re-imports those same extraction functions.

The practical consequence: during a benchmark run, telemetry flows to port 4320 and AgentMind on 4318 sees nothing. Operators cannot watch agent activity in real time during benchmarks. The same gap exists during normal spec work — telemetry goes to the recording collector on 4319, bypassing AgentMind entirely.

Claude Code can only export to one `OTEL_EXPORTER_OTLP_ENDPOINT`. Three receivers competing for that single endpoint is one receiver too many — arguably two too many.

## Decision

**AgentMind is the single OTLP receiver for all telemetry.** There is no separate headless collector. Port 4318 is the only OTLP port. AgentMind handles:

1. **Live visualization** (existing capability)
2. **NDJSON recording to disk** (new: `--output` flag)
3. **Benchmark collection** (new: auto-started by `bench run`)
4. **Per-spec recording** (new: auto-started by `record start` / `spec-init`)

When AgentMind is not running and telemetry arrives, the OTLP exporter silently drops events — this is standard OpenTelemetry behavior (connection refused, no retry storm, no visible errors). Nothing breaks.

### Key Changes from ADR-0010

| ADR-0010 Decision | ADR-0011 Replacement |
|:-------------------|:---------------------|
| Separate recording collector on :4319 | AgentMind on :4318 handles all collection |
| `record collect` as detached bg process | `agentmind serve --output <path>` as detached bg process |
| OTLP endpoint configured to :4319 | OTLP endpoint stays at :4318 |
| Bench collector on :4320 | Eliminated — bench uses AgentMind |
| No live viz during recording/bench | Live viz always available when AgentMind is running |

### What Stays the Same from ADR-0010

- **Spec-scoped recordings** at `docs/specs/<id>/recording/` — unchanged
- **Single append-only NDJSON file** per spec — unchanged
- **Phase markers** (`lifecycle.start`, `lifecycle.phase`, etc.) — unchanged
- **Manifest schema** — unchanged (PID now points to AgentMind process)
- **Zero-friction automatic capture** — unchanged (auto-start moves from `record collect` to `agentmind serve`)
- **SessionStart hook health-check** — unchanged (checks AgentMind PID instead)

### Auto-Start Behavior

Commands that need telemetry collection auto-start AgentMind if it is not already running:

| Command | Action |
|:--------|:-------|
| `spec-init` | Start AgentMind with `--output <spec-recording-dir>/events.ndjson` |
| `bench run` | Start AgentMind with `--output <bench-dir>/bench-events.jsonl` |
| `record start` | Start AgentMind with `--output <spec-recording-dir>/events.ndjson` |
| SessionStart hook | Health-check, restart if dead |

All print: `AgentMind started — watch live at http://localhost:8420`

This means operators always have a visualization URL available. They don't have to know about it or use it — but when they want to see what's happening, the UI is one click away.

## Consequences

### Positive

- **One port, one process** — eliminates port allocation conflicts and multiple-receiver complexity
- **Live viz during everything** — benchmarks, spec work, recording all visible in AgentMind
- **Simpler mental model** — "AgentMind receives all telemetry" vs. "three different receivers depending on context"
- **Free observability** — auto-started AgentMind means a viz URL is always available, even if the operator never explicitly asked for it
- **Less code** — removes the standalone collector process management for recording and bench

### Negative / Tradeoffs

- **AgentMind is heavier than a headless collector** — it maintains an in-memory graph, WebSocket hub, and embedded web UI even when no browser is connected. In practice the overhead is negligible (idle goroutines, no rendering without clients).
- **Single point of failure** — if AgentMind crashes, both visualization and recording stop. Mitigated by the existing SessionStart health-check that restarts it.
- **Port conflict with external tools** — if another tool claims :4318 (e.g., a real OTel Collector), AgentMind can't start. Mitigated by making the port configurable (`--otlp-port`).

## Alternatives Considered

### 1. Fan-Out from Headless Collector to AgentMind

Keep the headless collector as primary, add HTTP forwarding to AgentMind. Rejected because:
- Adds complexity (forward timeout, retry logic, error handling) instead of removing it
- Still two processes, two ports
- The collector and AgentMind share 90% of their code — maintaining both is redundant

### 2. Reverse: AgentMind Forwards to Headless Collector

AgentMind receives all events, forwards to a headless writer for NDJSON. Rejected because:
- Still two processes
- AgentMind already buffers events and can export NDJSON — adding a second writer is pointless

### 3. Keep Three Receivers, Add Fan-Out

Add fan-out to each receiver. Rejected because it multiplies complexity instead of reducing it. The root cause is too many receivers, not insufficient connectivity between them.

## Validation / Rollout

1. `agentmind serve --output /tmp/test.ndjson` writes NDJSON as events arrive
2. `bench run` auto-starts AgentMind, live viz shows benchmark activity, quantitative report generates correctly
3. `spec-init` starts AgentMind as recording collector, events appear in spec recording dir
4. AgentMind crash + SessionStart hook restart — recording continues with no data loss (append mode)
5. No references to port 4319 or 4320 remain in the codebase
6. `go test ./internal/viz/... ./internal/bench/... ./internal/recording/...` passes
