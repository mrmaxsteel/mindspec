# ADR-0009: AgentMind: Embedded Real-Time Agent Visualization

- **Date**: 2026-02-15
- **Status**: Accepted
- **Domain(s)**: viz, bench
- **Deciders**: Max
- **Supersedes**: n/a
- **Superseded-by**: n/a (amended by ADR-0011: AgentMind expanded to unified collector role)

## Context

MindSpec orchestrates spec-driven development with Claude Code agents. As agent workflows grow more complex — multiple concurrent agents, sub-agents spawned via Task tool, MCP server integrations — operators need real-time visibility into what agents are doing, what tools they're calling, and how resources are being consumed (tokens, cost, latency).

Existing observability tools (Grafana, Jaeger, Honeycomb) can ingest OTLP telemetry, but they present data as metrics dashboards and trace waterfalls — not as a live graph of agent activity. The mental model for agent orchestration is a graph (agents calling tools, tools reading files, agents spawning sub-agents), and no off-the-shelf tool renders that.

## Decision

Embed a real-time 3D agent activity visualizer ("AgentMind") directly in the MindSpec CLI as `mindspec agentmind serve`. AgentMind:

1. **Ingests OTLP/HTTP** on a configurable port (default 4318), reusing the same protocol Claude Code already exports telemetry to
2. **Normalizes events** into a typed graph model: nodes (agent, tool, mcp_server, data_source, llm_endpoint) and edges (model_call, tool_call, mcp_call, retrieval, write, spawn)
3. **Broadcasts** graph updates to connected browsers via WebSocket
4. **Renders** a 3D force-directed graph (Three.js / 3d-force-graph) with glow aesthetics, edge energy, particle effects, and floating token count animations
5. **Supports replay** of recorded NDJSON session files at variable speed
6. **Resolves multi-agent identity** from OTLP resource attributes and `session.id`, so multiple agents sharing a collector appear as distinct nodes

## Decision Details

### Graph Data Model

| Node Type | ID Pattern | Example |
|:----------|:-----------|:--------|
| agent | `agent:<name>` | `agent:claude-code`, `agent:research` |
| tool | `tool:<name>` | `tool:Read`, `tool:Bash` |
| mcp_server | `mcp:<name>` | `mcp:github` |
| data_source | `file:<path>` | `file:/src/main.go` |
| llm_endpoint | `llm:<model>` | `llm:claude-sonnet-4-5-20250929` |

| Edge Type | Meaning |
|:----------|:--------|
| model_call | Agent → LLM API request |
| tool_call | Agent → tool invocation |
| retrieval | Tool → file/data read |
| write | Tool → file/data write |
| mcp_call | Agent/tool → MCP server |
| spawn | Parent agent → child agent |

### Agent Identity Resolution

Agents are identified from OTLP resource attributes and event data in precedence order:

1. `agent.name` (resource attr) — explicit human-set label
2. `service.name` + `service.instance.id` (resource attrs) — OTLP standard
3. `service.name` + `session.id` (resource + data) — auto-differentiate instances
4. `service.name` alone
5. `session.id` alone — truncated to 8 hex chars for readable labels
6. Fallback: `claude-code`

Sub-agent hierarchy is expressed via `agent.parent` resource attribute, which creates `spawn` edges.

### Claude Code OTLP Field Inventory

As of Claude Code v1.x, the following fields are emitted in OTLP telemetry (verified Feb 2026):

**Resource attributes** (set once per process):

| Attribute | Value | Identity Use |
|:----------|:------|:-------------|
| `service.name` | Always `"claude-code"` | Used in precedence levels 2–4 |
| `service.version` | Semver string | Not used for identity |
| `os.type` | e.g. `"darwin"` | Not used for identity |
| `os.version` | OS version string | Not used for identity |
| `host.arch` | e.g. `"arm64"` | Not used for identity |

**Standard event (data) attributes** (per log record):

| Attribute | Value | Identity Use |
|:----------|:------|:-------------|
| `session.id` | UUID, unique per conversation | Used in precedence levels 3, 5 |
| `app.version` | Semver string | Not used for identity |
| `organization.id` | Org identifier | Not used for identity |
| `user.account_uuid` | User identifier | Not used for identity |
| `terminal.type` | e.g. `"iTerm2"` | Not used for identity |

**Notable absences**: No `process.pid`, no `parent.session.id`, no agent-type or sub-agent-role field. Claude Code does not distinguish parent from sub-agent in its standard telemetry.

**Custom resource attributes**: Users can inject arbitrary resource attributes via the `OTEL_RESOURCE_ATTRIBUTES` environment variable (e.g., `agent.name=research,agent.parent=orchestrator`). This is the only mechanism for explicit agent naming and hierarchy. However, Claude Code's Task tool does not currently expose a way to inject custom env vars into sub-agent processes, so sub-agents receive the same resource attributes as their parent.

**Practical consequence**: For automatic multi-agent differentiation, `session.id` is the only field that varies between parent and sub-agent. Each Claude Code session (including sub-agents spawned via Task tool) gets a unique UUID, which AgentMind truncates to 8 hex chars for readable node labels.

### OTLP as Ingestion Protocol

Claude Code already exports telemetry via OTLP (`OTEL_EXPORTER_OTLP_ENDPOINT`). By accepting OTLP directly, AgentMind requires zero additional agent configuration beyond pointing the exporter at the viz port. Standard `OTEL_RESOURCE_ATTRIBUTES` can set `agent.name` for multi-agent setups.

### Embedded Web UI

The web UI is embedded via Go's `embed` directive — a single `mindspec agentmind serve` command starts both the OTLP receiver and the web server. No separate frontend build step, no npm, no external dependencies.

## Consequences

### Positive

- Zero-config for single-agent: just `export OTEL_EXPORTER_OTLP_ENDPOINT=http://localhost:4318` and `mindspec agentmind serve`
- Graph mental model matches how agents actually work (tool calls, API calls, file I/O)
- Replay enables post-hoc analysis of recorded sessions
- Multi-agent identity is automatic via `session.id` — no manual setup for basic differentiation
- Embedded UI means single binary, no infrastructure to deploy

### Negative / Tradeoffs

- Duplicates some functionality that Grafana/Jaeger could provide (metrics, traces)
- 3d-force-graph is a heavyweight frontend dependency (~400KB CDN)
- Graph model is opinionated — may not map cleanly to non-Claude-Code agents
- No persistence — graph state is in-memory only, lost on restart (replay covers post-hoc needs)

## Alternatives Considered

### 1. Grafana + Custom Dashboard

Use standard OTLP → Grafana pipeline with custom panels. Rejected because Grafana excels at time-series metrics but doesn't render activity as an interactive graph. Building a Grafana panel plugin would be more effort than the embedded approach and would require users to deploy Grafana.

### 2. Jaeger/Tempo Trace Visualization

Use distributed tracing UI. Rejected because trace waterfalls show sequential call chains, not the concurrent graph topology of agent activity. Also requires deploying a trace backend.

### 3. Terminal-Only Visualization

Render agent activity in the terminal (TUI). Rejected because a 3D force-directed graph with particle effects requires a GPU-accelerated rendering context. Terminal UIs can show tables and trees but not spatial graphs.

## Validation / Rollout

1. `mindspec agentmind serve` starts cleanly, OTLP receiver accepts events
2. Single agent telemetry renders as expected graph topology
3. Two agents with different `session.id` values appear as separate nodes
4. Replay of recorded NDJSON produces identical graph to live observation
5. All tests pass: `go test ./internal/viz/... ./internal/bench/...`
