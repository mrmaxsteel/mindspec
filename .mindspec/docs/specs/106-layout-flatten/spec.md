# Spec 106-layout-flatten: Flatten .mindspec layout: promote specs/adr/domains, co-locate reviews, evict dogfood docs

## Goal

Promote the three lifecycle-artifact trees (`specs/`, `adr/`, `domains/`) and the authored `core/` reference out of the `.mindspec/docs/` wrapper to top-level `.mindspec/` children, co-locate panel review artifacts at `<spec-dir>/reviews/<panel-slug>/` (a tracked sibling of `recording/`), evict this repo's dogfood docs (`user/`, `installation/`, `research/`) to top-level `project-docs/`, and drop the vestigial `glossary.md` + `policies.yml`. The target user outcome is a `.mindspec/` whose top-level children are the artifact classes themselves â€” not one `docs/` bucket that conflates gated lifecycle artifacts, authored reference, and dogfood pollution â€” with reviews that travel with their spec, no orphaned repo-root `review/` tree (resolving the standing **adwu** friction), and reads of existing canonical/legacy projects that keep working byte-for-byte.

This is deliberately the careful, staged version: a 6-reviewer panel REJECTED a naive bundled "flatten everything now" proposal. The reorg ships as a no-move refactor first, then a net-new deterministic mover, then the moves â€” because the current `mindspec migrate` is a prompt emitter, not a transactional file mover, and the gate matchers, worktree resolution, and panel wiring all encode the physical layout in ways that a blind `git mv` would silently break.

## Background

`.mindspec/docs/` today conflates three content classes that have different lifecycles and different consumers:

1. **Lifecycle artifacts** â€” created and gated by `mindspec` commands; present in every user project: `docs/specs/`, `docs/adr/`, `docs/domains/`. These are read by the doc-sync lane (`internal/validate/docsync.go`), the ADR-divergence lane (`internal/validate/divergence.go`), and the ownership loader (`internal/validate/ownership.go`).
2. **Authored reference** â€” `docs/context-map.md`, `docs/core/` (ARCHITECTURE, CONVENTIONS, MODES, USAGE, DOCS-LAYOUTâ€¦).
3. **Dogfood pollution** â€” NOT part of the canonical layout; present only because this repo IS the mindspec tool: `docs/user/` (AGENTS.md, CLAUDE.md, README.md, ROADMAP.md), `docs/installation/`, `docs/research/` (verified on disk: `.mindspec/docs/{user,installation,research}/`).

Two further problems motivate this spec:

- **Homeless review artifacts.** Panel reviews live at repo-root `review/<slug>/` (42 tracked files today), with no owning spec and no owned home. `internal/panel/panel.go` `Scan` globs `review/*/panel.json` (panel.go:125) and the `mindspec complete` gate scans that repo-root tree; `internal/validate/divergence.go` `isProcessArtifact` hard-excludes `review/` (divergence.go:280). The rootless tree is the cause of the recurring lifecycle-approve deletion friction (the standing **adwu** bead).
- **Vestigial files.** `glossary.md` (`.mindspec/docs/glossary.md`) and `policies.yml` (`.mindspec/policies.yml`) survive even though ADR-0018 removed both subsystems; both have zero non-test binary consumers. A stale glossary reference also lingers at `internal/instruct/templates/spec.md:35` (`- Add glossary entries (\`GLOSSARY.md\`)`).

**Target layout.**

```
.mindspec/
  specs/<id>/{spec.md, plan.md, context-pack.md, recording/, reviews/<panel-slug>/...}
  adr/
  domains/
  core/                 # authored reference (flattened, name KEPT â€” not renamed to reference/ here)
  context-map.md
  config.yaml  session.json   # state/config (policies.yml DROPPED â€” vestigial)
  lineage/  migrations/
project-docs/           # dogfood docs evicted OUT of .mindspec â€” user/, installation/, research/
```

**Why this is net-new engineering, not a `git mv`.** `cmd/mindspec/migrate.go` is a prompt EMITTER (it scans markdown and prints an agent prompt; its own rubric still routes user-docs to `.mindspec/docs/user/` and agent files to `.mindspec/docs/agent/`, migrate.go:286-287). `internal/doctor/migration.go` only VERIFIES migration metadata. There is no transactional git-mv/link-rewrite framework to reuse, so the mover is new code. The single `workspace.DocsDir` join-point (workspace.go:116, falling back `CanonicalDocsDir â†’ LegacyDocsDir`) currently funnels `ADRDir`, `DomainDir`, and `ContextMapPath`; flattening dissolves that one join into per-artifact resolvers. `TreeRootForSpecDir` asserts `filepath.Base(docs) != "docs"` and returns `""` for a flat spec dir (workspace.go:214), which would regress the mindspec-ew79 cross-worktree ADR-visibility fix. The gate matchers are git-DIFF-STRING classifiers, not filesystem reads â€” `docsync.go` `HasPrefix` (isDocFile docsync.go:330, the `.mindspec/docs/specs/<id>/` literals at docsync.go:485-488/582/610, the cmd-docs accept-set docsync.go:534-538), `ownership.go` `TreeDirsAtRef(".mindspec/docs/domains")` (ownership.go:198), and `divergence.go` â€” and they cannot be absorbed by the accessors. `internal/contextpack/budgeter.go` hardcodes `.mindspec/docs/specs/<id>` and `.mindspec/docs/domains/<d>/<kind>` (budgeter.go:170/218), so a flat project would feed it empty dirs. Roughly 99 exact-slash-form `.mindspec/docs/{specs,adr,domains,core}` occurrences exist across 21 non-test Go files (~110 once the `filepath.Join(".mindspec","docs",â€¦)` forms are counted), plus 5+ harness scenario files; the true blast radius is in the low hundreds, not "15-20."

**Staged delivery (the spine the plan decomposes).**
- **Phase 1 â€” no-move refactor** (fully green, ZERO layout change, revertible): per-artifact accessors + a diff-path layout-classifier predicate + path-shape-aware `TreeRootForSpecDir`/`SpecDir` worktree tier + `guard.go`, all landed against the EXISTING two forms (canonical + legacy).
- **Phase 2 â€” net-new mover + link-rewriter**: transactional, history-preserving, idempotent git-mv driver with rollback; a finite-pattern markdown link-rewriter; a lineage writer; a run-state machine; a doctor link-existence check â€” all under golden-file tests over a captured copy of the real `.mindspec/docs` tree covering every cross-link class.
- **Phase 3 â€” the moves**: add the FLAT tier to the resolvers + `DetectLayout`; do dogfood-eviction + vestigial-drop FIRST as the low-stakes exercise of the mover, then the `specs/adr/domains/core` flatten + `context-map`, then reviews co-location; wire the merge-time guard; emit the deprecation Warn.
- **Governance**: amend DOCS-LAYOUT.md; author a layout-v2 ADR; amend ADR-0037 (reviews location); ADR-0018-consistent vestigial drops.

This is a forward-only migration: ADR-0023's lifecycle authority cannot rewind, so the design treats "flatten" as a one-way cut and enforces the drain at merge time rather than trusting git's heuristic rename detection.

## Requirements

1. **Per-artifact three-tier resolver; non-breaking reads.** Add a FLAT tier to every artifact accessor in `internal/workspace`, ordered FIRST in resolution PRECEDENCE (first-exists-wins): flat (`.mindspec/specs/<id>`, `.mindspec/adr/`, `.mindspec/domains/<d>`, `.mindspec/context-map.md`, `.mindspec/core/`) â†’ canonical (`.mindspec/docs/â€¦`) â†’ legacy (`docs/â€¦`). "FIRST" here is read-order precedence, NOT delivery order: the flat directories do not physically exist until the Phase-3 moves, so a Phase-1 tree (canonical or legacy) resolves exactly as it does today. The single `DocsDir` join-point (workspace.go:116) that today funnels `ADRDir` (workspace.go:188), `DomainDir` (workspace.go:235), and `ContextMapPath` (workspace.go:178) MUST dissolve into per-artifact resolvers; `SpecDir` (workspace.go:153) gains the flat tier alongside its existing worktree/canonical/legacy tiers. Reading an existing canonical OR legacy project MUST stay byte-for-byte non-breaking â€” a project with no flat tree resolves exactly as it does today.

2. **Whole-tree `DetectLayout(root) â†’ {flat | canonical | legacy | greenfield | mixed}`, computed once, drives the write-default.** `DetectLayout` MUST distinguish the two non-flat WRITE shapes â€” `canonical` (a `.mindspec/docs/â€¦` lifecycle tree exists) and `legacy` (a root `docs/â€¦` lifecycle tree exists) â€” because they are DISTINCT write targets, not one "legacy" bucket; collapsing them cannot drive the never-split write-default for a canonical-only vs a legacy-only project. The write-side "born flat" decision MUST key off this WHOLE-TREE classification ("no canonical/legacy lifecycle tree exists ANYWHERE under root"), never a per-ID existence probe â€” a per-ID probe cannot distinguish a greenfield project from a brand-new spec inside an existing legacy project, so it would split a project's layout. `greenfield` and post-migration `flat` projects write flat; a `canonical` or `legacy` project keeps writing to its existing form until migrated. `mixed` â€” ANY flat lifecycle tree coexisting with ANY canonical/legacy lifecycle tree (whether different spec ids OR different artifact classes, e.g. a flat `specs/` beside a canonical `adr/`) â€” is a HARD ERROR: first-exists-wins reads would mask the residue while writes go flat, reintroducing the exact half-old/half-flat split the panel rejected. The ONLY exception is a recorded in-progress migration recovery (a live `.mindspec/migrations/<run-id>/` run-state, Req 4), where a transient mixed tree is expected and the recovery path â€” not first-exists-wins â€” drives resolution. `mindspec init`/`bootstrap` create the flat directories for new projects (replacing the bootstrap.go:222-223 `.mindspec/docs/{domains,specs}` manifest entries).

3. **All root-enumerating `DocsDir` consumers become layout-tier-aware (incl. the context-pack budgeter).** Beyond the named accessors (Req 1), EVERY consumer that joins onto the `DocsDir` root or enumerates under it MUST resolve through a layout-aware accessor, or it silently reads an empty flat tree: `mindspec spec list`, `mindspec domain list`/`show`, context-map enrichment, the authored `core/` doc paths, and the doctor orphan/docs scans (`internal/doctor/docs.go`, also Req 9). In particular `internal/contextpack/budgeter.go` hardcodes `.mindspec/docs/specs/<id>` (budgeter.go:170) and `.mindspec/docs/domains/<d>/<kind>` (budgeter.go:218); on a flat project it feeds those joins empty directories and SILENTLY drops the spec + domain context from the assembled pack. The budgeter MUST consume the per-artifact accessors so a context pack assembles identically â€” same sections, same bytes â€” on a flat vs canonical vs legacy project.

4. **Net-new deterministic, transactional, idempotent mover with a DEFINED run-state machine + rollback semantics.** Build a history-preserving git-mv driver (routed through `internal/executor` per ADR-0030, not direct shell-out) that is idempotent (re-running on an already-flat tree is a no-op) and lands each artifact-group move as TWO commits: a pure 100%-similarity `git mv` first, then the link-rewrite (Req 5) second â€” so `git log --follow` and 3-way-merge rename detection stay reliable. It writes a lineage manifest under `.mindspec/lineage/` and a per-run state record under `.mindspec/migrations/<run-id>/` consistent with the existing doctor migration-metadata schema (`internal/doctor/migration.go`).
   **Run-state machine + crash recovery (DEFINED, not merely named).** The run records an ordered, durable checkpoint at every mutation boundary: (a) pre-run base ref captured; then for EACH artifact group â€” (b) before the `git mv`, (c) after the `git mv` / before the pure-move commit, (d) after the pure-move commit, (e) after the link-rewrite, (f) after the rewrite commit; then (g) the lineage/state finalize. Required RESUME behavior after a crash: the resumer reads the last durable checkpoint and â€” at (b)/(c) discards or re-stages the working-tree-only partial `git mv` (nothing is committed yet) and retries the group; at (d)/(f) detects the commit already landed and SKIPS it (idempotent); at (e) re-runs the deterministic link-rewrite (a pure function of the moved tree) and commits. **Rollback semantics (CHOSEN, in git terms): hard-reset-to-pre-run-ref while nothing is published.** On any failure or `--abort` BEFORE the run's commits are pushed/merged, the mover hard-resets the branch to the (a) pre-run ref and cleans the working tree â€” NO compensating revert commits. AFTER the run's commits are published, the mover REFUSES to auto-rollback and instead emits the forward-only recovery line (re-migrate / rebase), consistent with ADR-0023.

5. **Net-new finite-pattern markdown link-rewriter, gated by a doctor link-existence check that scans EVERY link.** Rewrite ONLY the closed, breaking subset of relative links â€” NOT a general markdown parser. Because the flatten is SYMMETRIC (specs and adr both shed the same `docs/` level), specâ†’ADR links like `../../adr/ADR-NNNN.md` are PRESERVED unchanged; the breaking set is small and enumerable: one-level `../adr/` cases, absolute `docs/core/` and `.mindspec/docs/core/`/context-map references, review-co-location links whose relative depth changed under the `<spec-dir>/reviews/` move, dogfood cross-links whose depth changed under eviction, and repo-root `README.md`/`AGENTS.md` references that point INTO the moved trees from OUTSIDE them. The GATING doctor link-existence check MUST scan EVERY markdown link in the post-migration moved tree AND the affected root docs â€” NOT merely the finite set the rewriter touched â€” and FAIL the migration (and the run-state machine, Req 4) if ANY link 404s, so a breaking link the finite rewriter failed to anticipate is caught at the gate rather than shipped.

6. **Gate matchers are git-DIFF-string classifiers, permanently multi-prefix.** The doc-sync, ownership, and ADR-divergence lanes match diff PATH STRINGS, not filesystem reads, so the accessors cannot absorb them. Introduce a relative-path layout-classifier predicate and make every git-ref matcher recognize ALL THREE prefixes (flat + canonical + legacy): `isDocFile`/`isSourceFile` (docsync.go:330/338), the `.mindspec/docs/specs/<id>/` literals in `validateSpecArtifactSync`/`specMDID` (docsync.go:561-622), the domain-doc prefixes in `checkInternalPackages` (docsync.go:485-488), the cmd-docs accept-set (docsync.go:534-538), `listDomainDirs`/`TreeDirsAtRef(".mindspec/docs/domains")` (docsync.go:352, ownership.go:198), `LoadOwnership`'s manifest path (ownership.go:79), AND â€” CRITICALLY â€” the ref-anchored ownership pair `LoadOwnershipAtRef` (ownership.go:150) + `domainManifestRelPath` (ownership.go:123-125), plus `isProcessArtifact` (divergence.go:277-281). The `complete` ADR-divergence gate loads ownership through the ref-anchored `LoadOwnershipAtRef`/`domainManifestRelPath` pair (NOT `LoadOwnership` at :79); updating only :79 leaves every domain manifest reading "missing" on a flat tree, so every 106 bead would hard-block `adr-divergence-unowned`. This multi-prefix posture is PERMANENT and DECOUPLED from the resolver-tier lifecycle: historical refs, old branches, and external forks never stop emitting legacy paths, so a "drop legacy tiers after N releases" deprecation applies ONLY to filesystem reads (Req 1), NEVER to git-ref matchers. The predicate MUST also classify `project-docs/**` as non-source docs (Req 14) so dogfood eviction does not trip `adr-divergence-unowned` or the doc-sync source lane; the cmd-docs accept-set must accept the post-flatten `core/USAGE.md` and the evicted `project-docs/user/**`.

7. **Worktree-aware fixes + regression tests.** `SpecDir`'s worktree tier must additionally probe the flat worktree shape (`.worktrees/worktree-spec-<id>/.mindspec/specs/<id>`). `TreeRootForSpecDir` (workspace.go:206-222) must recognize the flat `.mindspec/specs/<id>` shape â€” its current `filepath.Base(docs) != "docs"` assertion (workspace.go:214) returns `""` for a flat dir and would regress the mindspec-ew79 cross-worktree ADR-visibility fix. `guard.go` worktree-add recipes must be layout-aware. Regression tests MUST pin both worktree shapes.

8. **Reviews co-location wiring.** `internal/panel.Scan` and `PanelDirScanRoot` (panel.go:118-142, gate.go:420-422) move from the repo-root `review/*/panel.json` glob to the spec-scoped `<spec-dir>/reviews/*/panel.json` glob (a sibling of `RecordingDir`, workspace.go:244-250); the `mindspec complete` panel-gate scan becomes spec-scoped instead of repo-root. `divergence.go`'s `HasPrefix(path,"review/")` exclusion (divergence.go:280) becomes a `/reviews/`-segment exclusion under spec dirs. Update the `ms-panel-run` + `ms-panel-tally` skill bodies, their `historical_skills` snapshots (`internal/setup/skills.go`), and amend ADR-0037 for the new `panel.json` location. Reviews are TRACKED (durable audit, travel with the spec); the root `review/` tree is eliminated, resolving **adwu**.

9. **Doctor: real layout detection + dual-layout duplicate-ID error.** `mindspec doctor` MUST gain a real layout detector (reusing `DetectLayout`), a `would-migrate-layout` Warn (analogous to the existing `would-migrate: spec=â€¦` Warn, migration.go:101-106), and an ERROR when the SAME spec id exists under two layouts simultaneously (kills the stale-duplicate read hazard the first-exists-wins resolver would otherwise mask). `checkDryRunMigration` (migration.go:39) and the orphan/docs scans (docs.go via `workspace.DocsDir`) must become tier-aware rather than hardcoded to `.mindspec/docs/specs`.

10. **Merge-time layout-fingerprint HARD-FAIL.** `internal/complete` `MergeBranch` (and any specâ†’main / beadâ†’spec merge) MUST compute a layout fingerprint of source and target and HARD-FAIL a cross-layout merge (nested source onto flat target, or vice versa) with a "rebase onto post-flatten main" recovery line, instead of trusting git's heuristic rename / directory-rename detection. This makes the pre-flatten drain enforced, not advisory (ADR-0023 forward-only).

11. **`migrate layout` precondition: a feasible, testable pre-flatten branch/PR discovery algorithm.** The flatten command's precondition is the absence of any unmerged BRANCH or PR that predates the flatten â€” NOT "zero worktrees" (unreachable in this repo, and mis-attributed to impl-approve). DISCOVERY ALGORITHM (deterministic source-of-truth): scan (1) local branch refs, (2) remote-tracking refs (`refs/remotes/<remote>/*`), and (3) hosted PR metadata when a remote + token are available; compute a flatten base/fingerprint from the post-flatten layout SIGNATURE (does the ref's merge-base tree still carry the canonical/legacy `.mindspec/docs/` lifecycle layout, vs the flat shape?). A ref BLOCKS the migration iff it is unmerged into the target AND its fingerprint is pre-flatten. OFFLINE / no-remote behavior: the scan degrades to local + remote-tracking refs only and WARNS that hosted PRs could not be consulted (it does NOT silently pass). The command MUST require a clean idle working tree before mutating. Locked agent worktrees and long-lived external forks are explicitly TOLERATED â€” they cannot be drained, only fingerprint-guarded at merge (Req 10) â€” so the precondition MUST NOT count them as blockers.

12. **Skills + snapshots correctness on flat projects.** `ms-spec-grill`'s SKILL.md reads the domain set from `.mindspec/docs/domains/` (SKILL.md:30/38/40) and computes the next ADR number by scanning `.mindspec/docs/adr/` (SKILL.md:31); on a flat project these read empty dirs and mis-fire (reject-invented-domains and next-ADR-number both break) â€” this is the CENTRAL correctness claim of this requirement, so the skill MUST read the layout-correct path. Also update the `lifecycleSkillFiles` literals, the `setup/claude.go` and `setup/codex.go` generated text, the other path-bearing skill bodies (ms-panel-run/tally, ms-bead-impl/cycle/fix, ms-spec-final-review) and plugin README/FINDINGS + bootstrap text, and EVERY path-bearing `historical_skills` snapshot in `internal/setup/skills.go`. The plan MUST either update each path-bearing artifact or explicitly classify it out of scope with a recorded reason.

13. **The moves + the drops.** Flatten `specs/`, `adr/`, `domains/`, and `core/` to top-level `.mindspec/` children; move `context-map.md` to `.mindspec/context-map.md`. KEEP the name `adr` (load-bearing across `ADRDir`, `ADR-NNNN` filenames, `mindspec adr`, and ~150+ inbound links) and KEEP the name `core` (do NOT rename to `reference/` in this spec). Evict dogfood docs (`user/`, `installation/`, `research/`) to top-level `project-docs/`. - Renaming `adr` to `decisions` or `core` to `reference` â€” both names are KEPT (Req 13). A `reference/` rename is a separate future spec.

## Non-Goals

- This spec does not aim to remove the canonical/legacy resolver tiers; backward-compatible reads are a goal, not a casualty.
- This spec does not redesign the panel decision matrix or the ADR-0037 contract semantics; it relocates the artifact and the scan root only.
- This spec does not generalize the link-rewriter into a markdown parser; it rewrites a closed, enumerable breaking subset and gates on a 404 check.
- This spec does not claim a single atomic cut-over; the moves land staged (dogfood/vestigial first), behind a behavior-preserving Phase-1 refactor.
