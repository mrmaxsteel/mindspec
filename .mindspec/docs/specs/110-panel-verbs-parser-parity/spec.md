# Spec 110-panel-verbs-parser-parity: Panel lifecycle verbs (create/verify/tally) as the agent-neutral contract + spec-approve parser parity

## Goal

Make the review-panel lifecycle a set of **in-binary CLI verbs** instead of skill prose and hand-typed JSON, and close the **spec-approve parser asymmetry** that lets a mis-formatted spec pass its own gate and detonate the next two.

Two concrete outcomes:

1. **`mindspec panel create | verify | tally`** — the panel lifecycle that today lives in `ms-panel-run` step 0 and `ms-panel-tally`'s decision matrix becomes three agent-neutral commands. `create` writes the panel directory + BRIEF stub + `panel.json`, stamping `expected_reviewers` / `approve_threshold` from spec 109's config resolvers and capturing `reviewed_head_sha` **atomically with the round number** (the round+SHA co-bump invariant that `ms-panel-run` enforces only by prose today). `verify` is a read-only completeness/staleness report that prints the **same** PASS/BLOCK the complete-gate computes. `tally` renders the decision **from the binary** — verdict table, decision, exit 0 on Allow / non-zero on Block with recovery lines — so `internal/panel/gate.go` is the single home and the tally skill shrinks to consolidation judgment. This is spec 109's layering ratchet (ADR-0040) in action: a rule that lived in a skill proves load-bearing and ratchets **down** into a binary verb behind a stable artifact + CLI contract.

2. **Spec-approve parser parity** — `mindspec spec approve` additionally validates the spec with the **same canonical parsers the downstream gates consume**: `contextpack.ParseSpec` domain extraction resolved through the shared `normalizeImpactedDomains` (so every declared Impacted-Domain is a real, OWNERSHIP-resolvable name), and `## ADR Touchpoints` references resolved against the same `adr.Store` the plan-time citation gate reads (so a cited ADR actually exists). This catches the spec-108 Impacted-Domains formatting incident **at spec-approve**, where it is cheap, instead of at plan-approve and the bead-divergence gate, where it detonated two gates a phase later.

The empirical motivators are this week's 107/108/109 runs: `panel.json` was hand-written roughly ten times with the round+SHA co-bump enforced only by operator discipline; a `sed`-patched prompt nearly clobbered a verdict file; codex verdicts were lost to a sandbox-cwd mistake; the decision matrix was re-stated in tally-skill prose that could drift from `gate.go`; and a `## Impacted Domains` bullet whose formatting `ParseSpec` extracted as garbage sailed through spec-approve and only failed at plan-approve's `adr-coverage-missing` and the bead's `adr-divergence-unowned`. After this spec, each of those is a binary invariant or an at-source validation error.

## Background

MindSpec already enforces the panel gate in the binary: `internal/panel.PanelGateDecision` is the single, pure decision function that both `mindspec complete` and the (now-retired) PreToolUse hook invoke over identical facts (ADR-0037, relocated to `gate.go` in spec 099). What is **not** yet in the binary is the panel *lifecycle around* that decision — creating the registration, re-capturing the reviewed SHA on a re-panel, and rendering the tally. Those live in `ms-panel-run` / `ms-panel-tally` prose plus a hand-typed `panel.json`, which is exactly the "same rule in two places" drift ADR-0040 (spec 109) names: the decision matrix is stated in both `ms-panel-tally` prose **and** `gate.go`, and the round+SHA co-bump invariant is stated in `ms-panel-run` prose with no mechanical guard. Spec 109 landed the substrate these verbs consume — the `panel:` config block, the `PanelExpectedReviewers()` / `PanelApproveThresholdExpr()` resolvers, and the optional recorded `approve_threshold` field interpreted solely by `panel.ApproveThreshold()`. Spec 110 is the writer and the readers that call them.

**ADR-0040 is the *why*, not a citation.** ADR-0040 (the four-layer ratchet + the portability principle: agents integrate at the artifact + CLI contract level, never at the prompt-format level; runners are adapters behind those contracts) lands as Bead 1 of the **approved** spec 109 and governs this spec directly — the `panel.json` / verdict-JSON schemas and the `mindspec panel` verbs are precisely the artifact + CLI contract ADR-0040 says a runner targets. But ADR-0040 does **not exist as a file on `main`** until spec 109 merges, and this very spec introduces the rule (R6) that a `## ADR Touchpoints` reference must resolve to an existing ADR file. Citing ADR-0040 as a touchpoint here would fail the check this spec models, so 110 references ADR-0040 in prose (here and in the ADR-0037 touchpoint below) and cites only the already-Accepted enforceable set. Spec 109 is a hard prerequisite (see Open Questions), so ADR-0040 exists in 110's base by plan and implementation time; the plan may cite it in frontmatter for coverage, though the existing workflow ADRs already cover the sole impacted domain.

This spec deliberately stops at the verbs and the parity gate. It does **not** add the workflow-backed panel runner or any `runner:` adapter dispatch (spec 111), instruct-template generation from gate definitions (a dedicated future spec), or any change to `PanelGateDecision`'s semantics.

## Impacted Domains

- **workflow**: the sole impacted domain — every source edit lands under the `workflow` OWNERSHIP glob. `internal/panel` gains a leaf-safe `panel.json` **writer** (`Create`) and the verdict-file/slot **schema doc**, while remaining an import-clean, config-free leaf (the writer takes plain values, never `internal/config`). `cmd/**` gains the new `mindspec panel create | verify | tally` command tree, its git-and-config wiring, and the renderers. `internal/approve` (`spec.go` / `ApproveSpec`) and `internal/validate` (`spec.go` / `ValidateSpec`) gain the spec-approve parser-parity checks — the domain-resolution check reuses `internal/validate`'s existing `normalizeImpactedDomains` + `contextpack.ParseSpec`, and the ADR-Touchpoint check reuses the `adr.Store`. `plugins/mindspec/skills/ms-panel-run` and `.../ms-panel-tally` (mirrored under `.claude/skills/**`) shrink to remove the now-mechanized prose. All of these paths are claimed by `.mindspec/domains/workflow/OWNERSHIP.yaml`.

Note — three domains are consumed **read-only and are therefore NOT impacted** (no source in them changes), exactly as spec 109 consumed nothing outside its own two domains. This note is deliberately plain prose, not bullet points, so `contextpack.ParseSpec` extracts exactly one Impacted-Domain (`workflow`) and the parity check this spec introduces (R5) resolves it cleanly. **core** (`internal/config`): the `panel create` verb only *reads* the `PanelExpectedReviewers()` / `PanelApproveThresholdExpr()` resolvers and the recorded `approve_threshold` field spec 109 added — no `internal/config` source is edited. **context-system** (`internal/contextpack`): the parity check *calls* `contextpack.ParseSpec` unchanged — this is the established architecture where the parser stays dumb and domain resolution lives in the `internal/validate` gate layer (`ownership_resolve.go` pins it: "ParseSpec/contextpack is deliberately NOT made to depend on validate"). **execution** (`internal/executor`): the `panel create` verb resolves `reviewed_head_sha` via `newExecutor(root).RevParseRef` (the git-I/O boundary, ADR-0030), and consuming an executor method is not a source change. No path outside the `workflow` OWNERSHIP globs is edited as source, so no other domain claim is needed and none is trippable by `adr-divergence-unowned`.

## ADR Touchpoints

- [ADR-0037](../../adr/ADR-0037-panel-gate-enforced-contract.md): Panel Gate as Enforced Contract (Domains: workflow, execution). The spec's spine. `panel verify` and `panel tally` must render the **identical** decision `mindspec complete` enforces — they reuse `panel.PanelGateDecision` / `panel.ResolveGateFacts` (the single home §3 pins) and add no second interpreter, so "identical decision over identical facts" holds across the new call sites. `panel create` writes the `panel.json` the gate reads, stamping the optional `approve_threshold` field spec 109's ADR-0037 amendment introduced. The panel verbs are the executable form of this contract; per ADR-0040 (landing in spec 109) they are the artifact + CLI contract a non-Claude-Code runner targets. The `create` verb's target rev-parse routes through the executor git-I/O boundary (ADR-0030), keeping `internal/panel` a leaf. §§6/8 (fail-open/closed, trust boundary — plain reviewable files, no signing) are unchanged.
- [ADR-0035](../../adr/ADR-0035-agent-error-contract.md): Agent Error Contract — recovery lines and exit codes (Domains: workflow, execution, core). `panel tally`'s non-zero Block, and every new spec-approve validation error (unresolvable Impacted-Domain, missing ADR touchpoint), must be guard-style failures carrying a final recovery line — never a panic or a bare non-zero. The Block reuses the decision message's existing raw-`git merge` fence plus a genuine re-panel recovery command, exactly as `internal/complete.panelGate` already does.
- [ADR-0036](../../adr/ADR-0036-ownership-discovery.md): Ownership Discovery — zero framework cognition for OWNERSHIP.yaml (Domains: workflow, validation, doc-sync, ownership). Governs the domain-resolution parity check: an Impacted-Domains entry is resolved to its owning-domain NAME against every domain's OWNERSHIP `paths:` via the shared `normalizeImpactedDomains` (spec 100 R1), and zero/ambiguous owners is a hard error. Also governs the gate-forward doc-sync constraint — every bead editing `workflow` source appends to a `workflow` domain doc.
- [ADR-0032](../../adr/ADR-0032-adr-semantic-gates.md): Semantic ADR Coverage Gates (Domains: validation, adr, lifecycle, workflow). Governs the ADR-Touchpoint reference-resolution parity check (the `adr-cite-missing` shape, moved earlier to spec-approve over the touchpoints prose) **and** the honest boundary this spec draws: the touchpoint check verifies only *existence*; Accepted-status, domain-intersection (`adr-cite-irrelevant`), and coverage (`adr-coverage-missing`) stay at plan-approve. This spec does not resurrect touchpoints-as-citation-source (spec 097 R2 moved plan citations to plan frontmatter).
- [ADR-0034](../../adr/ADR-0034-ceremony-collapse.md): Ceremony Collapse — single-bead lifecycle (Domain: workflow). The panel verbs operate inside the single-bead lifecycle this ADR collapsed; `panel tally`'s pass hands off to the same `mindspec complete` merge terminal, and the verbs must not add a new ceremony step — they replace hand-typed JSON with one command, not add a gate.
- [ADR-0039](../../adr/ADR-0039-flat-layout-v2.md): Flat `.mindspec/` Layout v2 (Domains: core, workflow, execution, context-system). `panel create` must write the panel directory under the layout-aware location `panel.Scan` already reads (co-located `<spec-dir>/reviews/<slug>` on a flat tree, repo-root `review/<slug>` otherwise), resolved through `workspace.SpecDir` — it introduces no third convention.

## Requirements

1. **`mindspec panel create <slug> --spec <id> --target <ref> [--bead <id>] [--round N]` — atomic registration writer.** The command creates the panel directory (via `workspace.SpecDir`, honoring the flat/legacy layout), writes a `BRIEF.md` **stub** (the required section headers plus the machine-derivable fields — slug, round, branch, and the resolved commit SHA — with placeholders for the judgment prose the skill fills), and writes `panel.json` in a **single** write whose `expected_reviewers` comes from `config.PanelExpectedReviewers()` and whose `approve_threshold` comes from `config.PanelApproveThresholdExpr()` (spec 109's resolvers, read-only), with `bead_id` from `--bead` (null when absent), `spec`, `target`, `round` (default `1`), and `reviewed_head_sha` from `newExecutor(root).RevParseRef` of the live `--target` ref captured **at write time**. `round` and `reviewed_head_sha` are written **together, by construction** — there is no code path that writes one without the other. Re-invocation with `--round N+1` is the re-panel bump: the new round and a **freshly re-resolved** `reviewed_head_sha` are written in the same operation, and **prior-round verdict files (`<slot>-round-<K>.json`, `K < N+1`) are left untouched**. Falsified if any `panel.json` `create` writes omits `reviewed_head_sha`, if a re-panel bumps `round` without re-resolving `reviewed_head_sha` (or vice versa), if `expected_reviewers` / `approve_threshold` come from anywhere but the 109 resolvers, or if a re-panel deletes or overwrites a prior round's verdict file.

2. **`mindspec panel verify <slug>` — read-only completeness/staleness report, decision-identical to the gate.** The command prints, for the named panel: verdicts present vs `expected_reviewers`, per-slot parse status (malformed files named), `reviewed_head_sha` vs the live target tip (staleness), and a **PASS/BLOCK preview that is the same `panel.PanelGateDecision` outcome `mindspec complete` would compute** over the same facts — reusing `panel.ResolveGateFacts` + `panel.PanelGateDecision` (or the `instruct --panel-state` `PanelStateEntry` renderer that already wraps them), **not** a second decision implementation. It writes nothing and exits `0` (a read-only report is not itself a gate). Falsified if `panel verify` writes any file, if its PASS/BLOCK verdict diverges from `panel.PanelGateDecision` for identical facts, or if it introduces a second copy of the decision logic.

3. **`mindspec panel tally <slug>` — the decision, from the binary.** The command runs `panel.PanelGateDecision` over the recorded facts and prints (a) a per-slot verdict table (slot, verdict, hard_block), (b) the aggregate (APPROVE / REQUEST_CHANGES / REJECT counts and the resolved threshold), (c) the decision, and (d) the **aggregated `concrete_changes_required`** — every item read from the REQUEST_CHANGES/REJECT verdict JSONs, attributed to its slot (the mechanical union; the semantic dedup + criticality ranking stays skill judgment, see R8). It exits `0` on an Allow decision and **non-zero on a Block**, with the Block carrying an ADR-0035 recovery line. The decision and exit code are a pure function of `PanelGateDecision`; the `concrete_changes_required` read is presentation only and never feeds the decision. Falsified if `panel tally`'s exit code disagrees with `panel.PanelGateDecision`'s Action for the same facts, if a Block exit lacks a final recovery line (`guard.HasFinalRecoveryLine`), or if the tally introduces a second decision implementation.

4. **Verdict-file + slot naming is a documented schema (the portability contract).** The panel artifact contract — `panel.json` registration, `<slot>-round-<N>.json` verdict files (the `panel.verdictFileRE` shape, `N ≥ 1`), and `consolidated-round-<N>.md` (`panel.ConsolidatedName`) — is documented in the `workflow` domain doc as the agent-neutral surface other runners target, describing it as the artifact + CLI contract of ADR-0040's portability principle (runners are adapters behind it; degraded modes are the runner's concern, not the schema's). The documented schema must not contradict the code constants (`panel.FileName`, `panel.verdictFileRE`, `panel.ConsolidatedName`). Falsified if the schema doc is absent, or if it states a verdict-file pattern, round base, or consolidated-file name that disagrees with the `internal/panel` constants.

5. **Spec-approve Impacted-Domains parity — identical resolver, identical severity, merely earlier.** `mindspec spec approve` (through `internal/approve.ApproveSpec`, which already hard-fails on `validate.ValidateSpec` failures) additionally runs the **identical call** the plan-approve gates and the bead-divergence gate make: `contextpack.ParseSpec` for extraction, then `normalizeImpactedDomains(nil, root, "", domains)` (spec 100 R1, the working-tree read: `exec` nil, `ownerRef` ""). It surfaces **exactly** that resolver's returned errors under the **same** `impacted-domains-resolve` diagnostic code `plan.go` and `divergence.go` emit, adding **no stricter rule of its own**. Severity therefore matches the downstream gate precisely: a **path-like** entry (contains `/`) that no domain's OWNERSHIP `paths:` claims → error; an **ambiguous** entry claimed by more than one domain → error; and a **bare-name** entry that names no domain dir → **kept verbatim, NOT an error** (today's tolerance — a plain named-domain without an on-disk manifest survives normalization per Rule 2, and its uncovered state is reported later at plan-approve exactly as now). The check is thus behavior-identical to the downstream gate over the same spec — it only fires a phase earlier, catching the spec-108 formatting incident (a mangled path-like entry) at spec-approve instead of at plan-approve's `adr-coverage-missing` and the bead's `adr-divergence-unowned`. Falsified if the check rejects a bare-name-no-manifest spec that plan-approve tolerates today, if it surfaces the resolver's errors under any code other than `impacted-domains-resolve`, if it adds a severity rule `normalizeImpactedDomains` does not itself return, or if it extracts domains with any parser other than `contextpack.ParseSpec`.

6. **Spec-approve ADR-Touchpoint parity.** `spec approve` additionally resolves each ADR **referenced** in the spec's `## ADR Touchpoints` section against the **same** `adr.Store` the plan-time citation gate (`checkADRCitations`) consumes; a referenced ADR that is **absent from the store** is a spec-approve validation **error** with a recovery line naming the missing/typo'd reference. This check verifies **existence only** — it does **not** check Accepted-status, domain-intersection, or coverage (those stay at plan-approve, R7), and it does **not** re-derive plan citations from the touchpoints prose (spec 097 R2 boundary — touchpoints stay human prose whose references are integrity-checked, not a citation source). Falsified if a spec whose `## ADR Touchpoints` names a nonexistent ADR passes `spec approve`, if the check consults an ADR store different from the citation gate's, or if it emits an `adr-coverage-*` / `adr-cite-irrelevant` diagnostic at spec-approve.

7. **Honest boundaries held: single-home decision, config-free leaf, no plan-work moved.** (a) The panel verbs change **no** `PanelGateDecision` semantics — `gate.go`'s decision function keeps its exact inputs and outputs; the verbs are thin CLI adapters over it and `panel.ResolveGateFacts`. (b) `internal/panel` stays an import-clean, **config-free leaf**: the new `Create` writer imports no `internal/config` (and no git); the caller (`cmd/**`) resolves the config values and the target SHA and passes them as plain arguments, mirroring spec 109's "config reaches the leaf only as plain values." (c) The spec-approve parity does **not** re-run plan-level coverage — `adr-coverage-missing` and `adr-cite-irrelevant` remain plan-approve gates; spec-approve only guarantees the spec's machine-consumed surfaces (Impacted-Domains → resolvable names; ADR-Touchpoints → existing files) parse identically everywhere. Falsified if `PanelGateDecision` gains or loses an input, if `go list -deps ./internal/panel` shows `internal/config`, or if `spec approve` emits an `adr-coverage-missing` error.

8. **Skill de-duplication — the now-mechanized prose is removed, judgment stays.** In `ms-panel-run/SKILL.md` **§ Step 0**, the `mkdir` + hand-typed `panel.json` schema block + the "capture `reviewed_head_sha` NOW at fan-out" / "on every re-panel bump `round` AND `reviewed_head_sha` in the SAME write" invariant prose are **replaced** by a single `mindspec panel create <slug> --spec <id> --target <ref> [--bead <id>] [--round N]` invocation; the BRIEF composition (step 3) shrinks to filling the stub `create` wrote. The **Launch the panel**, **Codex failure detection**, **Working directory matters**, **Slot lens defaults**, and **Anti-patterns** sections **stay** (runner-specific launch orchestration + lens judgment = L4). In `ms-panel-tally/SKILL.md`, **Steps 1–3** (Load / Tabulate / the `| Condition | Action |` **decision-matrix table** with the N−1 threshold, REJECT/incomplete/hard_block rows) are **replaced** by `mindspec panel tally <slug>`, and the halt-recovery stale-verdict step is re-pointed at `panel create --round N+1`. **Step 4** (Consolidate — semantic dedup + criticality ranking), **§ Artifact gates** (the hard_block HARD-vs-soft *judgment* that SETS `hard_block`), **§ After a halt — recovery**, **§ Escape hatch**, and **§ Abandon procedure** **stay** (judgment / human-audited procedure). Falsified if the `| Condition | Action |` decision-matrix table survives in `ms-panel-tally`, if the hand-typed `panel.json` schema (`"reviewed_head_sha"`) survives in `ms-panel-run` step 0, if the skills do not reference the new verbs, or if any named judgment section (Artifact gates, Consolidate, Slot lens defaults) is deleted.

## Scope

### In Scope
- `internal/panel`: a leaf-safe `panel.json` writer (`Create` / equivalent) that takes plain values (resolved `expected_reviewers`, `approve_threshold` expression, `reviewed_head_sha`, round, ids) and writes the directory + `panel.json` in one operation — `internal/panel` stays import-clean of `internal/config` and of git. The verdict-file/slot schema documentation in the `workflow` domain doc. (R1, R4, R7b)
- `cmd/mindspec`: the `panel` command with `create` / `verify` / `tally` subcommands, wiring `config.Load` + the 109 resolvers + `newExecutor(root).RevParseRef` (create), and reusing `panel.ResolveGateFacts` + `panel.PanelGateDecision` (verify/tally); the testable renderer functions; the ADR-0035 guard failure on a `tally` Block. (R1, R2, R3, R7a)
- `internal/validate`: the spec-approve parser-parity checks folded into `ValidateSpec` (so both `mindspec validate spec` and `spec approve`, which calls `ValidateSpec`, catch them) — the Impacted-Domains resolution via `contextpack.ParseSpec` + `normalizeImpactedDomains`, and the `## ADR Touchpoints` reference resolution via `adr.Store`. (R5, R6, R7c)
- `internal/approve`: no logic change beyond inheriting the stricter `ValidateSpec` (it already hard-fails on `vr.HasFailures()`); the recovery-line wording for the new failures. (R5, R6, R7c)
- `plugins/mindspec/skills/ms-panel-run` + `ms-panel-tally` (and their `.claude/skills/**` mirrors): remove the mechanized prose, reference the new verbs, keep the judgment sections. (R8)
- Per-domain doc-sync (gate-forward, planning constraint): every bead that edits `workflow` source appends a **bead-unique** region to a `workflow` domain doc (`.mindspec/domains/workflow/*.md`) — the new CLI verbs' `cmd/**` edits, the `internal/panel` writer + schema, and the `internal/validate` parity checks are all `workflow`, so each such bead documents its verb / schema / check there; the skill edits are likewise `workflow`-owned and carry their own doc region.

### Out of Scope
- The workflow-backed panel runner and any `runner:` adapter dispatch (`claude-code-workflow` / `external`) — **spec 111**. 110 lands the verbs a runner drives; 111 is their orchestrating consumer.
- Instruct-template generation from gate definitions (the L3 "generated from gate definitions" mechanism) — a **dedicated future spec**. It requires generalizing the mode→gates surface (`gatesForMode` in `internal/instruct`) so instruct prose is emitted from the gate definitions, and is its own bounded change; not started here.
- Any change to `panel.PanelGateDecision`'s semantics or the panel gate's Allow/Block outcome for any `panel.json` — the verbs are adapters over the unchanged decision (R7a).
- Signing, hashing, or moving panel state out of the repo — ADR-0037 §8 forbids "fixing" perceived forgeability at this layer; the verbs keep panel state as plain reviewable files.
- `loop:` / `models:` enforcement, and any `mindspec config` change — those are spec 109's substrate, consumed read-only here.
- Re-running plan-level coverage at spec-approve — `adr-coverage-missing` / `adr-cite-irrelevant` stay plan-approve gates (R7c).

## Non-Goals

- No new decision semantics and no new gate. `mindspec complete` remains the authoritative panel-gate enforcer; `panel verify` and `panel tally` are read-side surfaces over the same decision, and `panel tally`'s non-zero exit is a *reporting* convenience, not a second enforcement point.
- No telemetry stream. Consistent with ADR-0027 (OTEL-only, pull surfaces), the panel verbs and the parity checks emit no metrics.
- No stricter *semantic* spec validation. The parity checks are deterministic parser/resolver runs; criterion-quality and requirement-vagueness judgment stays with the AI reviewer at spec-approve time (the ZFC boundary `validate.checkAcceptanceCriteria` already documents).
- No change to how a spec declares ADR touchpoints or citations — touchpoints stay human prose; plan citations stay plan frontmatter (spec 097 R2). R6 only integrity-checks the references already written.

## Validation Proofs

- `go test ./internal/panel/... -run 'TestCreate_WritesRegistrationAtomically'` and `! go list -deps ./internal/panel | grep -q internal/config`: the writer records the fields atomically and the leaf stays config-free.
