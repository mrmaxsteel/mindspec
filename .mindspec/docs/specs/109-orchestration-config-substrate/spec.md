# Spec 109-orchestration-config-substrate: Orchestration config substrate + the gate/config/instruct/skill layering ratchet (ADR-0040)

## Goal

Make MindSpec's four behavior-steering surfaces a **declared layering ratchet** instead of a blurred continuum, and give it a **machine-readable config substrate** so orchestration policy that today lives only in skill prose and operator memory becomes checked-in, parseable configuration.

Two concrete outcomes:

1. A new Accepted **ADR-0040** names the four layers — (L1) in-binary gates, (L2) declared config, (L3) mode-selected `instruct` advice, (L4) judgment-kernel skills — and the one-directional ratchet between them: a rule that proves load-bearing or gameable in a skill (L4) ratchets *down* into config (L2) or a binary gate (L1), never casually up. It also fixes MindSpec's **portability principle**: agents integrate at the artifact + CLI contract level, never at prompt-format level, and orchestration runners are adapters behind those contracts. ADR-0040 becomes the default answer to "where does this rule live?".
2. The `.mindspec/config.yaml` schema gains the orchestration blocks — `panel:` (reviewer mix, approve threshold, quota-substitution policy), `models:` (phase→model tiering), and `loop:` (governance skeleton: gate authority, halt conditions, budget, context policy) — plus a top-level `runner:` adapter selector, all parsed, defaulted, validated, and surfaced by a new read-only `mindspec config show`. The `panel:` block is wired into the panel machinery as the creation-time default for a fresh `panel.json`; the gate's decision stays a pure function of the recorded `panel.json` (config never changes an Allow/Block). `models:`, `loop:`, and `runner:` are declared, parsed, and surfaced now, and consumed by orchestration skills (and, for `runner:`, by spec 111); their in-binary enforcement is explicitly deferred to later specs.

The empirical motivators are the 107/108 delivery runs: the 6-reviewer panel mix and the N−1 threshold lived only in `SKILL.md` prose plus a hand-typed `panel.json`; model tiering (impl=opus, review=fable) lived in an operator memory file; the codex quota-wall substitution policy was improvised at 1am. After this spec each of those is a declared config key with a default that matches today's practice.

## Background

MindSpec already implements the hard part of an unattended agent loop — a forward-only, beads-derived state machine with unarguable in-binary gates (`internal/panel/gate.go`), fresh-context enforcement, and an audited escape-hatch record (ADR-0038). The 2026-07-02 loop-engineering research (`project-docs/research/loop-engineering-adaptation.md`) makes the case that what is missing is the *loop head*, and that its prerequisite is exactly this substrate: §3.2 L3 turns "the overnight autopilot grant" — which today exists only as prose in `bd` memories — into a config-selected governance profile, and §3.3 adds a controller context-policy alongside it. The research is emphatic on the invariant this spec must preserve: **the decision matrix stays in the binary** (as ADR-0037 did for the bead-merge gate — "identical decision over identical facts"), and config only selects *who is the authority*, never *what the evidence is*.

The three surfaces have blurred in practice. Deterministic gates are binary and exit non-zero. `mindspec instruct` emits binary-generated prose. But the `ms-*` skill surface mixes judgment, procedure, and *re-stated gate logic* — the panel decision matrix, for instance, exists in both `ms-panel-tally` prose AND `internal/panel/gate.go` code, and the panel mix / threshold exist in `SKILL.md` prose AND a hand-typed `panel.json`. When the same rule lives in two places, prose drifts from code and "prose under pressure fails" (the lola-f4a8 incident that motivated ADR-0037). The layering ratchet (ADR-0040) makes the direction of travel explicit — rules move *down* toward enforcement as they prove load-bearing — and the config substrate (this spec's `panel:`/`models:`/`loop:` blocks) is the middle rung that lets a prose rule become machine-readable *without* becoming code.

This spec deliberately stops at the substrate. It does **not** add the `mindspec panel create/verify/tally` verbs that will *write* `panel.json` from the config defaults (spec 110), the workflow-backed panel runner (spec 111), any change to skill files (they shrink in 110/111), or instruct-template generation from gate definitions (110 or later). It lands the config schema, the resolvers and the recorded-field plumbing the 110 writer will call, and a read-only surfacing command — the layer that everything downstream consumes.

## Impacted Domains

- **core**: `internal/config` gains the `panel:`, `models:`, and `loop:` blocks plus the top-level `runner:` key (new structs, `DefaultConfig` entries, `Load` backfill and validation, and the `PanelExpectedReviewers()` / `PanelApproveThresholdExpr()` resolver methods). This is the schema home of the substrate and the zero-config round-trip surface.
- **workflow**: `internal/panel` gains an optional recorded `approve_threshold` field on the `panel.json` schema struct, the single-home `ApproveThreshold()` method learns to interpret it, and a leaf-safe pure helper `ReviewerCountNote` renders the config-default-vs-recorded advisory; `cmd/mindspec` gains the read-only `config` / `config show` command and its renderer. `internal/panel` stays a dependency-clean leaf (it imports no `internal/config`); the config default reaches it only as plain values passed by the caller.

Note: the ADR work is documentation, not source. Creating `.mindspec/adr/ADR-0040-*.md` and appending an amendment note to `.mindspec/adr/ADR-0037-panel-gate-enforced-contract.md` touch only `.mindspec/adr/**`, which `internal/validate`'s divergence classifier treats as non-source lifecycle docs — so neither needs a domain claim and neither trips `adr-divergence-unowned`. No path outside the `core` and `workflow` OWNERSHIP.yaml globs is edited as source; the substrate touches `internal/config/**` (core), `internal/panel/**` + `cmd/**` (workflow), and nothing in `execution` or `context-system`.

## Requirements

1. **ADR-0040 — the layering ratchet + the portability principle (Accepted, Bead 1).** Add `.mindspec/adr/ADR-0040-orchestration-layering-ratchet.md` with `Status: Accepted` and `Domain(s): core, workflow`. It codifies **the four layers and their one-directional ratchet**: (L1) gates are in-binary invariants that exit non-zero with journaled escape hatches; (L2) config is declared policy consumed *by* gates and orchestration (the middle rung — how a prose rule becomes machine-readable without becoming code); (L3) `instruct` is mode-selected advice, generated from gate definitions where it describes gates; (L4) skills are judgment kernels and the **staging area** for procedure — a skill's procedural rule that proves load-bearing or gameable ratchets *down* into config or binary, never casually up. It must cite the precedent chain as already-shipped evidence (the PreToolUse hook → in-binary panel gate move in spec 102; the decision matrix mechanized in ADR-0037; the ceremony collapse in ADR-0034) and state the default answer to "where does this rule live?". It also codifies a **portability principle**: agent integration happens at the **artifact + CLI contract** level (the `panel.json` and verdict-JSON schemas, the future `mindspec panel` verbs, `mindspec instruct` output, and beads state) and **never** at the prompt-format level; orchestration runners (Claude Code skills/workflows today; opencode / codex-cli / copilot-cli later) are **adapters** behind those contracts. The ADR cites the in-repo precedent in both directions — `internal/setup`'s per-agent trio and the `Executor` / ADR-0030 boundary (an interface born with a real second consumer = healthy) versus the harness `Agent` interface accreting dead methods (abstraction ahead of a second implementation = rot) — and includes a **capability-tier** note: runners differ (no SessionStart hook in codex; no subagent spawning in some), so the contract defines **degraded modes** (human-run `instruct`, panel-less operation under `enforcement.panel_gate`) rather than assuming Claude Code's full surface. Falsified if the file is absent, is not `Accepted`, omits any of the four layers, omits the ratchet direction, omits the precedent chain, or omits the portability / artifact-contract principle.
2. **`panel:` config block with today's defaults.** `internal/config.Config` gains a `Panel` block: `reviewers` (a list of `{family: claude|codex, count: int}`), `approve_threshold` (a string expression, `"n-1"` or an integer), and `substitution.claude_sub_on_quota` (bool). `DefaultConfig` sets reviewers to `[{claude, 3}, {codex, 3}]`, `approve_threshold` to `"n-1"`, and `claude_sub_on_quota` to `true` — matching today's practice. An absent `panel:` key round-trips to those defaults (absent config file → `DefaultConfig` behavior unchanged for every pre-existing field). Falsified if a parsed reviewers list is dropped, or if the zero-config defaults differ from 3+3 / `"n-1"` / `true`.
3. **`models:` phase-tiering block.** `Config` gains `models` as a `map[string]string` keyed by phase, with the documented phases `implement`, `review`, `authoring`, `grill`, `final_review` and free-form model-id string values. The values are **free-form strings because model identifiers are runner-specific** (a `claude-code-skills` runner names Claude model ids; an `external` runner may name others), so the framework never validates them against a closed set. The default is an empty map (each phase inherits the ambient model). The block is parsed and surfaced by `mindspec config show`; it is consumed by orchestration skills (there is no in-binary agent spawner to enforce model selection — stated honestly, not wired here). Falsified if a populated `models:` map does not round-trip through `Load`, or if the default is anything other than empty/inherit.
4. **`loop:` governance skeleton — parsed, validated, surfaced, documented; not enforced in-binary.** `Config` gains a `Loop` block mirroring research §3.2 L3 / §3.3: `enabled` (bool, default `false` — loops are opt-in); `gate_authority` with exactly the four keys `spec_approve` / `plan_approve` / `bead_merge` / `impl_approve`, each `panel` or `human`, default `human`; `halt` with `max_rounds_per_bead` (default `3`), `panel_deadlock_rounds` (default `2`), `max_consecutive_impl_failures` (default `2`) — matching ms-spec-autopilot practice and research §3.2 — and `on_reject` (default `halt`); `budget` with `max_beads_per_wake` (int) and `token_budget` (an integer count of tokens), both defaulting to `0` meaning unset/unlimited; and `context` with `controller_handoff` (`per-spec` | `at-usage-threshold`, default `per-spec`). The block is parsed, validated (below), and surfaced by `mindspec config show`; its in-binary enforcement is future work (spec 110+) and this spec says so. `panel_skip` is **not** a `gate_authority` key and is documented as permanently `human` (`MINDSPEC_SKIP_PANEL` is env-only per ADR-0037 §7). Falsified if the four gate_authority keys are missing, if `enabled` defaults to anything but `false`, if any named `halt`/`budget`/`context` default differs from the values stated here, or if the block is silently unparsed.
5. **`Load` refuses the un-weakenable knobs and the threshold-skip loophole (research §3.4 "what deliberately stays human").** `config.Load` returns an error — with a recovery line naming the reason — for each of: a `panel_skip` key anywhere under `loop.gate_authority` (panel-skip is not delegable; it is permanently human, ADR-0037 §7); a `loop.halt.on_reject` set to any value other than `halt` (a REJECT halts the track; auto-fixing a rejection is verification debt); a `gate_authority` value other than `panel`/`human`; and a `controller_handoff` outside `{per-spec, at-usage-threshold}`. **Additionally — closing the loophole R5 itself would otherwise leave open — `panel.approve_threshold` is validated as `"n-1"` or an integer in `[1, sum(reviewers.count)]`:** a threshold of `0` (or negative) is a panel that always passes — a de-facto panel-skip achieved through the very substrate this requirement guards — and a threshold greater than the reviewer count is unsatisfiable; both are rejected with a recovery line. An absent `on_reject` defaults to `halt`; an absent `approve_threshold` defaults to `"n-1"`. This encodes the "stays human" floor and the no-always-pass floor as load-time refusals even before `loop:` enforcement exists. Falsified if any of those inputs loads without error.
6. **Config resolvers (creation-time defaults, resolution single-homed).** `internal/config` exposes `(*Config).PanelExpectedReviewers() int` (the sum of `reviewers[].count`, defaulting to `6` when unset) and `(*Config).PanelApproveThresholdExpr() string` (the raw `approve_threshold` expression, defaulting to `"n-1"`). `PanelApproveThresholdExpr` returns the **raw** expression and does **not** resolve it to an integer — expression resolution stays single-homed in `internal/panel.Panel.ApproveThreshold` (Requirement 7). These are the values the spec-110 `panel.json` writer will stamp; 109 lands and tests them directly. Falsified if a resolver resolves the expression to an int, or if the reviewer-count sum is wrong.
7. **Recorded `approve_threshold` on `panel.json`; `ApproveThreshold()` stays the single interpreter; zero-config byte-identical.** `internal/panel.Panel` gains an optional `ApproveThresholdExpr string` field (`json:"approve_threshold,omitempty"`). The existing `ApproveThreshold() int` method (the single home, `panel.go:76`) becomes the sole interpreter of that expression: an **absent/empty** field yields `ExpectedReviewers − 1` (byte-identical to today's behavior and to every legacy `panel.json`); `"n-1"` (case-insensitive) yields `ExpectedReviewers − 1`; an integer string **within `[1, ExpectedReviewers]`** yields that integer; an integer **outside `[1, ExpectedReviewers]`** (`0`, negative, or greater than the reviewer count) is treated as unparseable and falls back to `ExpectedReviewers − 1`, so a recorded `0` can never turn the gate into an always-pass (the record-side half of R5's threshold-skip defense); any other unparseable value likewise falls back to `ExpectedReviewers − 1`. No second interpreter is introduced anywhere. The one pre-existing edge is deliberately left untouched: today's `N=1 ⇒ N−1=0` behavior of an absent-field, single-reviewer panel predates this spec and is out of scope. ADR-0037's amendment note records this one-field schema addition. Falsified if an absent field ever yields a threshold other than `N−1`, if a recorded `0` yields a threshold of `0`, or if any package other than `ApproveThreshold()` interprets the expression.
8. **Precedence rule + leaf-safe advisory note (config never alters an Allow/Block).** The precedence is: the per-round `panel.json` is the **sole authority for the gate decision** — `PanelGateDecision` remains a pure function of the recorded facts and takes no `config` input, so `internal/panel` stays a config-free leaf and "identical decision over identical facts" is preserved verbatim. Config `panel:` supplies only the creation-time defaults (Requirement 6). When a recorded `expected_reviewers` differs from the current config default, the recorded value wins the decision (per-round facts beat defaults — a legitimately smaller substituted quorum must never be false-blocked), and the difference is surfaced as an **advisory** via a pure, config-free helper `panel.ReviewerCountNote(recorded, configDefault int) string` (empty string when equal). The note is rendered by the gate's caller (the workflow-side surface — `mindspec config show` and the in-binary complete-gate wiring), never by `PanelGateDecision`, and never changes `Allow`/`Block`. Falsified if `PanelGateDecision` gains a `config` dependency, if the note ever flips a decision, or if `internal/panel` imports `internal/config`.
9. **`mindspec config show` — read-only surfacing.** A new `config` command with a `show` subcommand loads the effective config (defaults merged with `.mindspec/config.yaml`) via `config.Load` and prints it — including the `panel:`, `models:`, `loop:`, and `runner:` keys — to stdout, exit `0`, writing nothing. The render logic lives in a testable function (`renderConfig(*config.Config) (string, error)`) so it is exercised without spawning a process. Falsified if the command mutates any file, if it omits any of the new keys, or if it exits non-zero on a valid config.

10. **`runner:` orchestration adapter key.** `internal/config.Config` gains a top-level `runner` string selecting which orchestration adapter drives panels and autopilot: `claude-code-skills` (today's default), `claude-code-workflow`, or `external`. `DefaultConfig` sets `claude-code-skills`; `Load` validates the value against that known set and errors on an unknown runner (a runner names an in-binary adapter, so an unknown one must fail loud, not silently default). It is parsed, surfaced by `mindspec config show`, and documented here; it is **consumed** by spec 111 (the first non-default runner) — no adapter dispatch is wired in this spec. This is the config expression of ADR-0040's portability principle: the runner is the adapter selector that sits behind the artifact / CLI contracts. Falsified if the default is not `claude-code-skills`, if an unknown runner value loads without error, or if the key is not surfaced by `config show`.

## Scope

### In Scope
- New `.mindspec/adr/ADR-0040-orchestration-layering-ratchet.md` (Accepted) and an amendment note appended to `.mindspec/adr/ADR-0037-panel-gate-enforced-contract.md` (R1, R7).
- `internal/config`: `Panel` / `Models` / `Loop` struct additions, the top-level `Runner` field, `DefaultConfig` entries, `Load` backfill (absent/empty → default) and validation (the R5 refusals: `panel_skip`, non-`halt` `on_reject`, the `gate_authority` / `controller_handoff` / `runner` enum checks, and the `approve_threshold` `[1, sum(reviewers.count)]` bounds), and the `PanelExpectedReviewers()` / `PanelApproveThresholdExpr()` resolver methods (R2–R6, R10).
- `internal/panel`: the optional recorded `approve_threshold` field, the `ApproveThreshold()` interpreter change, and the leaf-safe `ReviewerCountNote` helper — with `internal/panel` remaining import-clean of `internal/config` (R7, R8).
- `cmd/mindspec`: the read-only `config` / `config show` command and `renderConfig` (R9).
- Per-domain doc-sync (gate-forward, planning constraint): every bead that edits source in a domain also edits that domain's doc — a `core` bead touches `.mindspec/domains/core/*.md`, a `workflow` bead touches `.mindspec/domains/workflow/*.md`; the ADR edits are doc-classified `.mindspec/adr/**` and carry themselves. Bead 1 lands ADR-0040 (Accepted) first so the later config/panel beads can cite it for divergence coverage.
- New config keys round-trip zero-config: with no `.mindspec/config.yaml`, `Load` returns a `Config` whose every pre-existing field is unchanged and whose new blocks sit at their documented defaults.

### Out of Scope
- The `mindspec panel create` / `verify` / `tally` verbs, including the `panel.json` **writer** that will stamp the config defaults via the R6 resolvers — spec 110. (109 lands the resolvers and the recorded field; 110 wires the writer that calls them.)
- The workflow-backed panel runner and any non-default `runner:` adapter dispatch (the `claude-code-workflow` / `external` adapters) — spec 111. 109 declares and validates the `runner:` key; 111 is its first consumer.
- Any edit to skill files (`.claude/skills/ms-*`); they shrink in 110/111, not here.
- Instruct-template generation from gate definitions (the L3 "generated from gate definitions" mechanism) — spec 110 or later.
- In-binary **enforcement** of `loop:` (gate authority at the approve gates, halt-condition checking, budget ceilings, controller handoff) — declared and surfaced here, enforced in a later spec.
- In-binary **enforcement** of `models:` phase selection — there is no in-binary agent spawner; the map is consumed by orchestration skills.
- Signing, hashing, or moving panel state out of the repo — ADR-0037 §8 forbids "fixing" perceived forgeability at this layer; the substrate keeps panel state as plain reviewable files.

## Non-Goals

- No change to the `Allow`/`Block` outcome of the panel gate for any existing `panel.json`. Behavior is byte-identical for every panel that omits the new field, and config is never an input to the decision.
- No new telemetry stream. Consistent with ADR-0027 (OTEL-only, pull surfaces), `mindspec config show` is a pull surface over derived config; it emits no metrics.
- No trigger, scheduler, CI-wait, or live monitor (research L5–L7). This spec is the substrate those layers will later read.

- **RESOLVED — ADR-0040 lands Accepted, not Proposed, in Bead 1.** ADR-0040 codifies a layering that *already governs the codebase* — the precedent chain (spec 102's in-binary gate, ADR-0037's mechanized matrix, ADR-0034's ceremony collapse) has shipped; it proposes no new architecture awaiting validation. Codification ADRs land Accepted, matching ADR-0037 and ADR-0034, which are themselves Accepted-on-landing. Landing it Accepted in the first bead also lets the same spec's later config/panel beads cite it as a `core`/`workflow` coverage source for the ADR-divergence gate (ADR-0032), whereas a Proposed ADR would only emit an advisory `adr-divergence-proposed` warning and could not serve as coverage.
- **RESOLVED — a configurable threshold requires a recorded `panel.json` field.** For `panel.approve_threshold` config to genuinely reach the gate *and* keep the decision a pure function of the record, the resolved intent must live in `panel.json`, not be re-derived from config at decision time. So `panel.Panel` gains an optional `approve_threshold` field that the spec-110 writer stamps from config, and `ApproveThreshold()` (the single home) interprets it, defaulting to `N−1` when absent. The rejected alternative — letting config change `ApproveThreshold()`'s formula globally — was discarded because it breaks "identical decision over identical facts": the same `panel.json` would decide differently under different config.
- **RESOLVED — precedence: per-round record wins the decision; config is a creation-time default only.** The gate decides purely on the recorded `panel.json` (`internal/panel` stays a config-free leaf); a recorded `expected_reviewers` that contradicts the config default is allowed and surfaced as an advisory note (`ReviewerCountNote`) by the caller, never altering `Allow`/`Block`. This keeps a smaller substituted quorum from being false-blocked while making the divergence visible.
- **RESOLVED — `mindspec config show` is in scope; the writer is not.** A read-only surfacing command is the minimum viable "declared policy is visible," and it is the runnable proof that the new keys round-trip. The `panel.json` *writer* that consumes the R6 resolver defaults is spec 110; 109 lands the resolvers and the recorded field the writer will fill.
- **RESOLVED — `panel_skip` and `on_reject` are un-weakenable.** Rather than silently drop them, `config.Load` errors on a configurable `panel_skip` and on a non-`halt` `on_reject`, encoding research §3.4's "what deliberately stays human" as a load-time refusal even before `loop:` enforcement exists.
- **RESOLVED — `runner:` is a top-level key, validated, first consumed by spec 111.** The runner adapter selector is a top-level string (sibling to the `panel:`/`models:`/`loop:` orchestration blocks, matching the flat top-level style of the existing config), defaulting to `claude-code-skills` and validated against the known adapter set so an unknown value fails loud. It is the config expression of ADR-0040's portability principle (runners are adapters behind the artifact/CLI contracts); 109 declares and validates it, and spec 111 is its first non-default consumer. `models:` values stay free-form strings precisely because model identifiers are runner-specific.
