# Spec 042-copilot-cli-support: Copilot Integration Support (CLI + VS Code)

## Goal

Make MindSpec a first-class workflow for GitHub Copilot users — both Copilot CLI (terminal) and Copilot Chat in VS Code — so teams can run Spec -> Plan -> Implement -> Review with Copilot, and run brownfield migration classification without requiring Claude CLI.

## Background

MindSpec is CLI-first and already documents Claude Code and Codex workflows, but Copilot is not yet a first-class documented/bootstrap target.

GitHub Copilot ships on two surfaces that matter for MindSpec:

1. **Copilot CLI** — a terminal agent that can call `mindspec` commands directly, similar to Codex.
2. **Copilot Chat (VS Code)** — an IDE-integrated agent that reads `.github/copilot-instructions.md` for workspace context, can run terminal commands, and uses `@workspace` for codebase awareness.

Both surfaces share the same `.github/copilot-instructions.md` bootstrap file but differ in session lifecycle and command invocation patterns.

Three practical gaps currently block smooth Copilot adoption:

1. **Workflow onboarding gap**: there is no dedicated Copilot quick-start guide and no Copilot-specific bootstrap artifacts generated by `mindspec init` (for example `.github/copilot-instructions.md` and repository hooks).
2. **VS Code integration gap**: Copilot Chat users need workspace-level instruction that chains through the `AGENTS.md` universal layer and explains how to drive the workflow from the chat panel's terminal.
3. **Migration provider gap**: `mindspec migrate` LLM classification is effectively Claude-bound today (probe and executor paths), which blocks Copilot-only environments from using assisted classification.

This spec adds first-class Copilot onboarding (both surfaces) and migration-provider support while preserving existing Claude/Codex behavior and MindSpec's CLI-centric architecture.

## Impacted Domains

- **workflow**: Copilot operating guide and phase transitions using direct `mindspec` commands
- **core**: bootstrap artifacts and top-level docs/reference updates
- **brownfield migration subsystem**: provider resolution and LLM classification execution path

## ADR Touchpoints

- [ADR-0003](../../adr/ADR-0003.md): `mindspec instruct` remains the canonical runtime instruction source; Copilot bootstrap must defer to CLI-emitted guidance.
- [ADR-0012](../../adr/ADR-0012.md): Copilot integration must compose with external CLIs and repository-native config files instead of wrapping or replacing Copilot.
- [ADR-0017](../../adr/ADR-0017.md): `AGENTS.md` is the universal instruction layer; agent-specific files (`.github/copilot-instructions.md`, `CLAUDE.md`) point to it rather than duplicating workflow guidance.

## Requirements

### 1. Copilot workflow onboarding (CLI + VS Code)

1. Add a dedicated Copilot guide at `.mindspec/docs/user/guides/copilot.md` covering both Copilot CLI and Copilot Chat (VS Code).
2. The guide must include: setup prerequisites, session-start guidance pattern, direct workflow commands (`mindspec spec-init`, `mindspec approve ...`, `mindspec next`, `mindspec complete`), and known differences from Claude/Codex integrations.
3. The guide must document the VS Code Copilot Chat workflow: how `.github/copilot-instructions.md` points to `AGENTS.md`, using the integrated terminal for CLI commands, and how workspace context is provided automatically.
4. Update user-facing entry points (README and guide index references) so Copilot is listed alongside Claude and Codex.
5. Extend `mindspec init` scaffolding to add `.github/copilot-instructions.md` that points to `AGENTS.md` for shared workflow conventions (per ADR-0017), plus any Copilot-specific notes for both CLI and VS Code Chat usage.
6. Extend `mindspec init` scaffolding to add a repository hook template under `.github/hooks/` that runs `mindspec instruct` on session start (or equivalent supported hook event) for Copilot CLI.
7. `mindspec setup copilot` must create prompt files (`.github/prompts/*.prompt.md`) equivalent to Claude's slash commands (`.claude/commands/*.md`): `spec-init`, `spec-approve`, `plan-approve`, `impl-approve`, `spec-status`.
8. Bootstrap creation must remain additive and non-destructive: existing Copilot files are never overwritten.

### 2. Brownfield migrate provider support

9. Add explicit `copilot-cli` provider support to migration classification.
10. Classification execution must be provider-routed (not hardwired to `claude`), so the selected provider determines which CLI is executed.
11. When `MINDSPEC_LLM_PROVIDER=copilot-cli`, migrate plan/apply must not execute a Claude probe or Claude classifier call.
12. Provider auto-detection when `MINDSPEC_LLM_PROVIDER` is unset must support Copilot CLI availability (with deterministic precedence documented in code and tests).
13. Unsupported/missing provider errors must be explicit and actionable, including remediation via `MINDSPEC_LLM_PROVIDER` and `MINDSPEC_LLM_MODEL`.

### 3. Quality and compatibility

14. Existing Claude-based migrate behavior remains backward compatible when Claude is selected.
15. Existing deterministic-only mode (`MINDSPEC_LLM_PROVIDER=off`) remains unchanged.
16. New and updated tests cover provider resolution, probe behavior, and classification command routing.
17. `make test` passes.

## Scope

### In Scope
- Copilot documentation covering both CLI and VS Code Chat in user docs/README
- Bootstrap changes in `mindspec init` for Copilot instruction/hook artifacts
- `.github/copilot-instructions.md` content that serves both CLI and VS Code Chat surfaces
- `mindspec setup copilot` command creating prompt files (`.github/prompts/*.prompt.md`) with feature parity to Claude's slash commands
- Migration provider resolution and classifier execution path updates
- Unit/integration tests for migrate provider behavior

### Out of Scope
- Full AgentMind telemetry setup automation specifically for Copilot
- Redesign of MindSpec workflow modes, state model, or Beads lifecycle
- GitHub-hosted coding-agent policy configuration outside repository-controlled files
- VS Code extension development or Copilot Chat extension points beyond `.github/copilot-instructions.md` and `.github/prompts/`

## Non-Goals

- Replacing or regressing existing Claude/Codex integrations
- Introducing Copilot-specific workflow logic — prompt files are thin shims that call `mindspec` CLI, same as Claude's slash commands
- Building a generic multi-provider SDK beyond what migrate classification currently needs
- Building a VS Code extension for Copilot Chat — the integration is purely via repository-native config files and CLI commands

## Acceptance Criteria

- [ ] A Copilot guide exists at `.mindspec/docs/user/guides/copilot.md` covering both CLI and VS Code Chat workflows, linked from user entry-point docs.
- [ ] The guide documents how VS Code Copilot Chat users drive the workflow (integrated terminal for CLI commands, `.github/copilot-instructions.md` for workspace context).
- [ ] `mindspec init` creates Copilot bootstrap artifacts (`.github/copilot-instructions.md` and hook template) additively.
- [ ] `.github/copilot-instructions.md` points to `AGENTS.md` for shared conventions (per ADR-0017) and works for both Copilot CLI and VS Code Copilot Chat.
- [ ] `mindspec setup copilot` creates prompt files in `.github/prompts/` equivalent to Claude's slash commands (`spec-init`, `spec-approve`, `plan-approve`, `impl-approve`, `spec-status`).
- [ ] Running `mindspec migrate plan` with `MINDSPEC_LLM_PROVIDER=copilot-cli` routes LLM classification through Copilot CLI and does not invoke Claude.
- [ ] Running `mindspec migrate plan` with `MINDSPEC_LLM_PROVIDER=off` performs deterministic-only classification with no LLM CLI execution.
- [ ] Running migrate with unset provider on a Copilot-only environment can resolve/use Copilot provider per documented precedence.
- [ ] Existing Claude migrate tests remain green.
- [ ] New Copilot provider tests are present and passing.
- [ ] `make test` passes.

- `go test ./internal/brownfield -run 'ResolveLLMConfig|Copilot|Classify'`: provider detection/routing tests pass for Copilot, Claude, and off modes.

- `MINDSPEC_LLM_PROVIDER=off ./bin/mindspec migrate plan --run-id 042-off-smoke`: plan completes without invoking an LLM provider.
- `MINDSPEC_LLM_PROVIDER=copilot-cli ./bin/mindspec migrate plan --run-id 042-copilot-smoke`: low-confidence docs are classified via Copilot path (when Copilot CLI is available).
- `make test`: full suite passes.

## Open Questions

*None. Resolved decisions:*
- Copilot support covers both CLI and VS Code Chat, reusing `mindspec instruct` as canonical runtime guidance for both.
- Instruction layering follows ADR-0017: `.github/copilot-instructions.md` → `AGENTS.md` → `mindspec instruct`. Same pattern as `CLAUDE.md` → `AGENTS.md`.
- `.github/copilot-instructions.md` is the shared bootstrap artifact — it's read by both Copilot CLI and VS Code Copilot Chat.
- VS Code Copilot Chat users run `mindspec` commands via the integrated terminal; no VS Code extension is needed.
- Brownfield migrate gets explicit provider-level Copilot support instead of a Claude-only implicit path.
- Copilot bootstrap is repository-native (`.github/copilot-instructions.md` + `.github/hooks/`) and additive.
