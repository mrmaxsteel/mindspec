# Spec 111-workflow-panel-runner: Workflow-backed panel runner: the first orchestration adapter behind the panel contracts

## Goal

Replace the review panel's stochastic launch mechanics with the first deterministic **orchestration runner adapter** — a Claude Code dynamic workflow (`/ms-panel`) that runs a 6-reviewer panel behind the agent-neutral contracts spec 110 defines, selected by spec 109's `runner:` key.

Two concrete outcomes:

1. **A tracked, versioned workflow `.claude/workflows/ms-panel.js`** (embedded in the plugin and installed by `mindspec setup` to the Claude Code target). Given `{slug, spec, target, bead_id?, round, sha?, lenses[], mix}`, it: **registers** the panel through `mindspec panel create` (spec 110's verb — round + SHA co-bump and the 109 resolver stamps happen in the binary, not in the workflow); **fans out** reviewers per the config `panel:` mix (spec 109) — Claude slots as workflow `agent()` steps, Codex slots as **wrapper agents** that exec the codex CLI and write a schema-conformant verdict file themselves (codex never writes files); **substitutes** a `claude-sub` agent on a codex quota wall as a deterministic branch (per `panel.substitution.claude_sub_on_quota`, spec 109); and **returns** the `mindspec panel verify` report + `mindspec panel tally` preview as its result. It eliminates the two production failure classes — the codex sandbox-file-write failure and the verdict-parse failure — that `ms-panel-run`'s hand-driven launch + bash failure-detection existed to catch.

2. **`ms-panel-run` shrinks to lens composition (judgment) + one workflow invocation** when `runner: claude-code-workflow`, retaining its full manual launch path when `runner: claude-code-skills` (the default until the workflow path is proven). The decision stays in `internal/panel/gate.go` (via `mindspec complete`); the workflow is an adapter that writes the **same** artifacts, never a second decision authority, and never runs `mindspec complete`.

This is ADR-0040's **portability principle** (spec 109) made real: agents integrate at the artifact + CLI contract level (the `panel.json` / verdict-JSON schemas and the `mindspec panel` verbs), never at the prompt-format level; the runner is the adapter behind those contracts, and this is its first non-default instance.

## Background

MindSpec already enforces the panel gate in the binary — `internal/panel.PanelGateDecision` is the single pure decision function `mindspec complete` invokes (ADR-0037) — and spec 110 turns the panel *lifecycle around* that decision into agent-neutral CLI verbs (`mindspec panel create | verify | tally`) plus a documented verdict-file/slot schema. What is still stochastic is the **launch mechanics**: `ms-panel-run` hand-drives three `Agent` spawns and three `codex exec` background processes with bespoke failure detection, and the Codex slots depend on external ChatGPT-plan quota that periodically walls the panel, forcing a judged persona substitution. An untracked internal working paper on adapting MindSpec for unattended operation (`project-docs/research/loop-engineering-adaptation.md`, a working document **not** committed with this spec) identifies the same two problems this runner solves, and the design rationale below is stated in the spec's own terms: the panel's launch is shaped for an interactive operator, so its quota-wall substitution is decided by human judgment rather than as fixed policy; and a skill-driven panel round-trips six briefs and six verdict reads through the orchestrator's own context window, whereas a workflow-backed panel returns one compact structured result and keeps the reviewer transcripts out of the orchestrator entirely. Spec 111 makes the substitution a deterministic policy branch and the panel a single structured call.

**The Claude Code workflow feature — stated honestly, with its limits.** A Claude Code *dynamic workflow* is a saved JavaScript orchestration script under `.claude/workflows/` (project) or `~/.claude/workflows/` (user), invoked as `/<name>`, that coordinates subagents while the session stays responsive (docs: `https://code.claude.com/docs/en/workflows`). Three documented limits shape this spec's design and correct the naïve framing:

- **The workflow script has no direct shell or filesystem access.** The docs are explicit: "No direct filesystem or shell access from the workflow itself. Agents read, write, and run commands. The script coordinates the agents." Therefore every CLI touch (`mindspec panel create`, `codex exec`, `mindspec panel verify`, `mindspec panel tally`) and every verdict-file write is performed by an **agent step**, never by the script. This is *why* codex reviewers must be wrapper agents: an agent execs codex and writes the file with its own `Write` tool; codex itself never writes.
- **There is no documented output-schema / structured-output enforcement.** The workflows docs do not describe forcing an agent step to emit a validated JSON schema. So verdict conformance is **not** a platform guarantee this spec may assume; it is enforced by (a) the agent prompt specifying the 110 verdict shape and (b) **deterministic post-hoc validation** — `mindspec panel verify` (110 R2) reports per-slot parse status and names malformed files — with the wrapper re-prompting or substituting on a parse failure. The contract is the artifact schema + the `panel verify` check, exactly ADR-0040's "integrate at the artifact contract, never the prompt format."
- **Resume is session-scoped.** The docs support resuming a paused run within a session (completed agents return cached results); exiting Claude Code restarts the workflow fresh. So the re-round cache this spec relies on is **not** workflow resume — it is the persisted verdict files on disk (the 110 artifact contract): a re-panel bumps the round via `panel create --round N+1`, leaving prior-round `<slot>-round-<K>.json` files untouched, and only the current round's slots run.

The `.js` extension is the inference from the documented "JavaScript script" language, not an explicitly documented extension; the location (`.claude/workflows/`) and slash-command invocation (`/ms-panel`) are documented.

**Why this is an adapter, not architecture.** ADR-0040 (landing as Bead 1 of the **approved** spec 109) fixes that agent integration happens at the artifact + CLI contract level and that orchestration runners — Claude Code skills/workflows today; opencode / codex-cli / copilot later — are **adapters** behind those contracts, with **degraded modes** for hosts lacking a capability (no workflow engine → skills path; no subagent spawning → human-run). Spec 109 declares and validates the `runner:` key (`claude-code-skills` default, `claude-code-workflow`, `external`) as the config expression of that principle; spec 111 is its first non-default consumer. ADR-0040 does **not** exist as a file on `main` until spec 109 merges, and spec 110 introduces the rule (R6) that a `## ADR Touchpoints` reference must resolve to an existing ADR file — so this spec, like 110, **references ADR-0040 in prose** (here and inside the ADR-0037 touchpoint) and cites only the already-Accepted, workflow-intersecting enforceable set. Specs 109 and 110 are hard prerequisites (see Open Questions); ADR-0040, the `runner:` key, and the `mindspec panel` verbs all exist in 111's base by plan and implementation time.

This spec deliberately stops at the panel runner. It does **not** workflow-ize autopilot or the bead cycle (future), change `PanelGateDecision` or `mindspec complete`, make the workflow a tally-consolidation or decision authority, enforce `loop:` / `models:`, or implement a non-Claude-Code runner (those adapters arrive with real second consumers per ADR-0040).

## Impacted Domains

- **workflow**: the sole impacted domain — every source edit lands under the `workflow` OWNERSHIP glob. A new Claude Code dynamic workflow `.claude/workflows/ms-panel.js` (the installed dogfood copy) plus its embedded plugin source `plugins/mindspec/workflows/ms-panel.js` is the runner adapter; `plugins/mindspec/embed.go` gains a `WorkflowFiles()` accessor (a `//go:embed workflows/*` mirror of `SkillFiles()`) and `internal/setup/claude.go` installs the embedded workflow to `.claude/workflows/`; `plugins/mindspec/skills/ms-panel-run` and `.../ms-panel-tally` (mirrored under `.claude/skills/**`) gain the runner branch and the slimmed workflow-path prose; `internal/validate` gains a test pinning `attributeDomain` → `workflow` for the new path; and `.mindspec/domains/workflow/OWNERSHIP.yaml` gains the `.claude/workflows/**` claim (R9). All of these paths are claimed by `.mindspec/domains/workflow/OWNERSHIP.yaml` — including the `.claude/workflows/**` glob this spec adds — and `plugins/mindspec/workflows/**` is already covered by the existing `plugins/mindspec/**` glob (no new claim needed there).

Note — the substrate this spec consumes is read-only and therefore **NOT impacted**, exactly as spec 110 consumed its dependencies. This note is deliberately plain prose, not bullet points, so `contextpack.ParseSpec` extracts exactly one Impacted-Domain (`workflow`) and spec 110's spec-approve parity check resolves it cleanly. **core** (`internal/config`): the runner-dispatch reads the `runner:` key and the `panel:` mix / `substitution.claude_sub_on_quota` flag via `mindspec config show` (spec 109) — no `internal/config` source is edited. The `mindspec panel create | verify | tally` verbs and the verdict-file/slot schema doc (spec 110, `internal/panel` + `cmd/**`) are **invoked as an existing CLI + artifact contract, not modified** — `PanelGateDecision` and `mindspec complete` are untouched. The codex CLI is exec'd by a workflow *agent's own shell tool*, not through `internal/executor`, so **execution** is not impacted either. No path outside the `workflow` OWNERSHIP globs is edited as source, so no other domain claim is needed and none is trippable by `adr-divergence-unowned`.

## ADR Touchpoints

- [ADR-0037](../../adr/ADR-0037-panel-gate-enforced-contract.md): Panel Gate as Enforced Contract (Domains: workflow, execution). The spec's spine. The workflow is a **read-side + write-side adapter** that produces exactly the artifacts the gate reads — it registers via `mindspec panel create` (the `panel.json` §3 pins) and returns `mindspec panel verify` / `mindspec panel tally` output (both reusing `panel.PanelGateDecision`, adding no second interpreter), and it **never** runs `mindspec complete` — so "identical decision over identical facts" holds and `internal/panel/gate.go` stays the single home. Per ADR-0040 (landing in spec 109) the `panel.json` / verdict-JSON schemas and the `mindspec panel` verbs are precisely the artifact + CLI contract a runner adapter targets; this workflow is the first such adapter. **Honest boundary vs spec 110:** unlike 110's `panel create` (which rev-parses the target through the executor git-I/O boundary, ADR-0030), 111 adds no executor code — the codex CLI is exec'd by a workflow agent's own shell tool, outside `internal/executor`. §§6/8 (fail-open/closed, trust boundary — plain reviewable files, no signing) are unchanged; R3b **extends** §8's "plain reviewable files" to the codex transcription step by persisting each codex slot's raw stdout as a tracked audit artifact, so the wrapper's verdict is auditable against its source.
- [ADR-0035](../../adr/ADR-0035-agent-error-contract.md): Agent Error Contract — recovery lines and exit codes (Domains: workflow, execution, core). A codex wrapper agent's quota-wall detection, the substitution branch, and any workflow-surfaced failure must be guard-style, not silent: a malformed or missing verdict surfaces through `mindspec panel verify`'s named-malformed-file / incomplete report and through `mindspec panel tally`'s non-zero Block with a recovery line (both already ADR-0035-compliant in spec 110), never as a dropped slot the workflow hides. The workflow's own result carries the verify/tally output verbatim so the orchestrator sees the same recovery lines the CLI would print.
- [ADR-0036](../../adr/ADR-0036-ownership-discovery.md): Ownership Discovery — zero framework cognition for OWNERSHIP.yaml (Domains: workflow, validation, doc-sync, ownership). Governs the **new `.claude/workflows/**` claim**: `.claude/workflows/ms-panel.js` is governable source (`isDocFile` false, `isProcessArtifact` false — verified), so it must be claimed in the same diff as, or an earlier bead than, any edit under `.claude/workflows/**` (the exact spec-108 `internal/trace` / `.golangci.yml` same-diff-or-earlier invariant, R9). Also governs the gate-forward doc-sync constraint — every bead editing `workflow` source appends to a `workflow` domain doc, and the bead editing `OWNERSHIP.yaml` (itself workflow-owned non-doc source) does likewise.
- [ADR-0034](../../adr/ADR-0034-ceremony-collapse.md): Ceremony Collapse — single-bead lifecycle (Domain: workflow). The workflow adds **no new ceremony step and no new gate** — it replaces the launch mechanics of an existing panel with one invocation and hands off to the same `mindspec complete` merge terminal the skills path uses. The runner selection is a dispatch, not an additional lifecycle transition.

## Requirements

1. **The workflow as a tracked, versioned repo artifact.** Add `.claude/workflows/ms-panel.js` (the installed dogfood copy Claude Code reads) and `plugins/mindspec/workflows/ms-panel.js` (the plugin source of truth that R8 embeds), a Claude Code dynamic workflow invocable as `/ms-panel`. It accepts, via the documented `args` input, `{slug, spec, target, bead_id?, round, sha?, lenses[], mix}`, where `mix` is the resolved reviewer mix from config `panel:` (spec 109, e.g. `[{claude,3},{codex,3}]`) and `lenses[]` is the per-slot lens list the skill composes (R6). `sha?` is **advisory-only** — an optional display hint the BRIEF may show; the authoritative `reviewed_head_sha` is self-resolved by `mindspec panel create` from `--target` at write time (110 R1), so the workflow never uses `sha?` to set the recorded SHA (and works correctly when it is omitted). The two tracked copies are byte-identical (R8). The workflow orchestrates agents and does **not** itself exec shell or write files — the documented limit ("Agents read, write, and run commands. The script coordinates the agents"); every CLI call and file write in R2–R5 is an agent step. Falsified if the file is absent from either tracked location, if it is not a saved workflow invocable by name, if it derives its reviewer mix from anywhere but the passed `mix`, if `sha?` (rather than `panel create`) sets the recorded `reviewed_head_sha`, or if the script itself (rather than an agent step) is specified to exec a CLI or write a file.

2. **Deterministic registration through the 110 verb — the workflow never hand-writes `panel.json`.** The workflow's registration step invokes `mindspec panel create <slug> --spec <id> --target <ref> [--bead <id>] [--round N]` (spec 110 R1) through an agent step, so `panel.json` — with `round` and `reviewed_head_sha` co-bumped by construction and `expected_reviewers` / `approve_threshold` stamped from the 109 resolvers — is written by the binary. A re-panel invokes `panel create --round N+1` (re-resolving the SHA in the same write, prior-round verdict files untouched). The workflow contains **no** hand-typed `panel.json` schema and **no** re-implementation of the round+SHA co-bump. Falsified if the workflow writes `panel.json` by any path other than `mindspec panel create`, if it re-implements the co-bump invariant, or if it sources `expected_reviewers` / `approve_threshold` from anywhere but the create verb.

3. **Reviewer fan-out per mix: Claude `agent()` steps + Codex wrapper agents, all producing schema-conformant verdict files at the 110 contract path.** For each slot in `mix`, the workflow fans out one agent step:
   - A **claude** slot is an `agent()` step prompted with the BRIEF (path written by `panel create`) plus the 110 verdict-JSON shape (`reviewer_id`, `verdict`, `confidence`, `rationale`, `concrete_changes_required`, `findings`; optional `hard_block`) and its assigned lens, instructed to write its verdict to the 110 contract path `<spec-dir>/reviews/<slug>/<slot>-round-<N>.json`.
   - A **codex** slot is a **wrapper agent** that execs the codex CLI (`codex exec`) with the BRIEF prompt + lens, **persists codex's raw stdout verbatim** to the tracked audit artifact `<spec-dir>/reviews/<slug>/<slot>-round-<N>.codex.log` (R3b), parses that stdout into the verdict shape, and writes the verdict file **itself** via its `Write` tool. **Codex never writes files** — neither the verdict nor the log; the wrapper writes both.

   Because Claude Code workflows expose no documented output-schema enforcement (Background), verdict conformance is enforced by the prompt specifying the shape **and** deterministic post-hoc validation via `mindspec panel verify` (R5 / 110 R2), which names malformed files. **The parse-failure path is semantics-preserving and can never launder a verdict** (the anti-laundering ladder — gate integrity):
   1. **Parse failure on a *rendered* verdict** (the reviewer produced verdict content but it is not valid schema JSON) → the workflow re-prompts the **same** reviewer **exactly once**, feeding back that reviewer's own rendered output (for a codex slot, the persisted `.codex.log`) and instructing it to re-emit **that same verdict** as valid JSON **without re-reviewing**. This is a serialization retry, never a fresh review; the re-emitted verdict keeps the **same slot id and same family** `reviewer_id` (e.g. `R4 codex` stays `R4 codex` — **not** `claude-sub`).
   2. **Still unparseable after the single re-prompt** → the slot **fails CLOSED to a MISSING verdict** (no verdict file is written) → an incomplete panel → the gate **Blocks**. Such a slot is **never** substituted and **never** replaced by another reviewer's verdict.
   3. **Substitution (R4) is reserved EXCLUSIVELY for a quota-wall failure in which *no* verdict content was ever rendered** (codex hit its usage limit before producing any verdict) — and even then only when `claude_sub_on_quota == true`. A rendered verdict that fails to parse is handled solely by steps 1–2 above; it is out of substitution's reach.

   This design eliminates the two production failure classes `ms-panel-run`'s bash detection existed to catch: the **codex sandbox-file-write failure** (codex writing outside its workdir sandbox) cannot occur because the wrapper agent, not codex, writes the file; the **verdict-parse failure** is caught deterministically by `panel verify` and resolved by a semantics-preserving re-serialization rather than by log-scraping heuristics. Falsified if a codex slot has codex write its own verdict file, if any verdict path deviates from the 110 `<slot>-round-<N>.json` contract, if the spec relies on a platform-level output-schema guarantee the Claude Code workflow docs do not provide, **or if any path exists by which a rendered `REJECT` / `REQUEST_CHANGES` verdict can be replaced by a different reviewer's verdict or by a re-reviewed `APPROVE`** — a parse failure on a rendered verdict resolves only to the same reviewer's re-serialized verdict or to a MISSING slot, never to a substitution.

3b. **Codex raw stdout is a tracked audit artifact.** Every codex wrapper agent persists codex's unmodified stdout to `<spec-dir>/reviews/<slug>/<slot>-round-<N>.codex.log` before (or as) it transcribes the verdict, so the wrapper's transcription is auditable against its source — the ADR-0037 §8 "plain reviewable files" principle extended to the transcription step (a human or a later reviewer can diff the recorded verdict against the codex log it was derived from). The `.codex.log` sits in the panel dir alongside the verdict and is written by the wrapper agent, never by codex. Falsified if a codex slot writes a verdict without also persisting its `<slot>-round-<N>.codex.log`, or if the log is written anywhere but the panel dir.

4. **Quota-wall substitution is a deterministic catch branch, not judgment — and it applies ONLY when no verdict was rendered.** When a codex wrapper agent detects a quota wall (the codex CLI usage-limit signal) **before any verdict content was rendered** (the R3 step-3 precondition — an empty-handed slot, not a rendered-but-malformed one) **and** config `panel.substitution.claude_sub_on_quota == true` (spec 109 default), the workflow substitutes a **claude** agent for that slot in the same round, **keeping the slot id** and writing the verdict with `reviewer_id: "<slot> claude-sub"` (the `ms-panel-run` substitution contract, now a code branch rather than an operator's judged persona swap). When `claude_sub_on_quota == false`, the slot is left unfilled and surfaces as a missing verdict at `panel verify` (an incomplete panel → the gate Blocks) — the workflow does not silently pass a short panel. A parse failure on a *rendered* verdict is **out of this branch's scope** — it is handled solely by R3's same-reviewer re-serialize-or-MISSING ladder, so substitution can never displace a verdict a reviewer actually rendered. Substitution is a branch in the workflow script driven by the wrapper agent's returned status; it requires no human decision on the workflow path. Falsified if substitution on the workflow path requires an operator judgment call, if it is triggered by a parse failure on a rendered verdict (rather than a quota wall with no verdict), if the substituted verdict loses the slot id or the `claude-sub` `reviewer_id`, if it ignores the 109 `claude_sub_on_quota` flag, or if a `false` flag causes the workflow to fabricate or skip the slot rather than leave it missing.

5. **The workflow returns verify + tally-preview; it is neither the decision authority nor a lifecycle mutator.** After fan-out, the workflow invokes `mindspec panel verify <slug>` (110 R2, read-only completeness/staleness + PASS/BLOCK preview) and `mindspec panel tally <slug>` (110 R3, per-slot verdict table + aggregate + decision + the mechanical `concrete_changes_required` union) through agent steps, and returns their combined output **verbatim** (the CLI stdout passed through unmodified, not re-rendered or paraphrased) as the workflow **result** — the single compact structured result the context-economy rationale in Background calls for. To make "not a lifecycle mutator" structural rather than aspirational, the workflow constrains its agent steps to an explicit **command allowlist** (a declared `ALLOWED_CLI` set) containing exactly `mindspec panel create`, `codex exec`, `mindspec panel verify`, and `mindspec panel tally`; `mindspec complete` and every other lifecycle-mutating verb are **excluded from it and appear nowhere in the workflow file** (not as a command, not as a comment). The workflow does **not**: run `mindspec complete`; perform tally-**consolidation** (semantic dedup + criticality ranking — that is skill judgment, 110 R8); author `consolidated-round-<N>.md`; or mutate `panel.json` beyond what `panel create` wrote. `internal/panel/gate.go`'s `PanelGateDecision` (via `mindspec complete`) remains the sole gate authority. Falsified if the workflow runs or references `mindspec complete` or any lifecycle-mutating verb, if the `ALLOWED_CLI` allowlist admits any command beyond the four named, if it performs the semantic consolidation, if it writes `consolidated-round-<N>.md`, if it alters the verify/tally output rather than passing it through verbatim, or if it re-implements the decision rather than returning `panel tally`'s.

6. **Runner dispatch in `ms-panel-run` — the 109 `runner:` key selects the path.** `ms-panel-run/SKILL.md` reads the effective `runner:` value (via `mindspec config show`, spec 109) and branches: `claude-code-workflow` → **compose the slot lenses (the retained judgment step) then invoke the `/ms-panel` workflow once** with the resolved `{slug, spec, target, bead_id?, round, lenses[], mix}`; `claude-code-skills` (the default until the workflow path is proven) → the existing manual launch path runs **unchanged**; `external` → a documented out-of-scope **stub** (no adapter is shipped; the skill states the panel runs human/skills-path per ADR-0040 degraded modes). A host lacking workflow capability degrades to the skills path (ADR-0040 capability tiers). The skill explicitly labels which of its sections are workflow-path vs skills-path. Falsified if the skill hard-codes a single runner, if it invokes the workflow when `runner != claude-code-workflow`, if it fails to retain the skills-path launch mechanics for the default runner, or if it does not read the runner value from config.

7. **Skill slimming — the mechanized launch prose is superseded for the workflow path; judgment and the default path survive.** Under the `claude-code-workflow` branch of `ms-panel-run/SKILL.md`, the **Launch the panel**, **Codex failure detection (deterministic)**, **Working directory matters**, and the codex-specific **Anti-patterns** are **superseded** by the schema+wrapper mechanism (R3–R4) and the single workflow invocation (R6); the skill states this and does not re-state that bash detection for the workflow path. The **Slot lens defaults** section (lens composition = judgment, L4) **stays** and is the workflow path's sole authoring step. Those four superseded sections **remain in the skill as the `claude-code-skills` default path** — they are gated behind the runner branch, **not deleted** (the default path still hand-drives the launch). In `ms-panel-tally/SKILL.md`, a note records that on the workflow path the verdict table + decision arrive **pre-rendered in the workflow result** (an already-run `panel tally`), so the skill's job narrows to Step 4 consolidation + the merge terminal; the judgment sections (**Consolidate**, **Artifact gates**, **After a halt — recovery**, **Escape hatch**) are **unchanged**. Falsified if the skills-path launch mechanics are deleted rather than gated, if **Slot lens defaults** is removed, if the workflow path re-states the superseded bash detection, or if any named `ms-panel-tally` judgment section is deleted.

8. **Distribution: the workflow ships to the Claude Code target only (capability-tiered).** `plugins/mindspec/embed.go` gains a `//go:embed workflows/*` directive and a `WorkflowFiles()` accessor mirroring `SkillFiles()`; `internal/setup/claude.go` installs the embedded workflow(s) to `.claude/workflows/` (idempotent create/refresh, mirroring `installSkills`), so `mindspec setup` for the Claude target produces `.claude/workflows/ms-panel.js` byte-identical to the embedded `plugins/mindspec/workflows/ms-panel.js`. The **codex and copilot** setup targets (which install to `.agents/skills/`) do **not** receive the workflow — dynamic workflows are a Claude Code capability (ADR-0040 capability tiers); those runners stay skills-path. Falsified if the embedded and installed copies diverge, if `mindspec setup` (Claude target) does not produce `.claude/workflows/ms-panel.js`, or if a non-Claude setup target writes the workflow anywhere.

9. **Ownership claim lands first (`.claude/workflows/**`).** Add `.claude/workflows/**` to the `paths:` list of `.mindspec/domains/workflow/OWNERSHIP.yaml`. Because `.claude/workflows/ms-panel.js` is governable source (`isDocFile` and `isProcessArtifact` both return false for it — verified against `internal/validate`), editing it while unclaimed trips `adr-divergence-unowned`. The claim must be present **in the same diff as, or an earlier bead than,** any edit under `.claude/workflows/**` — divergence reads each domain's `OWNERSHIP.yaml` at the diffed ref, so the claim only needs to be visible at that ref (same-diff-or-earlier is the exact invariant, the spec-108 precedent). Once present, `validate.attributeDomain` returns `"workflow"` for `.claude/workflows/ms-panel.js`. Because `.mindspec/domains/workflow/OWNERSHIP.yaml` is itself workflow-owned non-doc source, the bead editing it must also touch a `workflow` domain doc in the same diff (gate-forwardness). Falsified if the claim is absent, if `attributeDomain` does not return `"workflow"` for the workflow file, or if any bead diff contains an unclaimed `.claude/workflows/**` path.

## Scope

### In Scope
- `.claude/workflows/ms-panel.js` (installed dogfood copy) and `plugins/mindspec/workflows/ms-panel.js` (embedded plugin source of truth): the Claude Code dynamic workflow implementing registration → fan-out (with per-codex-slot raw-stdout audit logs) → semantics-preserving parse retry / quota-wall substitution → verify/tally-return, under a declared `ALLOWED_CLI` command allowlist (R1–R5, R3b), byte-identical copies.
- `plugins/mindspec/embed.go`: `//go:embed workflows/*` + `WorkflowFiles()` (R8).
- `internal/setup/claude.go`: install the embedded workflow to `.claude/workflows/` on the Claude target, mirroring `installSkills`; codex/copilot targets unchanged (R8).
- `plugins/mindspec/skills/ms-panel-run` + `ms-panel-tally` (and their `.claude/skills/**` mirrors): the runner branch, the retained skills-path launch mechanics, the retained **Slot lens defaults**, and the superseded-for-workflow-path prose (R6, R7).
- `internal/validate`: a test pinning `attributeDomain` of `.claude/workflows/ms-panel.js` to `"workflow"` after the claim (R9), mirroring spec 108's `TestWorkflowOwnsTraceAndGolangci`.
- `.mindspec/domains/workflow/OWNERSHIP.yaml`: add the `.claude/workflows/**` claim (R9).
- Per-domain doc-sync (gate-forward, planning constraint): every bead that edits `workflow` source appends a **bead-unique** region to a `workflow` domain doc (`.mindspec/domains/workflow/*.md`) — the new-verb-consuming workflow artifact, the embed/setup plumbing, the skill edits, the validate test, and the OWNERSHIP claim are all `workflow`, so each such bead documents its change there.

### Out of Scope
- Workflow-izing **autopilot** or the **bead cycle** (`ms-spec-autopilot` / `ms-bead-cycle`) — a future spec; 111 is the panel runner only.
- Any change to `internal/panel.PanelGateDecision`, the `mindspec panel create | verify | tally` verbs (spec 110), or `mindspec complete` — 111 consumes them as an unchanged contract; it does **not** make the workflow a tally-consolidation, decision, or lifecycle authority.
- In-binary enforcement of `loop:` or `models:` (spec 109 substrate, inert here) and any `mindspec config` change — read-only consumed.
- The `claude-code-skills` default path's mechanics — **retained unchanged** (they are not the subject of this spec beyond being gated behind the runner branch).
- The `external` runner and any non-Claude-Code runner **implementation** — 111 ships only the `claude-code-workflow` adapter and an `external` documentation stub; other adapters arrive with real second consumers per ADR-0040.
- Signing, hashing, or moving panel/verdict state out of the repo — ADR-0037 §8; panel state stays plain reviewable files.

## Non-Goals

- No new decision semantics and no new gate. `mindspec complete` remains the authoritative panel-gate enforcer; the workflow returns `panel verify` / `panel tally` output (read-side surfaces over the unchanged decision) and never enforces or mutates the lifecycle.
- No claim of a platform output-schema guarantee. Verdict conformance is enforced by prompt + `mindspec panel verify`, not by a Claude Code workflow feature that the docs do not describe.
- No telemetry stream. Consistent with ADR-0027 (OTEL-only, pull surfaces), the workflow and its steps emit no metrics; the workflow result is the derived verify/tally output.
- No change to how a spec/plan/bead declares or runs its panel beyond the runner selection — the workflow is an adapter behind the existing artifact + CLI contracts, not a new authoring surface.

## Open Questions

_All open questions are resolved; every resolution is folded into the Requirements, Scope, and ADR Touchpoints above._

- **RESOLVED — the workflow is an adapter, not a second decision authority.** The decision stays in `internal/panel/gate.go` and is enforced solely by `mindspec complete`; the workflow writes the same `panel.json` + verdict artifacts and returns `panel verify` / `panel tally` output, but never runs `mindspec complete`, never consolidates, and never authors `consolidated-round-<N>.md` (R5). This is the ADR-0037 "single home / identical decision over identical facts" invariant held across the new adapter, and the ADR-0040 portability principle (integrate at the artifact + CLI contract, not the prompt format).
- **RESOLVED — verdict validity is enforced by prompt + `mindspec panel verify`, not a platform schema, and a parse failure can never launder a verdict.** The Claude Code workflows docs describe no output-schema/structured-output enforcement, so the spec does not assume one. Each verdict is specified in the agent prompt (the 110 shape) and validated deterministically by `mindspec panel verify`'s per-slot parse-status report (110 R2). On a parse failure of a *rendered* verdict the wrapper re-prompts the **same** reviewer once to re-serialize its **own** verdict verbatim (never a fresh review, never a different reviewer); a persistent failure fails **closed** to a MISSING slot — never a substitution (the R3 anti-laundering ladder; substitution is reserved for a quota wall with no verdict at all). This is more honest and more portable than a runner-specific schema feature, and it removes the verdict-parse failure class from the critical path without ever letting a rendered `REJECT`/`REQUEST_CHANGES` be replaced by an `APPROVE`.
- **RESOLVED — codex slots are wrapper agents; codex never writes files.** Because the workflow script has no filesystem/shell access (only its agents do — documented limit), a codex slot is an agent that execs `codex exec`, persists codex's raw stdout as a tracked audit artifact (`<slot>-round-<N>.codex.log`, R3b, so the transcription is reviewable against its source), parses that stdout, and writes the verdict itself. This eliminates the codex sandbox-file-write failure class (the `ms-panel-run` "Working directory matters" hazard) by construction — the file is written by an agent in the workflow's environment, not by codex in its sandbox (R3).
- **RESOLVED — the re-round cache is the persisted verdict files, not workflow resume.** Claude Code workflow resume is session-scoped (completed agents return cached results within a session; exiting restarts fresh). The durable re-round mechanism is the 110 artifact contract: `panel create --round N+1` bumps the round and leaves prior-round `<slot>-round-<K>.json` files untouched, so a re-panel runs only the current round's slots. State lives on disk (ADR-0023 beads-as-truth spirit), not in workflow-runtime memory (Background).
- **RESOLVED — quota-wall substitution is a deterministic branch honoring the 109 flag, and only for a no-verdict quota wall.** `panel.substitution.claude_sub_on_quota` (spec 109) drives a code branch in the workflow: `true` → substitute a `claude` agent keeping the slot id with `reviewer_id "<slot> claude-sub"`; `false` → leave the slot missing (an incomplete panel the gate Blocks). Substitution is policy, not judgment — no operator decision on the workflow path — and it fires **only** when codex hit its usage limit with *no* verdict rendered; a rendered-but-malformed verdict is handled solely by R3's same-reviewer re-serialize-or-MISSING ladder, so substitution can never displace a verdict a reviewer actually produced (R4).
- **RESOLVED — the `.claude/workflows/**` claim is required and lands same-diff-or-earlier.** `.claude/workflows/ms-panel.js` is governable source (`isDocFile` / `isProcessArtifact` both false — verified), and `workflow` today claims only `.claude/skills/**`, not all of `.claude/**`. So the new `.claude/workflows/**` claim is added to the workflow `OWNERSHIP.yaml` in the same diff as, or an earlier bead than, the workflow file — the exact spec-108 `internal/trace` / `.golangci.yml` invariant. `plugins/mindspec/workflows/**` needs no new claim (covered by the existing `plugins/mindspec/**` glob). (R9.)
- **RESOLVED — the workflow ships to the Claude Code target only.** Dynamic workflows are a Claude Code capability; codex/copilot installs (which write to `.agents/skills/`) stay skills-path per ADR-0040 capability tiers. The embed (`plugins/mindspec/embed.go`) + Claude-target install (`internal/setup/claude.go`) mirror the skills-distribution pattern so downstream `mindspec setup` (Claude) users can flip `runner: claude-code-workflow`; without this the adapter would only run in the mindspec checkout, contradicting the portability principle (R8).
- **RESOLVED — specs 109 and 110 are hard prerequisites; 111 rebases forward before plan-approve.** Spec 109 (approved) provides ADR-0040, the `runner:` key, and the `panel:` mix / `substitution.claude_sub_on_quota` config; spec 110 (in flight) provides the `mindspec panel create | verify | tally` verbs and the verdict-file/slot schema doc, and slims `ms-panel-run` step 0 / `ms-panel-tally` to the state 111 further branches. So 111 may enter Plan Mode (`plan approve`) only after **both** 109 and 110 have merged to `main` **and** `spec/111-workflow-panel-runner` has been rebased or merged-forward onto that post-109/110 `main` before `plan approve` — the spec-108→107 ordering pattern. The plan, authored on that post-109/110 base, **may** cite ADR-0040 in frontmatter for coverage, though the sole impacted domain (`workflow`) is already covered by ADR-0037/0036/0034.
