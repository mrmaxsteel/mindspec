# ADR-0020: Direct Phase Tracking — Drop Ceremony Molecules

- **Date**: 2026-02-28
- **Status**: Proposed
- **Domain(s)**: workflow, beads
- **Deciders**: Max
- **Supersedes**: ADR-0013, ADR-0015
- **Superseded-by**: n/a

## Context

MindSpec models the spec lifecycle (spec → plan → implement → review) using a Beads molecule with 7 issues: 1 epic + 6 steps. Each lifecycle command closes a molecule step, then the current mode is derived by querying which steps remain open.

This architecture was established by ADR-0013 (Use Beads Formulas) and ADR-0015 (Molecule-Derived State). In practice, 6 of the 7 molecule steps are **state machine nodes, not work items**. Nobody assigns, estimates, or comments on them. They are created at spec-init, sit idle, and get closed one-by-one as lifecycle commands run. Their sole purpose is to make `deriveMode()` work — a function that could be replaced by reading a single field.

Meanwhile, the **implementation beads** (created at plan-approve from plan sections) are genuine work items with dependencies, parallel execution, and tracked completion. These justify structured tracking.

The molecule ceremony imposes real costs:

- **Complexity**: `specmeta.EnsureFullyBound`, `recoverStepMapping`, `findMoleculeByConvention` — an entire recovery/reconciliation subsystem exists because molecule binding can get out of sync with spec.md frontmatter.
- **Overhead**: Every `state show` calls `bd mol show` to derive mode. Every `approve impl` iterates 7+ issues calling `bd close` individually.
- **Indirection**: Mode is derived (query molecule → map step statuses → compute phase) rather than read (open file → read field).
- **Issue bloat**: ~140 of 257 issues in `.beads/issues.jsonl` are molecule ceremony steps.

## Decision

**Drop the spec-lifecycle molecule. Track lifecycle phase as a direct field in mode-cache. Retain Beads only for implementation work items.**

### What changes

| Before | After |
|:---|:---|
| `spec-init` pours a 7-issue molecule | `spec-init` creates no Beads issues |
| Mode derived from molecule step statuses | Phase is a direct field in mode-cache |
| `spec.md` has `molecule_id` + `step_mapping` | `spec.md` has no molecule fields |
| `approve plan` closes molecule step, creates impl beads under molecule's implement step | `approve plan` creates a standalone epic + impl beads, writes `epic_id` to plan.md |
| `approve impl` closes 7+ molecule members | `approve impl` closes impl beads + parent epic only |
| `specmeta` package reconciles molecule binding | `specmeta` package deleted |

### What stays the same

- Implementation beads: created at plan-approve, tracked via `bd ready/close`, with real dependencies
- User-facing commands: `approve spec`, `approve plan`, `next`, `complete`, `approve impl` — same interface
- Branch/worktree lifecycle: unchanged
- Hook system: still reads mode-cache (already does, unchanged)
- Recording system: still emits lifecycle markers

## Decision Details

### Phase transitions become direct writes

Each lifecycle command writes the phase transition atomically:

| Command | Writes |
|:---|:---|
| `spec-init` | `phase: spec` |
| `approve spec` | `phase: plan` |
| `approve plan` | `phase: implement` |
| `complete` (last bead) | `phase: review` |
| `approve impl` | `phase: idle` |

No molecule query needed. The mode-cache file is the source of truth.

### Implementation beads are standalone

At plan-approve:
1. Create a parent epic: `bd create --title "[SPEC <id>] Implementation" --type epic`
2. Create child beads under it: `bd create --parent <epic-id> ...` (same as today)
3. Wire dependencies: `bd dep add` (same as today)
4. Write `epic_id` to plan.md frontmatter (replaces `molecule_id` + `step_mapping`)

At approve-impl:
1. Verify all impl beads are closed (read `bead_ids` from plan.md, check each)
2. Close the parent epic
3. Merge and clean up (same as today)

### `state show` reads directly

Instead of calling `bd mol show` + `deriveMode()`, `state show` reads mode-cache and returns it. For multi-spec detection, scan spec dirs for mode-cache presence rather than querying molecules.

## Consequences

### Positive

- **~500-800 lines of Go code deleted** — specmeta package, molecule derivation, recovery logic, formula embedding
- **Faster lifecycle commands** — no `bd mol show` queries, no multi-issue close loops
- **Simpler mental model** — phase is stored where you read it, not derived from a side system
- **Fewer Beads issues** — future specs create ~N+1 issues (N impl beads + 1 epic) instead of N+7
- **No more molecule binding recovery** — `EnsureFullyBound` and its error paths disappear

### Negative / Tradeoffs

- **Mode-cache is now authoritative, not a cache** — if it gets corrupted, there's no molecule to re-derive from. Mitigated by: the file is small, JSON, and written atomically by lifecycle commands.
- **ADR-0015's "single source of truth in Beads" property is lost** — the lifecycle state is now in a local file, not in Beads' database. But this was always aspirational; hooks already treated mode-cache as authoritative for performance.
- **Existing specs with molecule bindings need migration** — any in-flight specs will have `molecule_id`/`step_mapping` in frontmatter that becomes vestigial. The migration can strip these fields or ignore them.

## Alternatives Considered

### 1. Keep the molecule but skip ceremony step creation

Create only the `implement` step as a molecule, skip the other 5. Rejected because: a single-step molecule is just an epic, and `bd create --type epic` already does that without the formula machinery.

### 2. Keep full molecules but cache mode more aggressively

Make mode-cache writes more reliable and never query the molecule at runtime. Rejected because: this preserves all the complexity (specmeta, step_mapping, formula) for a "backup" that's never actually consulted. The molecule becomes write-only ceremony.

### 3. Do nothing

Live with the overhead. Rejected because: the complexity is actively harmful — every new contributor has to understand molecules, step mappings, and the derivation pipeline to modify lifecycle code.
