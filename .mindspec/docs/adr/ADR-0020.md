# ADR-0020: Per-Spec Lifecycle File — Drop Ceremony Molecules

- **Date**: 2026-02-28
- **Status**: Proposed
- **Domain(s)**: workflow, beads
- **Deciders**: Max
- **Supersedes**: ADR-0013, ADR-0015
- **Superseded-by**: n/a

## Context

MindSpec models the spec lifecycle (spec → plan → implement → review) using a Beads molecule with 7 issues: 1 epic + 6 steps. Each lifecycle command closes a molecule step, then the current mode is derived by querying which steps remain open.

This architecture was established by ADR-0013 (Use Beads Formulas) and ADR-0015 (Molecule-Derived State). In practice, 6 of the 7 molecule steps are **state machine nodes, not work items**. Nobody assigns, estimates, or comments on them. They are created at spec-init, sit idle, and get closed one-by-one as lifecycle commands run. Their sole purpose is to make `deriveMode()` work — a function that could be replaced by reading a single field.

The state architecture has evolved through three iterations, each fixing the previous:

1. **state.json (ADR-0005)**: Global file committed to git. Durable but caused merge conflicts, was noisy in diffs, and was a global singleton that couldn't support multiple specs.
2. **mode-cache (Spec 053, ADR-0015)**: Global file, gitignored. Fast and conflict-free, but ephemeral — safe only because the molecule served as backup for re-derivation. Removing the molecule (which this ADR does) would leave mode-cache authoritative but ephemeral.
3. **Per-spec lifecycle.yaml (this ADR)**: Per-spec file committed to git. Durable, versioned, and conflict-free because it lives on the spec branch where only one agent works. No molecule backup needed because the file itself is durable.

The molecule ceremony imposes real costs:

- **Complexity**: `specmeta.EnsureFullyBound`, `recoverStepMapping`, `findMoleculeByConvention` — an entire recovery/reconciliation subsystem for molecule binding drift.
- **Overhead**: `state show` calls `bd mol show`. `approve impl` iterates 7+ issues calling `bd close` individually.
- **Indirection**: Mode is derived (query molecule → map step statuses → compute phase) rather than read (open file → read field).
- **Issue bloat**: ~140 of 257 issues in `.beads/issues.jsonl` are molecule ceremony steps.

## Decision

**Drop the spec-lifecycle molecule. Track lifecycle phase in a per-spec `lifecycle.yaml` committed to git. Retain Beads only for implementation work items.**

### The lifecycle file

Each spec gets `docs/specs/<id>/lifecycle.yaml`:

```yaml
phase: implement
epic_id: mindspec-xyz   # set at plan-approve
```

This file is:
- **Created by `spec-init`** with `phase: spec`
- **Updated by lifecycle commands** (`approve spec` → `plan`, `approve plan` → `implement`, etc.)
- **Committed to git** on the spec branch — durable, versioned, survives session loss
- **Co-located with the spec** — lives in the same directory as `spec.md` and `plan.md`

### What changes

| Before | After |
|:---|:---|
| `spec-init` pours a 7-issue molecule | `spec-init` writes `lifecycle.yaml` with `phase: spec` |
| Mode derived from molecule step statuses | Phase read directly from `lifecycle.yaml` |
| `spec.md` has `molecule_id` + `step_mapping` | `spec.md` has no molecule fields |
| Global mode-cache stores phase | Focus file (`.mindspec/focus`) stores only `activeSpec` + `activeBead` |
| `approve plan` closes molecule step, creates impl beads under molecule's implement step | `approve plan` creates standalone epic + impl beads, writes `epic_id` to `lifecycle.yaml` |
| `approve impl` closes 7+ molecule members | `approve impl` closes impl beads + parent epic only |
| `specmeta` package reconciles molecule binding | `specmeta` package deleted |

### What stays the same

- Implementation beads: created at plan-approve, tracked via `bd ready/close`, with real dependencies
- User-facing commands: `approve spec`, `approve plan`, `next`, `complete`, `approve impl` — same interface
- Branch/worktree lifecycle: unchanged
- Recording system: still emits lifecycle markers

## Decision Details

### Phase transitions

Each lifecycle command writes the phase transition to `lifecycle.yaml`:

| Command | Phase written |
|:---|:---|
| `spec-init` | `spec` |
| `approve spec` | `plan` |
| `approve plan` | `implement` |
| `complete` (last bead) | `review` |
| `approve impl` | `done` |

### Mode-cache renamed to focus file

The global `.mindspec/mode-cache` is renamed to `.mindspec/focus` and demoted from "cached lifecycle state" to "which spec am I focused on":

```json
{"activeSpec": "054-foo", "activeBead": "mindspec-xyz.1"}
```

Phase is not stored here — it's read from the active spec's `lifecycle.yaml`. Commands can also auto-detect the spec from the current branch name (`spec/054-foo` → `054-foo`).

### Multi-spec detection

To find active specs, scan `docs/specs/*/lifecycle.yaml` for files where `phase` is not `done`. This replaces `ActiveSpecs()` which queried every molecule via `bd mol show`.

### Implementation beads are standalone

At plan-approve:
1. Create a parent epic: `bd create --title "[SPEC <id>] Implementation" --type epic`
2. Create child beads: `bd create --parent <epic-id> ...` (same as today)
3. Wire dependencies: `bd dep add` (same as today)
4. Write `epic_id` to `lifecycle.yaml`

At approve-impl:
1. Verify all impl beads are closed (read `bead_ids` from plan.md)
2. Close the parent epic
3. Merge and clean up (same as today)

## Consequences

### Positive

- **~500-800 lines of Go code deleted** — specmeta package, molecule derivation, recovery logic, formula embedding
- **Durable state without molecule overhead** — `lifecycle.yaml` is committed to git, survives session loss, no re-derivation needed
- **No merge conflicts** — file lives on the spec branch, only one agent writes to it, changes ~5 times per lifecycle
- **Simpler mental model** — phase is stored where you read it, co-located with the spec artifacts
- **Fewer Beads issues** — future specs create ~N+1 issues (N impl beads + 1 epic) instead of N+7
- **Natural multi-spec support** — each spec carries its own phase independently
- **No molecule binding recovery** — `EnsureFullyBound` and its error paths disappear

### Negative / Tradeoffs

- **Lifecycle state splits across two systems** — phase in git, bead statuses in Beads. But this split is clean: git tracks the macro lifecycle, Beads tracks the micro (implementation work items).
- **Existing specs need migration** — in-flight specs have `molecule_id`/`step_mapping` in frontmatter. Migration strips these and writes `lifecycle.yaml` from the molecule's current state.
- **Hooks need file I/O** — hooks that checked mode-cache for phase now read `lifecycle.yaml`. This is still a local file read (fast), but requires knowing the active spec's directory. The focus file's `activeSpec` field provides this.

## Alternatives Considered

### 1. Keep gitignored focus file as authoritative (no lifecycle.yaml)

Store phase in the focus file, derive from spec artifacts (spec.md approval + plan.md approval + bead closure) as fallback. Rejected because: the derivation logic reintroduces complexity, and ephemeral-but-authoritative state is fragile — if the focus file is lost, you need inference logic to recover.

### 2. Keep the molecule but skip ceremony step creation

Create only the `implement` step as a molecule. Rejected because: a single-step molecule is just an epic, and `bd create --type epic` already does that.

### 3. Return to global committed state.json

Like ADR-0005 but simplified. Rejected because: the per-spec approach avoids the merge conflict and multi-spec problems that motivated moving away from global state in the first place.

### 4. Do nothing

Live with the molecule overhead. Rejected because: the complexity is actively harmful — every new contributor has to understand molecules, step mappings, and the derivation pipeline to modify lifecycle code.
