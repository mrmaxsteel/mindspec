# ADR-0024: ADR Storage Abstraction — Interface-First, File-Based Default

- **Date**: 2026-03-08
- **Status**: Accepted
- **Domain(s)**: adr, context-system, workflow
- **Deciders**: Max
- **Supersedes**: n/a
- **Superseded-by**: n/a
- **Related**: ADR-0023 (beads as single state authority), Spec 075 (beads as context)

---

## Context

Spec 075's vision document ("Enterprise Knowledge Layer on Beads") proposes migrating ADRs from git-tracked markdown files to beads `decision` issues stored in Dolt, with a three-phase migration path ending in Backstage as the authoring UI and DoltLab as the system of record.

The vision is sound for enterprise scale (multi-repo federation, structured queryability, PM-accessible web UI). However, the current developer experience for browsing ADRs is excellent: open `.mindspec/docs/adr/ADR-NNNN.md` in VSCode, hit preview, read nicely rendered markdown. This workflow is fast, zero-infrastructure, works offline, and leverages the editor the developer is already in.

Moving ADRs to Dolt would break this workflow. The alternatives — terminal output via `bd show`, a VSCode extension, or a self-hosted DoltLab instance — are all worse for a solo developer or small team browsing decisions during development.

Meanwhile, multiple specs (075, 076) need to reference and validate ADRs. If these specs hardcode file-based access, the eventual Dolt migration becomes a rewrite. If they assume Dolt, the current UX regresses.

## Decision

ADR access throughout mindspec is mediated by an **interface** (Go interface or function signature), not by direct filesystem or beads calls. The default implementation reads from `.mindspec/docs/adr/` markdown files. A future implementation can read from beads `decision` issues via `bd list --type decision`.

### Design principles

1. **Consumers don't know the backend** — `contextpack`, `validate`, `instruct`, and `approve` call the interface, not `os.ReadFile` or `bd list` directly.
2. **File-based is the default** — no Dolt dependency for ADR access. The markdown-in-VSCode workflow is preserved.
3. **The interface is minimal** — `List()`, `Get(id)`, `Search(query)`. Don't over-abstract; add methods when a consumer needs them.
4. **Migration is additive** — when beads decisions are ready, add a new implementation behind the same interface. The switchover is a configuration change, not a code rewrite.

### What this does NOT decide

- When to migrate ADRs to Dolt (deferred to Spec 075 phasing)
- Whether to maintain a generated file mirror alongside Dolt (implementation detail of the Dolt backend)
- The Backstage integration surface (Phase 3+ concern)

## Consequences

### Positive

- **Current UX preserved** — developers continue browsing ADRs as markdown in their editor
- **Future-proof** — 075's Dolt migration slots in without rewiring consumers
- **Specs 075 and 076 can proceed** — both reference ADRs through the interface without coupling to either backend
- **Testable** — interface enables mock/stub backends in tests

### Negative / Tradeoffs

- **Indirection** — one more layer between consumers and ADR files. Mitigated by: the interface is thin (3 methods); the file-based implementation is trivial.
- **Two backends to maintain eventually** — when Dolt support is added, both implementations need to stay in sync. Mitigated by: the interface contract is the sync point; integration tests can verify both backends return equivalent results.

## Alternatives Considered

### 1. Migrate ADRs to Dolt now

Move all ADRs to beads `decision` issues immediately as part of Spec 075.

- Pros: Single source of truth from day one; enables structured queries (`bd list --type decision --domain payments`)
- Cons: Breaks the VSCode browse-and-preview workflow; requires DoltLab or a custom viewer for readable output; over-engineers for current single-developer scale
- Rejected: The UX regression is not justified at current scale. The interface approach preserves optionality without sacrificing the workflow that works today.

### 2. File-based only, no abstraction

Keep ADRs as files and let consumers access them directly. Deal with Dolt migration later.

- Pros: Simpler now; no interface to design
- Cons: Every consumer that reads ADR files becomes a migration liability; Spec 076's validator would hardcode file paths that need rewriting later
- Rejected: The abstraction cost is minimal and the migration cost of direct access is high.

### 3. Generated file mirror from Dolt

Use Dolt as SOT but auto-generate markdown files for browsing.

- Pros: Best of both worlds — structured queryability plus file browsing
- Cons: Two representations to keep in sync; stale mirror risk; adds complexity now for a backend that doesn't exist yet
- Rejected as premature: This may be the right approach when Dolt becomes the SOT, but designing the sync mechanism before the Dolt backend exists is speculative. The interface approach doesn't preclude adding a mirror later.

## Validation / Rollout

1. Define the ADR access interface in `internal/adr/` (or a new `internal/decision/` package if the naming better reflects the dual-backend future)
2. Implement the file-based backend (largely extracting existing code from `internal/adr/` and `internal/contextpack/adr.go`)
3. Update consumers (`contextpack`, `validate`, `instruct`, `approve`) to use the interface
4. Spec 076's validator uses the interface for ADR lookups — no direct file access
5. When Spec 075 Phase 2 adds beads decisions, implement the Dolt backend behind the same interface
