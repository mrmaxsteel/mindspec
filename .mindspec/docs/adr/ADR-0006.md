# ADR-0006: Protected Main Branch with PR-Based Merging

- **Date**: 2026-02-12
- **Status**: Proposed
- **Domain(s)**: workflow, git
- **Deciders**: MindSpec maintainers
- **Supersedes**: n/a
- **Superseded-by**: n/a

---

## Context

MindSpec's current workflow assumes direct commits to `main` at every mode transition:

- **Spec exit**: `spec(<bead-id>): ...` committed directly to main
- **Plan exit**: `plan(<bead-id>): ...` committed directly to main
- **Implementation exit**: `impl(<bead-id>): ...` committed directly to main

This is documented in AGENTS.md (Milestone Commits), MODES.md (Exit Gates), and the instruction templates emitted by `mindspec instruct`.

Direct-to-main works well for single-developer, local-first workflows. However, it has limitations:

- **No review gate**: Milestone commits land without review. For teams or when working with AI agents, a review step before merging to main provides an additional safety net.
- **No rollback unit**: Individual commits on main are harder to revert cleanly than merged PRs, which revert as a single unit.
- **No CI gate**: Branch protection enables required status checks (tests, linting, validation) before code reaches main.
- **Spec/plan phases unprotected**: Even markdown-only spec and plan changes land on main without review, which matters when specs define scope and acceptance criteria that govern subsequent implementation.

A decision is needed on whether MindSpec should adopt a protected-main model where all phases use branches merged via pull request.

## Decision

*To be decided.* This ADR captures the problem space and alternatives for future evaluation.

## Decision Details

### A) Scope of Change

Adopting branch protection would affect:

- **MODES.md exit gates**: Milestone commits become "merge PR to main" instead of "commit to main"
- **AGENTS.md commit conventions**: Branch naming conventions needed for spec/plan phases (currently only implementation has `bead/<bead-id>`)
- **Instruction templates**: `implement.md`, `spec.md`, `plan.md` templates would need PR creation steps
- **Worktree tooling**: Currently scoped to implementation beads (Spec 008). Under a protected-main model, spec/plan work would also need branches, though not necessarily worktrees (since that work is serial)
- **State file management**: `.mindspec/state.json` updates would happen on branches, not main — state on main only updates when a PR merges

### B) Proposed Branch Naming Conventions

If adopted, branch naming would follow:

- Spec phase: `spec/<spec-id>` (e.g., `spec/008-worktree-lifecycle`)
- Plan phase: `plan/<spec-id>` (e.g., `plan/008-worktree-lifecycle`)
- Implementation phase: `bead/<bead-id>` (already established)

### C) Worktree Implications

- **Implementation beads**: Continue using worktrees (parallel work, filesystem isolation)
- **Spec/plan phases**: Simple branch checkout is sufficient (serial work, markdown-only). Worktrees are unnecessary overhead here.
- If spec/plan work ever becomes parallelizable (e.g., multiple specs in flight), worktrees could be extended to cover those phases

### D) Workflow Comparison

**Current (direct-to-main):**
```
spec work on main → commit → plan work on main → commit → impl in worktree → commit to main
```

**Proposed (PR-based):**
```
spec work on spec/<id> → PR → merge → plan work on plan/<id> → PR → merge → impl in bead/<id> → PR → merge
```

## Consequences

### Positive

- **Review gate** at every phase transition — catches issues before they reach main
- **Clean revert unit** — each phase is a merged PR that can be reverted atomically
- **CI integration** — branch protection enables required status checks
- **Team-ready** — scales to multi-developer workflows without changing the model
- **Agent safety net** — AI-generated specs, plans, and code get reviewed before landing

### Negative / Tradeoffs

- **Ceremony overhead** — every mode transition requires creating a branch, opening a PR, and merging. For solo developers this may feel heavy.
- **Slower iteration** — PR review adds latency to each phase transition
- **Tooling changes** — instruction templates, exit gates, and potentially worktree tooling all need updates
- **State file complexity** — state updates on branches mean main's state only reflects the last merged phase, not work-in-progress
- **Local-only limitation** — PR-based workflows require a remote (GitHub/GitLab). The current local-only model has no remote configured.

## Alternatives Considered

### 1. Keep direct-to-main (status quo)

Simple, fast, works well for single-developer local-first workflows. No ceremony. The tradeoff is no review gate and no CI enforcement on main.

### 2. Protect main for implementation only

Spec/plan phases stay direct-to-main (markdown-only, low risk). Only implementation creates PRs. This is a middle ground that reduces ceremony for the low-risk phases while protecting main from code changes.

- Pros: Less overhead for spec/plan; code changes get review
- Cons: Inconsistent model; spec mistakes still land on main without review

### 3. Protect main with squash merging

Same as the full proposal but squash-merge each phase into a single commit on main. Keeps main history clean at the cost of losing per-commit granularity within a phase.

### 4. Git hooks only (no PR workflow)

Use pre-push hooks to run validation (`mindspec validate`) before allowing pushes to main. Provides a CI-like gate without the PR ceremony.

- Pros: Simpler than PRs; no remote required
- Cons: No human review step; hooks can be bypassed

## Validation / Rollout

1. Evaluate whether local-only workflow should gain a remote first (prerequisite for PRs)
2. If adopted, update MODES.md exit gates, AGENTS.md commit conventions, and instruction templates
3. Add branch naming conventions to CONVENTIONS.md
4. Extend `mindspec worktree` or add `mindspec branch` tooling for spec/plan phases
5. Dogfood on a single spec cycle before applying broadly
