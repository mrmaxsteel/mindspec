# ADR-0021: LLM Test Strategy — Workflow Validation via Minimal Prompts

- **Date**: 2026-03-01
- **Status**: Accepted
- **Domain(s)**: testing, harness, agent-behavior
- **Deciders**: Max
- **Supersedes**: n/a
- **Superseded-by**: n/a

## Context

MindSpec includes an LLM behavioral test harness (`internal/harness/`) that spawns real Claude Code sessions in isolated sandboxes and asserts the agent executed the correct mindspec workflow. The harness validates that mindspec's guidance layer (CLAUDE.md, instruct templates, CLI error messages, hooks) is sufficient to drive agent behavior through the spec lifecycle.

The central risk in LLM testing is **tautological tests**: if the test prompt tells the agent exactly which commands to run and in what order, a passing test proves only that the LLM can follow explicit instructions — not that the product's guidance actually works. A tautological test provides false confidence: the product could ship with broken guidance and every test would still pass.

This ADR records the testing strategy decisions that prevent tautological tests and ensure the harness validates the product, not the prompt.

## Decision

### 1. Fix Surface Rule

When an LLM test fails due to agent behavior, the fix MUST go into mindspec's own guidance — never into the test prompt.

**Guidance** means the artifacts the product ships: instruct templates (`internal/instruct/templates/*.md`), CLAUDE.md content, CLI error messages, and hook behavior. These are the same artifacts a real user's agent would consume.

**Test prompt** means the scenario's `Prompt` field — what the agent is told to do.

If the agent doesn't know to commit before running `mindspec complete`, the fix goes into the implement.md instruct template, not into the test prompt. The test prompt says *what* to build; the product says *how* to use mindspec.

### 2. Minimal Prompts — Task Only, Not Workflow

Prompts describe the **task** (what to build), not the **workflow** (how to use mindspec). The agent must discover the workflow from mindspec's guidance layer.

- **Good**: "Add a greeting feature that prints Hello."
- **Bad**: "Run `mindspec explore`, then `mindspec approve spec`, then write spec.md..."

Two narrow exceptions are permitted:
- **End-state marker**: "Run `mindspec complete` when done" — names the finish line, not the path.
- **Agent infrastructure**: "Do NOT respond conversationally. Execute NOW." — prevents Haiku's conversational mode, which is an agent-model quirk, not workflow guidance.

### 3. Deterministic Setup, Behavioral Assertions

Each scenario has two deterministic boundaries around a non-deterministic middle:

- **Setup** (`Setup func`): creates all infrastructure before the agent runs — config, state files, beads, specs, worktrees, git state. The agent's first action should be executing the task, not bootstrapping.
- **Assertions** (`Assertions func`): verify end-state — commands ran, files exist, git state is correct. Assertions check *what happened*, not *how* it happened.
- **Middle**: the agent runs freely, discovering workflow from guidance. This is intentionally non-deterministic — different models or prompt variations may take different paths to the same end state.

### 4. Haiku as Baseline Model

All LLM tests run against Claude Haiku (cheapest, fastest model). The reasoning: if Haiku can discover and execute the workflow from guidance alone, Sonnet and Opus certainly can. Haiku is the strictest test of guidance clarity because it requires the most explicit, well-structured guidance to succeed.

Tradeoffs:
- ~$0.01-0.05 per run (vs $0.50+ for Sonnet/Opus)
- 1-3 minute runs (vs 5-10 minutes)
- Requires imperative prompt framing to avoid conversational responses
- Higher retry rates on ambiguous CLI output — which surfaces guidance gaps faster

### 5. Workflow Tests Only — Enforcement Is Deterministic

The harness tests **workflow behavior** (agent discovering and executing the lifecycle). It does not test **enforcement logic** (hook dispatch, worktree guards, file guards).

Enforcement is tested deterministically in unit tests (`internal/hook/dispatch_test.go`, `internal/guard/`). These are pure functions — faster, cheaper, and more reliable than LLM tests. Using an LLM to test `if cwd != expected { return error }` wastes money and adds flakiness for no coverage gain.

## Consequences

### Positive

- **Tests validate the product**: a passing test means the guidance layer is sufficient, not just that the prompt was detailed enough.
- **Failures improve the product**: every fix goes into shipped guidance, making the real user experience better.
- **Tests are cheap and fast**: Haiku + minimal turns + deterministic setup keeps each run under $0.05 and 3 minutes.
- **Tests catch guidance regressions**: changing an instruct template or CLI error message can break a test, surfacing the impact before it ships.
- **Clear separation of concerns**: LLM tests for workflow discovery, unit tests for enforcement logic.

### Negative / Tradeoffs

- **Indirect debugging**: when a test fails, the fix location (guidance) differs from the failure location (agent behavior). Requires reading event logs to diagnose.
- **Haiku-specific workarounds**: imperative framing ("Execute NOW") is needed for Haiku but wouldn't be needed for Sonnet/Opus. These are clearly marked as agent infrastructure, not workflow guidance.
- **Non-deterministic middle**: two runs of the same scenario may take different paths. Assertions must be end-state checks, not sequence checks.
- **Slower feedback than unit tests**: even at 1-3 minutes, LLM tests are orders of magnitude slower than unit tests. Reserve them for what only they can test.

## Alternatives Considered

### 1. Scripted prompts (step-by-step instructions in the test)

Tell the agent exactly what commands to run. Rejected because: this produces tautological tests. A passing test proves the LLM can follow instructions, not that the product works. Guidance could be completely broken and tests would still pass.

### 2. Record-and-replay (mock the LLM)

Capture a successful session, replay the same API responses. Rejected because: this tests the harness infrastructure, not the agent's ability to discover workflows. Any guidance change invalidates all recordings. Useful for infrastructure testing but not for behavioral validation.

### 3. Sonnet/Opus as test model

Use a more capable model that needs fewer retries. Rejected as default because: Haiku is the stricter test of guidance quality (if Haiku can do it, anything can), and the cost/speed difference is 10-50x. Sonnet/Opus runs are valuable as occasional spot-checks but not as the CI baseline.

### 4. Test enforcement via LLM (hook dispatch, file guards)

Have the agent trigger guard violations and assert the correct error. Rejected because: enforcement logic is deterministic and already has comprehensive unit tests. Using an LLM to test pure functions adds cost and flakiness for no coverage gain.
