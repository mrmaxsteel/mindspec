# ADR-0018: Lean Bootstrap — Remove Glossary and Policies Subsystems

- **Date**: 2026-02-22
- **Status**: Accepted
- **Domain(s)**: core, context-system
- **Deciders**: Max
- **Supersedes**: n/a
- **Superseded-by**: n/a

---

## Context

`mindspec init` created several files and subsystems that were not consumed by any core workflow command or provided no meaningful value:

1. **Glossary** (`GLOSSARY.md`, `internal/glossary/`): A standalone lookup tool for mapping terms to doc sections. It was parsed by `mindspec doctor` and included in context packs via the context-system domain. In practice, nobody used `mindspec glossary list/match/show`, and the glossary entries added no value to context packs — domain docs already provide the same information more naturally.

2. **Policies** (`.mindspec/policies.yml`, `internal/contextpack/policy.go`): Machine-readable policy rules included in context packs. These duplicated guidance already present in instruct templates and AGENTS.md. The policies were never enforced programmatically — they were informational text rendered into context packs.

3. **Bootstrap bloat**: `mindspec init` also pre-created `.mindspec/docs/core/`, `.mindspec/docs/adr/`, and `.mindspec/docs/context-map.md`. The `core/` directory is for mature projects and unhelpful when empty. The `adr/` directory is created on demand by `mindspec adr create`. The context map is created by `mindspec domain add`.

Meanwhile, `mindspec migrate` spent its prompt on file classification rubrics but gave no guidance on the hard part: identifying domains from the codebase and populating domain docs with real content.

## Decision

### 1. Remove glossary entirely

Delete the `internal/glossary/` package (6 files), `cmd/mindspec/glossary.go`, `workspace.GlossaryPath()`, the `checkGlossary()` doctor check, and `GLOSSARY.md` from the bootstrap manifest. Remove glossary from the `isDocFile()` classifier in validation.

### 2. Remove policies entirely

Delete `internal/contextpack/policy.go` and its test. Remove policies loading, filtering, and rendering from the context pack builder. Remove `workspace.PoliciesPath()`, `workspace.LegacyPoliciesPath()`, and `.mindspec/policies.yml` from the bootstrap manifest.

### 3. Lean the bootstrap manifest

Remove from `mindspec init`:
- `GLOSSARY.md`
- `.mindspec/docs/context-map.md` (created by `mindspec domain add`)
- `.mindspec/policies.yml`
- `.mindspec/docs/core/` directory (created manually as needed)
- `.mindspec/docs/adr/` directory (created by `mindspec adr create`)

Keep:
- `.mindspec/` directory
- `.mindspec/docs/domains/` directory
- `.mindspec/docs/specs/` directory
- `.mindspec/state.json`
- `AGENTS.md` and `CLAUDE.md`

### 4. Rewrite migrate prompt for domain discovery

Replace the file-classification-only prompt with a 5-phase guided workflow:

1. **Codebase analysis** — scan source structure, identify natural boundaries
2. **Domain identification** — propose domains, create via `mindspec domain add`
3. **Context map population** — map relationships and contracts
4. **Domain doc population** — fill scaffolded files with real codebase content
5. **File classification** — move stray docs to canonical locations (simplified rubric without glossary/policies categories)

## Consequences

### Positive

- Leaner `mindspec init` — fewer files to understand, less noise for new adopters
- Context packs are simpler — domain docs + ADRs are sufficient context without policies or glossary
- `mindspec migrate` now guides the agent through the valuable work (domain identification and doc population) instead of just file shuffling
- Fewer subsystems to maintain — ~600 lines of glossary code and ~100 lines of policy code removed

### Negative / Tradeoffs

- Projects that used `mindspec glossary` commands lose that capability. No known users.
- `policies.yml` as a machine-readable policy format is gone. If programmatic policy enforcement is needed in the future, it should be reimplemented with actual enforcement (linting, CI checks), not informational context-pack rendering.
- Existing projects with `GLOSSARY.md` or `.mindspec/policies.yml` will have orphaned files. These are harmless and can be deleted manually.

## Alternatives Considered

### 1. Keep glossary but remove from bootstrap

Could have kept `internal/glossary/` and `cmd/mindspec/glossary.go` for projects that want it, while removing it from `mindspec init`. Rejected because the glossary commands themselves add no value — domain docs serve the same purpose with better structure.

### 2. Keep policies with enforcement

Could have kept policies but added actual enforcement (e.g., a `mindspec lint` command that checks policy compliance). Rejected as premature — the instruct templates already convey behavioral guidance, and adding a policy enforcement engine is a separate concern that should be driven by actual need.

## Validation / Rollout

1. `make build && make test` passes — all tests updated
2. `mindspec init --dry-run` produces the leaner manifest
3. `mindspec migrate` emits the improved multi-phase prompt
4. Existing projects are unaffected — orphaned glossary/policies files are inert
