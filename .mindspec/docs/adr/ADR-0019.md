# ADR-0019: Deterministic Worktree and Branch Enforcement for Agent Workflows

- **Date**: 2026-02-26
- **Status**: Accepted
- **Domain(s)**: workflow, git, agent-integration
- **Deciders**: Max
- **Supersedes**: n/a
- **Superseded-by**: n/a

---

## Context

ADR-0006 establishes that zero changes happen on main — all work happens on branches in worktrees, with a single PR per spec lifecycle merging `spec/NNN-slug` to main. However, ADR-0006 does not specify *how* this is enforced.

Currently, the only mechanism is prompt-based guidance: the `implement.md` instruction template tells agents to "work in the bead-specific worktree" and not to "make changes outside the assigned worktree." This is advisory, not deterministic.

In practice, agents regularly violate worktree isolation:

1. **`mindspec next` creates a worktree and prints `cd <path>`**, but the agent's CWD does not change. The agent continues writing files in the main worktree.
2. **No git hook blocks commits to main** during any mindspec mode. The beads pre-commit hook handles JSONL sync, not branch protection.
3. **No agent hook (PreToolUse) checks CWD** against the expected worktree before file writes.
4. **`mindspec complete` does not verify** that the agent is actually in the correct worktree.
5. **`mindspec approve spec/plan`** writes to `docs/specs/` from any directory without checking context.

The result: changes intended for an isolated worktree land directly on main, bypassing the branching model that ADR-0006 requires.

### Why prompt-based enforcement fails

AI agents do not reliably follow advisory instructions when:
- The instruction competes with the user's request ("just make the change")
- Context window pressure causes earlier instructions to be deprioritized
- The agent has no feedback that it's violating the constraint (no error, no warning)

Deterministic enforcement — where the *system* prevents the violation rather than asking the agent not to do it — is the only reliable approach.

### Beads provides plumbing, not policy

Beads has comprehensive worktree support (`bd worktree create/info/list/remove`, `RepoContext.IsWorktree`, shared database via redirect files, hook chaining). But beads deliberately does not enforce a 1:1 mapping between issues and worktrees, nor does it protect branches. Beads is a tracker, not an orchestrator. The enforcement policy is mindspec's responsibility, built on beads' plumbing.

## Decision

**Enforce worktree isolation and branch protection through three independent, layered mechanisms that operate deterministically without relying on prompt guidance.**

### Layer 1: Git pre-commit hook (branch protection)

A mindspec-managed pre-commit hook, chained with the existing beads pre-commit hook, that:

- Reads `.mindspec/state.json` to determine whether mindspec is active
- When any mindspec mode is active (spec, plan, implement, review): blocks commits if the current branch is in the protected branches list (default: `main`, `master`)
- Emits a clear error: `"mindspec: commits to protected branch '<name>' are blocked. Switch to your worktree: cd <path>"`
- Configurable via `.mindspec/config.yaml`: `protected_branches: [main, master]`
- Override escape hatch: `MINDSPEC_ALLOW_MAIN=1 git commit` for emergencies
- Works for ALL tools — agents, humans, any git client

### Layer 2: CLI guard rails (mindspec commands refuse to act)

Mindspec commands that write files or change state validate their execution context:

- **`mindspec complete`**: refuses to run if CWD is the main worktree. Requires CWD to be inside a bead worktree.
- **`mindspec next`**: after creating/finding the worktree, emits a hard instruction to `cd` and validates on subsequent calls that the agent has moved.
- **`mindspec spec-init`**: creates the `spec/NNN-slug` branch and worktree, instructs the agent to `cd` into it.
- **`mindspec approve spec/plan`**: validates CWD is the spec worktree, not main.
- **All modes enforced**: since zero changes are allowed on main, CLI guards apply in all modes (spec, plan, implement).

### Layer 3: Agent PreToolUse hooks (deterministic CWD and path enforcement)

For Claude Code and Copilot, PreToolUse hooks that enforce worktree isolation at the tool level:

- **`Edit` and `Write` hooks**: Check whether the target file path is within the active worktree. Block the tool call if the path is outside.
- **`Bash` hook**: Check whether the agent's CWD is the active worktree (not main). **Block ALL bash execution from the wrong CWD.** This is the primary deterministic enforcement for CWD — the agent literally cannot execute any command from main while a worktree is active. This forces the agent to `cd` to the worktree before doing anything.
- Blocked tool calls emit: `"mindspec: blocked — your working directory is the main worktree. Run: cd <worktree-path>"`
- Active in all mindspec modes (since zero changes on main)

### Soft layer: `mindspec instruct` CWD redirect

In addition to the three deterministic layers, `mindspec instruct` provides a strong soft signal:

- When `mindspec instruct` runs from main CWD but a worktree is active, it **replaces** the normal mode template (spec/plan/implement guidance) with ONLY a CWD redirect instruction: `"You are in the main worktree. Run: cd <path> — then run mindspec instruct for guidance."`
- No operating instructions are emitted until the agent moves to the worktree
- This fires on every `SessionStart` hook, so it's the first thing the agent sees
- This is prompt-based (not deterministic) but combined with Layer 3's bash blocking, the agent has no choice: it receives no guidance AND cannot execute commands until it moves

### How the layers interact

```
Session starts → mindspec instruct runs from main CWD
  → Soft layer: "You are in the wrong worktree. cd /path/to/worktree" (no other guidance)
  → Agent tries to run a bash command from main
  → Layer 3 (PreToolUse on Bash): BLOCKED — CWD is main, not the worktree
  → Agent tries to edit a file on main
  → Layer 3 (PreToolUse on Edit): BLOCKED — path outside worktree
  → Agent runs cd /path/to/worktree
  → Agent tries to write code (now in correct CWD)
  → Layer 3: ALLOWED
  → Agent tries to commit
  → Layer 1 (pre-commit hook): checks branch — allowed (on spec/* or bead/*)
  → Agent runs mindspec complete
  → Layer 2 (CLI guard): checks CWD — allowed (in bead worktree)
```

Each layer is independent. If one is bypassed (e.g., Copilot doesn't support PreToolUse), the others still catch the violation. Defense in depth.

## Decision Details

### A) Worktree binding in state

`state.json` tracks the active worktree for the current phase:

```json
{
  "activeSpec": "046-worktree-enforcement",
  "activeBead": "mindspec-xxx",
  "activeWorktree": "/path/to/worktree-spec-046-worktree-enforcement"
}
```

During spec/plan phases, `activeWorktree` points to the spec worktree. During implementation, it points to the bead worktree. This binding is the reference point for all three enforcement layers.

### B) Enforcement applies to ALL modes

Per ADR-0006's "zero changes on main" decision, enforcement is not mode-selective. The pre-commit hook blocks commits to protected branches whenever mindspec state is non-idle. CLI guards check worktree context in all modes. Agent hooks intercept writes in all modes.

### C) Configuration

`.mindspec/config.yaml` controls enforcement behavior:

```yaml
# Branch protection
protected_branches:
  - main
  - master

# Merge strategy for final spec → main merge
merge_strategy: auto    # "pr" | "direct" | "auto" (PR if remote exists)

# Enforcement layers (all on by default)
enforcement:
  pre_commit_hook: true     # Layer 1
  cli_guards: true          # Layer 2
  agent_hooks: true         # Layer 3
```

### D) Hook installation

`mindspec setup claude` and `mindspec setup copilot` are extended to install/update the enforcement hooks:

- **Pre-commit hook**: installed via beads' chain mechanism (`bd hooks install --chain`), so the mindspec hook runs alongside the beads hook
- **PreToolUse hooks**: added to `.claude/settings.json` and `.github/hooks/` alongside the existing plan-mode gate

### E) Relationship to ADR-0015 (molecule-derived state)

The enforcement layers read `state.json` as a fast hint for the active worktree path. They do NOT derive mode from the molecule (that would require a `bd mol show` call in a git hook, which is too slow). The `state.json` cursor is authoritative enough for enforcement — if it says non-idle with an `activeWorktree`, enforce it. If state is stale, `mindspec instruct` corrects it on next session start.

### F) The `CheckWorktree()` false alarm fix

The existing `CheckWorktree()` function in `internal/instruct/worktree.go` fires a false alarm when called from the instruct-tail of `mindspec next` (because CWD is still the main repo). This is fixed as part of this work: `mindspec next` updates `state.json` with `activeWorktree` BEFORE emitting the instruct tail, and `CheckWorktree()` checks the binding rather than sniffing CWD.

### G) Interaction with ADR-0006's branch topology

The enforcement layers support ADR-0006's topology where:
- Spec/plan work happens on `spec/NNN-slug` in a worktree
- Impl beads branch from `spec/NNN-slug` into their own worktrees
- Impl beads merge back to `spec/NNN-slug`
- One final PR merges `spec/NNN-slug` to main

Layer 1 (pre-commit hook) blocks commits to `main` and `master`. It does NOT block commits to `spec/*` or `bead/*` branches — those are the intended destinations.

Layer 2 (CLI guards) validates that the CWD matches the expected worktree for the current phase:
- During spec/plan: CWD must be in the spec worktree
- During implement: CWD must be in the bead worktree

Layer 3 (agent hooks) validates file write paths against the active worktree, regardless of which branch it's on.

## Consequences

### Positive

- **Deterministic protection** — agents cannot accidentally write to main, regardless of prompt pressure or context loss
- **Defense in depth** — three independent layers; any single layer catches the violation
- **Works for all tools** — pre-commit hook protects against any git client; agent hooks protect Claude Code and Copilot specifically
- **Configurable** — teams can tune enforcement (protected branches list, merge strategy, which layers are active)
- **Builds on existing infrastructure** — beads hook chaining, Claude Code PreToolUse, existing `state.json` cursor
- **Supports ADR-0006 topology** — enforcement is aware of the spec-branch-as-integration-point model

### Negative / Tradeoffs

- **Pre-commit hook adds latency** — reading `state.json` and checking branch name adds a few ms to every commit. Negligible.
- **Escape hatch exists** — `MINDSPEC_ALLOW_MAIN=1` bypasses Layer 1. This is intentional (emergencies happen) but means enforcement is not absolute.
- **Agent hook coverage is IDE-specific** — Layer 3 only works for Claude Code and Copilot (where PreToolUse is supported). Other agents rely on Layers 1 and 2. This is acceptable because the git hook (Layer 1) is universal.
- **State.json as enforcement hint** — if state.json is stale or corrupt, enforcement may be wrong (too strict or too lenient). Mitigated by `mindspec instruct` correcting state on session start.
- **All-modes enforcement is stricter** — even trivial spec edits require a worktree. Mitigated by: `mindspec spec-init` creates the worktree automatically; the overhead is one `cd` command.

## Alternatives Considered

### 1. Prompt-only enforcement (status quo)

Continue relying on instruction template text to tell agents to use worktrees.

- Pros: No code changes; no configuration
- Cons: Demonstrably fails — agents write to main regularly. Not deterministic.
- Rejected because this is the problem we're solving.

### 2. Beads-level enforcement (bd worktree create --bind-issue)

Add bead-to-worktree binding and enforcement to beads itself, so beads rejects operations when the wrong worktree is active.

- Pros: Enforcement at the tracker level; works for any beads user
- Cons: Violates beads' design as a generic tracker (not an orchestrator); couples beads to a specific workflow model; beads deliberately doesn't enforce worktree-per-issue because not all teams want that
- Rejected because enforcement policy belongs in the orchestrator (mindspec), not the tracker (beads).

### 3. Server-side branch protection only (GitHub branch rules)

Rely entirely on GitHub/GitLab branch protection to prevent pushes to main.

- Pros: No local tooling needed; platform-enforced
- Cons: Doesn't prevent local commits (only push); requires a remote; doesn't help during development (agent makes 10 commits on main before pushing); no protection for local-only workflows
- Rejected as insufficient on its own, though it complements this approach nicely.

### 4. Single enforcement layer (pre-commit hook only)

Only implement Layer 1 (git pre-commit hook). Skip CLI guards and agent hooks.

- Pros: Simpler; single mechanism to maintain
- Cons: Catches violations late (at commit time, after files are written); agent has to undo work. PreToolUse hooks catch violations early (before file write). CLI guards catch violations at the command level. Defense in depth is worth the modest complexity.

### 5. Implementation-only enforcement

Only enforce worktree isolation during implementation mode; allow spec/plan to commit directly to main.

- Pros: Less ceremony for markdown-only phases
- Cons: Inconsistent invariant ("sometimes main is OK, sometimes not"); agents must understand the distinction; ADR-0006 decided on zero-on-main
- Rejected because "zero changes on main" is a cleaner invariant.

## Validation / Rollout

1. Implement the pre-commit hook (Layer 1) as a shell script, installed via `mindspec setup` using beads' chain mechanism
2. Add `activeWorktree` field to `state.json` schema; update `mindspec spec-init` and `mindspec next` to populate it
3. Extend `mindspec spec-init` to create `spec/NNN-slug` branch + worktree
4. Add CLI guards (Layer 2) to `mindspec complete`, `mindspec next`, and `mindspec approve`
5. Add PreToolUse hooks (Layer 3) to `mindspec setup claude` and `mindspec setup copilot`
6. Add `.mindspec/config.yaml` with `protected_branches`, `merge_strategy`, and `enforcement` fields
7. Fix `CheckWorktree()` false alarm (use `activeWorktree` binding instead of CWD sniffing)
8. Update instruction templates (`spec.md`, `plan.md`, `implement.md`) to reference worktree workflow
9. Dogfood on next spec cycle — verify all three layers trigger correctly
