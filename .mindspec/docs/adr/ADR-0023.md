## Context

MindSpec currently maintains two state files per spec:

1. **`.mindspec/focus`** â€” JSON tracking active spec, bead, mode, worktree, branch. Written on every transition. ADR-0022 Â§4.5 says it lives on main; in practice the code writes it to worktrees.
2. **`lifecycle.yaml`** â€” YAML in each spec directory with `phase` and `epic_id`. Written on transitions, lives in the spec worktree on the branch.

Both are denormalized caches of state that already lives authoritatively in beads:

| Focus field | Beads equivalent |
|:------------|:----------------|
| `mode` | Derived from epic/bead statuses |
| `activeSpec` | Derived from epic `metadata.spec_num` + `metadata.spec_title` |
| `activeBead` | `bd list --status=in_progress --parent <epicID>` |
| `specBranch` | Convention: `spec/<specID>` |
| `activeWorktree` | Convention: `.worktrees/worktree-<beadID>` or `.worktrees/worktree-spec-<specID>` |

| lifecycle.yaml field | Beads equivalent |
|:---------------------|:----------------|
| `phase` | Derived from epic/bead statuses (same as focus `mode`) |
| `epic_id` | Epic with matching `metadata.spec_num` |

These caches create three problems:

**Consistency drift** â€” focus and lifecycle.yaml can disagree with beads. The LLM test harness exposed this: `approve spec` exits 0 but focus still says `spec` instead of `plan`, because the file was written to the wrong location.

**Parallel agent contention** â€” focus on main is a single cursor. Two agents working on different specs create a last-writer-wins race. Per-worktree focus avoids this but fragments state.

**Fragile discovery** â€” `ActiveSpecs()` scans filesystem for `lifecycle.yaml`, but those files only exist on worktree branches (not merged to main). Discovery fails when worktrees aren't checked out.

Beads solves all three. It runs a Dolt server daemon (MySQL-compatible) with sub-millisecond local socket queries. The `.beads/` directory is shared across all worktrees via git's common directory mechanism. It handles concurrent access natively. It supports structured metadata on issues (`--metadata='{"spec_num":60,"spec_title":"example"}'`). And it already tracks everything both files cache.

## Decision

### 1. Eliminate both focus and lifecycle.yaml

No `.mindspec/focus` file anywhere â€” not on main, not in worktrees. No `lifecycle.yaml` in spec directories. All state queries go through beads.

CLI commands that currently write focus or lifecycle.yaml stop writing them. Commands that read them derive the same information from beads instead.

The spec directory contains only human-readable artifacts:
```
docs/specs/NNN-slug/
  spec.md        # specification document
  plan.md        # implementation plan
  context-pack.md  # generated context
  proofs/        # optional proof outputs
```

### 2. Epic metadata replaces lifecycle.yaml

Each spec's beads epic stores two metadata fields:

```bash
bd create --title="[SPEC 060-example] Feature Name" \
  --type=epic \
  --metadata='{"spec_num":60,"spec_title":"example"}'
```

- **`spec_num`** (int) â€” the numeric prefix, used for collision detection and ordering
- **`spec_title`** (kebab-case string) â€” the slug, used with `spec_num` to derive all paths

All identifiers and paths are derived from these two fields:

| Derived value | Formula |
|:-------------|:--------|
| `spec_id` | `fmt.Sprintf("%03d-%s", spec_num, spec_title)` |
| Spec branch | `spec/<spec_id>` |
| Spec worktree | `.worktrees/worktree-spec-<spec_id>` |
| Spec directory | `.mindspec/docs/specs/<spec_id>/` |
| Epic title | `[SPEC <spec_id>] <Human-Readable Title>` |

The `[SPEC NNN-slug]` title convention is retained for human readability but is not the primary lookup mechanism.

Discovery query:
```bash
bd list --type=epic --status=open --json
```
Then filter by `metadata.spec_num` for targeted lookups.

### 3. Phase derivation from bead statuses

The lifecycle phase for any spec is derived from its epic and bead statuses. Epic creation is the spec-approval gate â€” `spec approve` creates the epic, so its existence is the durable approval record.

| Condition | Derived phase |
|:----------|:-------------|
| No epic with matching `spec_num` | **spec** (draft, not yet approved) |
| Epic exists, no children | **plan** (spec approved, plan being drafted) |
| Epic exists, all children open (none claimed) | **plan** (plan approved, beads ready) |
| Epic exists, any child in_progress | **implement** |
| Epic exists, all children closed, epic open | **review** |
| Epic closed | **done** |
| No open epics for any spec | **idle** |

Edge cases:
- Some children closed, some open, none in_progress â†’ **plan** (next bead ready to claim)
- Some children closed, some open, some blocked â†’ **plan** (only blocked beads remain; agent must unblock or replan)
- Multiple children in_progress â†’ **implement** (parallel beads, multiple agents)

#### Amendment (spec 095, mindspec-ry73): `review` is derived over LIFECYCLE children only

The `review`/`implement`/`plan` derivation above counts only **lifecycle**
children of the epic. A spec epic can also carry **non-lifecycle follow-up**
children â€” most commonly a `bug` filed against the spec after implementation
(the natural place to attach it). Counting those follow-ups as blocking
children stranded the spec short of `review` forever: every lifecycle bead
could be closed, yet one open follow-up bug forced `implement`, and
`impl approve` failed `expected review mode` until the operator manually
detached the child and ran `repair phase` (mindspec-ry73; observed on
spec 094 / cdk8.5).

**Discriminator (`issue_type`).** A child is a lifecycle bead iff its
`issue_type == "task"`. `plan approve` creates every lifecycle implementation
bead with `--type task` (invariant: `internal/approve/plan.go:341`); only the
spec container itself uses `--type epic`. Therefore any non-`task`, non-`epic`
child (e.g. `bug`) is a non-lifecycle follow-up. An **untyped** child (`""`)
is treated as lifecycle â€” the conservative default, so a real lifecycle bead
is never mistaken for a follow-up. A nested `epic` child is structural and
ignored entirely (neither lifecycle nor follow-up).

**Refined rule.** Partition children by `issue_type`, then derive over the
LIFECYCLE subset only:

| Lifecycle-subset condition | Derived phase |
|:---------------------------|:--------------|
| **Empty lifecycle set** (no children, OR only follow-up / epic children) | **plan** |
| Any lifecycle child in_progress | **implement** |
| All lifecycle children closed (â‰Ą1 lifecycle child) | **review** |
| Some lifecycle closed + some open, none in_progress | **implement** |
| All lifecycle children open (none claimed) | **plan** |

A non-lifecycle open child never forces `implement` and never blocks `review`.

**Empty-lifecycle-set guard (critical).** `review` is NEVER derived over an
empty lifecycle set. A bug filed as an epic child while the spec is still in
plan mode (zero lifecycle beads) must not derive `review` on an
unimplemented spec â€” the empty lifecycle set is treated exactly like the
no-children case and falls back to `plan`. This closes the vacuous-review
hazard that a naive "all non-in_progress children closed â†’ review" rule would
open once `bug` children are excluded from the in_progress/open tally.

**Advisory guard hint.** When `review` is derived while a non-lifecycle child
remains open, `mindspec impl approve` emits an advisory recovery line
(ADR-0035 convention) naming the offending child so the operator can re-file
it as standalone backlog or leave it attached. The hint does NOT block (the
gate has already passed). It deliberately does NOT recommend
`bd update <id> --parent ""`: that detach is buggy (mindspec-bk5t â€” not
reflected in `bd list --parent`).

Examples (placeholder IDs):
- Children `task#a` (closed), `task#b` (closed), `bug#c` (open) â†’ **review**
  (every lifecycle bead closed; the open bug is ignored, hint names `bug#c`).
- Children `bug#c` (open) only â†’ **plan** (empty lifecycle set â€” the
  vacuous-review guard; impl approve does not fire on an unimplemented spec).
- Children `task#a` (in_progress), `bug#c` (open) â†’ **implement** (the open
  bug neither forces nor blocks).

### 4. Worktree-to-spec resolution by convention

Worktree paths are deterministic from IDs â€” no state file needed to record them:

| Context | Path convention |
|:--------|:---------------|
| Spec worktree | `.worktrees/worktree-spec-<specID>` |
| Bead worktree | `.worktrees/worktree-<beadID>` |
| Spec branch | `spec/<specID>` |
| Bead branch | `bead/<beadID>` |

When an agent starts on main, `mindspec instruct` queries beads for open epics, derives the phase, and determines which worktree(s) exist by convention. It then emits the CWD redirect.

When an agent is already in a worktree, the worktree name encodes the spec or bead ID. The CLI extracts it from the path or branch name.

### 5. Beads as the single state store

| Concern | Source of truth | Location |
|:--------|:---------------|:---------|
| Which specs are active | Beads epics (open) | `.beads/` (Dolt, shared) |
| Spec-to-epic binding | Epic `metadata.spec_num` + `metadata.spec_title` | `.beads/` (Dolt, shared) |
| Spec lifecycle phase | Derived from bead statuses | `.beads/` (Dolt, shared) |
| Which bead is claimed | Bead status = in_progress | `.beads/` (Dolt, shared) |
| Bead dependencies/blocking | Beads dependency graph | `.beads/` (Dolt, shared) |
| Spec artifacts | spec.md, plan.md | Per-worktree filesystem (on branch) |

No state files on the filesystem. Beads (Dolt) is the single state authority. Spec artifacts (spec.md, plan.md) are versioned documents on branches, not state.

### 6. Updated invariants (supersedes ADR-0022 Â§4)

1. **Main is read-only** â€” no direct commits, no file writes. Only receives merges. *(unchanged)*
2. **One worktree per unit of work** â€” specs, beads, and bugfixes each get their own worktree. *(unchanged)*
3. **Worktrees are disposable** â€” created by `mindspec next`/`spec create`, removed by `mindspec complete`/`cleanup`. *(unchanged)*
4. **Context switch = worktree switch** â€” each worktree has its own filesystem state. *(unchanged)*
5. **No state files on filesystem** â€” no focus, no lifecycle.yaml. State is derived from beads + path conventions. *(changed)*
6. **Beads is the single state store** â€” lifecycle discovery, phase derivation, spec-epic binding, and work-readiness queries all go through beads. *(new)*

## Consequences

### Positive

- **Parallel agents work without contention** â€” no shared mutable state files; Dolt handles concurrent reads/writes natively
- **Single source of truth** â€” eliminates the consistency bug class where focus/lifecycle.yaml disagree with beads
- **Lifecycle discovery works from any context** â€” beads is always available via the Dolt daemon, no worktree checkout or filesystem scan required
- **Less code, fewer files** â€” remove `WriteFocus`/`ReadFocus`, `ReadLifecycle`/`WriteLifecycle`, `state.Focus`, `state.Lifecycle` structs, and all callers; remove lifecycle.yaml generation from every transition command
- **Cleaner spec directories** â€” only human-readable artifacts (spec.md, plan.md), no machine state files mixed in
- **Structured epic-spec binding** â€” `metadata.spec_num` + `metadata.spec_title` are queryable and type-safe, replacing string parsing of epic titles

### Negative / Tradeoffs

- **Beads daemon must be running** â€” if the Dolt server is down, no state queries work. Mitigated by: beads auto-starts on first `bd` call; `mindspec doctor` checks daemon health; the daemon is already required for all lifecycle operations.
- **More beads queries per command** â€” commands that previously read a local file now query Dolt. Mitigated by: Dolt daemon queries over local socket are sub-millisecond; total overhead is negligible.
- **Migration** â€” existing repos have focus and lifecycle.yaml files that must be cleaned up. Existing epics need `spec_num`/`spec_title` metadata backfilled. Mitigated by: `mindspec doctor` can detect stale files and offer cleanup; epic title convention `[SPEC NNN-slug]` provides fallback for parsing during migration.
- **No offline state** â€” without the Dolt daemon, `mindspec instruct` can't derive phase. Mitigated by: the daemon is lightweight and always-on; JSONL export provides a degraded fallback.

## Alternatives Considered

### 1. Keep lifecycle.yaml, drop only focus

Retain lifecycle.yaml as the per-spec state file on the branch.

- Pros: Human-readable state co-located with spec artifacts; works without beads
- Cons: Still a denormalized cache that drifts; filesystem scan needed for discovery; two fields that are both derivable from beads
- Rejected: lifecycle.yaml stores exactly `phase` and `epic_id` â€” both are derivable from beads. The file adds maintenance cost for zero unique information.

### 2. Per-worktree focus files (no main focus)

Keep focus files but only in worktrees.

- Pros: Faster local reads; familiar pattern
- Cons: Still a denormalized cache; still must be written on every transition; fragments state across worktrees
- Rejected: if beads is authoritative, the cache provides no value.

### 3. Keep focus as a performance cache with beads as fallback

Read focus first, fall back to beads if missing/stale.

- Pros: Fast path for common case
- Cons: Two code paths to maintain; "which is authoritative?" confusion; stale cache bugs are the current problem
- Rejected: performance gain is negligible (sub-ms Dolt queries) and consistency cost is high.

## Validation / Rollout

1. Move epic creation from `plan approve` to `spec approve` â€” epic existence is the spec-approval gate
2. Add `metadata.spec_num` (int) + `metadata.spec_title` (kebab-case) to epic creation (`--metadata='{"spec_num":60,"spec_title":"example"}'`)
3. Add `bd dolt pull` before epic creation in `spec approve` to prevent `spec_num` collisions across agents
4. Add `DiscoverActiveSpecs()` function that queries `bd list --type=epic --status=open --json` and derives phase from bead statuses
5. Add `ResolveContext(root)` function that returns `(specID, beadID, phase, worktreePath)` by combining beads query with path conventions
6. Update `instruct` to use `ResolveContext()` instead of `ReadFocus()`
7. Update `next`, `complete`, `approve` commands to derive context from beads instead of reading/writing focus or lifecycle.yaml
8. Remove `state.Focus`, `state.Lifecycle`, `ReadFocus()`, `WriteFocus()`, `ReadLifecycle()`, `WriteLifecycle()`, and all callers
9. Remove lifecycle.yaml generation from `spec create`, `approve spec`, `approve plan`
10. Backfill `spec_num`/`spec_title` metadata on existing epics (parse from title convention as migration)
11. Update LLM test harness: replace focus/lifecycle assertions with beads-based assertions; update sandbox setup to use `bd update` instead of `WriteFocus()`/`WriteLifecycle()`
12. `mindspec doctor` detects and removes stale `.mindspec/focus` and `lifecycle.yaml` files
13. Run full LLM test suite to verify
