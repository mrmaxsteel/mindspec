# ADR-0023: Eliminate Focus and Lifecycle Files — Beads as Single State Authority

- **Date**: 2026-03-03
- **Status**: Accepted
- **Domain(s)**: workflow, git, state
- **Deciders**: Max
- **Supersedes**: ADR-0022 §4 invariant 5 ("focus on main"), ADR-0020 (per-spec lifecycle file), ADR-0015 §state.json-as-cursor
- **Related**: ADR-0022 (worktree-aware resolution)

---

## Context

MindSpec currently maintains two state files per spec:

1. **`.mindspec/focus`** — JSON tracking active spec, bead, mode, worktree, branch. Written on every transition. ADR-0022 §4.5 says it lives on main; in practice the code writes it to worktrees.
2. **`lifecycle.yaml`** — YAML in each spec directory with `phase` and `epic_id`. Written on transitions, lives in the spec worktree on the branch.

Both are denormalized caches of state that already lives authoritatively in beads:

| Focus field | Beads equivalent |
|:------------|:----------------|
| `mode` | Derived from epic/bead statuses |
| `activeSpec` | Epic metadata `spec_id` or title convention `[SPEC NNN-slug]` |
| `activeBead` | `bd list --status=in_progress --parent <epicID>` |
| `specBranch` | Convention: `spec/<specID>` |
| `activeWorktree` | Convention: `.worktrees/worktree-<beadID>` or `.worktrees/worktree-spec-<specID>` |

| lifecycle.yaml field | Beads equivalent |
|:---------------------|:----------------|
| `phase` | Derived from epic/bead statuses (same as focus `mode`) |
| `epic_id` | Epic with `metadata.spec_id` matching the spec ID |

These caches create three problems:

**Consistency drift** — focus and lifecycle.yaml can disagree with beads. The LLM test harness exposed this: `approve spec` exits 0 but focus still says `spec` instead of `plan`, because the file was written to the wrong location.

**Parallel agent contention** — focus on main is a single cursor. Two agents working on different specs create a last-writer-wins race. Per-worktree focus avoids this but fragments state.

**Fragile discovery** — `ActiveSpecs()` scans filesystem for `lifecycle.yaml`, but those files only exist on worktree branches (not merged to main). Discovery fails when worktrees aren't checked out.

Beads solves all three. It runs a Dolt server daemon (MySQL-compatible) with sub-millisecond local socket queries. The `.beads/` directory is shared across all worktrees via git's common directory mechanism. It handles concurrent access natively. It supports structured metadata on issues (`--metadata='{"spec_id":"NNN-slug"}'`). And it already tracks everything both files cache.

## Decision

### 1. Eliminate both focus and lifecycle.yaml

No `.mindspec/focus` file anywhere — not on main, not in worktrees. No `lifecycle.yaml` in spec directories. All state queries go through beads.

CLI commands that currently write focus or lifecycle.yaml stop writing them. Commands that read them derive the same information from beads instead.

The spec directory contains only human-readable artifacts:
```
docs/specs/NNN-slug/
  spec.md        # specification document
  plan.md        # implementation plan
  context-pack.md  # generated context
  proofs/        # optional proof outputs
```

### 2. Epic metadata replaces lifecycle.yaml

Each spec's beads epic stores the spec ID as structured metadata:

```bash
bd create --title="[SPEC 060-example] Feature Name" \
  --type=epic \
  --metadata='{"spec_id":"060-example"}'
```

The `spec_id` metadata field provides a reliable, queryable binding between specs and their beads epics — no title string parsing required. The `[SPEC NNN-slug]` title convention is retained for human readability but is not the primary lookup mechanism.

Discovery query:
```bash
bd list --type=epic --status=open --json
```
Then filter by `metadata.spec_id` for targeted lookups.

### 3. Phase derivation from bead statuses

The lifecycle phase for any spec is derived from its epic and bead statuses:

| Epic state | Derived phase |
|:-----------|:-------------|
| Epic open, no children | spec |
| Epic open, all children open (none claimed) | plan |
| Epic open, a child is in_progress | implement |
| Epic open, all children closed | review |
| Epic closed | done |

Edge cases:
- Some children closed, some open, none in_progress → **plan** (next bead ready to claim)
- Some children closed, some open, some blocked → **plan** (only blocked beads remain; agent must unblock or replan)
- Multiple children in_progress → **implement** (parallel beads, multiple agents)

### 4. Worktree-to-spec resolution by convention

Worktree paths are deterministic from IDs — no state file needed to record them:

| Context | Path convention |
|:--------|:---------------|
| Spec worktree | `.worktrees/worktree-spec-<specID>` |
| Bead worktree | `.worktrees/worktree-<beadID>` |
| Spec branch | `spec/<specID>` |
| Bead branch | `bead/<beadID>` |

When an agent starts on main, `mindspec instruct` queries beads for open epics, derives the phase, and determines which worktree(s) exist by convention. It then emits the CWD redirect.

When an agent is already in a worktree, the worktree name encodes the spec or bead ID. The CLI extracts it from the path or branch name.

### 5. Beads as the single state store

| Concern | Source of truth | Location |
|:--------|:---------------|:---------|
| Which specs are active | Beads epics (open) | `.beads/` (Dolt, shared) |
| Spec-to-epic binding | Epic `metadata.spec_id` | `.beads/` (Dolt, shared) |
| Spec lifecycle phase | Derived from bead statuses | `.beads/` (Dolt, shared) |
| Which bead is claimed | Bead status = in_progress | `.beads/` (Dolt, shared) |
| Bead dependencies/blocking | Beads dependency graph | `.beads/` (Dolt, shared) |
| Spec artifacts | spec.md, plan.md | Per-worktree filesystem (on branch) |

No state files on the filesystem. Beads (Dolt) is the single state authority. Spec artifacts (spec.md, plan.md) are versioned documents on branches, not state.

### 6. Updated invariants (supersedes ADR-0022 §4)

1. **Main is read-only** — no direct commits, no file writes. Only receives merges. *(unchanged)*
2. **One worktree per unit of work** — specs, beads, and bugfixes each get their own worktree. *(unchanged)*
3. **Worktrees are disposable** — created by `mindspec next`/`spec-init`, removed by `mindspec complete`/`cleanup`. *(unchanged)*
4. **Context switch = worktree switch** — each worktree has its own filesystem state. *(unchanged)*
5. **No state files on filesystem** — no focus, no lifecycle.yaml. State is derived from beads + path conventions. *(changed)*
6. **Beads is the single state store** — lifecycle discovery, phase derivation, spec-epic binding, and work-readiness queries all go through beads. *(new)*

## Consequences

### Positive

- **Parallel agents work without contention** — no shared mutable state files; Dolt handles concurrent reads/writes natively
- **Single source of truth** — eliminates the consistency bug class where focus/lifecycle.yaml disagree with beads
- **Lifecycle discovery works from any context** — beads is always available via the Dolt daemon, no worktree checkout or filesystem scan required
- **Less code, fewer files** — remove `WriteFocus`/`ReadFocus`, `ReadLifecycle`/`WriteLifecycle`, `state.Focus`, `state.Lifecycle` structs, and all callers; remove lifecycle.yaml generation from every transition command
- **Cleaner spec directories** — only human-readable artifacts (spec.md, plan.md), no machine state files mixed in
- **Structured epic-spec binding** — `metadata.spec_id` is queryable and type-safe, replacing string parsing of epic titles

### Negative / Tradeoffs

- **Beads daemon must be running** — if the Dolt server is down, no state queries work. Mitigated by: beads auto-starts on first `bd` call; `mindspec doctor` checks daemon health; the daemon is already required for all lifecycle operations.
- **More beads queries per command** — commands that previously read a local file now query Dolt. Mitigated by: Dolt daemon queries over local socket are sub-millisecond; total overhead is negligible.
- **Migration** — existing repos have focus and lifecycle.yaml files that must be cleaned up. Existing epics need `metadata.spec_id` backfilled. Mitigated by: `mindspec doctor` can detect stale files and offer cleanup; epic title convention provides fallback for epics without metadata.
- **No offline state** — without the Dolt daemon, `mindspec instruct` can't derive phase. Mitigated by: the daemon is lightweight and always-on; JSONL export provides a degraded fallback.

## Alternatives Considered

### 1. Keep lifecycle.yaml, drop only focus

Retain lifecycle.yaml as the per-spec state file on the branch.

- Pros: Human-readable state co-located with spec artifacts; works without beads
- Cons: Still a denormalized cache that drifts; filesystem scan needed for discovery; two fields that are both derivable from beads
- Rejected: lifecycle.yaml stores exactly `phase` and `epic_id` — both are derivable from beads. The file adds maintenance cost for zero unique information.

### 2. Per-worktree focus files (no main focus)

Keep focus files but only in worktrees.

- Pros: Faster local reads; familiar pattern
- Cons: Still a denormalized cache; still must be written on every transition; fragments state across worktrees
- Rejected: if beads is authoritative, the cache provides no value.

### 3. Keep focus as a performance cache with beads as fallback

Read focus first, fall back to beads if missing/stale.

- Pros: Fast path for common case
- Cons: Two code paths to maintain; "which is authoritative?" confusion; stale cache bugs are the current problem
- Rejected: performance gain is negligible (sub-ms Dolt queries) and consistency cost is high.

## Validation / Rollout

1. Add `metadata.spec_id` to epic creation in `plan approve` and `spec create` (`--metadata='{"spec_id":"..."}'`)
2. Add `DiscoverActiveSpecs()` function that queries `bd list --type=epic --status=open --json` and derives phase from bead statuses
3. Add `ResolveContext(root)` function that returns `(specID, beadID, phase, worktreePath)` by combining beads query with path conventions
4. Update `instruct` to use `ResolveContext()` instead of `ReadFocus()`
5. Update `next`, `complete`, `approve` commands to derive context from beads instead of reading/writing focus or lifecycle.yaml
6. Remove `state.Focus`, `state.Lifecycle`, `ReadFocus()`, `WriteFocus()`, `ReadLifecycle()`, `WriteLifecycle()`, and all callers
7. Remove lifecycle.yaml generation from `specinit`, `approve spec`, `approve plan`
8. Backfill `metadata.spec_id` on existing epics (parse from title convention as migration)
9. Update LLM test harness: replace focus/lifecycle assertions with beads-based assertions; update sandbox setup to use `bd update` instead of `WriteFocus()`/`WriteLifecycle()`
10. `mindspec doctor` detects and removes stale `.mindspec/focus` and `lifecycle.yaml` files
11. Run full LLM test suite to verify
