# Spec 098-lifecycle-correctness-2: Lifecycle correctness 2 (harness ADR coverage, close-verify, validation auto-fill, gate hardening)

## Goal

Close the five genuinely-open lifecycle-correctness defects that survived a re-audit of the spec-091/096 cleanup: an LLM-harness impl-approve scenario that exits 1 on a spurious ADR-divergence block, a `complete` close-verify that cannot detect a non-persisting `bd close`, an over-strict plan-approve gate that hard-blocks on a missing-but-defaultable `version`, a pre-complete hook matcher that fails open on wrapper-prefixed `mindspec complete`, and a direction-blind harness merge-to-main assertion. After this spec, each lifecycle path either succeeds on the merits, fails with a clear recoverable error, or correctly distinguishes safe from unsafe — never silently slips through and never spuriously blocks a clean run.

## Background

A prior re-audit of the open lifecycle beads against current `main` closed three items that had already been fixed in flight (the agent-facing merge-to-main root cause in `implement.md:98` / commit `d5d8ee54`, the now-present `default`-case hard error on a close status mismatch, and the relaxed `StepsCount`/`adr_citations` plan checks). The five requirements below are what remained genuinely open after that re-confirmation, each re-cited against current code in its requirement. Three are correctness gaps in shipping lifecycle commands (`complete`, `approve plan`, the pre-complete hook); two are test-infrastructure defects in `internal/harness` that make the LLM-harness suite either falsely fail (`ScenarioImplApprove`) or falsely pass-blind (the direction-blind assertion). They are grouped because they share the lifecycle-correctness theme, the same two impacted domains (workflow + execution), and the same ADR error/gate contracts; none is large enough to warrant its own spec.

## Requirements

1. **(myn3 — M) Wire ADR/ownership coverage into the fixture-based LLM-harness impl-approve scenario (`ScenarioImplApprove`) so a clean impl-approve succeeds.** Failure mode today: `ScenarioImplApprove` (`internal/harness/scenario_spec_lifecycle.go`, the builder around lines 360-444) commits a source `done.go` at the sandbox root (written to `wt.SpecWtDir+"/done.go"`, landing as `done.go` on `main` after merge) but ships NO `.mindspec/domains/*/OWNERSHIP.yaml` coverage and leaves the fixture plan's `adr_citations` empty. `CheckADRDivergence` runs unconditionally in impl-approve (`internal/approve/impl.go:277`) and, absent `--override-adr`/`--supersede-adr`, returns a hard error when `adrResult.HasFailures()` (`internal/approve/impl.go:302-305`) — so the committed-but-unowned `done.go` triggers `adr-divergence-unowned` and a perfectly clean approve exits 1. The shared helper `writeSandboxDomainCoverage` (`internal/harness/scenario_contract_hardening.go:61`) already produces the minimal ownership + Accepted-ADR-0001 + plan-`adr_citations` triple and is wired into 6 other scenarios (stale_phase, gate_discovery, etc.) but is NOT applied to this legacy scenario. Change: apply the proven recipe to `ScenarioImplApprove` — call `writeSandboxDomainCoverage(sandbox, "done.go")` and `sandbox.Commit(...)` at the sandbox ROOT BEFORE `setupWorktrees` forks the worktree (so the OWNERSHIP.yaml + Accepted ADR-0001 are present on the base ref the divergence gate diffs against), with the path claim (`done.go`) matching where the impl file lands on `main` after merge, and add `adr_citations:\n- ADR-0001` to the fixture plan's frontmatter (currently absent at the inline plan around lines 386-395). **What must be true after:** `ScenarioImplApprove` passes with a clean impl-approve (exit 0, spec branch merged + deleted, worktree removed, impl content on main) and NO spurious `adr-divergence-unowned` block.

   1a. **(DEFERRED — follow-up bead) `ScenarioSpecToIdle` is NOT in scope for this requirement.** `ScenarioSpecToIdle` (`internal/harness/scenario_spec_lifecycle.go:10-56`) has an EMPTY `Setup` and is FULLY agent-authored: the haiku agent picks the impl filename (`hello.go`), runs the whole lifecycle live, and authors `spec.md`/`plan.md` itself. The fixture therefore cannot pre-inject `adr_citations` into the agent-generated plan. Even though ownership path-matching DOES support globs (`internal/validate/ownership.go` `GlobMatch`, so a wildcard claim could cover an agent-chosen filename), `attributeDomain` + the `adr-divergence-uncovered` branch require the covering ADR to be **plan-CITED** (`IsDomainCovered`, `internal/validate/plan.go:577`) — and a fixture cannot cite an ADR in a plan the agent writes, nor is the agent prompted to. So a Setup-committed wildcard ownership claim would merely flip the failure from `adr-divergence-unowned` to `adr-divergence-uncovered`, not make the clean approve pass. Because no fixture-only mechanism makes `ScenarioSpecToIdle`'s agent-authored impl-approve pass cleanly, it is DEFERRED to a named follow-up bead (`mindspec-098-spectoidle-coverage`, to be filed at spec-approve) which will solve the distinct agent-prompting problem (prompt the agent to author the ownership + ADR-citation coverage triple, or relax the harness assertion for this agent-driven scenario). This spec's Req 1 / AC-1 scope to `ScenarioImplApprove` only.

2. **(9n2h — M) Harden `complete` close-verify so a non-persisting `bd close` is reliably DETECTED via a forced commit + committed-state verification, never silently completed.** Failure mode today: after `closeBeadFn` returns nil, `complete` re-reads status via `fetchBeadByIDFn` → `next.FetchBeadByID` → `bd show --json` (`internal/next/beads.go:172`) — the SAME bd/Dolt session path the close wrote through, with no forced commit and no committed-state read. The post-close retry loop (`internal/complete/complete.go:401-409`) retries only on a read ERROR; it never confirms DURABILITY, so a close that lands in-session but does not persist to committed Dolt state reads back `"closed"` (case (a) at `complete.go:432`) and `complete` proceeds to merge + worktree removal on an unverified close. (Observed recurring this session; the `default` case (b) at `complete.go:435` already hard-errors when the re-read shows open/in_progress, but does not catch the in-session-only "reads closed" case.) Note: the missing boundary is NOT process separation — the re-read is already a second `bd` subprocess. The missing boundary is an independent COMMITTED-state verification.

   **Change (concrete, testable durability mechanism):** after `closeBeadFn` returns nil, force durability with `bd dolt commit` — the strongest available bd durability primitive, confirmed present via `bd dolt commit --help` ("Create a Dolt commit from any uncommitted changes in the working set … useful before push operations that require a clean working set"). Add a new `internal/bead` wrapper (`bdcli.go`, e.g. `DoltCommit()`, running `bd dolt commit`) per ADR-0012 rather than ad-hoc shelling. Then perform a committed-state verification re-read. If EITHER the `bd dolt commit` step FAILS or the verification read shows the bead is not `closed`/errors, emit a clear, recoverable ADR-0035 error (via the existing `guard.NewFailure` recovery-line path) that KEEPS the worktree, does NOT set `BeadClosed`, and is safe to re-run (both the close step and `bd dolt commit` are idempotent) — mirroring the existing case-(b) PRE-MERGE return at `complete.go:435-444`. The command must NEVER print `closed` + exit 0 on an unverified close.

   **Injectable test seam (required):** add a new function-variable seam alongside the existing `closeBeadFn` / `fetchBeadByIDFn` package-level seams (`internal/complete/complete.go:28,34`) for the durability/verification step — e.g. `doltCommitFn = bead.DoltCommit` and a `verifyCommittedFn` (the committed-state verifier) — so a RED-on-revert regression test can simulate the make-or-break case: `closeBeadFn` returns nil, the normal `fetchBeadByIDFn` returns `"closed"`, but the committed-state verifier (`verifyCommittedFn`) reports not-closed/error → assert a non-zero error, the worktree is RETAINED, and `BeadClosed == false`. This is the exact failure today's seams cannot simulate (today they can only re-read open/error), so the new seam is what makes the bug testable.

   **HONESTY CLAUSE (implementation fallback, stated up front so R2 is shippable either way):** if implementation finds that `bd` exposes NO committed-state read distinct from `bd show` (i.e. after `bd dolt commit`, `bd show` still reads the same session state and a genuinely-in-session-only close still reads `"closed"`), then `verifyCommittedFn` cannot be grounded as a true committed-vs-session discriminator. In that case R2 is scoped to: (a) force `bd dolt commit` after every close, (b) DETECT and error on `bd dolt commit` FAILURE (non-zero exit / wrapper error) — never silently proceed on an unverified close, (c) emit the recoverable ADR-0035 error keeping the worktree / `BeadClosed=false`, and (d) FILE an upstream-bd bead (`mindspec-098-bd-committed-read`) for the residual durability-read gap. Under this fallback the RED test asserts the `bd dolt commit`-failure path (`doltCommitFn` returns an error → recoverable error, retained worktree, `BeadClosed == false`). Either path is implementable, testable RED-on-revert, and shippable. **What must be true after:** after `bd close`, `complete` forces `bd dolt commit` and (where a committed-state read exists) verifies the committed status; a close whose durability cannot be confirmed — commit failure OR a committed-state mismatch — is detected and surfaced as a clear recoverable ADR-0035 error with the worktree retained and `BeadClosed` unset; `complete` never prints `closed` + exit 0 on an unverified close.

3. **(e6qq — S) Relax the plan-approve `version` gate by AUTO-FILLING a canonical default of `"1"`.** Failure mode + behavior change (this requirement LOOSENS a gate): `internal/validate/plan.go:304` raises a HARD error `frontmatter-version: missing required field: version` when `fm.Version == ""`, with no auto-default-fill — even though sibling checks were already relaxed for ZFC alignment (`StepsCount < 3` demoted to a warning at `plan.go:797`; empty `adr_citations` demoted to a warning when a `## ADR Fitness` section is present at `plan.go:137-147`). An agent-generated plan that is otherwise valid but omits only `version` is hard-blocked on a field that can be deterministically defaulted, and nothing downstream reads `PlanFrontmatter.Version` (verified: the only consumers — `internal/approve/plan.go`, `internal/validate/divergence.go` — read `ADRCitations`/`WorkChunks` only), so any well-defined default is safe. Change (ONE pinned behavior): when `fm.Version == ""`, AUTO-FILL it to the canonical default `version` value `"1"` (the string `"1"`) instead of raising `frontmatter-version`, so the plan validates. Auto-fill is preferred over demote-to-warning because it makes agent plans Just Work. **What must be true after:** a plan missing only `version` (all other required fields present) no longer fails `mindspec validate plan` / `approve plan`; the `version` field is treated as / written with the exact value `"1"`. (This is an intentional gate loosening; `status` and `spec_id` remain hard-required.)

4. **(7eup — S) Make the pre-complete gate matcher catch unquoted wrapper-prefixed `mindspec complete`.** Failure mode + behavior change (this requirement TIGHTENS a gate): `segmentInvokesComplete` (`internal/hook/precomplete.go:366-394`) strips only leading env-assignments (`isEnvAssignment`) and `cd`/`pushd` prefixes before checking `fields[0]` is the mindspec binary and `fields[1] == "complete"`. Unquoted wrapper commands therefore fail OPEN: `env VAR=x mindspec complete …`, `timeout 30 mindspec complete …`, `xargs … mindspec complete`, and `command mindspec complete …` are not recognized as invoking complete, so the pre-complete gate does not fire. Change: extend the matcher to recognize and skip the catchable unquoted wrapper prefixes (`env`, `timeout`, `xargs`, `command`) so the wrapped `mindspec complete` is still detected. The matcher's terminal check is PRESERVED unchanged — after stripping wrapper prefixes it still requires `isMindspecBinary(fields[0]) && fields[1] == "complete"`, so the tightening fires ONLY when `complete` is genuinely the wrapped command and introduces no false positives (`timeout 30 go test` strips to `go test` → not mindspec → false; `env FOO=bar mindspec next` → `fields[1] != "complete"` → false). The exact per-wrapper arg-skip rules (`timeout`: skip the duration plus its `-s SIG`/`--preserve-status`/`-k` flags; `command`: skip `-p`/`-v`/`-V`; `xargs`: skip its flags; bare `env`: skip leading `VAR=x` assignments after the keyword) are a plan-phase implementation detail to enumerate precisely so the matcher neither under-skips (misses the wrapped complete) nor over-skips into a wrong `fields[0]`. The QUOTED `sh -c '…'` / `eval '…'` forms are structurally unreachable by a non-executing tokenizer and remain an explicitly-documented accepted ADR-0037 residual — do NOT attempt to match them; document that boundary in code/comment. **What must be true after:** `env VAR=x mindspec complete`, `timeout 30 mindspec complete`, `xargs … mindspec complete`, and `command mindspec complete` are all detected by `segmentInvokesComplete` and trigger the pre-complete gate; the quoted `sh -c`/`eval` forms remain a documented residual.

5. **(pi24 — S) Make the harness merge-to-main assertion direction-aware and fix its message.** Failure mode today (test-layer only): `preApproveImplMainMergeOrPRViolation` (`internal/harness/asserts.go`, the merge check around lines 199-207) flags any `git merge` whose args contain both `merge` and `main` as a pre-approve "merge-to-main occurred before approve impl" violation, regardless of DIRECTION. This is direction-blind: `git merge main` (merging main INTO a feature/spec branch — a legitimate update) matches the same as the genuinely-bad `git merge <branch>` while on `main`, and the message ("merge-to-main occurred") is misleading for the safe direction. (The agent-facing root cause was already fixed in commit `d5d8ee54` / `implement.md:98`; this is the residual test-layer defect.) Change: make the assertion direction-aware — only flag the genuinely-bad direction (a merge that lands content ONTO `main`, i.e. run while the checkout is on `main` / `git merge <spec-branch>` into main), allowing `git merge main` into another branch — and correct the violation message to describe the actual bad direction. **What must be true after:** the assertion flags only merges that land onto `main` before approve-impl (with an accurate message) and no longer flags a legitimate `git merge main` into a feature/spec branch.
