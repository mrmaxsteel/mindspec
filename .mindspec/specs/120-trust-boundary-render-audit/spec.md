# Spec 120-trust-boundary-render-audit: Trust-Boundary / Agent-Writable Render + Derivation Audit

## Goal

Make the repo-wide **provenance taxonomy** — agent-writable free text ⇒ escape at render; structured IDs ⇒ **an untrusted string must pass `idvalidate` before it is promoted to an ID at every authority-bearing consumer** (composition, executable operand, render, semantic lookup, persistence), after which clean IDs stay raw — **and no value is ever *presented as* an ID unless it validated** (invalid values render forcibly quoted); agent-writable path components ⇒ predicate-validated + shell-safe — actually **enforced, not assumed**. Round 2 proved the prior refuse-at-every-render-sink approach unsound at its root (`slugify` preserves shell metacharacters from agent-writable bd epic metadata; `idvalidate` gates only `workspace.SpecDir`); round 3 proved two prerequisites of the resulting validate-at-derivation spine: **(1) the validators' grammar is wrong** — `idvalidate.BeadID` (`internal/idvalidate/ids.go:29`) rejects **489 of 774 live bd IDs** (every dotted epic-child like `mindspec-9cyu.1` and every ≤3-char-suffix legacy ID like `mindspec-0ke`/`mindspec-mol-015`), and `idvalidate.SpecID` (`ids.go:19`) rejects the live spec dirs `008b-human-gates`/`008c-prime-compose` — so applying them unchanged would brick `complete`/`next`/worktree-detect for most beads; and **(2) enumerating derivation sources spirals** exactly as enumerating sinks did — round 3 found five more ingresses (CLI `--spec`, `impl approve args[0]`, lifecycle-scan bd enumeration, inline `"bead/"+` concat, direct `filepath.Join`) the D1/D2/D3 enumeration missed. This revision therefore (1) **corrects the `idvalidate` grammar to the framework's real ID format first** (contract WIDENING: dotted children, short suffixes, letter-suffixed spec dirs — pinned by a live-inventory fixture; every security rejection preserved), and (2) moves the hard guarantee to the **composition waist**: the ten `workspace` composition helpers validate their ID argument internally, so every caller — CLI, metadata, dir-parse, lifecycle-scan — is covered **by construction**, and D1/D2 remain only the degrade-vs-error *policy* points; round 4 surfaced the waist's mirror image — **reverse derivation**, parsing an ID back OUT of an agent-writable string — and round 5 then proved derivation-shape enumeration cannot converge either (four parser shapes in two rounds: `TrimPrefix`, `ReadDir`, `ReadFile`+`Unmarshal`, `Index`-slice), so (3) the reverse class closes at the **FINITE authority-bearing CONSUMER boundary, not the unbounded derivation shapes**: every consumer where a string is *treated as* an ID — the five classes: **structural composition** (the waist + gated concats), **executable operands**, **render/display** (the new forced-safe `idrender` helper), **semantic lookup**, **structured persistence** — validates before the string exercises ID authority; the derivation-site gates and the R6 reverse scans remain as labeled defense-in-depth. The **convergence stopping rule** follows: the only admissible future reverse-derivation finding is an unvalidated value exercising ID authority at a consumer missing from the five-class inventory (or a gate failing its AC); a new parser shape alone is a non-blocking ratchet-hygiene note, because its hostile product is contained at every consumer class. Downstream `cd`/`git merge`/`git branch -d`/template operands then carry only `[a-z0-9./-]` bytes by construction. Retained around the spine: the **display sweep** (`termsafe.Escape` for genuinely-free-text fields; the forced-safe `idrender.Spec`/`idrender.Bead` for ID-typed positions — byte-identity on every validated ID, `strconv.Quote` on anything else), the **config `worktree_root` predicate** (charset + lexical pin + symlink-aware containment **with an explicit check-at-use and an honestly bounded TOCTOU residual** + shell-safe emit), the **composition ratchet re-aimed at inline bypass** (helper-call allowlist + inline-concat/Join scans + template classification), and the three bundled hygiene items (`mindspec-0wxh`, `mindspec-4xzi`, `mindspec-gne7`). Clean inputs render byte-identically to today; no gate or command decision changes for any well-formed input — a claim now genuinely true against the corrected patterns (fixture-verified: 774/774 live bd IDs and 120/120 spec dirs pass them).

## Background

**Why now, and why this is not 116 again.** Spec 116 shipped the escaping *mechanism* (`internal/termsafe.Escape`: printable ASCII `[0x20,0x7e]` unchanged, else single-line `strconv.Quote`) and closed the panel-gate path. Three review rounds on this spec then established, verified against this worktree, that (i) the raw-render *class* spans `cmd/` + `internal/` far beyond the panel gate, (ii) escaping is structurally insufficient for **executable** renders — `termsafe.Escape` is the identity on printable ASCII, so a hostile-but-printable operand survives it byte-identically and executes on agent compliance — and (iii) neither sink-enumeration (round 1→2) nor source-enumeration (round 2→3) converges: each pass found channels the previous matcher could not see. The structural answers are upstream of both spirals: correct the ID grammar, then validate where IDs are **composed**.

**The root cause (round-2, orchestrator-verified).** `slugify` (`derive.go:112`) only lowercases, hyphenates spaces/underscores, and collapses hyphens — it PRESERVES `;`, `|`, `$`, `#`, `&`, `(`, `)`, backticks, and `..`. `spec_title` is agent-writable bd epic metadata (`ExtractSpecMetadata`, `derive.go:646`, with an equally-writable `[SPEC NNN-slug]` title fallback at `:666`). Every metadata-derived specID flows through the single constructor `SpecIDFromMetadata` (`derive.go:106`) — callers: `specIDForEpicWithCache` (`:225`), `DiscoverActiveSpecsWithCache` (`:442`), `FindEpicForBeadWithCache` (`:754/:777`, the lineage `complete` trusts per spec 119), `cache.go:189`, `internal/lifecycle/scan.go:100`, `internal/approve/plan.go:223`. None validates the result. Bead IDs are bd-sourced or CLI-supplied and never pass `idvalidate.BeadID` on the live paths — and bd-sourced is NOT safe: `bd create --force --id=<arbitrary>` writes any id directly to Dolt (round-9 empirical proof, below), so bd ids are agent-writable like every other ingress. Worktree-directory-parsed IDs (`workspace.DetectWorktreeContext`, `workspace.go:655` — a raw `strings.TrimPrefix` of an on-disk dir name) are a third un-gated ingress. Round 3 added five more (below). The taxonomy promised "ingress-validated ⇒ stays raw" — this spec makes that promise TRUE instead of assumed.

**The corrected grammar (R1 — round-3 Blocker A, the prerequisite).** The current patterns are empirically wrong about the framework's own IDs. Live inventory, verified in this worktree: `.beads/issues.jsonl` holds 774 unique bd IDs, of which **489 FAIL** `beadIDPattern` (`ids.go:29`, `^[a-z][a-z0-9]*(?:-[a-z0-9]+)*-[a-z0-9]{4,}$`): 265 dotted epic-children (`mindspec-9cyu.1`, multi-level up to `mindspec-69y.2.2`, child indexes up to `.11`) — bd's CURRENT minting scheme for every spec's beads — plus 224 flat IDs whose final segment is shorter than the pattern's `{4,}` floor (`mindspec-0ke`, and the three-segment `mindspec-mol-015` family). `.mindspec/specs/` holds 120 dirs of which 2 FAIL `specIDPattern` (`ids.go:19`): `008b-human-gates`, `008c-prime-compose` (letter-suffixed number). The corrected patterns, validated against the full live inventory (0 failures):
- `specIDPattern` → `^[0-9]{3,}[a-z]?-[a-z0-9]+(?:-[a-z0-9]+)*$` — one optional letter suffix after the number; everything else unchanged.
- `beadIDPattern` → `^[a-z][a-z0-9]*(?:-[a-z0-9]+)+(?:\.[0-9]+)*$` — (i) any-length alnum segments (drops the false `{4,}` floor; still ≥2 segments, non-empty, no leading/trailing/consecutive hyphens), (ii) zero or more dotted **numeric** child suffixes, any depth (`<parent>.<n>[.<m>…]`).
This is a **contract widening**: every ID the old patterns accepted still passes (old-accept ⊂ new-accept); the security rejections are all preserved — empty, `.`, `..`, path separators `/\`, glob metacharacters `*?[]{}` (explicit checks in `SpecID`/`BeadID`, unchanged), and by charset every shell metacharacter (`;|$&()#`, backtick, quotes), whitespace, control byte, and uppercase. Traversal stays impossible: each `.` must be followed by digits, so `..` cannot occur and an ID cannot begin or end with `.`. The `ids.go:29` and `:83-89` doc comments (which explicitly disclaim the hierarchical format bd actually mints) are corrected; the package stays a stdlib-only leaf (regex + tests only); the SEC-1 header hazard (`ids.go:1-9`) is unchanged. Pinned by a **live-inventory fixture** (AC-1): a committed testdata snapshot of every current `.mindspec/specs/` dir name and every live bd ID must pass the corrected validators — RED today (489+2 failures), and RED again if a future legacy form is added without widening.

## Non-Goals

- **Not tamper-resistance.** Repo artifacts, config, and bd fields stay agent-writable; this spec stops them forging output lines or executable operands — derivation validation + output hygiene, not artifact authentication.
- **Not atomic filesystem containment.** The R5 symlink containment is check-then-use twice over (composition time AND immediately before each use); the window between the check-at-use and the kernel/git operation remains — `git worktree add` offers no `O_NOFOLLOW` equivalent. An adversary who can WIN that race holds concurrent local write access inside the working tree, i.e. the same capability plane as editing hooks or config directly — already outside this spec's threat model (see tamper-resistance above). The residual is accepted, named, and bounded to that plane; AC-11 pins that the check-at-use exists, not that the race is unwinnable.
- **Not full prompt-injection safety.** The spine + escaping close the hostile-operand, control-byte, forged-line, and traversal classes; a persuasive printable string in a fenced payload still renders in its labeled context.
- **Not a decision change.** No gate, guard, or command changes its Action/exit for any well-formed input; refusals fire only on inputs invalid under the CORRECTED `idvalidate` patterns / the R5 predicate. Every currently-existing spec ID, bead ID, and default config in this repo is well-formed **under the corrected grammar** — verified against the live inventory (774/774 bd IDs, 120/120 spec dirs incl. `008b`/`008c`) and pinned by the AC-1 fixture — so the spine is behavior-invisible on this repo today. (Under the OLD patterns this claim was false — 489 bd IDs and 2 spec dirs failed — which is exactly why R1 precedes everything.) (A space-bearing operator root is CLEAN: root is not charset-validated, and the quoted `cd` it now gets replaces an already-broken unquoted one.)
- **Not a config-surface change.** No new `.mindspec/config.yaml` knobs; the taxonomy is an L1 invariant.
