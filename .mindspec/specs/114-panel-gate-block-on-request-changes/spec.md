# Spec 114-panel-gate-block-on-request-changes: Panel gate blocks on any unresolved REQUEST_CHANGES, with audited-refutation escape

## Goal

Close the last silent out-vote in the authoritative `mindspec complete` panel gate: today a reviewer's REQUEST_CHANGES finding can be out-voted by the approve count (the default N−1 threshold tolerates one dissent), so a real, evidenced finding is droppable without anyone recording why. Two operator decisions, already made, are encoded here:

1. **R1 (gate change)** — any **unresolved REQUEST_CHANGES blocks `mindspec complete`, exactly like a REJECT**. The gate passes only when every expected reviewer's latest-round verdict is APPROVE, or is a REQUEST_CHANGES that has been explicitly refuted (Decision 2). This is a genuine change to the `PanelGateDecision` matrix and amends **ADR-0037**.
2. **R2 (spurious-finding escape)** — because a mechanical gate cannot judge whether an RC is correct, the operator/orchestrator can clear ONE specific reviewer slot's REQUEST_CHANGES by recording an **audited refutation** (slot id + reason + evidence), following the existing abandon precedent: always audited on bead metadata, never silent. A refuted RC no longer blocks; the audit entry persists. This is the only agent-settable escape for a specific RC — the human-only `MINDSPEC_SKIP_PANEL` hatch and the whole-panel abandonment procedure are unchanged.

Target user outcome: an operator or orchestrating agent running `mindspec complete` **cannot silently out-vote a reviewer's REQUEST_CHANGES by hitting the count**; a genuinely-spurious RC is cleared only via an explicit, audited, per-slot refutation that leaves a permanent record on the bead — the same "legitimate precisely because it is always audited, never silent" property ADR-0037 §7 already grants abandonment.

## Background

**Where the out-vote lives today.** The single decision home is `panel.PanelGateDecision` (`internal/panel/gate.go:131-263`). Its short-circuit legs (comment block at gate.go:110-127) end with:

- leg (9), gate.go:245-253 — `f.Res.Rejects > 0 || len(f.Res.HardBlocks) > 0` → Block ("no vote count overrides an evidence-bearing gate");
- leg (10), gate.go:255-262 — `f.Res.Approves >= threshold && threshold > 0` → **Allow**, where `threshold` is `Panel.ApproveThreshold()` (`internal/panel/panel.go:113-127`): default **N−1**, optionally overridden per-panel by the recorded `approve_threshold` expression (spec 109 / ADR-0040).

A REQUEST_CHANGES verdict is counted as *neither* an APPROVE nor a REJECT: the tally switch at `internal/panel/tally.go:296-301` increments `Approves` only for `VerdictApprove` and `Rejects` only for `VerdictReject`; per the `Result` doc (tally.go:107-111), "REQUEST_CHANGES (and anything unrecognized) is neither." So on the default 6-reviewer panel (threshold 5): **5 APPROVE + 1 REQUEST_CHANGES, complete verdicts, fresh SHA → `Approves` = 5 ≥ 5 → Allow at gate.go:257-258.** The RC's `concrete_changes_required` never has to be addressed or even acknowledged; the merge proceeds and no record of the dissent survives on the bead. (The same arithmetic lets one *unrecognized* verdict string ride through — it is also "neither".) The vote-only advisory twin `Result.VoteDecision` (`internal/panel/tally.go:180-214`, threshold legs at 206-213) reproduces the identical pass, by design ("can never disagree on the vote portion").

**Every surface shares this decision, so fixing it once fixes them all.** The authoritative in-binary gate is `panelGate` (`internal/complete/panel_advisory.go:140-215`), invoked at complete.Run step 2.25 (`internal/complete/complete.go:337-378`) before any mutation; on Block it returns a `guard.NewFailure` whose final line is an ADR-0035 recovery command (panel_advisory.go:194-207). The read-only surfaces — `mindspec panel verify`/`panel tally` (`cmd/mindspec/panel.go:7-10`, pinned by `TestPanelVerbs_DecisionIsPanelGateDecision`) and `mindspec instruct --panel-state` (`internal/instruct/panelstate.go:121`) — delegate to the same `PanelGateDecision`, and complete's warning-only `panelAdvisory` (panel_advisory.go:48-77) renders `VoteDecision`.

**The audit precedent to follow (Decision 2).** Abandonment is the existing "legitimate exit" shape: `"abandoned": true` + mandatory `abandon_reason` recorded on `panel.json` (`internal/panel/panel.go:58-68`, deliberately parse-lenient), honored by the gate as Warn/Allow at leg (3) (gate.go:160-170), and **audited at complete time**: `writePanelAuditMetadata` (`internal/complete/panel_advisory.go:304-342`) writes `panel_abandoned` + `panel_abandoned_at` + `panel_abandoned_reason` (and, for the env hatch, `panel_gate_skipped` + `panel_gate_skipped_at`) to bead metadata via `completeMergeMetadataFn` → `bead.MergeMetadata` (`internal/bead/bdcli.go:263-266`), called at step 5.6 (`internal/complete/complete.go:778-785`) only after the terminal mutation succeeds, best-effort. The operator procedure is documented in `plugins/mindspec/skills/ms-panel-tally/SKILL.md:84`. The refutation record follows this exact pattern: a recorded repo-file structure the gate reads, plus a persistent audit entry on the bead.

**Trust boundary is (almost entirely) unchanged.** Per ADR-0037 §8, every gate input is an agent-writable repo artifact; the refutation *record* on `panel.json` is exactly as agent-writable as `abandoned` is today. This spec adds an anti-footgun (a finding can no longer vanish *by arithmetic*, without a trace) and ONE narrow, operator-authorized **audit-durability** guarantee (§8 amendment): once a refutation has durably cleared an RC, its audit persists across retries even if the panel is later removed. This is deliberately NOT general anti-adversary tamper-resistance — no signing, no hashing, no out-of-repo store, the record stays agent-writable and an empty `reason` still clears — only that an applied refutation's clearance always leaves a durable trace on bead metadata (the single state authority, ADR-0023). The general "do not fix perceived forgeability" posture §8 sets is otherwise intact.

## Impacted Domains

- **workflow**: `internal/panel/**` (the decision matrix in `gate.go`, the tally/`VoteDecision` in `tally.go`, the `panel.json` schema in `panel.go`), `internal/complete/**` (the authoritative gate wiring + the new refutation audit write in `panel_advisory.go`), `cmd/**` (the panel verbs inherit the new decision; any refutation-recording surface), and `plugins/mindspec/**` (the /ms-panel-tally refutation procedure documentation) — all owned by workflow per `.mindspec/domains/workflow/OWNERSHIP.yaml` (lines 2, 14, 19-20).
- **execution**: `internal/bead/**` (`bead.MergeMetadata` in `bdcli.go` + a read helper for bead metadata) — owned by execution per `.mindspec/domains/execution/OWNERSHIP.yaml` (line 3). **Amendment note (2026-07-10, plan-approve round-3, operator-authorized):** an earlier draft scoped `internal/bead` read-only ("no change lands there"). The refutation-audit *durability* guarantee (R2/AC11) requires the durable pending-obligation on bead metadata to be un-eraseable, which requires making `bead.MergeMetadata` **fail-closed** on a metadata read error — an execution-domain companion change to the workflow-domain gate change. This is a deliberate, good-reason scope addition so the complete-time ADR-divergence gate (which derives its candidate domains from THIS section) attributes the `internal/bead/**` edit to a declared domain rather than hard-failing. ADR-0037 (Domain(s): workflow, execution) and ADR-0035 (workflow, execution, core), both cited by the plan, cover execution, so no new ADR is introduced.

## ADR Touchpoints

- [ADR-0037](../../adr/ADR-0037-panel-gate-enforced-contract.md): **amended by this spec** — the panel-gate enforced contract. **§1** (registration schema) gains the optional `refutations` array on `panel.json` (entries `{slot, round, reason, evidence}`, parse-lenient), the same schema-addition precedent §1 already records for 112's `gate` field and 109's `approve_threshold`. **§3** ("Threshold: `expected_reviewers − 1`, defined once" — "one dissent tolerated") gains the new rule: the threshold remains a necessary floor on genuine APPROVE count, but is no longer sufficient — every expected reviewer's latest-round verdict must be APPROVE or an explicitly-refuted REQUEST_CHANGES; an unresolved RC blocks exactly like a REJECT, and a refuted RC never counts toward the floor (so refutation cannot buy past a sub-threshold panel — ADR-0040's recorded `approve_threshold` stays meaningful). **§7** (escape hatches) gains the **audited refutation** alongside skip/abandonment/config-toggle, with the "legitimate precisely because it is always audited, never silent" contract — and, because a refutation produces a non-empty applied-refutation set (the grounding for the asymmetry — NOT any "terminal non-merge exit" claim, since abandonment and env-skip both Warn→Allow→MERGE and merely produce no applied refutations), its audit is **durable**: the durable-obligation protocol (a `refutation_pending` marker whose successful persistence is what makes a refutation "applied" at all, a satisfying `panel_refuted` or a `refutation_discharged` audit reconciled BEFORE close on every completion path, and fail-closed metadata reads/writes) so an applied refutation can never clear an RC that later merges un-audited. Like the `MINDSPEC_SKIP_PANEL` hatch, the block message must NOT advertise a paste-able refutation incantation; the escape is documented (§7 + /ms-panel-tally) but not reflex-suggested at the block. **§6** (fail-open without a panel) gains one affirmative-evidence refusal branch: a bead carrying a durable, unsatisfied refutation obligation on its metadata does not fail open — it refuses until the obligation is satisfied or discharged; a genuinely pristine bead (no obligation) still fails open exactly as today. **§8** is **AMENDED** by a narrow, operator-authorized **audit-durability** carve-out: a refutation actively clears a specific reviewer's finding, so the AUDIT of an applied refutation is made durable across retries — this is not general tamper-resistance (the record stays agent-writable; an empty `reason` still clears; a bead with no refutation is freely completable; no signing/hashing is added), only a guarantee that an applied refutation's clearance leaves a permanent trace. The amendment text must state the carve-out and re-affirm the otherwise-unchanged §8 posture.
- [ADR-0035](../../adr/ADR-0035-agent-error-contract.md): agent error contract. The new RC Block message must name the unresolved slot(s) and end with a genuine `recovery:` line (the existing Block path already routes through `guard.NewFailure`, `internal/complete/panel_advisory.go:194-207`; the convention test `internal/guard/recovery_convention_test.go` enforces the final-line rule).
- [ADR-0040](../../adr/ADR-0040-orchestration-layering-ratchet.md): the recorded `approve_threshold` override (spec 109) layers under this change — a recorded threshold below N−1 still sets the approve *floor* but no longer licenses unrefuted dissent (see R1's chosen model and OQ1). The amendment must state this interaction explicitly so 109's extension and 114's rule cannot be read as contradicting.
- [ADR-0023](../../adr/ADR-0023.md): beads as single state authority. The refutation record on the review artifact is a *review artifact, not workflow state* (ADR-0037 §8 closing paragraph); the durable audit lives on bead metadata exactly like `panel_abandoned`.

## Requirements

1. **Any unresolved REQUEST_CHANGES blocks `mindspec complete`, exactly like a REJECT.** `panel.PanelGateDecision` (`internal/panel/gate.go:131-263`) gains a new blocking condition, **layered on top of** the existing matrix (chosen model — layer, not replace; OQ1 resolved): the `approve_threshold`/N−1 floor stays a **necessary** condition on the count of **genuine `APPROVE`** verdicts, and *in addition* the gate Blocks unless **every** latest-round verdict among the complete set is either (a) canonical `APPROVE`, or (b) canonical `REQUEST_CHANGES` covered by a recorded refutation for that slot at that round (R2). A refuted RC is removed from the *blocking* set but **never** increments `Approves` (tally.go must not count it as APPROVE), so refutation can never buy past the approval floor: an **at-threshold** panel unblocks on refutation (e.g. 7 APPROVE + 1 refuted RC, threshold 7 → passes), but a **sub-threshold** panel still Blocks on the floor even after every RC is refuted (e.g. 6 APPROVE + 2 refuted RC, threshold 7 → 6/7 still blocks; the recourse is a re-panel for genuine approvals, not refutation). This keeps the recorded `approve_threshold` (spec 109 / ADR-0040) meaningful. An unrefuted REQUEST_CHANGES — and equally an unrecognized verdict string, which is likewise "not an APPROVE" (tally.go:107-111) and is NOT refutable — Blocks. All other blocking conditions are byte-preserved: incomplete panels (leg 8), stale SHA (leg 6), round mismatch (leg 4), dirty tree (leg 7), malformed registration (leg 2), REJECT/hard_block (leg 9), and the `Approves >= ApproveThreshold()` floor (leg 10) all still block exactly as today. **Message format (OQ4 resolved — do NOT advertise the escape)**: the Block message names the offending unresolved slot(s) by slot id, never prints the skip variable (HC-7), carries the raw-merge fence in its body, and ends with an ADR-0035 `recovery:` line whose PRIMARY advice is fix-and-re-panel (`/ms-bead-fix` + `/ms-panel-run step 0`); it does **not** print a paste-able refutation command/incantation — exactly as the gate never prints a paste-able `MINDSPEC_SKIP_PANEL` prefix. The refutation escape is discoverable in ADR-0037 §7 and the /ms-panel-tally skill, not reflex-advertised at the block, so it stays a deliberate looked-up action. The vote-only twin `Result.VoteDecision` (`internal/panel/tally.go:180-214`) moves in lockstep so complete's advisory line, `panel verify`/`panel tally`, and `instruct --panel-state` — all of which delegate to the same decision (`cmd/mindspec/panel.go:7-10`, `internal/instruct/panelstate.go:121`) — can never disagree with the gate on the vote portion. **Falsified if**: a panel with complete verdicts and a fresh SHA carrying N−1 APPROVE + 1 unresolved REQUEST_CHANGES still yields Allow from `PanelGateDecision` or a zero exit from `mindspec complete`; if a **sub-threshold** panel whose RCs are ALL refuted (e.g. 6 APPROVE + 2 refuted RC, threshold 7) yields Allow instead of blocking on the floor; if any OTHER blocking condition (REJECT, hard_block, incomplete, stale SHA, round mismatch, dirty tree, malformed registration, sub-threshold) stops blocking; if a refuted RC increments `Approves`; if the Block message prints a paste-able refutation incantation; if an unrecognized verdict string can pass the gate or be cleared by a refutation; or if `VoteDecision` reports the vote portion would pass while `PanelGateDecision` blocks on it (or vice versa) for the same tally.
2. **Audited refutation escape — per-slot, always recorded, never silent.** An operator/orchestrator clears ONE specific reviewer slot's REQUEST_CHANGES by **hand-editing `panel.json`** (OQ3 resolved — a plain repo-file edit, abandon-parallel; the gate does the validation, not the recording surface), adding to a new **`refutations` array** (OQ2 resolved — array on `panel.json`, NOT per-slot files) one entry with the exact schema `{"slot": <string>, "round": <int>, "reason": <string>, "evidence": <string>}`, where `slot` is the filename-derived `Verdict.Slot` (`internal/panel/tally.go:43-59`) and `round` is the round it refutes. The array is parse-lenient like `abandon_reason` (`internal/panel/panel.go:58-68`): a missing/empty field is a consumer concern, never a parse error that would tip the gate fail-open. Effects, all mandatory and enforced **in the gate** (`internal/panel` + `internal/complete`), not at record time: (a) a refutation clears a slot's RC only when that slot's **latest-round** verdict is canonical `REQUEST_CHANGES` and the entry's `round` matches that latest round — the panel then unblocks iff no other blocking condition remains; (b) **DURABLE audit — not best-effort, and durable ACROSS RETRIES**: because a refutation CHANGES the gate outcome (it actively clears an evidence-bearing block and produces a non-empty applied-refutation set — the grounding for the asymmetry is that AppliedRefutations, NOT any "non-merge exit" claim: abandonment and env-skip both Warn→Allow→MERGE and simply produce no applied refutations), a refutation that EVER durably clears an RC MUST NOT let its bead merge — across any number of `mindspec complete` retries, including one where the `panel.json` is later removed — without a durable audit on bead metadata. Durable-obligation protocol, all writes fail-closed and BEFORE the bead close (a closed bead is one commit from a merged one, so the obligation must be settled while the bead is still `in_progress`): **(i) "applied" ≡ durably-persisted.** A refutation is "applied" (clears its RC) ONLY once its `refutation_pending` marker (the applied slot(s)+round(s)) is durably on bead metadata; the marker write is PART of the allow-decision, so if it FAILS the refutation is NOT applied — the RC is NOT cleared — and the gate returns a **BLOCK** (bead stays `in_progress` with the RC unresolved; no abort-with-refutation-applied). This makes "applied ⟹ durable obligation" definitional. **(ii) reconcile on EVERY completion path, before close.** Every path (panel-present AND the §6 no-panel fail-open) must reconcile any recorded unsatisfied `refutation_pending` — never silently ignore it — by exactly one of: **satisfy** (the current run's applied set covers it → write the `panel_refuted` audit); **discharge** (a panel is present and the gate passed, so the pending slot is no longer an unresolved latest-round RC — resolved naturally by a re-panel APPROVE or an advanced round → write a `refutation_discharged` audit recording the natural resolution); or **refuse** (the resolution cannot be verified — the no-panel path, or a `bead.GetMetadata` read/parse error which is itself fail-closed → a recoverable hard failure: restore the panel so the gate can satisfy/discharge, or restore the audit). A genuinely pristine bead (no recorded obligation) reconciles to a no-op and fails open exactly as today. Neither the satisfy nor the discharge write may be swallowed as the warning `writePanelAuditMetadata` uses today (`internal/complete/panel_advisory.go:316-342`, best-effort) for the abandon/skip audits; and `bead.MergeMetadata` (`internal/bead/bdcli.go:263-266`) is itself made **fail-closed on a metadata read error** so no later best-effort write can erase a recorded `panel_refuted`/`refutation_discharged`. **Hatches excepted:** `MINDSPEC_SKIP_PANEL` and `enforcement.panel_gate: false` bypass the panel gate entirely (their own `panel_gate_skipped` audit records the bypass), apply no refutation, and record no obligation — the invariant is scoped to non-hatch completions. To record only genuine clears, the set of refutations the gate ACTUALLY APPLIED (each matched a latest-round `REQUEST_CHANGES` — never a stale, invalid, REJECT-targeting, or unused entry; deduplicated by slot+round) is plumbed OUT of the decision (`PanelGateDecision`/`panelGate`, `internal/panel/gate.go` + `internal/complete/panel_advisory.go:140-215`) into both writes; the `panel_refuted` entry carries its key (parallel to `panel_abandoned`) plus the applied slot(s)/round(s)/reason(s) and a timestamp, written via `completeMergeMetadataFn` → `bead.MergeMetadata`. (c) a refutation clears ONLY canonical `REQUEST_CHANGES` — never a `REJECT`, never a `hard_block: true`, never an unrecognized verdict, and never any non-vote condition (staleness, incompleteness, round mismatch, dirty tree, sub-threshold floor); (d) one refutation clears exactly the one named slot at the named round — no wildcard, no whole-panel form, and a newer-round re-RC of the same slot is not cleared; (e) the refutation procedure is documented in `plugins/mindspec/skills/ms-panel-tally/SKILL.md` beside the abandon procedure (line 84), with the same "do NOT refute to dodge a finding you have not actually disproven; legitimate precisely because always audited, never silent" framing abandonment carries — and crisply that a refutation's **reason + evidence carry its legitimacy**: an empty `reason` still clears (parity with the accepted empty-`abandon_reason` footgun, ADR-0037 §8), but that is a documented footgun, not an endorsed use. The trust boundary is unchanged: the record is agent-writable by design (ADR-0037 §8), legitimate because audited. **Falsified if**: recording a `refutations` entry for the sole unresolved-RC slot at the latest round does NOT unblock `mindspec complete` (all else clean); if an applied refutation lets `mindspec complete` finish successfully WITHOUT the `panel_refuted` bead-metadata entry being durably written (a swallowed metadata-write failure yielding a silent, un-audited gate-clearing completion); **if a refutation that ever durably cleared an RC lets its bead merge across ANY number of retries — including a retry after the `panel.json` is removed, or a live-panel retry where the RC resolved naturally — without a durable `panel_refuted` (satisfied) or `refutation_discharged` (moot) audit**; if a `refutation_pending` marker-write failure lets the RC clear (it must instead BLOCK, leaving the RC unresolved); if a `bead.GetMetadata` read/parse error on the reconciliation path fails OPEN (it must refuse); if a later transient `MergeMetadata` read failure erases a previously-written `panel_refuted`/`refutation_discharged`; if the audit records a refutation that did NOT actually clear a latest-round `REQUEST_CHANGES` (stale/invalid/unused); if a refutation can clear a REJECT, a hard_block, an unrecognized verdict, or any non-vote blocking condition; if it clears a slot whose latest-round verdict is not REQUEST_CHANGES; or if a single refutation entry clears more than the one named slot+round.
3. **Round-scoping and multiplicity (scenario probes).** (a) *Multiple RCs*: with two or more unresolved REQUEST_CHANGES in the latest round, each must be refuted individually; refuting one leaves the gate blocked naming the other(s). (b) *Natural resolution*: a re-panel in which the dissenting reviewer flips to APPROVE at the new round needs no refutation — the tally already reads only the filename-derived latest round (`internal/panel/tally.go:273-276`), so the old RC simply stops being tallied; this path must remain zero-ceremony. (c) *Refute-then-re-RC*: a refutation is bound to its recorded round — if the reviewer files REQUEST_CHANGES again at a newer round, the earlier refutation does NOT carry forward and the gate blocks again until that new RC is refuted or resolved. **Falsified if**: refuting one of two unresolved RC slots unblocks the gate; if a re-panel whose latest round is all-APPROVE still requires a refutation record for a previous round's RC; or if a refutation recorded against round N clears the same slot's REQUEST_CHANGES filed at round N+1.

## Scope

### In Scope
- `internal/panel/gate.go` — the new unresolved-RC blocking condition in `PanelGateDecision`, its Block message (slot-naming, HC-7, fence), and the refutation facts' plumbing through `GateFacts`/`ResolveGateFacts`.
- `internal/panel/tally.go` — surfacing per-slot RC state and refutation coverage in `Result`; `VoteDecision` lockstep change.
- `internal/panel/panel.go` — the refutation record schema (per OQ2's resolution), parse-lenient like `abandon_reason`.
- `internal/complete/panel_advisory.go` + `complete.go` — the **durable-obligation protocol**: the `refutation_pending` marker whose successful fail-closed persistence is what makes a refutation "applied" at all (a marker-write failure returns a BLOCK, leaving the RC unresolved); a pre-close reconciliation reached on EVERY completion path (panel-present AND no-panel) that satisfies (`panel_refuted`), discharges (`refutation_discharged`, when the RC resolved naturally), or refuses; all keyed off the gate's actually-applied refutations plumbed out of `panelGate`/`PanelGateDecision`. Unlike the best-effort `panel_abandoned`/`panel_gate_skipped` writes, a satisfy/discharge metadata-write failure must fail completion pre-close, not warn-and-continue.
- `internal/bead/bdcli.go` (execution domain) — `bead.MergeMetadata` made **fail-closed** on a metadata read error (so a later transient read cannot erase a recorded audit), plus a `GetMetadata` read helper the reconciliation consults (a read/parse error refuses, never fail-opens).
- `plugins/mindspec/skills/ms-panel-tally/SKILL.md` (and any mirrored skill copies) — the documented refutation procedure.
- `cmd/mindspec/panel.go` — the verbs inherit the decision change (OQ3 resolved to hand-edit: no new CLI recording verb in this spec).
- `.mindspec/adr/ADR-0037-panel-gate-enforced-contract.md` — the amendment (§1 `refutations` schema addition + §3 rule extension + §7 new durable escape + §8 narrow operator-authorized audit-durability carve-out + §6 affirmative-evidence refusal branch; the general anti-adversary posture otherwise unchanged).
- **RC-tolerance fixtures that MUST change semantically (R1's intended effect, in-scope updates)** — `internal/panel/panel_decision_test.go` (the "5 APPROVE + 1 dissent → Allow" case, ~line 184-193), `internal/complete/panel_advisory_test.go` (`TestPanelAdvisory_Passing_WouldPass`, 2 APPROVE + 1 REQUEST_CHANGES → "would PASS", ~line 68-82), and `cmd/mindspec/panel_test.go` (the "passing (at-threshold) → exit 0" case that carries an implicit dissent, ~line 386-389): each currently pins the exact RC-tolerated pass R1 removes and must be updated to the new Block/would-BLOCK expectation.
- Tests across `internal/panel`, `internal/complete`, `cmd/mindspec` pinning R1-R3 and every new AC below.

### Out of Scope
- `internal/bead` is NOT consumed read-only: `bead.MergeMetadata` is made **fail-closed** on a metadata read error and a `GetMetadata` read helper is added (see In Scope + Impacted Domains: execution). What stays out is any change to the underlying `bd`/Dolt storage, the `bd show`/`bd update` CLI surface, or the `completeMergeMetadataFn` seam signature — the durable-obligation reads/writes ride the existing seam shape.
- `internal/config` — no new config keys; per-gate reviewer counts and thresholds (spec 112) and the recorded `approve_threshold` interpreter (spec 109, `panel.go:113-127`) are read, not changed.
- The retired PreToolUse hook (removed by spec 102), `MINDSPEC_SKIP_PANEL`, the `enforcement.panel_gate` toggle, and the abandonment procedure — all byte-unchanged.

## Non-Goals

- **Not changing per-gate reviewer counts or threshold configuration** — that is spec 112's per-gate config and spec 109's recorded `approve_threshold`; this spec only adds a blocking condition on top of whatever threshold those resolve.
- **Not changing bead-panel mechanics or the final-review flow** — panel creation, round derivation, verdict-file conventions, `panel.ForBead`'s non-bead exclusion, and the /ms-panel-run//ms-spec-final-review choreography are untouched.
- **No auto-refuting** — a refutation is always an explicit, evidenced, audited record; no code path may synthesize one (not from a fix-commit, not from a tally heuristic, not from an orchestrator default).
- **No `mindspec panel refute` CLI verb in this spec (OQ3)** — the recording surface is a hand-edit of `panel.json`'s `refutations` array, exactly parallel to the abandon procedure; the gate (`internal/panel` + `internal/complete`) carries the whole validation burden. A dedicated `mindspec panel refute <slug> --slot --round --reason --evidence` verb in the spec-110 panel-verb family (which would validate at record time) is a deliberate follow-up, not part of this spec.
- **Not touching `MINDSPEC_SKIP_PANEL`** — the human-only env hatch (ADR-0037 §7 spawn-topology rationale) keeps its exact semantics, its HC-7 never-printed rule, and its `panel_gate_skipped` audit.
- **No tamper-resistance** — per ADR-0037 §8 the refutation record is deliberately a plain agent-writable artifact; signing/hashing it is explicitly out.

## Acceptance Criteria

- [ ] **AC1 (R1 — the out-vote is closed)**: a table-driven `internal/panel` decision test (e.g. `TestPanelGateDecision_UnresolvedRequestChangesBlocks`) proves: 6 expected reviewers, complete latest-round verdicts, fresh SHA, clean tree, 5 APPROVE + 1 REQUEST_CHANGES, no refutation → Block, message names the RC slot and ends with a `recovery:` line; and an end-to-end `internal/complete` gate test (e.g. `TestPanelGate_RequestChangesBlocksComplete`, alongside the existing `TestPanelGate_*` suite) proves `complete.Run` returns the guard failure having mutated nothing. Proof: `go test ./internal/panel -run 'PanelGateDecision' && go test ./internal/complete -run 'PanelGate'`.
- [ ] **AC2 (R2 — refutation unblocks + audit persists)**: with the same 5+1 panel plus a recorded refutation naming that slot and round, `PanelGateDecision` → Allow, `mindspec complete` proceeds, and on success bead metadata carries the `panel_refuted` audit entry (applied slot, round, reason, timestamp) — pinned by a `panel_refuted` audit test beside `TestWritePanelAuditMetadata_Abandoned` (`internal/complete/panel_advisory_test.go:109-133`) plus a gate-side Allow test.
