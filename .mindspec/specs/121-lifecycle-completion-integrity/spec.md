# Spec 121-lifecycle-completion-integrity: Lifecycle Completion & Finalization Integrity

## Goal

Close the residual correctness and convergence gaps that specs 119 (gate-before-mutate, ADR-0041) and 120 (trust-boundary/provenance, ADR-0042) surfaced in the lifecycle **completion path** ŌĆö `mindspec complete` and `mindspec impl approve` ŌĆö and automate the last **manual dance** those specs left in the protected-`main` finalize flow. Four clusters, every one a filed, reproduced bug or a designed automation of a reproduced operator burden: **(1)** the `chore/finalize-<specID>` carrier that `impl approve` pushes on protected `main` today ends in a NOTE telling the operator to hand-open and hand-merge a second PR (`cmd/mindspec/impl.go:186-198`; spec 120 needed PR #201 then PR #202) ŌĆö this spec auto-opens that PR (templated, degrade-to-NOTE when `gh` is unavailable) and, behind an explicit config opt-in, auto-merges it on green checks with a true merge commit so the shipped finalize-orphan doctor suppression (`internal/lifecycle/finalize_orphans.go:130`, SHA-ancestry) still sees it land (`mindspec-uxl4`); **(2)** the protected-main already-merged probe (`internal/executor/mindspec_executor.go:711-726`) and that same doctor suppression are both blind to SQUASH-merged branches ŌĆö a documented known blind spot (`mindspec_executor.go:700-704`) under which the wu7t silent-revert class recurs ŌĆö this spec makes "already landed on `origin/main`" a single, content-aware **net-effect** predicate consumed by both (`mindspec-3xqm` item 1); **(3)** two convergence holes in ADR-0041 ┬¦2's "re-invocation is always safe" guarantee are closed: the landed-merge identity predicate (`internal/lifecycle/landed.go:FindLandedMerge`) can positively identify a merge whose work was subsequently REVERTED, and accepts a subject-only match with zero corroboration when no panel is registered and the branch is deleted (`mindspec-q9ea`); and the orphaned-sibling preflight (`internal/complete/complete.go:491-499`) mutually-refuses two simultaneously-orphaned closed siblings ŌĆö a deadlock with no non-manual exit (`mindspec-tpjn`); **(4)** `gitutil.WorktreeRemoveForce` (`internal/gitutil/gitops.go:935`) composes a destructive `git worktree remove --force` path with no containment gate, unlike its `WorktreeAddDetach`/`WorktreeAdd` siblings (`gitops.go:895`, `:920`) ŌĆö the one wrapper spec 120's check-at-use sweep scoped out (`mindspec-17bd`). The convergence doctrine these fixes complete ŌĆö recovery commands form a deadlock-free, converging graph whose every refusal names a genuine forward exit; landed-work evidence is durably corroborated, never subject-only, and net-effect/revert-aware; already-merged detection is content-aware by current-state net effect; the machine-owned tracker-only finalize carrier may be machine-opened and, opt-in, machine-merged ŌĆö is recorded as an AMENDMENT to ADR-0041.

This spec is deliberately scoped against what 119/120 already shipped (verified in this worktree, see Background): lineage-authoritative spec resolution, never-`main` tracker commits, the merged-unclosed/branch-less reconcile, the doctor stale-OPEN and finalize-orphan checks, epic-scoped `FinalizeEpic` enumeration, exact-refname `RemoteHeadSHA`, and the corrected `PushBranchForceWithLease` doc are SHIPPED baseline, consumed as-is, and not re-specified here.

These four clusters are one doctrine seen from four angles: ADR-0041 ┬¦2 promises forward convergence, but the promise currently has a deadlock (tpjn), an evidence predicate weaker than the anti-bypass it carries (q9ea), an already-merged detector that a common GitHub workflow defeats (3xqm-1), and a terminal manual step that the machine has all the information to perform (uxl4) ŌĆö plus one destructive wrapper outside 120's containment discipline (17bd).

## Requirements

1. **Auto-open the finalize PR (`mindspec-uxl4`).** When `impl approve`'s finalize lands the epic-close export on a `chore/finalize-<specID>` carrier (`result.FinalizeBranch` non-empty, `internal/approve/impl.go:562`) and the `gh` CLI is available, the verb MUST open a PR from that branch into `main` with the templated title `chore(beads): finalize epic <epicID> for spec <specID>` (the existing commit-message shape, `mindspec_executor.go:937`; note today's NOTE-suggested title at `cmd/mindspec/impl.go:196` OMITS the epicID ŌĆö the automation's title includes it) and a templated body naming the spec, the epic, and the staleness consequence the NOTE states today (`impl.go:197`). Because the cmd-side automation home consumes `approve.ImplResult`, which today carries NO epic ID, the result surface MUST additionally expose the finalizing epic's ID (populated wherever `FinalizeBranch` is set) ŌĆö a bounded result-surface addition, not new behavior. The PR URL MUST be surfaced in the verb's output escaped as free text, and any PR title or other remote-provided string echoed back renders escaped likewise (ADR-0042). The operation MUST be idempotent across re-runs, with a pinned adoption rule: an already-open PR is adopted as success (its URL re-surfaced, never a duplicate or an error) ONLY when its base is `main` AND its head is exactly `chore/finalize-<specID>`; a same-head PR with any other base is NOT adopted ŌĆö the automation degrades (warning + NOTE) and such a PR is never auto-merged under Requirement 2. Every ID operand embedded in `gh` argv passes `idvalidate` at each construction site (ADR-0042 gate-all-ids); the `gh` invocations route through an injectable seam so the harness `gh` shim (`internal/harness/recorder.go:122`) can pin them. Governed by a new flat config key `auto_open_finalize_pr` (bool, default **true**), beside `auto_finalize` (`internal/config/config.go:21`), rendered by `mindspec config` (`cmd/mindspec/config.go:139` area).
2. **Opt-in auto-merge on green checks, with a merge commit (`mindspec-uxl4`).** Behind a second flat config key `auto_merge_finalize_pr` (bool, default **false** ŌĆö merging to `main` without a human is opt-in policy, never default), after opening or ADOPTING a PR under Requirement 1 the verb MUST wait for its checks (the `gh pr checks --watch` equivalent through the same seam, bounded per Requirement 3) and, on affirmative all-green evidence, merge it with a TRUE MERGE COMMIT (`gh pr merge --merge`) ŌĆö never squash or rebase, so the ancestry/net-effect consumers (the doctor merged-carrier suppression, `finalize_orphans.go:130`, and the Requirement 4 predicate) observe the carrier as landed. Three boundary semantics are pinned: **(a) merge only what R1 vetted** ŌĆö auto-merge operates ONLY on a PR Requirement 1 opened or adopted (head/base pinned there); with `auto_open_finalize_pr: false` no PR is opened or adopted, so `auto_merge_finalize_pr: true` is INERT ŌĆö no merge is attempted against any pre-existing operator PR, and a warning names the inert key combination. **(b) zero checks is not green** ŌĆö a checks query reporting NO checks configured/reported (the conventional `gh pr checks` failure on a checkless PR) is NOT green: the PR is left open with the NOTE; auto-merge fires only on positive green evidence, never on absence of evidence. **(c) post-merge ref freshness, for BOTH finding kinds** ŌĆö after a successful merge (including a merge observed via Requirement 3's reconcile query), the automation MUST refresh `origin/main` (a best-effort `git fetch origin main`, mirroring the probe's own fetch discipline inside `mindspec_executor.go:711-726`) so the suppression and predicate evaluate against real refreshed refs, not stale local ancestry; a fetch failure degrades to a warning (the doctor finding then clears on a later fetch ŌĆö detection is delayed, never lost). The fetch alone only cures the `finalize_branch` finding: the sibling `stale_tracker` classifier today reads LOCAL `main`'s committed export (`finalize_orphans.go:186`), which no fetch advances ŌĆö so post-auto-merge it would emit a factually-false "the finalize export never reached main" whose recovery line (`finalize_orphans.go:80`, `mindspec impl approve <spec>`) self-loops, violating this spec's own ┬¦2(i) doctrine. The classifier MUST therefore consult the refreshed `origin/main` committed export for this determination, aligning with the `finalize_branch` sibling's never-possibly-stale-local-`main` discipline (`finalize_orphans.go:99` area); a residual local-`main` lag behind an export that HAS reached `origin/main` may be surfaced only as a pull advisory whose recovery is a pull/update of local `main` ŌĆö never a re-finalize of an epic whose export already landed. On success the finalize-orphan finding for that spec MUST no longer be reported by `lifecycle.ScanIntegrityFindings`. With the key false, no merge is attempted and the PR is left open for the operator (and doctor keeps flagging it until merged ŌĆö the `mindspec-3xqm` item-2 surfacing interop preserved by construction).
3. **The PR automation is strictly post-finalize, best-effort, and evidence-honest about partial failure ŌĆö it never fails or un-finalizes the verb.** Both legs run only AFTER the finalize mutation chain has durably completed (branch committed and pushed, epic closed). A nonzero `gh` result is NOT proof that nothing landed ŌĆö GitHub may create or merge server-side and the client lose the response ŌĆö so after ANY failed leg (`pr create`, the existing-PR lookup, `pr checks`, `pr merge`, or a timeout on any of them) the automation MUST reconcile by querying the exact headŌåÆbase PR state through the same seam (itself bounded): an observed OPEN PR is create-success (its URL surfaced), an observed MERGED PR is merge-success (Requirement 2's post-merge path applies), and the stranded-carrier NOTE asserts "unmerged" ONLY when that query confirms no such open/merged PR exists ŌĆö when the query itself fails, the warning names the state UNDETERMINED and the NOTE falls back without asserting either way. Every `gh` leg carries a bounded timeout; a timeout follows the same degrade-and-reconcile path. Every failure ŌĆö `gh` absent from PATH, a failed leg, checks non-green, a watch timeout ŌĆö MUST degrade to today's behavior: the NOTE with the manual `gh pr create` command (`cmd/mindspec/impl.go:194-197`), a warning naming the failed leg and (when one exists) the open PR, and **exit 0** ŌĆö the finalize itself succeeded, the stranded-carrier state remains fully detected by the shipped doctor check, and tracker/git state is byte-identical to a no-`gh` run. `gh` stderr/error text embedded in any warning, and remote check names surfaced in any output, render `termsafe`-escaped (ADR-0042). These legs are classified DOCUMENTED-FORWARD-SAFE under ADR-0041 ┬¦3 (their errors are absorbed by design; the audit trail is the doctor finding), and the fault-injection suite MUST record that classification rather than fabricate kill tests against them.
4. **Content-aware already-merged detection ŌĆö net-effect, one predicate, both consumers (`mindspec-3xqm` item 1).** The "this branch's work already landed on `origin/main`" decision MUST become a single exported predicate (home: `internal/lifecycle` or `internal/gitutil` ŌĆö plan-level) whose meaning is NET-EFFECT equivalence against `origin/main`'s CURRENT state, not historical patch presence: when SHA ancestry fails, the content-level fallback (mechanism plan-level ŌĆö e.g. a cherry/patch-id or committed-tree comparison evaluated against the current tree) reports landed iff the branch introduces NO content absent from `origin/main` as it stands NOW. Whether ancestry ALONE suffices is decided per consumer, because the two consumers answer different questions: for the PROBE, SHA ancestry remains sufficient as-is (the probe decides carrier ROUTING ŌĆö an ancestor spec branch is a spent PR carrier regardless of later history, since nothing pushed to it can reach `main` again; a later revert of its work on `main` is a `main`-side state the doctor/stale-tracker surface detects, not a routing input); for the DOCTOR suppression, ancestry alone is NOT sufficient ŌĆö the suppression decision is the net-effect/current-content test even when ancestry holds, so a carrier that was truly (non-squash) merged and then REVERTED on `origin/main` is flagged again (its tracker content is once more absent from `main` ŌĆö suppressing it would hide a genuinely-stranded export behind a historical merge). Pinned consequences: a squash-merge whose content was later REVERTED on `origin/main` is NOT landed; a branch only PART of whose content is present (partially-landed-plus-novel) is NOT landed; a squash-merge followed by unrelated later `main` changes IS landed. BOTH existing SHA-only sites consume the one exported symbol, with the observable contract stated PER CARRIER: the protected-main probe in `FinalizeEpic` (`mindspec_executor.go:711-726`) evaluates a SPEC branch (code + tracker; net effect over the branch's full diff against its merge-base with `origin/main`) ŌĆö a squash-merged spec branch routes to the `chore/finalize-` carrier instead of pushing the epic-close commit onto the dead carrier, retiring the known-blind-spot comment at `:700-704`; the doctor merged-carrier suppression (`finalize_orphans.go:126-130`) evaluates a `chore/finalize` carrier (tracker-JSONL-only; net effect over its single regenerated `.beads/issues.jsonl` payload) ŌĆö a carrier whose epic-close content `main`'s committed export already carries, including via a LATER superseding export, is suppressed, while one whose JSONL still carries novel content keeps being flagged. A genuinely-unmerged branch carrying novel work MUST NOT be misdetected as landed (the normal push path is unchanged). Each consumer's infra-error fail-posture is pinned EXPLICITLY: the probe, on a content-fallback infra failure, WARNS naming the undetermined probe and proceeds on the ancestry answer alone ŌĆö a fail-open (`orphaned=false` can push the epic-close onto a possibly-dead carrier) retained DELIBERATELY because that stranded outcome remains fully detected by the shipped doctor finalize-orphan finding, the same absorb that already covers the probe's fetch-failure leg; the suppression, on an infra failure, SKIPS the branch asserting nothing, surfacing the first error through the existing mixed-list contract (`finalize_orphans.go:130-135`).
5. **Landed-merge identity: durably corroborated, never subject-only (`mindspec-q9ea`).** `lifecycle.FindLandedMerge` MUST NOT positively identify a candidate on subject text alone, and no heuristic over the candidate's CONTENTS substitutes for identity ŌĆö that the second parent carries non-empty work is explicitly NOT corroboration (it proves somebody merged something, not that THIS bead's work landed). (a) **Durable corroboration only**: a candidate is positively identified only when at least one independent identity datum CONFIRMS it (equals or is an ancestor of the candidate's second parent) and no available datum contradicts it. The admissible data are: a registered panel's `reviewed_head_sha` (existing, `landed.go:136-146`); a surviving bead-branch tip (existing, `:147-153`); and the lifecycle-recorded landed-binding of (b). (b) **The binding is recorded at merge time, FAIL-CLOSED before cleanup** (the small in-scope addition that makes (a) livable): both in-binary beadŌåÆspec merge producers ŌĆö `CompleteBead`'s merge leg (`mindspec_executor.go:410`) and `FinalizeEpic`'s auto-merge leg (`:622`) ŌĆö MUST, after `gitutil.MergeInto` succeeds and BEFORE any branch/worktree cleanup for that bead, durably record the beadŌåÆmerge binding (the merge commit SHA and its second-parent SHA) in bd metadata through the existing `bead.MergeMetadata` surface (`internal/bead/bdcli.go:322`; Dolt is the state authority, ADR-0023; in-tree precedent: the reconcile-evidence write at `complete.go:1043`). The write's FAILURE posture is part of the contract, not an implementation choice: a failed binding write MUST suppress that bead's branch/worktree cleanup and refuse recoverably with an ADR-0035 recovery line naming the verb's own re-run ŌĆö mirroring the refuse-recoverably shape of the cited reconcile-evidence precedent (`complete.go:1032-1055`), NOT the warn-and-continue style of the adjacent cleanup legs (`mindspec_executor.go:437-450`) ŌĆö so the branch survives as the corroborating datum and re-invocation converges (a re-attempted `MergeInto` of an already-ancestor branch is a no-op). On that re-invocation the binding write MUST locate the already-landed merge BY IDENTITY (the subject-scan candidate corroborated by the surviving branch tip) and record THAT merge's SHAs ŌĆö never assume the current tip or `HEAD` is the merge. The invariant that closes the evidence gap for every lifecycle-produced merge holds BECAUSE write-failure suppresses cleanup, not merely because of ordering: at any interruption OR failure point at least one corroborating datum exists ŌĆö before the record lands, the bead branch still survives (cleanup has not run, and a failed write keeps it that way); after it lands, the binding itself corroborates. A no-datum state is therefore reachable only for out-of-band merges and pre-existing (pre-121) history. (c) **Fail-closed with a genuine forward exit**: a candidate with NO confirming datum is `ErrLandedMergeNotFound` ŌĆö the reconcile refuses via 119's existing no-evidence path (119 AC-8) and NEVER closes on subject text. That refusal's recovery MUST be a forward exit that is NOT bare re-invocation (re-running cannot change the refusing fact): it names the candidate merge and second-parent SHAs the subject scan found and the exact attested-restore command ŌĆö restore the bead branch ref at the candidate's second parent (`git branch bead/<id> <second-parent SHA>`) ŌĆö whose execution is an explicit operator ATTESTATION that the merge carried this bead's work, converting the state into the surviving-branch corroboration leg; re-running then converges. Because orchestrators follow recovery lines mechanically, this refusal line MUST carry an explicit human-verification marker (to the effect of "running this command attests that merge <SHA> carried THIS bead's work ŌĆö verify before running; do not execute blindly") distinguishing it from the mechanically-safe recovery lines every other refusal in this spec names. This keeps ADR-0041 ┬¦2 deadlock-free (Requirement 8 pins it in the amendment): the machine never self-loops, and never closes without evidence. (d) **Revert- and reapply-aware, defined as net effect SINCE the merge**: a candidate M whose merged work was subsequently reverted on the spec branch's first-parent chain and NOT re-applied MUST NOT be identified as landed (the refusal names the revert evidence). The contract is pinned against the mechanism shape that would otherwise deadlock it: a reapply (a revert-of-the-revert, or a cherry-pick of the work) creates NO new `Merge bead/<id>` merge commit ŌĆö the newest-first subject scan still finds only M ŌĆö so revert-awareness MUST be evaluated as the NET EFFECT of the first-parent chain SINCE M (a revert later canceled by a reapply does not disqualify M; the same doctrine as Requirement 4, detection mechanism plan-level). A reverted-then-reapplied M therefore IS identified and the reconcile closes; "reject any M that was ever reverted" is a non-conforming implementation (it would manufacture a permanent refusal for an honest landed state, violating the ┬¦2 deadlock-free rule of (c)). (e) **Regression floor, per-datum**: 119's AC-5/AC-7/AC-9 fixtures ŌĆö genuine landed merges, panel-free and panel-registered, branch deleted ŌĆö MUST still reconcile and close (assertions unchanged), plus new honest multi-commit and conflict-resolution merge fixtures that likewise reconcile and close. Each admissible datum of (a) MUST be pinned as INDEPENDENTLY sufficient: the panel-registered variant stays binding-ABSENT (its real-world counterpart is pre-121 history that has no binding ŌĆö a binding-REQUIRED implementation must fail it), a surviving-branch-only binding-absent variant closes, and a binding-only variant closes (AC-12).
6. **Orphan-preflight convergence ŌĆö no mutual refusal (`mindspec-tpjn`).** `complete`'s step-1.6 preflight already excludes the invoked bead from the orphan enumeration (`findOrphanedClosedBeadsFn(specID, root, beadID)` passes `excludeBeadID = beadID`, `complete.go:491`) ŌĆö the deadlock is two beads each refusing on the OTHER. When that self-orphan determination hits an infra/ancestry error, the invoked bead is treated as NOT self-orphaned and the refusal is RETAINED ŌĆö a retryable preflight refusal with nothing mutated (ADR-0041 ┬¦1), never a demotion on unproven evidence. (b) when the invoked bead is NOT itself orphaned, the refusal is retained but MUST name ALL orphaned siblings (not just `orphans[0]`) with a single recovery sequence (`mindspec complete <A>`, `mindspec complete <B>`, then re-run) ŌĆö so the full non-manual exit is knowable from one refusal. (c) the WARN-demotion path deliberately admits an INTERMEDIATE state ŌĆö completing A lands A while WARN'd siblings linger orphaned ŌĆö and that is safe by construction: each step strictly shrinks the orphan set (the sequence is monotone and finite), and the shipped fail-closed impl-approve orphan gate (`ScanOrphanedClosedBeads`, `orphans.go:142`, 119 Bead 2) backstops it by refusing to finalize the epic while ANY orphan remains. IDs in the new warning/refusal text render via `idrender`/`termsafe` per the shipped conventions at this site.

In-tree, plainly-named removals are byte-identical in behavior. The caller-level gate spec 120's final review added before the finalize self-heal remains (defense-in-depth); this closes every OTHER present and future caller at the wrapper, per ADR-0042's check-at-use discipline.

## Non-Goals

- No hard dependency on `gh`: every automation leg degrades to the shipped NOTE + doctor surfacing; a `gh`-less environment loses convenience, never correctness or detection.
- No auto-merge by default, and no new merge authority: `auto_merge_finalize_pr` defaults false; enabling it is an explicit, config-recorded operator policy for a tracker-only carrier the automation itself opened or adopted ŌĆö the panel-gated review path for code is untouched, and no pre-existing operator PR is ever machine-merged.
- No rollback machinery, per ADR-0041: convergence is completed forward (the tpjn WARN-demotion, the q9ea refuse-not-close with the attested-restore exit), never by undoing landed state.
- No content-forensics identity attribution: landed-merge identity comes from recorded identity data (panel SHA, the merge-time binding, a surviving ref), never from heuristics over what the second parent's commits "look like" ŌĆö a spoof-resistance floor, not an authorship analyzer.
- No general squash-workflow support beyond the two Requirement-4 consumers: `mindspec` still merges beads and specs with merge commits; net-effect awareness is a detection floor, not a workflow change.
- No relaxation of the orphan gate for NORMAL beads: a non-orphaned bead still refuses to complete past orphaned siblings; only the enumeration (all, not first) and the invoked-self-orphan case (WARN) change.
- No new config surface beyond the two flat finalize-PR keys (the attested-restore exit is a named refusal recovery line, not a flag or key).
