# Spec 113-panel-verb-workflow-followups: Panel-verb & workflow followup hardening (110/111/112 fallout wave)

## Goal

Harden the panel-verb and workflow surface that specs 110/111/112 just shipped, with four small, bounded, independently-landable fixes — no new architecture, no gate-decision change for `mindspec complete`:

1. **R1** — `mindspec panel verify`/`panel tally` tell the truth about a **non-bead** panel's staleness instead of unconditionally reporting PASS-advisory with a malformed message (bead `mindspec-4d9m`, P2 — the load-bearing one).
2. **R2** — `ms-panel.js`'s `SHELL_METACHAR_RE` rejects a bare `$`, closing the `$HOME`-style variable-expansion survivor (bead `mindspec-lczt`, P3).
3. **R3** — `mindspec panel create` gains `--gate <name>`, the writer-side stamping of the decision-inert `gate` field and per-gate creation-time defaults that spec 112 explicitly deferred to the writer (bead `mindspec-zw81`, P3).
4. **R4** — the `model: ""`-with-`family` validation ambiguity spec 112 left between its R1 and R4 texts is reconciled and pinned by test (bead `mindspec-cthj`, P3).

Target user outcome: an operator (or orchestrating agent) who runs `panel verify`/`panel tally` on a spec-approve or final-review panel gets a report that actually reflects the reviewed ref's staleness; workflow input hardening has no `$`-expansion gap; CLI-created panels carry the gate identity 112's advisories key on; and config validation behaves exactly as documented, with a test pinning it.

## Background

The 110/111/112 batch built the panel-verb CLI (`panel create|verify|tally`, spec 110), the workflow panel runner (`ms-panel.js`, spec 111), and per-gate panel config plus the recorded decision-inert `gate` field (spec 112). Four followups fell out of those panels and post-merge probing:

- **R1 (`mindspec-4d9m`)**: `cmd/mindspec/panel.go`'s `resolvePanelGateFacts` (line ~302) resolves `beadID` only when `reg.Panel.IsBead()`; for a non-bead panel (`bead_id` null — spec-approve / final-review / PR panels) `beadID` stays `""` and `panel.ResolveGateFacts` (internal/panel/gate.go:372) rev-parses the literal ref `bead/`. `git rev-parse --verify --quiet "bead/^{commit}"` exits 1 (verified live in this repo), which `gitutil.RevParseRef` (internal/gitutil/gitops.go:542) maps to `ErrRefNotFound`, so `facts.MissingRef` is always true and `PanelGateDecision` short-circuits at leg (5) with the Warn `"panel for  references branch bead/, which no longer exists — assuming the merge already landed …"` (gate.go:186-189, note the empty interpolations). Consequences, all observable today: (a) `panel verify` prints `PASS (advisory: …)` and `panel tally` exits 0 for a non-bead panel **even after its `--target` ref advances past `reviewed_head_sha`**; (b) because leg (5) returns before legs (8)-(10), a non-bead panel with zero verdicts, or with a REJECT/hard_block on file, **still** reports PASS-advisory — the CLI verbs can never Block on a non-bead panel; (c) the message renders the malformed `references branch bead/,` fragment. `mindspec complete` is NOT affected: its gate matches panels via `panel.ForBead` (internal/panel/panel.go:232-243), which excludes non-bead panels by construction (`internal/complete/panel_advisory.go:140-153`), and spec 110 fbel.3 deliberately made `internal/instruct`'s snapshot leave non-bead staleness facts zero/inert (internal/instruct/panelstate.go:108-119) — that package evaluates the vote legs but never claims a merge landed. The CLI verbs are the only lying surface.
- **R2 (`mindspec-lczt`)**: `plugins/mindspec/workflows/ms-panel.js:138` defines `SHELL_METACHAR_RE = /[\x60;|&\n]|\$\(/` — it matches `` ` ``, `;`, `|`, `&`, newline, and `$(`, but **not** a bare `$`. `buildCommand` assembles a shell-executed template string (`` `${verb} ${slug} --spec ${specId} --target ${target}` ``, line 78), so a slug/spec/target/bead_id value containing `$HOME`, `${x}`, or `$x` passes `validateShellSafe` and would undergo variable expansion when the command string is shell-executed. (Defense in depth: `validatePathElement`'s whitespace/path-separator rejections narrow the blast radius, but `$HOME` contains none of those.) The mirror copy `.claude/workflows/ms-panel.js` is byte-identical today (verified with `cmp`) and must stay so.
- **R3 (`mindspec-zw81`)**: spec 112 R6 added the decision-inert `Panel.Gate` field (`json:"gate,omitempty"`, internal/panel/panel.go:91) and R7's `PanelGateAdvisoryDefault(recordedGate, isBead)` selection rule (internal/config/config.go:832), consumed at `internal/complete/complete.go:390` and `cmd/mindspec/config.go:541`. But 112's Out of Scope explicitly deferred "the `panel.json` **writer** that stamps these creation-time defaults (including the `gate` field)" to the spec-110 writer — i.e. `panel create`. Today `panel.CreateInput` (internal/panel/create.go:51-74) has no Gate field, `panel create` has no `--gate` flag (cmd/mindspec/panel.go:210-218), and it stamps only the **global** defaults (`cfg.PanelExpectedReviewers()` / `cfg.PanelApproveThresholdExpr()`, panel.go:130-131), never 112 R3's gate-scoped resolvers. A CLI-created non-bead panel therefore has empty `gate` + `isBead == false`, so `PanelGateAdvisoryDefault` returns `(0, false)` and the per-gate reviewer-count advisory is skipped entirely.
- **R4 (`mindspec-cthj`)**: spec 112 R4's text lists "a reviewer entry with neither `model` nor `family`, **or an empty-string `model`**" among the `Load` refusals, but 112 R1 says "an entry with `family` and no `model` is valid and resolves its model to the family string", and 112's AC only tests the neither-set case. The implementation (internal/config/config.go:114-144, 540-550) uses a plain `string` for `Model`, so `model: ""` is indistinguishable from an absent `model`: today `{model: "", family: codex}` is **accepted** and resolves to the family (`Reviewer.model()`, config.go:139-144), while `{model: ""}` with no family is refused via the neither-set branch. The spec text and the code disagree for exactly the empty-string-model-with-family case; this wave reconciles them (OQ2).

Explicitly deferred out of this wave: the lifecycle-finalize bugs `mindspec-blp6` and `mindspec-zty3` — they change `impl-approve`/`complete` behavior and belong to a separate spec.

## Impacted Domains

- **workflow**: `cmd/mindspec/panel.go` (R1 fact-gathering + R3 `--gate` flag), `internal/panel` (R1 non-bead message rendering + R3 `CreateInput.Gate`; owned by workflow per `.mindspec/domains/workflow/OWNERSHIP.yaml` — `internal/panel/**`, `cmd/**`), `plugins/mindspec/workflows/ms-panel.js` + `.claude/workflows/ms-panel.js` (R2). Constraint carried forward, not weakened: `internal/panel` stays a dependency-clean, config-free leaf — any R1/R3 change reaches it only as plain values / GateIO seams, never an `internal/config` or git import.
- **core**: `internal/config` (R4 — `validateReviewerEntries`/`Reviewer.model()` reconciliation and its pinning test; owned by core per `.mindspec/domains/core/OWNERSHIP.yaml` — `internal/config/**`). R3 *consumes* `config.PanelGateKeys` and the 112 R3 gate-scoped resolvers read-only from `cmd/mindspec`; no core code change is required for R3.

## ADR Touchpoints

- [ADR-0037](../../adr/ADR-0037-panel-gate-enforced-contract.md): Panel Gate as Enforced Contract. R1 must not move the decision authority: `PanelGateDecision` stays the single decision home ("identical decision over identical facts"), and the fix changes only the **facts** the CLI verbs gather for a non-bead panel (plus message rendering) — never the Allow/Block matrix for bead panels or for `mindspec complete`. R3 stamps the `gate` field 112's second amendment note already records as decision-inert metadata; its decision-inertness (§§3/6/8) is preserved and re-pinned here.
- [ADR-0035](../../adr/ADR-0035-agent-error-contract.md): Agent Error Contract. R1's corrected non-bead messages and Block recovery text, and R3's invalid-`--gate` rejection, carry genuine recovery lines (no malformed `bead/<empty>` fragments, no `mindspec complete <bead>` advice for a panel that is not complete-gated); R4's surviving neither-set refusal keeps the same contract.
- [ADR-0040](../../adr/ADR-0040-orchestration-layering-ratchet.md): Orchestration Layering Ratchet. The panel verbs are the ADR-0040 portability contract's CLI half (cmd/mindspec/panel.go header); R1 keeps that surface truthful for the non-bead panels the per-gate config (112) makes first-class. R2 hardens the L3 workflow runner without changing its L2 contract. R3 is the ratchet's writer-side completion: the per-gate defaults move from operator memory into the recorded `panel.json`.
- [ADR-0034](../../adr/ADR-0034-ceremony-collapse.md): Ceremony Collapse. R3's `--gate` values are 112 R1's closed five-key enum (`spec_approve`, `plan_approve`, `bead`, `final_review`, `adhoc`) — review-event vocabulary, deliberately distinct from `loop.gate_authority`'s approval-act vocabulary; the rejection message preserves that disambiguation.
- [ADR-0030](../../adr/ADR-0030-executor-boundary.md): Executor Boundary. R1's added non-bead staleness rev-parse routes through the executor seam `cmd/mindspec/panel.go` already uses (`newExecutor(root).RevParseRef`), staying inside the existing per-verb subprocess budget (one rev-parse for staleness; the worktree/dirty legs stay bead-only).

No dedicated config-validation ADR exists in `.mindspec/adr/`; R4's authority is spec 112's R1/R4 requirement texts (quoted in Background) plus ADR-0035 for any refusal's recovery line — cited here in prose rather than as a link.

## Requirements

1. **Non-bead panel staleness in `panel verify`/`panel tally` (bead `mindspec-4d9m`).** For a registered **non-bead** panel (`bead_id` null), `mindspec panel verify <slug>` and `mindspec panel tally <slug>` must stop deriving staleness from the empty-bead ref `bead/` and instead resolve it from the panel's own recorded target: rev-parse `panel.json.target` in the panel's scan root (through the existing executor seam) and compare against `reviewed_head_sha`, feeding the result through the SAME `panel.PanelGateDecision` legs a bead panel uses — stale target ⇒ Block, genuinely-deleted target ref ⇒ the missing-ref Warn, transient git error ⇒ the honest transient Warn, and (because the false missing-ref short-circuit is gone) the incomplete/REJECT/hard_block/threshold legs (8)-(10) become reachable for non-bead panels. *(Resolved at the grill: OQ1 → resolve-from-target.)* Message hygiene: no rendered output for a non-bead panel — verify report, tally body, Warn advisory, Block message, `RawMergeFence`, or the tally recovery line — may interpolate an empty bead ID into a `bead/<id>` slot (today's `"panel for  references branch bead/, which no longer exists"`, gate.go:186-189) or advise `mindspec complete <bead>` for a panel that is not complete-gated; non-bead messages must name the recorded target ref instead. Consistency fence: `mindspec complete`'s gate semantics are untouched — `panel.ForBead` keeps excluding non-bead panels (they remain non-complete-gated), `PanelGateDecision`'s outcome for every bead-panel fact set is byte-identical, and `internal/instruct`'s fbel.3 snapshot behavior (non-bead staleness facts left zero/inert, one git call per **bead** panel only) does not change in this wave — this fix changes what the CLI verbs *gather and render*, not what any gate *decides*. **Falsified if**: after `mindspec panel create <slug> --spec <id> --target <ref>` (no `--bead`) and then advancing `<ref>` by one commit, `panel verify <slug>` reports `PASS` (with or without advisory) or `panel tally <slug>` exits 0; if a non-bead panel with a REJECT verdict or with fewer verdicts than `expected_reviewers` (at an un-advanced target) still yields a PASS report; if any non-bead verify/tally rendering emits `references branch bead/,`, `git merge bead/ ` (empty interpolation), or a `mindspec complete <bead>` recovery instruction; if any bead-panel decision or message changes; or if `internal/complete`'s or `internal/instruct`'s non-bead handling changes.
2. **`SHELL_METACHAR_RE` rejects a bare `$` (bead `mindspec-lczt`).** Extend `SHELL_METACHAR_RE` in `plugins/mindspec/workflows/ms-panel.js` (line ~138) so `validateShellSafe` rejects any value containing `$` — not merely the `$(` digraph — closing `$HOME` / `${x}` / `$x` variable-expansion survival in the shell-executed command string `buildCommand` assembles (line ~78). No legitimate slug, spec ID, target ref, or bead ID in this repo's conventions contains `$`; the tightening is strictly monotone (everything previously rejected stays rejected). The mirror `.claude/workflows/ms-panel.js` is updated in the same commit and stays byte-identical (`cmp` clean), and the existing floor tests — `TestWorkflowFiles_EmbedsMsPanel` (embed == plugin source) and `TestMsPanelWorkflow_AllowedCLIExactSet` (spec 111's exact-set/chokepoint pins) in `plugins/mindspec/workflow_test.go` — stay green. **Falsified if**: `validateShellSafe` accepts any of `$HOME`, `${x}`, `$x` (or any value containing `$`); if any input the old regex rejected now passes; if the two `ms-panel.js` copies differ byte-for-byte after the change; or if either named floor test regresses.
3. **`panel create --gate <name>` — the deferred writer-side stamping (bead `mindspec-zw81`).** `mindspec panel create` gains an optional `--gate <name>` flag validated against the closed enum `config.PanelGateKeys` (`spec_approve`, `plan_approve`, `bead`, `final_review`, `adhoc` — the single declaration, internal/config/config.go:101; never a second copy). When given: (a) `panel.CreateInput` gains a `Gate` field and `panel.Create` stamps `panel.json`'s `"gate"` with the value (the field's spec-112-R9 stable contract — name `gate`, type string, `omitempty`, parse-lenient — is unchanged; an omitted flag writes no `gate` key); and (b) the creation-time defaults are resolved through spec 112 R3's gate-scoped resolvers — `ExpectedReviewers` from `cfg.PanelGateExpectedReviewers(gate)` and `ApproveThresholdExpr` from `cfg.PanelGateApproveThresholdExpr(gate)` — instead of the global `PanelExpectedReviewers()`/`PanelApproveThresholdExpr()`, exactly the writer-side stamping 112's Out of Scope deferred ("the `panel.json` writer that stamps these creation-time defaults (including the `gate` field)"). When `--gate` is omitted, behavior is byte-identical to today: global defaults, no `gate` key. A value outside the enum is rejected before any write with a `guard.NewFailure` whose recovery line enumerates all five valid keys (ADR-0035; same control-byte discipline as `--bead`/`--target`, `rejectControlBytes`). The field stays decision-inert end to end: `PanelGateDecision` and `ApproveThreshold()` never read it (112 R6); the recorded `expected_reviewers`/`approve_threshold` remain the sole decision inputs, exactly as for every panel today. The advisory read-through works: a stamped `gate` is what `PanelGateAdvisoryDefault` (config.go:832) keys on at its two call sites (`internal/complete/complete.go:390`, `cmd/mindspec/config.go:541`), so a CLI-created `final_review` panel selects the `final_review` per-gate default instead of being advisory-skipped. **Falsified if**: `panel create <slug> --spec <id> --target <ref> --gate final_review` does not yield `"gate": "final_review"` in `panel.json`; if a `--gate` value outside the five-key enum is accepted (exit 0) or its rejection lacks a recovery line naming all five keys; if `--gate <g>` on a config with a per-gate mix does not stamp that gate's resolved `expected_reviewers`/raw `approve_threshold`; if omitting `--gate` changes any byte of the `panel.json` produced today; if the field's presence or value changes any `PanelGateDecision` Allow/Block or any `ApproveThreshold()` result; or if the R9 stable contract (name/type/`omitempty`/parse-lenience) is altered.
4. **Reconcile the empty-string-model validation to resolve-to-family (bead `mindspec-cthj`).** Resolve the spec-112 R1-vs-R4 disagreement for `{model: "", family: <f>}` (Background) as **resolve-to-family** *(resolved at the grill: OQ2)*: `model: ""` with a `family` present stays valid and resolves to the family string, because 112 R1 explicitly blesses family-fallback, the typed `string` field cannot distinguish absent from empty without a `*string`/custom `UnmarshalYAML` (a YAML-shape risk with zero operator benefit), and 112's own AC never tested the empty-string case. No struct or unmarshal change. The reconciliation is documented where the ambiguity lives — a clarifying comment on `Reviewer.model()`/`validateReviewerEntries` (internal/config/config.go:139-144, 540-550) recording that this supersedes 112 R4's "or an empty-string `model`" phrase — and pinned by a new `internal/config` test asserting `{model: "", family: "codex"}` loads and slot-expands to model `codex`. Unchanged: `{model: ""}` with **no** family stays refused (today's neither-set branch), and the refusal keeps its ADR-0035 recovery line. **Falsified if**: `{model: "", family: "codex"}` fails `Load`, or slot-expands to any model other than `codex`, or no test pins that behavior, or the superseding comment is absent; or if `{model: ""}` with no family loads without error.

## Scope

### In Scope
- `cmd/mindspec/panel.go` — R1 (non-bead fact gathering + rendering + tally recovery text), R3 (`--gate` flag, enum validation, per-gate default resolution).
- `internal/panel` (`gate.go`, `create.go`, `panel.go`) — R1 non-bead message rendering (mechanism is plan territory: e.g. a message-only fact field or non-bead message variants — the leaf invariant and decision matrix hold either way), R3 `CreateInput.Gate` stamping.
- `plugins/mindspec/workflows/ms-panel.js` + `.claude/workflows/ms-panel.js` — R2 regex tightening, mirrors byte-identical.
- `internal/config` — R4 comment + pinning test (no struct or unmarshal change).
- New/extended tests in `cmd/mindspec`, `internal/panel`, `plugins/mindspec`, `internal/config` pinning each fix.

### Out of Scope
- `mindspec complete` / `PanelGateDecision` semantics for bead panels, `panel.ForBead`'s non-bead exclusion, and `internal/instruct`'s fbel.3 snapshot behavior (all explicitly fenced by R1).
- `ms-panel.js` `--gate` pass-through: the workflow runner's bead panels already resolve advisories via `PanelGateAdvisoryDefault`'s `isBead` fallback, and its `ALLOWED_CLI`/`buildCommand` surface is a spec-111 pinned floor — extending the workflow's arg schema is a deliberate separate change if ever needed.
- Any change to the 112 `--gate --json` machine contract, the gate-scoped resolvers, or the inheritance chain.

## Non-Goals

- The lifecycle-finalize bugs `mindspec-blp6` and `mindspec-zty3` are **explicitly excluded** — they change `impl-approve`/`complete` behavior and are deferred to a separate spec.
- No broader panel or config redesign: no new gates, no new config keys, no dispatch/spawning, no change to panel decision authority, thresholds, or the trust boundary (ADR-0037 §8 — these remain anti-footgun devices, not anti-adversary ones).
- No re-opening of approved specs 110/111/112; where this spec supersedes a single 112 phrase (R4), it says so explicitly rather than editing 112.
