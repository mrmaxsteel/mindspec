# Spec 115-impl-approve-panel-gate: impl-approve panel gate ‚Äî FinalizeEpic re-gates raw-bd-close'd beads before auto-merge

## Goal

Close the silent-out-vote hole spec 114 closed for `mindspec complete` on the one remaining verb that can still merge a bead branch un-gated: `mindspec impl approve`. Today `ApproveImpl` ‚Üí `FinalizeEpic` auto-merges any unmerged `bead/<id>` branch into the spec branch (`internal/executor/mindspec_executor.go:381-422`) ‚Äî including a bead that was closed via a raw `bd close`, skipping `mindspec complete`'s authoritative panel gate entirely. So an operator or agent can `bd close` a bead carrying an unresolved REQUEST_CHANGES and then run `mindspec impl approve`, and the RC-bearing branch merges to the spec branch with no gate, no reconciliation, and no audit ‚Äî exactly the out-vote 114 made impossible at `complete` (114's own final-review scope note, `.mindspec/specs/114-panel-gate-block-on-request-changes/spec.md:40-42`, filed this as `mindspec-o4fd`).

Target user outcome: **no lifecycle verb in the binary merges a bead branch that lacks proof of panel settlement.** `mindspec impl approve` REFUSES (exit non-zero, mutating nothing) when any closed bead under the spec's epic was closed without `mindspec complete` ‚Äî naming the bead and its unmerged branch, and (best-effort) the unresolved RC slot(s) ‚Äî and the recourse converges through the ONE existing gate home: `mindspec complete <bead>` (which tolerates an already-closed bead, re-runs the full layered gate, and blocks until the RC is resolved or durably refuted). A spec whose beads all went through `mindspec complete` finalizes exactly as today.

## Background

**The hole, code-traced.** `FinalizeEpic` (`internal/executor/mindspec_executor.go:349`) contains the "Auto-merge unmerged bead branches into spec branch before cleanup" loop (`:381-422`): it lists worktrees (`:383`), filters on the `bead/` branch prefix (`:385`), auto-commits leftovers (`:390`), skips branches already an ancestor of the spec branch (`:394`), and merges the rest via `gitutil.MergeInto` (`:402`). Its explicit purpose comment (`:382`) is *"This handles beads that were closed via `bd close` without `mindspec complete`."* The ONLY guard in front of that merge is `guardMergeLayout` (`:399`, defined `internal/executor/layout_guard.go:74`) ‚Äî a DIRECTIONAL layout-regression guard, nothing else. The comment at `:396-398` ("same regression-only rule as CompleteBead; mutates nothing on a block") refers to that layout guard only; o4fd's "verify whether any gate actually fires there" verifies as: **no panel gate, no obligation reconciliation, and no orphan check fire anywhere on this path** ‚Äî not in the loop, and not in its sole production caller `ApproveImpl` (`internal/approve/impl.go:132-428`), whose bead-status gate (`:220-242`) checks only that every plan bead's status is `"closed"` ‚Äî a status a raw `bd close` satisfies.

**What `complete` does that this path skips.** `complete.Run` runs, before any mutation: the sibling bd_close lifecycle-bypass guard (step 1.6, `internal/complete/complete.go:307-321`), the AUTHORITATIVE in-binary panel gate (step 2.25, `:345-405`, calling `panelGate` at `:387` ‚Üí `internal/complete/panel_advisory.go:162-262` ‚Üí `panel.PanelGateDecision`, `internal/panel/gate.go:143`; ADR-0037, amended by 114 so any unresolved REQUEST_CHANGES Blocks), the layered per-bead gates (doc-sync/ADR, steps 3.5), and the durable-obligation reconciliation (step 3.75, `:558-572` ‚Üí `reconcilePendingRefutations`, `panel_advisory.go:519-584`) ‚Äî and only then closes (tolerating an already-closed bead: step 4, `:574-582`) and merges. Raw `bd close` + `impl approve` reaches the same merge having run none of it.

**The detection signal already exists.** `lifecycle.FindOrphanedClosedBeads` (`internal/lifecycle/orphans.go:74-115`, bead mindspec-4gsz) is the ONE shared predicate for "closed via raw `bd close`": a closed bead whose `bead/<id>` branch EXISTS (cheap trigger) and is NOT an ancestor of the spec branch (confirmation ‚Äî so a benign merged-but-undeleted branch is never flagged). Three verbs already consume it: `mindspec complete` (step 1.6), `mindspec next` (`cmd/mindspec/next.go:459`), and `mindspec doctor` (`internal/doctor/orphaned_beads.go:48-79`, which reports each orphan with the `mindspec complete <id>` recovery and a `--fix` that re-invokes it). `mindspec impl approve` is the one lifecycle verb that does NOT consume it ‚Äî and it is the verb that MERGES. Its canonical recovery is already defined: `Orphan.RecoveryCommand()` (`orphans.go:54`).

**Chosen enforcement model ‚Äî REFUSE, not RUN (OQ1, RESOLVED at grill).** This spec makes `impl approve` REFUSE and route to `mindspec complete <bead>`, rather than running the panel gate + reconciliation itself inside `ApproveImpl`/`FinalizeEpic`. Rationale: (a) **one gate home** ‚Äî 114's three bead-panel review rounds each found a distinct false-discharge hole in a *second* settlement surface; duplicating the gate/reconcile ordering (gate before CommitAll, reconcile before close, audit writes after the terminal mutation) into a second verb re-opens that drift surface; (b) **ADR-0030 layering** ‚Äî `internal/executor` is the git-I/O boundary and must not house enforcement, and it cannot import `internal/complete` (import cycle: complete ‚Üí executor); (c) **the recourse already converges** ‚Äî `complete.Run` tolerates an already-closed bead (step 4) and re-runs the FULL layered gate (RC-block, stale-SHA, incomplete, dirty-tree, round-mismatch, doc-sync, ADR) plus the durable-refutation reconciliation, then merges and cleans the branch, after which `impl approve` passes. The refusal is a 4gsz-class lifecycle-bypass guard, sibling to `complete`'s own step 1.6 ‚Äî not a second panel gate.

**Adjacent, deliberately out of scope:** the auto-merge loop iterates ALL `bead/*` worktree branches, not just this epic's ‚Äî the cross-spec merge bug is `mindspec-blp6` (P2) and stays a separate fix (the refusal here is scoped to this spec's epic; see Non-Goals).

## ADR Touchpoints

- [ADR-0037](../../adr/ADR-0037-panel-gate-enforced-contract.md): **amended by this spec** ‚Äî the panel-gate enforced contract, as amended by 114, still names `mindspec complete` as the enforcement point. Add a dated amendment: the contract's reach is **every lifecycle verb that can merge a bead branch** ‚Äî `mindspec impl approve` refuses to finalize while any bead under the epic lacks proof of panel settlement (closed-without-`complete`, or carrying an unsettled durable refutation obligation), and the single settlement surface remains `mindspec complete` (no second gate home). The ¬ß7 hatches keep their exact semantics: they except the GATE DECISION inside the recovery `complete` run, never the durable obligation (114 R2), and they do NOT bypass this refusal (it is a lifecycle-bypass guard, not the gate decision).
- [ADR-0030](../../adr/ADR-0030-executor-boundary.md): cited, not amended ‚Äî the reason the refusal lives in the enforcement package `internal/approve` (pre-mutation, alongside the existing gates 1/3-3/3) and NOT inside `FinalizeEpic`: the executor is the git/process I/O boundary and cannot import `internal/complete` (cycle).
- [ADR-0035](../../adr/ADR-0035-agent-error-contract.md): the refusal is a `guard.NewFailure` whose FINAL line is the genuine recovery `mindspec complete <bead>` (`lifecycle.Orphan.RecoveryCommand()`), matching `complete` step 1.6's existing message shape (`internal/complete/complete.go:317-320`); `internal/guard/recovery_convention_test.go` enforces the final-line rule.
- [ADR-0023](../../adr/ADR-0023.md): beads as single state authority ‚Äî a bead's `closed` STATUS is not proof of settlement; the durable `refutation_pending_entries` / `panel_refuted_entries` obligation record on bead metadata (114 R2) is authoritative and must be honored by this verb too (R3).

## Requirements

1. **`mindspec impl approve` refuses while any closed bead under the spec's epic was closed without `mindspec complete`.** `ApproveImpl` (`internal/approve/impl.go:132`) gains a new PRE-TERMINAL enforcement gate that calls the shared predicate `lifecycle.FindOrphanedClosedBeads(specID, root, "")` (`internal/lifecycle/orphans.go:74`) ‚Äî the SAME predicate `complete` step 1.6, `next`, and `doctor` already use, with the same merged-but-undeleted exemption (an ancestor branch is never flagged). **Infra-error posture (OQ2, resolved): fail-open, parity preserved.** The predicate's deliberate best-effort contract (`orphans.go:71-73`: an absent epic, `bd` failure, or ancestry error yields NO orphans) is kept byte-unchanged for this fourth consumer ‚Äî a bd/git infra error is not evidence of an unsettled RC, and fail-closing here would wedge finalize (and false-block `complete`/`next`/`doctor`, which share the predicate) on transient Dolt hiccups. The compensating asymmetry is deliberate and load-bearing: R3's durable-obligation leg is independently FAIL-CLOSED (an unreadable obligation store cannot prove settlement), so infra fail-open on detection never extends to fail-open on a recorded obligation. Any orphan ‚Üí return a `guard.NewFailure` that (a) names the bead ID and its unmerged `bead/<id>` branch and the spec branch, (b) states it was closed without `mindspec complete`, and (c) ends with the recovery line `mindspec complete <bead>` (ADR-0035). **Ordering (HC-4)**: the gate runs among the pre-terminal gates ‚Äî after the phase gate and BEFORE the Spec 092 deferred phase-reconcile write (`impl.go:327-340`), MUTATION (1/3) epic close (`:343-348`), the `mindspec_phase=done` write, the CommitCount preflight, and `exec.FinalizeEpic` (`:372`) ‚Äî so a refusal mutates NOTHING (no epic close, no phase write, no merge, no push). `TestApproveImplCallOrder` (`internal/approve/impl_test.go:714`, AST-based) is extended to pin this placement RED-on-revert. **Hatches**: `MINDSPEC_SKIP_PANEL` (`internal/panel/gate.go:30`) and `enforcement.panel_gate: false` do NOT bypass this refusal ‚Äî it is a 4gsz-class lifecycle-bypass guard, not the panel-gate decision; the hatches keep their exact 114 semantics inside the recovery `complete` run (gate decision bypassed, pre-existing durable obligation still reconciled at step 3.75). **Falsified if**: with an orphaned closed bead present (branch exists, not an ancestor), `mindspec impl approve` exits 0, or `FinalizeEpic` merges that branch, or any mutation (epic close / phase metadata write / merge / push) lands before the refusal; or if a merged-but-undeleted bead branch (ancestor) triggers the refusal; or if setting `MINDSPEC_SKIP_PANEL=1` or `enforcement.panel_gate: false` makes the refusal pass.
2. **Best-effort slot naming ‚Äî advisory, read-only, never load-bearing.** When the refused orphan bead has a registered panel readable via the same layout-aware scan the gate uses (`panelGateRoots` / `panel.ForBead` + `panel.Tally`, `internal/complete/panel_advisory.go:52-81, 624-633`), the refusal message ADDITIONALLY names the unresolved REQUEST_CHANGES slot(s) from the vote-state (114 R1 keeps `VoteDecision` in lockstep with the gate, so this can never contradict it). This is strictly advisory: an unreadable, missing, or removed panel changes NOTHING about the refusal (R1 fires on the orphan predicate alone ‚Äî the panel is agent-writable, ADR-0037 ¬ß8, and its absence must never un-refuse). No gate decision is computed here, no metadata is written, and the message never prints the skip variable (HC-7) nor a paste-able refutation incantation (114 OQ4). **Falsified if**: removing/corrupting the panel directory makes the refusal pass or crash; if the slot line appears without a readable registered panel; if the message names `MINDSPEC_SKIP_PANEL` or contains a paste-able refutation command; or if the advisory disagrees with `PanelGateDecision` on the vote portion for the same tally.
3. **Durable-obligation backstop: an unsettled refutation obligation blocks `impl approve` even when the orphan predicate cannot see it.** For every plan bead (`readPlanBeadIDs`, `impl.go:452`), the new gate reads bead metadata (via the fail-closed `bead.GetMetadata`, 114) and refuses if any `refutation_pending_entries` entry is not covered ‚Äî (slot, round)-exact ‚Äî by a durable `panel_refuted_entries` record, using the SAME decode + coverage discipline as `reconcilePendingRefutations` (`internal/complete/panel_advisory.go:519-584`; the check-only core is extracted/exported from `internal/complete` rather than re-implemented). This covers the case R1's ancestry predicate exempts: a bead whose branch was manually merged (the forbidden raw `git merge`) or already deleted while its obligation is unsettled. **Fail-closed** exactly as 114 R2: a metadata read error, a present-but-corrupt `refutation_pending_entries`/`panel_refuted_entries` value, or a shape-invalid entry (empty slot / round < 1) REFUSES ‚Äî never decodes-as-empty. Hatches do not bypass this leg ("hatches except the gate, not the obligation", 114). A bead with no recorded pending entries is a no-op (fail-open for pristine beads preserved). Recovery line: `mindspec complete <bead>` (whose step 3.75 reconciliation settles the obligation). **Deleted-branch recovery ergonomics (OQ3, resolved): manual recourse accepted for this spec** ‚Äî for a branch-less bead, `complete.Run` errors at the step-3.5 merge-base (`complete.go:492-495`) before reaching step 3.75; the R3 refusal still fires and names the bead, and the operator restores the branch ref or settles the obligation out-of-band. Reaching this compound state requires the fenced raw-`git merge` anti-pattern (or manual branch deletion) ON TOP of an unsettled obligation. Teaching `complete` to reconcile an already-closed, branch-less bead is the deferred follow-up `mindspec-h4n5` (P3, filed at grill) ‚Äî a `complete`-flow change, Out of Scope here. **Falsified if**: a closed plan bead carrying an unsettled `refutation_pending` entry, with its branch merged or deleted, lets `impl approve` exit 0 without a covering `panel_refuted` record; if a metadata read/decode error on this leg fails OPEN; if a (slot, round)-covered obligation refuses (false positive); or if a hatch bypasses this leg.
4. **Recovery convergence ‚Äî the recourse is `mindspec complete <bead>`, and it still gates.** Pinned as behavior (mostly existing, from 114 + step-4 tolerance): running `mindspec complete <bead>` on the already-closed orphan re-runs the FULL layered gate ‚Äî including 114's unresolved-RC Block naming the slot, stale-SHA, incomplete, round-mismatch, dirty-tree, doc-sync, ADR-divergence ‚Äî and BLOCKS until the RC is resolved (re-panel) or durably refuted (114 R2's audited, fail-closed marker + reconciliation); once settled it warns "already closed", merges the bead branch, cleans up, and a subsequent `mindspec impl approve` passes the R1 gate and finalizes. **Falsified if**: under default configuration (`MINDSPEC_SKIP_PANEL` unset, `enforcement.panel_gate: true`), `mindspec complete` on the already-closed RC-bearing orphan exits 0 without the RC resolved or durably refuted (a hatched recovery run legitimately bypasses the GATE DECISION per 114 ¬ß7 ‚Äî that is sanctioned, audited behavior, not a falsification ‚Äî while any pre-existing durable obligation still reconciles at step 3.75); or if, after a legitimate settle-and-complete, `impl approve` still refuses (the refusal must be exactly as recoverable as `complete` step 1.6's).
5. **No regression on the normal path.** A spec whose beads were all completed via `mindspec complete` (branches merged ‚Äî ancestor or deleted) finalizes byte-identically to today: the auto-merge loop's ancestor skip (`mindspec_executor.go:394`), `guardMergeLayout` (`:399`), the Spec 092 Req 14(a) semantic conflict-abort (`:402-417`), the wu7t protected-main finalize flow (`:432-489`, `finalize_orphan_test.go`), and the merged-but-undeleted no-op all keep their exact behavior; existing `internal/executor` and `internal/approve` tests pass without semantic modification (mechanical fixture additions only). The misleading comment at `mindspec_executor.go:396-398` is corrected to name the layout guard specifically (o4fd's NOTE). **Falsified if**: any existing `TestApproveImpl*`, `TestFinalize*`, layout-guard, or merge-conflict test needs a semantic expectation change; or if a clean all-completed spec fails `impl approve`.

### Out of Scope

- `MINDSPEC_SKIP_PANEL`, `enforcement.panel_gate`, abandonment, refutation recording ‚Äî all keep their exact 114 semantics.

## Non-Goals

- **Not a second panel gate.** `impl approve` never computes an Allow/Block panel decision, never writes panel audit metadata, never applies refutations ‚Äî the REFUSE design routes all settlement through `mindspec complete` (the anti-drift point of 114's single-gate-home architecture).
- **Not general tamper-resistance.** Per ADR-0037 ¬ß8 the inputs stay agent-writable; a raw `git merge bead/<id>` (the fenced anti-pattern) remains outside the binary's reach. This spec closes the BINARY's last un-gated merge path, not git's.
- **Not fixing blp6** (cross-spec in_progress bead merged during finalize) ‚Äî separate P2 bead. OQ4 resolved NO executor seam for this spec, so blp6's future fix gains no hook here; nothing in 115 precludes it (the loop at `:381-422` is untouched behaviorally).
- **Not blocking `bd close`** ‚Äî beads may still be raw-closed (e.g. non-lifecycle work); they simply cannot ride an `impl approve` merge without settlement.
