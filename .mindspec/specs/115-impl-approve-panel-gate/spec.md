# Spec 115-impl-approve-panel-gate: impl-approve panel gate — FinalizeEpic re-gates raw-bd-close'd beads before auto-merge

## Goal

Close the silent-out-vote hole spec 114 closed for `mindspec complete` on the one remaining verb that can still merge a bead branch un-gated: `mindspec impl approve`. Today `ApproveImpl` → `FinalizeEpic` auto-merges any unmerged `bead/<id>` branch into the spec branch (`internal/executor/mindspec_executor.go:381-422`) — including a bead that was closed via a raw `bd close`, skipping `mindspec complete`'s authoritative panel gate entirely. So an operator or agent can `bd close` a bead carrying an unresolved REQUEST_CHANGES and then run `mindspec impl approve`, and the RC-bearing branch merges to the spec branch with no gate, no reconciliation, and no audit — exactly the out-vote 114 made impossible at `complete` (114's own final-review scope note, `.mindspec/specs/114-panel-gate-block-on-request-changes/spec.md:40-42`, filed this as `mindspec-o4fd`).

Target user outcome: **the binary refuses to merge, via any lifecycle verb, a bead branch that lacks proof of panel settlement wherever its detection signal or a recorded obligation can see the gap.** `mindspec impl approve` REFUSES (exit non-zero, performing no epic close, no phase write, no merge, no push) when any closed bead under the spec's epic was closed without `mindspec complete` — naming the bead and its unmerged branch, and (best-effort) the unresolved RC slot(s) — and the recourse converges through the ONE existing gate home: `mindspec complete <bead>` (which tolerates an already-closed bead, re-runs the full layered gate, and blocks until the RC is resolved or durably refuted). A spec whose beads all went through `mindspec complete` finalizes exactly as today. Two disclosed residuals stay out of reach by design (see Non-Goals): a bead branch already raw-`git merge`d whose unresolved RC never had a refutation applied leaves no durable obligation for R3 to see, and cross-spec orphans riding the auto-merge loop's global iteration are `mindspec-blp6`'s separately-tracked fix.

## Background

**The hole, code-traced.** `FinalizeEpic` (`internal/executor/mindspec_executor.go:349`) contains the "Auto-merge unmerged bead branches into spec branch before cleanup" loop (`:381-422`): it lists worktrees (`:383`), filters on the `bead/` branch prefix (`:385`), auto-commits leftovers (`:390`), skips branches already an ancestor of the spec branch (`:394`), and merges the rest via `gitutil.MergeInto` (`:402`). Its explicit purpose comment (`:382`) is *"This handles beads that were closed via `bd close` without `mindspec complete`."* The ONLY guard in front of that merge is `guardMergeLayout` (`:399`, defined `internal/executor/layout_guard.go:74`) — a DIRECTIONAL layout-regression guard, nothing else. The comment at `:396-398` ("same regression-only rule as CompleteBead; mutates nothing on a block") refers to that layout guard only; o4fd's "verify whether any gate actually fires there" verifies as: **no panel gate, no obligation reconciliation, and no orphan check fire anywhere on this path** — not in the loop, and not in its sole production caller `ApproveImpl` (`internal/approve/impl.go:132-428`), whose bead-status gate (`:220-242`) checks only that every plan bead's status is `"closed"` — a status a raw `bd close` satisfies. That gate is also SKIPPED entirely when `plan.md`'s bead IDs are unreadable (`:226-227`, `if planErr == nil`) — a fail-open R3 inherits unless pinned otherwise (R3 pins it fail-closed for the new gate).

**What `complete` does that this path skips.** `complete.Run` runs, before its terminal close (step 4) and bead→spec merge (step 5): the sibling bd_close lifecycle-bypass guard (step 1.6, `internal/complete/complete.go:307-321`) and the AUTHORITATIVE in-binary panel gate (step 2.25, `:345-405`, calling `panelGate` at `:387` → `internal/complete/panel_advisory.go:162-262` → `panel.PanelGateDecision`, `internal/panel/gate.go:143`; ADR-0037, amended by 114 so any unresolved REQUEST_CHANGES Blocks) — both BEFORE even the step-2.5 auto-commit — then the layered per-bead gates (doc-sync/ADR, steps 3.5) and the durable-obligation reconciliation (step 3.75, `:558-572` → `reconcilePendingRefutations`, `panel_advisory.go:519-584`), which run after that auto-commit but still before anything terminal — and only then closes (tolerating an already-closed bead: step 4, `:574-582`) and merges. Raw `bd close` + `impl approve` reaches the same merge having run none of it.

**The detection signal already exists — and is epic-scoped by construction.** `lifecycle.FindOrphanedClosedBeads` (`internal/lifecycle/orphans.go:74-112`, bead `mindspec-4gsz`) is the ONE shared predicate for "closed via raw `bd close`": a closed bead whose `bead/<id>` branch EXISTS (cheap trigger) and is NOT an ancestor of the spec branch (confirmation — so a benign merged-but-undeleted branch is never flagged). Its enumeration is scoped to the given spec's epic (`orphans.go:75-83`: epic lookup by specID, then `bd list --parent <epic> --status=closed`), so — unlike `FinalizeEpic`'s global `bead/*` merge loop — it can never flag a different spec's orphan. Three verbs already consume it: `mindspec complete` (step 1.6), `mindspec next` (`cmd/mindspec/next.go:459`), and `mindspec doctor` (`internal/doctor/orphaned_beads.go:48-79`, which reports each orphan with the `mindspec complete <id>` recovery and a `--fix` that re-invokes it). `mindspec impl approve` is the one lifecycle verb that does NOT consume it — and it is the verb that MERGES. Its canonical recovery is already defined: `Orphan.RecoveryCommand()` (`orphans.go:54`). This spec is the impl-approve leg of `mindspec-4gsz`'s detect-and-block floor: 4gsz identified the shared predicate and its consumers; the complete/next/doctor legs shipped there, and 4gsz's remaining leg (a CI check) stays 4gsz's own scope — 115 delivers only the fourth, merging consumer.

**Chosen enforcement model — REFUSE, not RUN (OQ1, RESOLVED at grill; validated by the round-1 panel).** This spec makes `impl approve` REFUSE and route to `mindspec complete <bead>`, rather than running the panel gate + reconciliation itself inside `ApproveImpl`/`FinalizeEpic`. Rationale: (a) **one gate home** — 114's three bead-panel review rounds each found a distinct false-discharge hole in a *second* settlement surface; duplicating the gate/reconcile ordering (gate before CommitAll, reconcile before close, audit writes after the terminal mutation) into a second verb re-opens that drift surface; (b) **ADR-0030 layering** — `internal/executor` is the git-I/O boundary and must not house enforcement, and it cannot import `internal/complete` (import cycle: complete → executor); (c) **the recourse already converges** — `complete.Run` tolerates an already-closed bead (step 4) and re-runs the FULL layered gate (RC-block, stale-SHA, incomplete, dirty-tree, round-mismatch, doc-sync, ADR) plus the durable-refutation reconciliation, then merges and cleans the branch, after which `impl approve` passes. The refusal is a 4gsz-class lifecycle-bypass guard, sibling to `complete`'s own step 1.6 — not a second panel gate.

**Adjacent, deliberately out of scope:** the auto-merge loop iterates ALL `bead/*` worktree branches, not just this epic's — the cross-spec merge bug is `mindspec-blp6` (P2). Decision (round-1 panel item 4): epic-scoped DETECTION fully closes the refuse correctness this spec needs — the refusal enumerates the finalizing epic's own beads and can neither refuse on nor be silenced by another spec's orphan — so `blp6`'s merge-loop global-iteration fix stays DEFERRED as its own P2 (see Non-Goals).

## ADR Touchpoints

- [ADR-0037](../../adr/ADR-0037-panel-gate-enforced-contract.md): **amended by this spec** — the panel-gate enforced contract, as amended by 114, still names `mindspec complete` as the enforcement point. Add a dated amendment: the contract's reach is **every lifecycle verb that can merge a bead branch** — `mindspec impl approve` refuses to finalize while any bead under the epic lacks proof of panel settlement (closed-without-`complete`, or carrying an unsettled durable refutation obligation), and the single settlement surface remains `mindspec complete` (no second gate home). The §7 hatches keep their exact semantics: they except the GATE DECISION inside the recovery `complete` run, never the durable obligation (114 R2), and they do NOT bypass this refusal (it is a lifecycle-bypass guard, not the gate decision).
- [ADR-0030](../../adr/ADR-0030-executor-boundary.md): cited, not amended — the reason the refusal lives in the enforcement package `internal/approve` (pre-terminal, alongside the existing gates 1/3-3/3) and NOT inside `FinalizeEpic`: the executor is the git/process I/O boundary and cannot import `internal/complete` (cycle).
- [ADR-0035](../../adr/ADR-0035-agent-error-contract.md): the refusal is a `guard.NewFailure` whose FINAL line is a GENUINE recovery — `mindspec complete <bead>` (`lifecycle.Orphan.RecoveryCommand()`) for the orphan-with-branch and merged-but-undeleted cases, matching `complete` step 1.6's existing message shape (`internal/complete/complete.go:317-320`); for the R3 branch-less case the final recovery line instead names the restoration-prerequisite recourse (restore the `bead/<id>` branch ref, then `mindspec complete <bead>`), because a bare `mindspec complete <bead>` is NOT runnable there (step-3.5 merge-base error, `complete.go:492-495`) and ADR-0035 requires the recovery to be truthful. `internal/guard/recovery_convention_test.go` enforces the final-line rule.
- [ADR-0023](../../adr/ADR-0023.md): beads as single state authority — a bead's `closed` STATUS is not proof of settlement; the durable `refutation_pending_entries` / `panel_refuted_entries` obligation record on bead metadata (114 R2) is authoritative and must be honored by this verb too (R3).

## Requirements

1. **`mindspec impl approve` refuses while any closed bead under the spec's epic was closed without `mindspec complete` — and refuses when it cannot prove otherwise.** `ApproveImpl` (`internal/approve/impl.go:132`) gains a new PRE-TERMINAL enforcement gate built on the SAME epic-scoped detection `complete` step 1.6, `next`, and `doctor` already use, with the same merged-but-undeleted exemption (an ancestor branch is never flagged). **Infra-error posture (OQ2, resolved by the round-1 panel): asymmetric by consumer role — advisory fail-open, gate fail-CLOSED.** `internal/lifecycle` is refactored so the enumeration core is error-preserving — an exported variant (e.g. `ScanOrphanedClosedBeads(specID, workdir, excludeBeadID string) ([]Orphan, error)`) that returns the epic-lookup, `bd`-list, **branch-existence-probe**, or ancestry error instead of swallowing it — while `FindOrphanedClosedBeads` remains as the fail-open wrapper with byte-identical behavior for the three existing consumers (`complete`/`next`/`doctor`; their parity contract, `orphans.go:71-73`, is unchanged). The impl-approve GATE calls the error-preserving core and **REFUSES on any infra error**: an unreadable store cannot prove the epic is settled, and a merge gate must not fail open on "can't prove settlement". **Branch-existence probe is one of the fail-closed legs (round-2 G2)**: today's trigger `gitutil.BranchExists` (`internal/gitutil/gitops.go:94-100`, `return cmd.Run() == nil`, consumed via `branchExistsFn`, `orphans.go:40`, at `:95`) maps EVERY `git rev-parse --verify` error to `false` — an infra error is indistinguishable from a cleanly-deleted branch, and since `FinalizeEpic`'s worktree-listing merge loop does not use `BranchExists`, a probe error would let the gate pass while the loop still merges the orphan branch (the o4fd class re-opened via infra noise). Fix: a new error-preserving git/ref seam `gitutil.BranchExistsE(name string) (bool, error)` beside the unchanged `BranchExists` (`gitops.go:94`); the error-preserving core consumes `BranchExistsE` and propagates its error like the other three legs; the gate REFUSES on it; the fail-open wrapper `FindOrphanedClosedBeads` keeps swallowing it (treating error as "no branch" exactly as `BranchExists` does today — complete/next/doctor byte-identical). **Epic-scoped detection (round-1 item 4)**: the core enumerates ONLY closed beads under the finalizing spec's epic (`orphans.go:75-83`), so an orphan belonging to a DIFFERENT spec never triggers — or suppresses — this refusal; `blp6`'s global merge-loop iteration stays a separate deferred P2. Any orphan → return a `guard.NewFailure` that (a) names the bead ID and its unmerged `bead/<id>` branch and the spec branch, (b) states it was closed without `mindspec complete`, and (c) ends with the recovery line `mindspec complete <bead>` (ADR-0035); any infra error → a `guard.NewFailure` naming the failed enumeration step. **Ordering (HC-4)**: the gate runs among the pre-terminal gates — after the phase gate and BEFORE the Spec 092 deferred phase-reconcile write (`impl.go:327-340`), MUTATION (1/3) epic close (`:343-348`), the `mindspec_phase=done` write, the CommitCount preflight, and `exec.FinalizeEpic` (`:372`) — so a refusal performs **no epic close, no phase write, no merge, no push**. (The one write that legitimately precedes it is `phase.EnsureMigrated`'s one-shot migration reconcile at `impl.go:140`, which precedes EVERY `ApproveImpl` gate today — the refusal guarantee is scoped to close/phase-write/merge/push, not to that pre-existing migration step.) `TestApproveImplCallOrder` (`internal/approve/impl_test.go:714`, AST-based) is extended to pin this placement RED-on-revert. **Hatches**: `MINDSPEC_SKIP_PANEL` (`internal/panel/gate.go:30`) and `enforcement.panel_gate: false` do NOT bypass this refusal — it is a 4gsz-class lifecycle-bypass guard, not the panel-gate decision; the hatches keep their exact 114 semantics inside the recovery `complete` run (gate decision bypassed, pre-existing durable obligation still reconciled at step 3.75). **Falsified if**: with an orphaned closed bead present (branch exists, not an ancestor), `mindspec impl approve` exits 0, or `FinalizeEpic` merges that branch, or an epic close / phase metadata write / merge / push lands before the refusal; or if an epic-lookup/`bd`-list/branch-existence-probe/ancestry infra error on this gate path lets `impl approve` exit 0 (fail-open); or if a merged-but-undeleted bead branch (ancestor) triggers the refusal; or if an orphaned closed bead under a DIFFERENT spec's epic triggers this spec's refusal; or if any existing `internal/lifecycle` consumer (`complete`/`next`/`doctor`) changes behavior on infra errors; or if setting `MINDSPEC_SKIP_PANEL=1` or `enforcement.panel_gate: false` makes the refusal pass.
2. **Best-effort slot naming — advisory, read-only, never load-bearing.** When the refused orphan bead has a registered panel readable via the same layout-aware scan the gate uses (`panelGateRoots` / `panel.ForBead` + `panel.Tally`, `internal/complete/panel_advisory.go:52-81, 624-633`), the refusal message ADDITIONALLY names the unresolved REQUEST_CHANGES slot(s) from the vote-state (114 R1 keeps `VoteDecision` in lockstep with the gate, so this can never contradict it). This is strictly advisory: an unreadable, missing, or removed panel changes NOTHING about the refusal (R1 fires on the orphan predicate alone — the panel is agent-writable, ADR-0037 §8, and its absence must never un-refuse). The R1 fail-closed posture does NOT extend to this leg: a panel read error here degrades to omitting the slot line, never to a pass and never to a crash. No gate decision is computed here, no metadata is written, and the message never prints the skip variable (HC-7) nor a paste-able refutation incantation (114 OQ4). **Falsified if**: removing/corrupting the panel directory makes the refusal pass or crash; if the slot line appears without a readable registered panel; if the message names `MINDSPEC_SKIP_PANEL` or contains a paste-able refutation command; or if the advisory disagrees with `PanelGateDecision` on the vote portion for the same tally.
3. **Durable-obligation backstop: an unsettled refutation obligation blocks `impl approve` even when the orphan predicate cannot see it.** For every plan bead (`readPlanBeadIDs`, `impl.go:452`), the new gate reads bead metadata (via the fail-closed `bead.GetMetadata`, 114) and refuses if any `refutation_pending_entries` entry is not covered — (slot, round)-exact — by a durable `panel_refuted_entries` record, using the SAME decode + coverage discipline as `reconcilePendingRefutations` (`internal/complete/panel_advisory.go:519-584`; the check-only core is extracted/exported from `internal/complete` rather than re-implemented). **Enumeration is fail-CLOSED (round-1 item 2)**: if the plan's bead IDs cannot be read (missing `plan.md`, missing/corrupt frontmatter, empty `bead_ids` — the `readPlanBeadIDs` error cases), the gate REFUSES naming the unreadable plan path, rather than inheriting the existing gate-1/3 silent skip (`impl.go:226-227`) — a corrupt `plan.md` must not make this leg's unique coverage silently vanish. This leg covers what R1's ancestry predicate exempts — a bead whose branch was manually merged (the forbidden raw `git merge`) or already deleted while its obligation is unsettled — **but only when a durable obligation was recorded** (a refutation was applied at some prior gate run); a raw-merged bead whose RC was never refuted leaves no `refutation_pending` marker and is a disclosed residual (Non-Goals). **Fail-closed** exactly as 114 R2: a metadata read error, a present-but-corrupt `refutation_pending_entries`/`panel_refuted_entries` value, or a shape-invalid entry (empty slot / round < 1) REFUSES — never decodes-as-empty. Hatches do not bypass this leg ("hatches except the gate, not the obligation", 114). A bead with no recorded pending entries is a no-op (fail-open for pristine beads preserved). **Recovery line — truthful per branch state (round-2 G3)**: when the bead's `bead/<id>` branch still exists (merged-but-undeleted), the recovery line is `mindspec complete <bead>` (whose step 3.75 reconciliation settles the obligation — that command IS runnable). When the branch does NOT exist, a bare `mindspec complete <bead>` would error at the step-3.5 merge-base (`complete.go:492-495`) BEFORE reaching the step-3.75 reconciliation — so the branch-less refusal carries a DISTINCT recourse that names the restoration prerequisite: restore the `bead/<id>` branch ref (so `complete` can reach reconciliation) and then run `mindspec complete <bead>`, or settle the obligation out-of-band; the message must never present a bare command known to fail. **Deleted-branch recovery ergonomics (OQ3, resolved): manual recourse accepted for this spec** — the R3 refusal still fires and names the bead; the operator performs the restoration or settles out-of-band. Reaching this compound state requires the fenced raw-`git merge` anti-pattern (or manual branch deletion) ON TOP of an unsettled obligation. Teaching `complete` to reconcile an already-closed, branch-less bead is the deferred follow-up `mindspec-h4n5` (P3, filed at grill) — a `complete`-flow change, Out of Scope here. **Falsified if**: a closed plan bead carrying an unsettled `refutation_pending` entry, with its branch merged or deleted, lets `impl approve` exit 0 without a covering `panel_refuted` record; if a metadata read/decode error on this leg fails OPEN; if an unreadable-plan-bead-IDs state lets the gate silently skip this leg instead of refusing; if a (slot, round)-covered obligation refuses (false positive); if the branch-less refusal's recourse is a bare `mindspec complete <bead>` with no restoration prerequisite named (a command known to fail at the step-3.5 merge-base); or if a hatch bypasses this leg.
4. **Recovery convergence — the recourse is `mindspec complete <bead>`, and it still gates.** Pinned as behavior (mostly existing, from 114 + step-4 tolerance): running `mindspec complete <bead>` on the already-closed orphan re-runs the FULL layered gate — including 114's unresolved-RC Block naming the slot, stale-SHA, incomplete, round-mismatch, dirty-tree, doc-sync, ADR-divergence — and BLOCKS until the RC is resolved (re-panel) or durably refuted (114 R2's audited, fail-closed marker + reconciliation); once settled it warns "already closed", merges the bead branch, cleans up, and a subsequent `mindspec impl approve` passes the R1 gate and finalizes. **Falsified if**: under default configuration (`MINDSPEC_SKIP_PANEL` unset, `enforcement.panel_gate: true`), `mindspec complete` on the already-closed RC-bearing orphan exits 0 without the RC resolved or durably refuted (a hatched recovery run legitimately bypasses the GATE DECISION per 114 §7 — that is sanctioned, audited behavior, not a falsification — while any pre-existing durable obligation still reconciles at step 3.75); or if, after a legitimate settle-and-complete, `impl approve` still refuses (the refusal must be exactly as recoverable as `complete` step 1.6's).
5. **No regression on the normal path.** A spec whose beads were all completed via `mindspec complete` (branches merged — ancestor or deleted) finalizes byte-identically to today: the auto-merge loop's ancestor skip (`mindspec_executor.go:394`), `guardMergeLayout` (`:399`), the Spec 092 Req 14(a) semantic conflict-abort (`:402-417`), the wu7t protected-main finalize flow (`:432-497`, `finalize_orphan_test.go`), and the merged-but-undeleted no-op all keep their exact behavior; existing `internal/executor` and `internal/approve` tests pass without semantic modification (mechanical fixture additions only). The misleading comment at `mindspec_executor.go:396-398` is corrected to name the layout guard's directional layout-regression scope specifically, dropping the "same regression-only rule as CompleteBead" parity phrasing (o4fd's NOTE). **Falsified if**: any existing `TestApproveImpl*`, `TestFinalize*`, layout-guard, or merge-conflict test needs a semantic expectation change; or if a clean all-completed spec fails `impl approve`.

## Non-Goals

- **Not a second panel gate.** `impl approve` never computes an Allow/Block panel decision, never writes panel audit metadata, never applies refutations — the REFUSE design routes all settlement through `mindspec complete` (the anti-drift point of 114's single-gate-home architecture).
- **Not general tamper-resistance.** Per ADR-0037 §8 the inputs stay agent-writable; a raw `git merge bead/<id>` (the fenced anti-pattern) remains outside the binary's reach — including its worst case: a raw-merged bead carrying an unresolved RC that never had a refutation applied leaves NO durable obligation, so R1's ancestry exemption and R3's marker scan both legitimately pass it (disclosed residual; the Goal's coverage claim is scoped accordingly). This spec closes the BINARY's last un-gated merge path, not git's.
- **Not fixing blp6** (cross-spec in_progress bead merged during finalize) — separate P2 bead. OQ4 resolved NO executor seam for this spec, so blp6's future fix gains no hook here; nothing in 115 precludes it (the loop at `:381-422` is untouched behaviorally). A cross-spec orphan on that global loop is the second disclosed residual scoped out of the Goal.
