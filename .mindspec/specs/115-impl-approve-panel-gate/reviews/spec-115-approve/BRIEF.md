## What this spec is (the follow-up to spec 114, which just shipped to main)
Spec 114 closed the silent out-vote at the `mindspec complete` panel gate. The 114 final-review panel (codex, code-traced) found the SAME class of hole still open on a different verb: `mindspec impl approve` → `ApproveImpl` → `FinalizeEpic` (`internal/executor/mindspec_executor.go:381-422`) **auto-merges bead branches closed via raw `bd close`** (skipping the panel gate) — guarded only by a directional layout guard, no gate, no reconciliation. So an operator can `bd close` an RC-bearing bead and `impl approve` merges it un-gated. Filed as `mindspec-o4fd` (P1); this spec is its fix.

**The design (grill-resolved OQ1): REFUSE, not RUN.** `mindspec impl approve` REFUSES (exit non-zero, mutating nothing — before epic close / phase write / `FinalizeEpic`) when any closed bead under the epic was closed via raw `bd close` (detected by the existing shared predicate `lifecycle.FindOrphanedClosedBeads` — branch exists + not an ancestor of the spec branch), naming the bead/branch with recovery `mindspec complete <bead>`. Recovery converges through the ONE gate home: `complete.Run` tolerates already-closed beads and re-runs every layered gate leg + the durable-refutation reconciliation, then merges — after which `impl approve` passes. Rationale for REFUSE over RUN: **ADR-0030** makes the executor the git-I/O boundary that cannot import `internal/complete` (import cycle), so RUN would re-implement the gate in a second package — exactly the duplicated-settlement-surface anti-pattern that spec 114's discharge path proved dangerous (three panel rounds, a new spoof hole each time). One gate home.
