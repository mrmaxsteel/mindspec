# Spec 124-impl-readiness-gate: Impl-Readiness Gate — reject under-specified beads before coding

## Goal

Make bead readiness an evaluated fact instead of an assumption. Today the impl-dispatch flow checks only a bead's **status** before spending a full implementation + panel + fix cycle on it: `/ms-bead-impl` Phase A captures the bead and asserts "Status must be `open` or `in_progress`" (`plugins/mindspec/skills/ms-bead-impl/SKILL.md:29`) — nothing anywhere judges whether the bead is **ready to implement** (plan section concrete, claimed Rs/ACs resolvable, dependencies actually merged, no blocking open questions). An under-specified bead therefore consumes the most expensive path in the whole loop — impl, 8-reviewer panel, fix, re-panel, sometimes fix-of-fix — before the ambiguity surfaces.

This spec adds a two-layer readiness gate, deliberately split along the ADR-0040 layering ratchet:

1. **A deterministic mechanical floor in the binary** — a new read-only verb, `mindspec bead ready-check <bead-id>`, evaluating four mechanical signals (plan section present and non-placeholder; claimed R/AC tokens resolve in spec.md by an exact whole-token/base-of-sub-lettered rule; declared dependencies closed AND landed-merged by the repo's real branch-deletion-tolerant detector; no genuine blocking marker) — wired **gate-before-mutate** (ADR-0041) into `mindspec next`'s claim path (with a durable `--allow-not-ready` override) and re-run as an **unconditional dispatch ingress** by `/ms-bead-impl` on **every** dispatch path.
2. **A semantic readiness review in the impl subagent's prompt** — a mandatory Phase 0 in the staged prompt: before any file edit, the subagent evaluates five judgment signals and either proceeds or returns a structured, commit-less `NOT READY: <reasons>` report. `/ms-bead-cycle` then chooses one of two dispositions — ACCEPT (halt → plan/spec revision) or, once per bead, a bounded, audited, grounded **Readiness Clarification** (R8) for re-evaluation.

Target outcome: a deliberately under-specified bead is refused (mechanically at claim/dispatch, or semantically at Phase 0) with actionable, span-quoting reasons and **zero commits spent**, while a well-specified bead — carrying the repo's ordinary scaffold checklists and bare-base R/AC claims — passes with no new interactive step and no refusal. Both directions are pinned by committed fixtures modeled faithfully on real shipped beads, so the gate is falsifiable both ways and cannot decay into a subjective ceremony that refuses valid work.

## Background

### The cost this spec attacks — the ROOT of spec 123's fix chain

Spec 123 (greenfield-first-run-integrity) took three consecutive final-review fix rounds before it merged: `e28db191` (r1 seam fixes), `9bca51b3` ("fix-of-fix (final-review r2 codex)"), and `ab12d8cb` (r3). The causal chain runs through the **root**: r1's findings (negation-aware handling, heal breadth — SR-1-class *unstated-behavior* ambiguities plausibly refusable pre-code at Phase 0), and rounds 2–3 were fixes **of r1's own fix** — churn in the `/ms-bead-fix` dispatch path, which this spec deliberately leaves ungated. This gate does not reach fix-round churn directly; it attacks the pre-code-visible ambiguity that *produced* r1, and when that root is caught the downstream r2/r3 collapse with it. Fix-round dispatch quality itself remains ungated (Out of Scope) — spec 123 is existing evidence that it can hurt, deferred here deliberately, not a claim that it never does. A 9-slot decision panel independently ranked this gate the highest-leverage next move for the same root-cause reason: one cheap `NOT READY` collapses an entire downstream cycle.

### The gap, precisely (bead `mindspec-p8t5`)

- `/ms-bead-impl` Phase A resolves the bead (`bd show`), the plan section, the spec Rs/ACs, and the dependency chain — as **prompt inputs**, never as a gate. The only precondition is the status check (`SKILL.md:29`). Phase B dispatches unconditionally, and a supplied `prompt-path` skips Phase A entirely (`SKILL.md:17,23`).
- `mindspec next` claims the selected bead, creates `bead/<id>`, and creates the worktree (`cmd/mindspec/next.go`, `internal/next/`) with no readiness evaluation — the claim mutation lands before any fact about the bead's implementability is checked, the ordering defect class ADR-0041 exists to name.
- `/ms-bead-cycle`'s only brakes are post-damage: `max-rounds` (default 3) and the Spec 109 loop-halt skeleton `loop.halt.max_consecutive_impl_failures` (`internal/config/config.go:382`, default 2 at `config.go:635-636`), which stop the loop only **after** failed implementations have been paid for.
- Plan-approve's structural validation (`internal/validate/plan.go` — `bead-acceptance-criteria` hard error, `bead-depends` warning; scaffold contract in `internal/approve/spec.go`) fires once, at approval time. It cannot see temporal facts (is the dependency *merged yet*?), cross-document facts (does claimed `AC-7` exist in spec.md?), or post-approval drift (plan hand-edited after a bead was claimed). And its structural floor is satisfiable by the scaffold's own placeholders (`path/to/file.go`, the angle-bracketed AC template literal) — exactly what an under-specified bead ships.

### Grounding the mechanical floor against real merged state (the r1-panel corrections)

The first-round spec-gate panel found the naive floor would reject nearly every real bead; the floor below is re-specified against actual shipped artifacts:

- **Merged-ness is branch-deletion-tolerant.** `mindspec complete` merges the bead with `--no-ff`, records `mindspec_landed_*` metadata, then **deletes** the `bead/<id>` branch (`internal/executor/mindspec_executor.go:468`). A branch-ref reachability check is therefore *unevaluable* for a normally-completed dependency and would false-negative every one of them. The repo's real landed-merge detector is `internal/lifecycle.FindLandedMerge` (`internal/lifecycle/landed.go:249`) — a first-parent merge subject-scan corroborated by landed-binding metadata / a surviving branch tip, with a `LandedMergeNoEvidence` failure mode (`landed.go:48`) — which tolerates the deleted branch. MF-3 is defined against it, not an ad-hoc ancestry check.
- **Routine checklists are not blocking markers.** The plan scaffold emits `**Verification** / - [ ] make test passes` inside *every* `## Bead N` section (`internal/approve/spec.go` `scaffoldPlan`), and spec 123 Bead 1 — the shape the POSITIVE fixture must model — carries ~17 unchecked `- [ ]` Verification/AC-mapping entries (`.mindspec/specs/123-greenfield-first-run-integrity/plan.md:448-467`). These are the convention for WELL-specified beads. MF-4 detects only genuine blockers (`TBD` / `OPEN QUESTION` as **whole-word/phrase tokens scanned OUTSIDE inline-code spans and fenced code blocks** — verified zero such hits across shipped plans 119–123; a naïve case-insensitive substring scan over-matches the `Open Questions` section convention, 5 hits in plans 121–123, and any bead that quotes the literals as fixture data — and unchecked items in a designated blocking region), never routine checklists.
- **Token resolution is exact both ways.** Spec 123's plan Bead 2 claims bare `R5` ("R5 in full", "R5(b)", "R5(c)") while its spec.md carries only `R5a/R5b/R5c` — so a whole-token-only rule would refuse a shipped, merged bead, and a substring rule would let a dangling `AC-1` resolve via `AC-19`. MF-2 pins the exact rule that passes the former and refuses the latter.

## ADR Touchpoints

- [ADR-0041: Gate-Before-Mutate](../../adr/ADR-0041-gate-before-mutate.md) — **AMENDED by this spec** (Requirement 9): ADR-0041 §1's three-phase contract enumerates a **closed set of three** mutating lifecycle verbs (`complete`, `plan approve`, `impl approve`; `ADR-0041:55-56`, with "all three verbs" recurring through its fault matrix). This spec brings `mindspec next` — which claims in bd, writes session/recording state, and creates a branch + worktree — under the discipline, so leaving the ADR "unchanged" would leave its recorded scope false. The amendment is small and PARTIAL — a **preflight-leg-only addition**: it adds ONLY the readiness preflight leg to `next` — readiness is a fact derivable before any mutation, evaluated before `bd update --claim`, branch, or worktree; a NOT-READY refusal leaves state byte-identical ("exit non-zero implies nothing changed"); the distinct `--allow-not-ready` override is recorded. The amendment makes **no claim** about `next`'s existing success-path claim/branch/worktree mutation chain — that chain retains whatever commit/forward-reconcile contract it already carries; this spec neither grants nor removes it, and the amendment is explicitly NOT a certification that `next` needs no commit/reconcile phase (its success path DOES mutate). ADR-0041's own Alternatives §1 (`:336-343`) explicitly anticipates "a fourth verb … from silently re-introducing the same defect shape," so the ADR authorizes this application; the amendment records it. **Amendment lifecycle** (spec 117/122 precedent): the amendment text is PRE-DRAFTED at plan time and finalized by the same bead that lands the `next` gate, so the ADR-divergence gate sees the declared touchpoint (AC-16 pins its presence).
- [ADR-0035: Agent Error Contract](../../adr/ADR-0035-agent-error-contract.md) — unchanged, APPLIED: every ready-check failure and every `next` refusal routes through `internal/guard` and ends in `recovery:` lines — per failing signal, a copy-pastable next step (edit the named plan section; `mindspec complete <dep-id>` for an unmerged dependency; `mindspec bead ready-check <id>` for the full report; `--allow-not-ready` for the deliberate override). Raw `bd update --metadata` is never emitted (the ADR-0035 ban); the clarification/override writes go through a thin verb, not pasted bd argv.
- [ADR-0040: Orchestration Layering Ratchet](../../adr/ADR-0040-orchestration-layering-ratchet.md) — unchanged, APPLIED as the design's spine: deterministic invariants live in the binary (the mechanical floor), judgment lives in prompts (the Phase 0 semantic review), and no gate logic is re-stated as skill prose (the skills *invoke* the verb and *route* its outcomes; content-pin tests keep them honest). The decision-matrix invariant is respected: the semantic layer produces a structured verdict the orchestrator routes, but no skill decides what the mechanical evidence *is*.
- [ADR-0023: Beads/Dolt as Single State Authority](../../adr/ADR-0023.md) — unchanged, RESPECTED: the R8 readiness-attempt/clarification record and the `--allow-not-ready` override marker are ADVISORY audit annotations persisted via existing `internal/bead` metadata helpers (no schema change), NOT lifecycle state. Bead lifecycle state stays derived from bd statuses; the readiness verdict never becomes an authority the lifecycle reads.
- [ADR-0034: Ceremony Collapse](../../adr/ADR-0034-ceremony-collapse.md) — unchanged, RESPECTED: this spec adds two verbs (`bead ready-check`, `bead clarify`) and one flag (`--allow-not-ready`) and pays for each explicitly — the spec-122 ceremony non-inflation baseline (`cmd/mindspec/ceremony_guard_test.go`) is updated in the same bead that adds each surface, so the inflation is deliberate and reviewed, never silent (AC-9).
- [ADR-0036: Ownership Discovery — Zero Framework Cognition](../../adr/ADR-0036-ownership-discovery.md) — unchanged, RESPECTED: the mechanical floor reads only repo artifacts and mindspec's **own** scaffold literals; it proposes no content and guesses nothing about the consumer's semantics.
- [ADR-0037: Panel Gate as Enforced Contract](../../adr/ADR-0037-panel-gate-enforced-contract.md) — unchanged, PROTECTED: readiness gains no merge authority. `mindspec complete` and the panel gate never consult readiness state; a NOT-READY bead that was force-claimed and implemented anyway is judged by the panel exactly as today (AC-9's isolation clause). `next` does not merge a bead branch, so ADR-0037's "every lifecycle verb that can merge a bead branch" reach is not triggered; MF-3 only READS git merged-ness.

## Requirements

Each requirement carries a hard falsifier. "Readiness fixture pair" below means the two committed test fixtures of R6: NEGATIVE (deliberately under-specified) and POSITIVE (well-specified, carrying the repo's ordinary benign features).

1. **A deterministic, read-only readiness verb: `mindspec bead ready-check <bead-id>`.** The verb resolves the bead's owning spec from its lineage (never from cwd — the ADR-0041 lesson from `complete`), evaluates the four mechanical signals of R2, and prints a per-signal report: one line per signal with a stable signal ID (`MF-1`..`MF-4`), `PASS`/`FAIL`, and for each FAIL the evidence (file path + the offending or missing element). Exit 0 when all four pass; on any FAIL, exit non-zero via `internal/guard` with one `recovery:` line per failing signal. The verb performs **no mutation**: no bd writes, no git writes, no file writes — two consecutive runs against unchanged state produce identical output and leave `git status` and bd state untouched. Malformed bead IDs are refused at ingress via `idvalidate.BeadID` (the existing `cmd/mindspec/bead.go` pattern). All bead/plan-derived text rendered in the report passes through `termsafe.Escape` before reaching the terminal (bead descriptions and plan prose are agent-authored free text — the spec 113/120 trust-boundary rule).
   *Falsified if*: the NEGATIVE fixture exits 0 or the POSITIVE fixture exits non-zero; or any FAIL line lacks a signal ID or a recovery line; or a run mutates bd state, git state, or any file; or a bead description containing terminal-escape bytes reaches the report unescaped.

2. **The mechanical floor — four signals, all mechanical, each grounded against shipped artifacts, none judging prose quality.** The bead-index `N` is the bead ID's numeric suffix, mapping positionally to `work_chunks[N-1]`/`bead_ids[N-1]` per the spec 097 contract in `internal/validate/plan.go`.
   - **MF-1 — plan section concrete-by-structure**: the plan's `## Bead N` section exists; its `**Acceptance Criteria**` block contains at least one entry that is not the scaffold's angle-bracketed AC placeholder (the "Specific, measurable criterion for this bead" template literal emitted by `scaffoldPlan`, `internal/approve/spec.go`); and `work_chunks[N-1].key_file_paths` is non-empty and contains no scaffold placeholder (`path/to/file.go`). Authoritative files-in-scope is the frontmatter, not prose (spec 097 R3/R4) — MF-1 never parses prose "Files:" lines.
   - **MF-2 — claimed Rs/ACs resolve, by an EXACT rule**: MF-2 harvests only the bead's OWN claimed `R<n>` / `AC-<n>` tokens — those declared as its criteria (its bd-description claim context and its `## Bead N` plan section), NOT tokens appearing inside a prose citation of another spec (a named foreign-spec reference like "the spec 123 AC-17 pattern" is a citation, not a claim, and is excluded from harvesting; harvest heuristic plan-level, pinned by the POSITIVE fixture's benign feature (iv)). Each harvested token MUST resolve in the owning `spec.md` by this pinned rule: **(letterless claim)** a claimed letterless token `X` (`R5`, `AC-1`) resolves iff spec.md contains, as a **whole token** (identifier-boundary delimited), either `X` itself OR any sub-lettered form `X<letter>` (`R5` resolves via `R5a`; a parenthesized claim `R5(b)` normalizes to `R5b` before matching); **(sub-lettered claim)** a claimed sub-lettered token `X<letter>` (`R5a`, `AC-2b`) resolves iff spec.md contains that EXACT sub-lettered token — never the bare `R5`, never a sibling `R5b`. **Numeric-prefix matches NEVER resolve**: `AC-1` does not resolve via `AC-19`, `R1` not via `R10` (boundary-enforced; `AC-1` and `AC-19` are distinct tokens). A claimed token with no resolving counterpart is a dangling promise → FAIL naming it.
   - **MF-3 — dependencies closed AND landed-merged (branch-deletion-tolerant)**: for every bd dependency edge of the bead (the authoritative source — bd edges wired from `work_chunks[].depends_on`, spec 097 R3; the prose `**Depends on**` section is explicitly non-authoritative and is not parsed), the dependency bead MUST be **closed** in bd AND its work MUST be a **landed merge into the spec branch** as determined by `internal/lifecycle.FindLandedMerge` — which tolerates the deleted `bead/<id>` branch that `mindspec complete` leaves behind (`mindspec_executor.go:468`) and corroborates via `mindspec_landed_*` binding / branch-tip / first-parent subject scan. The general rule is **PASS iff `FindLandedMerge` returns a positive `*LandedMerge`; FAIL (fail-closed) on ANY error** — a dependency that is open, or closed but for which `FindLandedMerge` returns `ErrLandedMergeNotFound` (the 2u0u closed-but-unmerged split state — no first-parent merge positively identified, `landed.go:35`) OR `*LandedMergeNoEvidence` (the spec-121 case: a subject-scan candidate exists but no admissible corroboration, `landed.go:48`), FAILs with a state-specific message and a recovery line; no error is ever silently treated as "merged" (never a false PASS). **Helper-locatability is NOT a mechanical file-existence check** — `key_file_paths` is the dependency's *context surface*, not a produced-artifact contract (spec 097 R4), and a valid dep may delete/rename/merely-inspect a path; whether the helper API is actually usable is the semantic judgment SR-3 (the subagent reads the files the prompt names). **Drift scope (recorded):** MF-3 evaluates bd-edge state only; a `work_chunks[N-1].depends_on` entry hand-added after approval that was never wired to a bd edge is out of MF-3's scope by design — the named backstops are SR-3 (an absent helper surfaces at Phase 0) and the cycle's step-0.1 `bd ready`-vs-plan reconciliation.
   - **MF-4 — no GENUINE blocking marker (routine checklists AND quoted literals excluded)**: FAIL only on (a) the token `TBD` (whole word, case-insensitive) or the phrase `OPEN QUESTION` (whole-word/phrase, case-insensitive) appearing in the bead's `## Bead N` plan section or bd description **outside** any inline-code span (`` `…` ``) or fenced code block (` ``` `) — so a bead that QUOTES those literals as fixture data or described behavior is not false-refused, and the whole-phrase qualifier stops `OPEN QUESTION` from substring-matching the `Open Questions` section convention (5 such hits in plans 121–123); or (b) an unchecked `- [ ]` item appearing under a **designated blocking region** — a `**Blocking Questions**`, `**Open Questions**`, or `**Blocked on**` sub-block inside the bead section. Routine `**Verification**` and `**Acceptance Criteria**` checklists (the scaffold convention; spec 123 Bead 1 carries ~17 such entries) are EXPLICITLY EXCLUDED and are never blocking. Spec-level open questions remain gated at spec-approve (`internal/validate/spec.go` `open-question` error) and are not re-scanned here.
   *Falsified if*: any signal PASSes on the NEGATIVE fixture's planted defect for that signal (placeholder AC; a dangling `AC-99` AND a prefix-dangling `AC-1` present only as `AC-19`; a closed-but-unmerged dependency; a genuine `TBD`/blocking-region marker); or any signal FAILs on the POSITIVE fixture (which carries scaffold Verification/AC checkboxes, a bare-base `R`/`AC` claim resolving via a sub-lettered spec token, and a prose pattern-citation); or MF-3 false-PASSes any `FindLandedMerge` error state (`ErrLandedMergeNotFound` or `*LandedMergeNoEvidence`), or false-FAILs a dependency completed normally via `mindspec complete` (branch deleted); or MF-2 resolves `AC-1` against only `AC-19`, or resolves a sub-lettered claim `R5a` against a bare `R5` or a sibling `R5b`; or MF-4 FAILs a routine Verification/AC checkbox or a `TBD`/`OPEN QUESTION` quoted inside an inline-code or fenced span. (A review-time obligation, not a mechanical falsifier: the bead panel confirms no signal's implementation reads prose *quality* rather than presence/resolution/merged-ness — the POSITIVE-fixture pin is the mechanical backstop for that class.)

3. **`mindspec next` gates the claim on the mechanical floor (gate-before-mutate), with a durable override.** After bead selection and before **any** mutation (no `bd update --claim`, no `bead/<id>` branch, no worktree), `next` evaluates R2 against the selected bead. On NOT READY it refuses: prints the per-signal report, exits non-zero, and leaves bd state, branches, and worktrees byte-identical to before the invocation; the refusal's recovery lines name the per-signal fixes plus the two escape hatches (`mindspec bead ready-check <id>` for the standalone report; re-run with `--allow-not-ready` to claim deliberately). `--allow-not-ready` proceeds past a failing floor AND records a **durable override marker** on the bead (a dedicated bd metadata key naming the overridden signals) so the later dispatch-ingress re-check (R4) honors the deliberate claim instead of re-blocking it; a warning naming every failing signal is emitted (the override is visible in the transcript, never silent). `--allow-not-ready` is distinct from the existing `--force` (session-freshness only, `cmd/mindspec/next.go`), which MUST remain orthogonal and gains no readiness authority. A passing floor adds no interactive step and no output beyond a single OK line.
   *Falsified if*: `next` against the NEGATIVE fixture's bead claims it, creates a branch, or creates a worktree without `--allow-not-ready`; or the refusal path leaves any mutation behind; or `--allow-not-ready` proceeds without recording the durable override marker or without naming the failing signals; or `--force` alone bypasses the readiness gate; or `next` against the POSITIVE fixture's bead is refused or prompts interactively.

4. **`/ms-bead-impl` runs the ready-check as an UNCONDITIONAL dispatch ingress and stages the semantic Phase 0.** (a) The `mindspec bead ready-check <bead-id>` re-check MUST run on **every** dispatch path as an ingress step **before** the Phase-A-skip decision — so a supplied `prompt-path` (which today skips Phase A, `SKILL.md:23`) and the documented manual `bd update`/`git worktree add` fallback cannot bypass it. This covers the stale-claim window: a bead claimed before this spec ships, or a plan/spec edited after the claim, is caught at dispatch. On FAIL the dispatch STOPS (no prompt staged, no dispatch) UNLESS the bead carries the R3 durable override marker for the failing signals, in which case dispatch proceeds with a warning naming the overridden signals — so a deliberately force-claimed bead has a coherent path to implementation rather than dead-ending. (b) The staged-prompt skeleton gains a mandatory **`## Phase 0 — readiness review (before any edit)`** section instructing the subagent to evaluate five semantic signals against the quoted spec/plan material (and any `## Readiness Clarification` entries surfaced into the prompt per R8) before touching any file: **SR-1** the plan steps + ACs are implementable without inventing unstated behavior; **SR-2** each claimed AC is decidable (a concrete pass/fail check exists or is directly constructible); **SR-3** every helper the prompt says to reuse actually exists at the stated import path with the stated shape; **SR-4** the bead does not contradict the spec or a sibling bead's landed work; **SR-5** no ambiguity remains that would force the subagent to choose between materially different implementations. If any signal fails, the subagent makes **zero commits** and returns a report whose first line is exactly `NOT READY: <bead-id>` followed by reasons numbered by **ordinal** (1, 2, …; each unique within the report), each tagged with its signal ID and quoting the offending or missing span verbatim (the grill's anchoring discipline) plus the concrete unblocking question. The Phase 0 block MUST also carry the clarification-handling rule (R8): on a re-dispatch, any cited NOT-READY reason ordinal lacking a mapped, span-grounded clarification entry MUST be re-reported NOT READY, and the reviewer judges whether the clarification RESOLVES the ambiguity against a cited authoritative span — a bare "it's ready" never suffices. If all pass, Phase 0 costs one line ("Phase 0: READY") and coding proceeds.
   *Falsified if*: the shipped `ms-bead-impl` SKILL.md runs the ready-check inside a phase that a `prompt-path` or manual-fallback dispatch can skip (it must be unconditional ingress); or dispatch proceeds on a mechanical FAIL with no override marker; or the prompt skeleton lacks any SR signal ID, the zero-commit rule, the `NOT READY: <bead-id>` first-line + ordinal-numbered verbatim-span shape, or the clarification-handling rule (pinned by AC-5/AC-6).

5. **`/ms-bead-cycle` routes NOT READY as its own outcome — pre-damage triage, not an impl failure.** A subagent return whose first line is `NOT READY: <bead-id>` is handled distinctly from every existing outcome: **no panel round is consumed** (nothing was implemented), and the outcome does **not** count toward `loop.halt.max_consecutive_impl_failures` (that brake, `internal/config/config.go:382`, stops *repeated failed implementations* — post-damage; NOT READY is the pre-damage refusal that prevents them, and conflating the two would punish the gate for working). On a NOT READY the cycle chooses exactly one of two dispositions: **ACCEPT** (the default) — **halt the bead and surface the reasons** with the routing instruction (revise the plan/spec section the reasons quote, then re-dispatch); or **DISAGREE/clarify** — the bounded R8 loop. Either way it **never routes a NOT READY into `/ms-bead-fix`** (there is no implementation to fix), and the bead worktree remains intact (zero commits). Autopilot treats an ACCEPTed NOT READY as the existing bead-level halt class.
   *Falsified if*: the shipped `ms-bead-cycle` SKILL.md routes a NOT-READY return into the panel or the fix loop, counts it toward the impl-failure halt counter, or lacks the ACCEPT halt-and-surface routing (pinned by AC-7's content test).

6. **The gate is falsifiable in both directions via a committed readiness fixture pair modeled on real shipped beads.** The repo carries, under the test tree, (a) a NEGATIVE fixture with exactly one planted defect per mechanical signal: a placeholder-only AC block (MF-1); a claimed `AC-99` absent from spec.md AND a prefix-dangling `AC-1` whose spec.md contains only `AC-19` (MF-2, both false-negative-and-false-positive classes); a dependency bead closed-but-unmerged with its branch **present** (MF-3, the 2u0u split); and a genuine blocking marker — a `TBD` token and/or an unchecked item in a `**Blocking Questions**` region (MF-4). (b) a POSITIVE fixture: a fully-specified bead section modeled faithfully on spec 123 Bead 1 that MUST carry, and still PASS, the benign features that distinguish a real shipped section from a sterile ideal — (i) an unchecked `**Verification**` checklist in scaffold shape; (ii) at least one AC entry in `- [ ] AC-n — …` checkbox form; (iii) a bare-base `R`/`AC` claim whose spec.md counterpart is sub-lettered only (exercising the MF-2 base-resolution rule); (iv) a prose citation of another spec's pattern (e.g. "the spec 123 AC-17 pattern") that must NOT be harvested as a dangling claim; with named files, resolvable ACs, and a merged (branch-deleted) dependency. CI pins the NEGATIVE fixture to FAIL naming all four planted signals and the POSITIVE fixture to PASS with zero findings — a sanitized positive fixture (stripped of the benign features) is itself a falsification.
   *Falsified if*: either fixture is absent; or the NEGATIVE fixture passes while any planted defect goes unflagged; or the POSITIVE fixture passes while the verb reports any finding against it; or the POSITIVE fixture lacks any of the four enumerated benign features (the sanitized-fixture trapdoor).

7. **Surface inflation is paid for, and prose cannot drift from the binary.** (a) The spec-122 ceremony non-inflation baseline (`cmd/mindspec/ceremony_guard_test.go`) is updated in the same bead that adds each new surface — `bead ready-check`, `bead clarify`, and `--allow-not-ready` — the guard trips red on each and the baseline update is the deliberate, reviewed acknowledgment; the guard test itself is never weakened or deleted (guard-weakening detection is a review-time obligation of the bead panel — AC-9(i) provides the git-diff evidence). (b) Skill↔binary contract pins (the spec 123 AC-17 pattern): content tests assert the shipped `ms-bead-impl` SKILL.md contains the unconditional-ingress ready-check invocation, the Phase 0 block with the five SR IDs, the `NOT READY:` ordinal-report contract, AND the clarification-handling rule; and the shipped `ms-bead-cycle` SKILL.md contains the NOT-READY routing of R5 and the R8 loop contract — so a future skill edit cannot silently detach the prose layer from the verb.
   *Falsified if*: any new surface lands without a same-bead ceremony-baseline update; or the `ms-bead-impl` clarification-handling rule (R8 anti-browbeat) can be removed from the shipped prompt with every pinned test still green; or either skill's mandated content can be removed without a test going red.

8. **The semantic clarification loop — a bounded, layer-scoped, GROUNDED, audited DISAGREE disposition (not a bypass).** When Phase 0 returns a semantic `NOT READY: <bead-id>` and the orchestrator judges the concern resolvable, `/ms-bead-cycle` MAY answer it **once per bead** instead of halting:
   - **(a) Carrier OFF the mechanical scan surface.** The clarification is written to a **dedicated bd metadata key** (via a thin `mindspec bead clarify <id>` verb — or equivalent single new surface — wrapping the existing `internal/bead` `MergeMetadata` helper; a skill LLM cannot call Go helpers and raw `bd update --metadata` is ADR-0035-banned). The key is one MF-2/MF-4 **never scan** (they read the bd description and plan section, not this key), so a clarification whose prose legitimately contains an `AC-<n>` token, `TBD`, or `- [ ]` cannot flip a mechanical PASS→FAIL — invariance holds by construction, both directions.
   - **(b) Grounded, reason-keyed entries.** The record is an append-only **readiness attempt**: the original NOT-READY report (each reason's **ordinal**, verbatim reason/question, and signal tag), plus one clarification entry per cited ordinal, each carrying `{ordinal, verbatim reason, concrete answer, source span in spec.md / plan.md / landed code}`. `bead clarify` REJECTS an entry that names an ordinal absent from the recorded report or that omits a source span — this is a **presence** check only (the entry must cite a span); whether the cited span actually SUPPORTS the answer is a semantic judgment left to the fresh Phase-0 subagent (ADR-0040: the binary validates structure, the model judges meaning), not verified by the verb. A clarification may DISAMBIGUATE existing authority but may not CREATE new normative behavior; when no authoritative span supports the answer, the correct disposition is ACCEPT (revise the plan/spec), not clarify.
   - **(c) Surfaced into re-dispatch, and JUDGED not just mapped.** `/ms-bead-impl`'s dispatch ingress reads the attempt record and injects its clarification entries (keyed by ordinal) into the re-dispatched prompt's Phase 0 section. The fresh Phase-0 subagent re-reports NOT READY for any cited ordinal lacking a mapped, span-grounded entry, AND judges whether each clarification actually RESOLVES the ambiguity against its cited span — a syntactically reason-keyed but unsupported "ignore it / it's ready" does not flip the verdict.
   - **(d) Bounded — one round PER BEAD, then forced revision.** The cap is categorical and durable: the presence of ANY prior readiness-attempt record on the bead means the single clarification round is already consumed — the next NOT READY (regardless of whether it repeats an addressed reason or raises a NEW one) MUST escalate to the ACCEPT disposition (plan/spec revision). No second `clarify` write is permitted; `bead clarify` refuses when an attempt record already exists (recovery: escalate to revision). This survives orchestrator restart / context loss because the marker is durable bd state, not transcript memory. The cap assumes a **single writer per bead** — the honest, grounded assumption, since `/ms-bead-cycle` is serial per bead by design (it claims and drives exactly one bead at a time; `ms-bead-cycle` SKILL.md "Don't claim multiple beads at once"). Two concurrent orchestrators racing `MergeMetadata` on one bead (an ordinary lost-update window) is out of scope for the cap's guarantee and is not defended against here; the serial-per-bead cycle is the enforced invariant that makes the single-writer assumption hold in practice.
   - **(e) Layer boundary + DERIVED audit resolution.** A clarification affects only the semantic Phase-0 SR signals; the mechanical floor (`ready-check` MF-1..MF-4, run by `next` and by the R4 dispatch ingress) is evaluated as if the record were absent — a bead failing any mechanical signal stays NOT READY regardless of any clarification (a missing AC, dangling `AC-n`, or unmerged dependency never passes via clarification; the blunt override for a known-acceptable mechanical FAIL is `--allow-not-ready`, recorded as a bypass). The append-only **attempt record** (original ordinal-keyed report + the grounded clarification entries) is the single durable audit anchor and is written exactly once — consistent with the (d) one-write cap and the AC-9 two-verbs accounting. The **terminal disposition is DERIVED, not a second write**: READY-after-clarification = a successful re-dispatch occurring after the record exists; escalated-to-revision = the forced-revision path a repeat NOT-READY takes (the cycle transcript + the bead's subsequent state show which). A fresh `bd show` reveals the NOT-READY → clarification trail durably; the outcome is inferable from it plus the subsequent re-review, so no additional write surface (a `--finalize` verb or a two-phase idempotent `clarify`) is introduced — the concern is resolved-with-evidence or acted-on, never silently bypassed (the "findings never out-voted" discipline applied to the readiness verdict).
   *Falsified if*: a clarification record changes any mechanical signal's verdict in either direction (fail-stays-fail AND pass-stays-pass are both pinned); or Phase-0 re-review accepts a reason-keyed but span-ungrounded / unsupported clarification, or flips READY on an entry that creates new normative behavior; or a second clarification round is authored after any first-round NOT-READY re-return (same OR new reason); or the attempt record — original report and clarification entries — does not survive worktree teardown / a fresh `bd show`; or a second write surface for the terminal disposition is added (derive-don't-write is the contract); or `bead clarify` is absent from the AC-9 ceremony baseline.

9. **ADR-0041 amendment.** ADR-0041 §1 MUST be amended with the fourth-verb clause named in ADR Touchpoints, carrying the distinctive anchor phrase **"preflight-leg-only addition"** and naming `mindspec next`: `next` adopts the gate-before-mutate **preflight leg** (readiness refusal before any mutation; byte-identical-on-refusal; the distinct `--allow-not-ready` override). The amendment is a SCOPE-DEFERRAL, not a governance certification: it adds only the preflight leg and makes no claim about `next`'s existing success-path claim/branch/worktree mutation chain, which retains whatever commit/forward-reconcile contract it already carries (this spec neither grants nor removes it); the "three verbs" wording is reconciled to name `next` as a preflight-leg fourth member. The amendment is pre-drafted at plan time and lands in the same bead as the `next`-gate code, so the ADR-divergence gate sees the declared touchpoint.
   *Falsified if*: the code landing the `next` readiness gate ships without an ADR-0041 amendment carrying the "preflight-leg-only addition" fourth-verb clause naming `next`, or the amendment certifies `next`'s success path as commit/reconcile-exempt, or the code cites a clause the amendment does not contain — operationally pinned by AC-16.

- Any bd/beads binary or schema change; dependency edges, statuses, and metadata are read and written via the existing `internal/bead` helpers only (the readiness-attempt/override records are advisory metadata, ADR-0023-respecting).

Spec 123's rounds 2–3 are existing evidence that fix-round churn can hurt; gating it is deferred here deliberately to keep this spec at the claim/dispatch layer, not because it never hurts.
- MF-3 dependency-artifact contracts — `key_file_paths` is context, not a postcondition; MF-3 does not assert declared files exist at HEAD (helper usability is SR-3).
- Plan-approve-time validation (`internal/validate/plan.go`) — already covers the approval-time structural floor; this spec adds the claim/dispatch-time temporal and cross-document layer, not a second approval gate.
- Auto-remediation: NOT READY never auto-dispatches a plan-refinement subagent; the R8 clarification is orchestrator-authored, not machine-generated (see Non-Goals).

## Non-Goals

- **Not Spec 6's runtime brakes.** `max-rounds`, `panel_deadlock_rounds`, and `max_consecutive_impl_failures` stop the loop after damage; this gate refuses before it. The two mechanisms stay independent: this spec neither modifies the halt skeleton nor feeds it.
- **Not a semantic-conformance gate.** The binary never judges prose quality, spec fitness, or design merit — mechanical presence/resolution/merged-ness only. All judgment lives in the subagent's Phase 0, whose output is structured but whose evaluation is explicitly not deterministic and is never treated as binary truth.
- **No new merge or approval authority.** READY/NOT READY influences claiming and dispatching only. It appears nowhere in `mindspec complete`, panel tally, or any lifecycle approval; the readiness-attempt record is advisory, never lifecycle state (ADR-0023).
- **No automatic plan-fixer.** Deciding whether to ACCEPT a NOT READY (revise the plan) or DISAGREE (author a grounded clarification and re-dispatch, R8) is the orchestrator's judgment; the binary never auto-rewrites the plan or auto-generates the clarification.
- **Bounded, not unbounded, retry.** The R8 clarification loop is capped at ONE round per bead, categorically (any prior attempt record forces the next NOT READY to revision — new reasons included). There is no infinite clarify↔re-dispatch loop and no mechanism to wear the gate down by re-asserting readiness.
- **Clarification is not a mechanical-floor bypass.** A `## Readiness Clarification` resolves only semantic (Phase-0 SR) concerns; MF-1..MF-4 ignore it entirely (dedicated key off the scan surface). A genuinely under-specified bead — missing AC, dangling dep, unmerged/absent dependency, blocking marker — stays NOT READY regardless. The blunt override for a known-acceptable mechanical FAIL is `--allow-not-ready`, recorded as a bypass, never a resolution.
- **No readiness scoring, thresholds, or config.** Signals are individually pass/fail; there is no aggregate score, no tunable strictness knob, no config key. A knob would be ceremony's front door.
