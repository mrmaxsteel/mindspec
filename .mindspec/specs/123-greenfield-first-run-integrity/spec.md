# Spec 123-greenfield-first-run-integrity: Greenfield First Run Integrity

## Goal

Make the first hour of MindSpec in a fresh consuming repo **correct, convergent, and self-checking**. Five shipped verbs ŌĆö `init`, `setup claude|codex|copilot`, `domain add`, `adr create`, `panel create` ŌĆö plus the `models:` config key each break, mislead, or dead-end a greenfield consumer today, and every break is a filed, reproduced issue against v0.12.0: **(1)** `init` promises "'mindspec doctor' passes and the spec-driven workflow is immediately usable" (`cmd/mindspec/init.go:14-16`) but does not scaffold `.mindspec/context-map.md`, so the very first `mindspec domain add` errors after writing the domain docs ŌĆö a silent partial state (domain scaffolded, unmapped) that a re-run refuses to repair and `doctor` never flags (GH #207); **(2)** the runtime-state files `doctor` itself insists must stay untracked (`.mindspec/session.json`, `.mindspec/focus` ŌĆö `internal/doctor/git.go:17-20`) are never gitignored when the repo already has a `.gitignore`, so a routine `git add .mindspec/` commits them and only THEN does doctor turn red (GH #208); **(3)** `adr create` emits bare `ADR-NNNN.md` filenames that contradict the framework's own ADR convention (this repo's ADR-0025+ are all slugged), and a slugged filename makes the full stem the reported ID (GH #206); **(4)** `models:` is a declared-but-unenforced config key with no schema doc, no populate prompt, and no doctor nudge ŌĆö unlike its sibling `source_globs:`, which has all three (GH #210); **(5)** `init` and `setup codex` write mindspec-THE-FRAMEWORK's own identity and Go build commands (`make build` / `make test`, "# AGENTS.md ŌĆö MindSpec Project") into the CONSUMING repo's managed `AGENTS.md` block, so every agent in every consuming repo reads actively wrong build guidance with no supported override (GH #211); and **(6)** the shipped `ms-panel-run` skill documents an ad-hoc/no-owning-spec panel path (`.mindspec/reviews/<slug>/`) that `panel create` cannot scaffold ŌĆö a skillŌåöbinary contract drift that also blocks the deterministic tally for exactly the design-review panels the skill invites (GH #209).

Target user outcome: an operator (or agent) in an empty directory runs `mindspec init`, `mindspec setup <agent>`, `mindspec domain add <name>`, and `mindspec adr create "<title>"` in any order the docs suggest, and every command succeeds, every artifact matches the framework's own published conventions, nothing framework-specific leaks into the consumer's agent-facing docs, `git add .mindspec/` cannot capture runtime state, and `mindspec doctor` detects ŌĆö with a named recovery ŌĆö every one of these states instead of passing silently.

## Background

All citations verified in this worktree (fork of `main` at `66c2946e`, post-spec-117).

### #207 ŌĆö `init` does not scaffold `context-map.md`; the first `domain add` half-fails; doctor is silent

- `mindspec init` runs `bootstrap.Run` (`cmd/mindspec/init.go:24-30`). The manifest (`internal/bootstrap/bootstrap.go:218-235`) creates `.mindspec/`, `.mindspec/domains/`, `.mindspec/specs/`, `AGENTS.md`, `CLAUDE.md`, `.github/copilot-instructions.md`, and `.gitignore` ŌĆö and nothing else. There is no `.mindspec/context-map.md` item.
- `domain add` (`internal/domain/scaffold.go:31-82`) refuses when the domain dir exists (`scaffold.go:40-42`), then writes the four template docs plus `OWNERSHIP.yaml`, and only THEN calls `appendContextMap` (`scaffold.go:75`), which `os.ReadFile`s `workspace.ContextMapPath(root)` (`scaffold.go:96-100`; path resolution `internal/workspace/workspace.go:506-508`) and fails with "reading context map: ŌĆ” no such file or directory" when the file is absent. Net: on a fresh `init`, the FIRST `domain add` scaffolds the domain then errors; the domain has no context-map entry; and because the dir now exists, a re-run hits the `scaffold.go:42` refusal and can never backfill the entry. The insertion logic itself (`scaffold.go:118-150`) tolerates an empty file (no `## Bounded Contexts` heading ŌåÆ append at end), so the operator workaround is `touch .mindspec/context-map.md` ŌĆö plus hand-writing the first domain's entry.
- `doctor` checks only the `domains/` and `specs/` directories for existence (`internal/doctor/docs.go:19-39`). No check asserts `context-map.md` exists or that scaffolded domains have entries; the link lane merely scans it IF present (`internal/doctor/links.go:49`).

### #208 ŌĆö runtime state is one `git add .mindspec/` from being committed

- Bootstrap's `.gitignore` manifest item (`bootstrap.go:232`, content `starterGitignore` at `bootstrap.go:267-271`: `.mindspec/session.json` + `.mindspec/focus`) is create-only: the manifest write loop appends to an existing file only when `item.appendBlock != ""` (`bootstrap.go:150-160`), and the `.gitignore` item sets no `appendBlock` ŌĆö so any repo that already has a `.gitignore` (i.e. nearly every real repo) gets `Skipped` and no ignore rules. `setup claude|codex|copilot` (`internal/setup/claude.go:148`, `codex.go:12`, `copilot.go:12`) never touch `.gitignore` at all, and `setup` is the onboarding verb for repos that never ran the greenfield-only `init`.
- `doctor`'s runtime-file check (`internal/doctor/git.go:24-46`) fires only when the file is already TRACKED (`Error`, with a `--fix` that untracks and appends the ignore entry, `git.go:50-83`). An unignored-but-untracked runtime file ŌĆö the pre-accident state the check exists to prevent ŌĆö produces `OK: not tracked by git`. The fix machinery (`appendGitignoreEntry`, `git.go:66-83`) already exists; nothing invokes it proactively.

### #206 ŌĆö `adr create` contradicts the framework's own ADR filename convention; slugged stems become IDs

- `create` writes `filepath.Join(adrDir, fmt.Sprintf("ADR-%s.md", id))` (`internal/adr/create.go:139`) ŌĆö a bare filename, no slug, no `--slug` flag. Meanwhile mindspec's own ADRs since ADR-0025 are all `ADR-NNNN-kebab-slug.md` (see `.mindspec/adr/`), and `idvalidate.ADRID` already accepts the slugged form (`internal/idvalidate/ids.go:33`: `^ADR-[0-9]{4,}(?:-[a-z0-9]+(?:-[a-z0-9]+)*)?$`).
- `ParseADR` derives the ID as the full filename stem (`internal/adr/parse.go:47`: `id := strings.TrimSuffix(base, ".md")`), so a slugged file reports `ID: ADR-0001-integrate-at-contracts` in `list`/`show` ŌĆö a long stem-ID that ADR-citation surfaces and `--supersedes` references (validated as `ADR-NNNN[-slug]` but resolved by exact join, `internal/workspace/workspace.go:578-583`) do not expect.
- The numbering side is already slug-aware ŌĆö `maxADRNum` counts the leading digit run of both bare and slugged stems (`parse.go:260-300`) ŌĆö so this spec's change is filename emission + ID normalization + resolution, not numbering.

### #210 ŌĆö `models:` is declared-but-unenforced with zero agent-facing guidance

- `Config.Models` is a free-form `map[string]string` declared inert by spec 109 (`internal/config/config.go:54-66`), rendered by `mindspec config` with the annotation `declared, not yet enforced` (`cmd/mindspec/config.go:22`, `:204-215`, plus per-model advisory warnings via `renderKnownModelWarnings`, `:344`).
- Contrast `source_globs:`, which has the full ZFC guidance stack from spec 091 / ADR-0036: a literal commented schema block scaffolded by `doctor --fix` (`internal/doctor/config.go:18-29` `sourceGlobsBlock`; three-state byte-preserving scaffolder `scaffoldSourceGlobs`, `config.go:98-145`), a prompt emitter `mindspec source populate` (`cmd/mindspec/source.go:23-45`), and a `missing-source-globs` doctor Warn when empty (`checkSourceGlobs`, `config.go:63-96`). `models:` has none of these: no schema doc, no populate prompt, no doctor nudge ŌĆö so an agent captures the model protocol out-of-band (the filed failure mode) and never learns the key exists or whether populating it does anything.

### #211 ŌĆö `init`/`setup codex` write mindspec's own identity and build into consumers

- `init`'s `starterAgentsMD` (`bootstrap.go:275`) is titled `# AGENTS.md ŌĆö MindSpec Project` and hardcodes a `## Build & Test` section with `make build` / `make test` (`bootstrap.go:287-288`) ŌĆö mindspec's own Go build. The append-variant block (`appendAgentsBlock`, `bootstrap.go:353-365`) hardcodes the same commands (`:364-365`).
- `setup codex` writes `AGENTS.md` via `ensureAgentsMD` (`internal/setup/codex.go:16`, `:49-50`); its managed block (`agentsMDManagedBlock`, `codex.go:53-104`) hardcodes the same `make build` / `make test` (`codex.go:62-63`). A consuming repo with no Makefile and no go.mod receives agent instructions to run commands that do not exist. `setup claude` and `setup copilot` are clean of the build hardcode ŌĆö their managed docs delegate to AGENTS.md (`claude.go:789`, `copilot.go:140`) ŌĆö but `setup codex` and `init` both plant it.
- There is no config key, flag, or prompt to declare the consumer's real build/test (confirmed against `internal/config/config.go` ŌĆö no such field), and hand-editing the managed block is unsafe by design: `ensureManagedDoc` REPLACES the BEGIN/END block wholesale on the next `setup` run (`internal/setup/claude.go:573-582`). So a repo-specific fact is baked to mindspec's own value with no override ŌĆö the same declared-nowhere class as #210, and precisely the drift ADR-0040's layering doctrine exists to prevent (a repo fact belongs in L2 declared config, not hardcoded prose).

### #209 ŌĆö the ad-hoc panel path is documented but unreachable

- The shipped `ms-panel-run` skill (installed into consumers via `pluginmindspec.SkillFiles()`, `internal/setup/claude.go:669`) documents ad-hoc reviews ŌĆö "no owning spec ŌĆ” a repo-wide review, a pre-release review, a design/proposal review, or a standalone bug-fix" ŌĆö at `<repo>/.mindspec/reviews/<panel-slug>/`, explicitly NOT scanned by the `complete` gate (`plugins/mindspec/skills/ms-panel-run/SKILL.md:43-52`).
- But `panel create` hard-requires `--spec` (`cmd/mindspec/panel.go:113-115`) resolving to an existing `<NNN>-<slug>` spec dir (`panelDirFor` ŌåÆ `workspace.SpecDir`, `panel.go:352-366`), even with `--gate adhoc` ŌĆö a gate key that exists in the enum (`--gate` flag `panel.go:286`; `internal/panel/disposition.go:93` `CanonicalGateKeys` includes `"adhoc"`; the disposition store already aggregates an `adhoc` gate). `panelDirFor` has NO ad-hoc branch, so the documented `.mindspec/reviews/` location is unreachable, and with it the deterministic `panel tally` (which reads a `panel.json` only `panel create` produces) ŌĆö the consumer must hand-tally exactly the design reviews the skill invites (the workaround pain GH #212 independently reports).

### Related, deliberately out (see Non-Goals)

GH #212 asks for spec/plan-DOCUMENT adversarial panels, `panel_gate` coverage of `spec_approve`/`plan_approve`, and escalation cues in Spec/Plan-mode `instruct`. That is panel-GOVERNANCE design (gate authority, lens design, escalation policy), not first-run scaffolding integrity; folding it here would balloon this spec past its bead budget and couple a fast bug-fix spec to an open governance debate. This spec ships #212's mechanical substrate ŌĆö Requirement 8 makes ad-hoc (including design/proposal) panels creatable and deterministically talliable ŌĆö and leaves the governance layer to its own spec (candidate: the governed-autonomy kernel roadmap item).

## Requirements

Each requirement carries a hard falsifier. "Greenfield fixture" below means: an empty directory, `git init`, then the named mindspec verbs ŌĆö no manual file creation.

1. **`init` scaffolds a first-run-complete workspace, including the context map (#207).** The `bootstrap.Run` manifest MUST additionally scaffold `.mindspec/context-map.md` ŌĆö a skeleton with a title and a `## Bounded Contexts` section followed by a `---` separator, so `appendContextMap`'s existing insertion scan (`scaffold.go:118-150`) inserts entries at the intended place, not the file's tail. The item follows the manifest's existing additive discipline: never overwrites an existing file. This restores `init`'s own stated contract (`cmd/mindspec/init.go:14-16`) that the workflow is immediately usable.
   *Falsified if*: a greenfield fixture running `mindspec init` then `mindspec domain add alpha` still errors, or produces a context map without an `### Alpha` entry under `## Bounded Contexts`, or `init` overwrites a pre-existing `context-map.md`.

2. **`domain add` converges from every partial state (#207).** (a) When `context-map.md` is absent, `appendContextMap` MUST create the Requirement-1 skeleton before appending ŌĆö `domain add` succeeds even in a pre-123-initialized repo. (b) The already-exists refusal (`scaffold.go:40-42`) MUST become a convergence check: when the domain dir exists but its context-map entry is MISSING, `domain add <name>` backfills the entry (and any missing standard file: the four templates + `OWNERSHIP.yaml`, each written only if absent ŌĆö existing files are never overwritten) and succeeds; only when the domain is fully scaffolded AND mapped does it refuse with "already exists". (c) The write order or failure handling MUST guarantee no terminal partial state: any failure after the dir is created leaves a state a bare re-run of the same command repairs.
   *Falsified if*: any sequence of `domain add <name>` invocations (including against the #207 legacy state: dir present, entry absent) reaches a state where the domain dir exists, its context-map entry is missing, and re-running `mindspec domain add <name>` does not produce the entry; or a re-run duplicates an existing entry; or a re-run overwrites operator-edited domain docs.

3. **`doctor` detects the missing/incomplete context map (#207).** Two new checks in the docs lane: (a) **missing-context-map** ŌĆö `context-map.md` absent at the layout-resolved path ŌåÆ `Missing`, with a `--fix` that scaffolds the Requirement-1 skeleton (mechanical, ZFC-safe: structure only, no invented content); (b) **unmapped-domain** ŌĆö a directory under `domains/` whose name has no corresponding entry heading in `context-map.md` ŌåÆ `Warn` naming each unmapped domain with the recovery line `mindspec domain add <name>` (the Requirement-2 backfill). Detection uses the same entry-shape `appendContextMap` writes (the `### <Title>` heading for the domain), so the check and the writer cannot disagree about what "mapped" means (single shared helper, not two parsers).
   *Falsified if*: `doctor` exits green on a repo with domain dirs and no `context-map.md`, or on a repo with a scaffolded-but-unmapped domain; or the two checks' notion of "has an entry" is a second independent parse that can drift from `appendContextMap`'s emission.

4. **Runtime state is gitignored by scaffolding, and doctor checks ignore-ness, not just tracked-ness (#208).** (a) `init` MUST ensure the two runtime entries (`.mindspec/session.json`, `.mindspec/focus`) are present in the root `.gitignore` even when the file already exists ŌĆö an idempotent, entry-granular append (exact-line presence check, the `appendGitignoreEntry` discipline at `git.go:66-83`) that never reorders or rewrites existing content; the greenfield create-from-scratch path (`bootstrap.go:232`) is unchanged. (b) `setup claude`, `setup codex`, and `setup copilot` MUST each ensure the same entries (setup is the onboarding verb for repos that never ran the greenfield-only `init`). (c) `doctor` MUST add a proactive check per runtime file: not tracked AND `git check-ignore` misses ŌåÆ `Warn` ("runtime file not gitignored ŌĆö one `git add .mindspec/` from being committed") with a `--fix` that appends the entry (reusing `appendGitignoreEntry`); ignored ŌåÆ `OK`; the existing tracked ŌåÆ `Error` + untrack `--fix` (`git.go:24-46`) is unchanged and takes precedence.
   *Falsified if*: after `mindspec init` (or any one `setup` verb) in a repo with a pre-existing `.gitignore`, `git check-ignore .mindspec/session.json` or `git check-ignore .mindspec/focus` fails to match; or running the verb twice yields a different `.gitignore` than running it once; or `doctor` reports all-OK on a repo where a runtime file is unignored-untracked.

5. **`adr create` emits slugged filenames; every surface reports and accepts the canonical `ADR-NNNN` ID (#206).** (a) `create` MUST derive a kebab slug from the title (lowercase; non-alphanumeric runs collapse to single hyphens; leading/trailing hyphens trimmed; length-capped ŌĆö cap value plan-level) and write `ADR-NNNN-<slug>.md` (`create.go:139` today); a `--slug` flag overrides the derivation; both derived and supplied slugs MUST yield a filename stem that passes `idvalidate.ADRID` (`ids.go:33`), and a title that derives an empty slug falls back to the bare form rather than an invalid name. (b) `ParseADR` MUST derive `ADR.ID` as the canonical `ADR-<digits>` prefix of the stem (`parse.go:47` today), never the full slugged stem ŌĆö `list`/`show` report `ADR-0001` for `ADR-0001-integrate-at-contracts.md`. (c) IDŌåÆfile resolution (`show`, `--supersedes` via `workspace.ADRFilePath` and its callers) MUST resolve a canonical `ADR-NNNN` to whichever file carries that number ŌĆö bare or slugged ŌĆö and MUST also accept the full slugged form; when BOTH a bare and a slugged file share one number, resolution errors naming both paths (never a silent pick). (d) Backward compatibility: existing bare files keep their IDs and behavior; number allocation (`maxADRNum`, `parse.go:260-300`) is already slug-aware and MUST remain covered against mixed directories.
   *Falsified if*: `adr create "<multi-word title>"` still writes a bare `ADR-NNNN.md`; or `adr show ADR-NNNN` against a slugged file reports an ID other than `ADR-NNNN` or fails to resolve; or `adr create --supersedes ADR-NNNN` cannot resolve a slugged predecessor; or a bare/slugged number collision resolves silently.

6. **`models:` reaches guidance parity with `source_globs:` ŌĆö schema doc, populate prompt, doctor nudge, honest enforcement status (#210).** (a) A literal commented schema block for `models:` (mirroring `sourceGlobsBlock`, `internal/doctor/config.go:18-29`) documenting: the free-form phaseŌåÆmodel-id map shape (`config.go:63`), the canonical advisory phase keys (`authoring`, `implementation`, `review` ŌĆö documented vocabulary, not an enforced enum), and ŌĆö verbatim honesty ŌĆö its enforcement status: today the key is declared-and-inert (`cmd/mindspec/config.go:22` `inertAnnotation`), nothing in the binary changes behavior based on it, and the authoritative consumers of the model protocol remain the orchestration skills; the block says exactly that and names wiring enforcement as a follow-up. (b) `mindspec models populate` ŌĆö a ZFC prompt emitter mirroring `mindspec source populate` (`cmd/mindspec/source.go:23-45`): prints an agent prompt to declare the per-phase protocol in `.mindspec/config.yaml`; writes nothing. (c) A `missing-models` doctor `Warn` when `len(cfg.Models) == 0` (mirroring `checkSourceGlobs`, `config.go:63-96`), disclosing the inert status, hinting `mindspec models populate`, with a `--fix` that scaffolds the schema block three-state byte-preservingly (file absent / key absent / key present ŌĆö the `scaffoldSourceGlobs` contract, `config.go:98-145`). (d) `mindspec config` output for `models:` keeps the inert annotation until an enforcement spec removes it.
   *Falsified if*: an empty-`models:` repo produces no doctor nudge; or `mindspec models populate` is absent or writes files; or the scaffolded block omits schema or overstates enforcement (claims any binary behavior reads the key); or the `--fix` scaffold rewrites operator-authored config bytes.

7. **Managed/scaffolded content carries the CONSUMER's identity, never the framework's (#211).** (a) All content mindspec writes into a consuming repo's agent docs ŌĆö `init`'s `starterAgentsMD` (`bootstrap.go:275-351`) and `appendAgentsBlock` (`:353-`), `setup codex`'s `agentsMDManagedBlock` (`codex.go:53-104`), and every other managed block ŌĆö MUST contain NO mindspec-repo-specific facts: no `make build` / `make test` (`bootstrap.go:287-288`, `:364-365`, `codex.go:62-63`), no "# AGENTS.md ŌĆö MindSpec Project" title (`bootstrap.go:275`); identity framing is "This project uses MindSpec" (already the body phrasing), with a neutral `# AGENTS.md` title. (b) A new declared L2 config surface for the consumer's build/test guidance (exact key shape plan-level; rendered by `mindspec config`): when POPULATED, `init`/`setup` render its commands as the managed block's Build & Test section, and re-running `setup` (whose wholesale block replacement, `claude.go:573-582`, is retained) re-renders from config ŌĆö the operator's declaration survives every refresh; when UNSET, the Build & Test section is OMITTED entirely (never a placeholder that reads as runnable). (c) The unset state gets the ADR-0036 stack: a doctor `Warn` nudging the declaration with a ZFC populate-prompt path (analogous to Requirement 6's; command naming plan-level), so a consumer learns the declared home exists. (d) `mindspec setup --check` and re-runs stay idempotent under both states.
   *Falsified if*: a greenfield `init` or `setup codex` in a repo with no Makefile produces an `AGENTS.md` whose managed content tells an agent to run `make build`/`make test` or titles the repo a "MindSpec Project"; or a populated build declaration is lost by a subsequent `setup` refresh; or the unset state renders framework or placeholder build commands; or doctor gives no cue that the declaration exists and is empty.

8. **`panel create` can scaffold the ad-hoc panel the shipped skill documents (#209).** (a) `mindspec panel create <slug> --gate adhoc --target <ref>` MUST succeed WITHOUT `--spec`, writing the panel dir + `panel.json` to `<repo>/.mindspec/reviews/<slug>/` ŌĆö exactly the location `ms-panel-run` documents (`plugins/mindspec/skills/ms-panel-run/SKILL.md:43-52`) ŌĆö via a new ad-hoc branch in `panelDirFor` (`panel.go:352-366`); the existing slug/target/round validation and `panel.json` stamping (expected reviewers, threshold, gate `adhoc`) apply unchanged. (b) `--spec` remains REQUIRED for every non-adhoc gate (`panel.go:113-115` retained); supplying `--spec` WITH `--gate adhoc` is refused with a recovery line naming both valid forms (an ad-hoc panel has no owning spec by definition ŌĆö a silent ignore would misfile the panel). (c) `panel tally` (and `panel show`-class readers) MUST operate on the ad-hoc dir, restoring the deterministic tally for design/proposal reviews. (d) Ad-hoc panels stay OUTSIDE the lifecycle gate: `mindspec complete`'s panel scanning never consults `.mindspec/reviews/` (the skill's stated contract), and the `ms-panel-run` SKILL.md ad-hoc note is updated in the same change to state the now-real `panel create --gate adhoc` invocation instead of a path the binary can't produce.
   *Falsified if*: the #209 repro (`mindspec panel create adr-review --gate adhoc --target "commit <sha>"`, no `--spec`) still errors; or the panel lands anywhere other than `.mindspec/reviews/<slug>/`; or `panel tally` cannot read it; or an ad-hoc `panel.json` influences any `mindspec complete` gate decision; or the shipped skill still documents an invocation the binary rejects.

9. **ADR-0040 amendment.** ADR-0040 MUST be amended with the consumer-identity clause named in ADR Touchpoints (managed/scaffolded consumer content is framework-generic or L2-sourced, never framework-repo-specific; declared-inert keys are surfaced honestly with their guidance stack naming today's authoritative consumer), landing in the same bead as the first code that cites it, so the ADR-divergence gate sees the declared touchpoint.

10. **Regression pinning.** Every failure class fixed here MUST be pinned by an automated test that reproduces the original issue's repro and goes RED if the fix is reverted (per-AC mapping below). The Requirement-3 "mapped" predicate MUST be one shared helper consumed by both `appendContextMap`'s emission side and doctor's detection side, pinned by a test that fails if either side is rewired to a private reimplementation (the established anti-drift pattern).

## Non-Goals

- **No build-system auto-detection.** The framework never guesses the consumer's build/test commands (ZFC, ADR-0036): unset means omitted plus a nudge, never an inferred `npm test`.
- **No new gate authority.** Ad-hoc panels remain stored artifacts: creatable, talliable, and irrelevant to `mindspec complete` and every lifecycle gate. #212's question ŌĆö whether spec/plan documents should be panel-GATED ŌĆö is explicitly not answered here.
- **No model-registry validation of `models:` values** beyond the existing advisory `renderKnownModelWarnings`; values stay free-form strings.
- **No doctor hard-failures for the new advisory states.** Missing context-map is `Missing` (structural, like the sibling dir checks); unmapped-domain, not-ignored, missing-models, and unset-build-guidance are `Warn` ŌĆö first-run nudges, not CI breakers.
- **No managed-block opt-out mechanism.** The wholesale-replacement refresh contract stands; this spec changes what the block CONTAINS (config-sourced, framework-generic), not the marker machinery.
- **No slug backfill for `--supersedes` chains or citation sites** ŌĆö canonical `ADR-NNNN` remains the reference currency everywhere; slugs are filename ergonomics only.
