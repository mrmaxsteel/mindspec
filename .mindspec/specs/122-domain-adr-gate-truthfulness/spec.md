# Spec 122-domain-adr-gate-truthfulness: Domain/ADR Gate Truthfulness

## Goal

Make the three domain/ADR gates — `adr-divergence` (bead/impl-approve time), `doc-sync`'s domain lane, and the plan-time `adr-coverage`/`adr-citations` pair — **truthful in both directions**. Today they lie both ways: they **false-positive on correct inputs** (a correct, present OWNERSHIP.yaml and a genuinely-governing cited ADR still fail the gate, forcing `--override-adr` on every source-touching bead — which is corrosive, because it trains operators to normalize gate bypass until the gate carries zero signal), and they **false-negative on unresolvable inputs at authoring time** (a spec whose `## Impacted Domains` labels can never resolve passes `spec approve` and `validate plan` clean, then fails `adr-divergence` on every single bead, far from the authoring mistake that caused it). Four filed issues and one bead ground the fix: **#178** (Rule-2 bare labels silently kept → per-bead `adr-divergence-unowned`), **`mindspec-6ou2`** (ADR `Domain(s)` literal-string intersection blocks specs whose paths the cited ADR itself declares), **#147** (the original file-path-vs-domain-name mismatch — its resolver core shipped in spec 100, but the full repro chain is still red at the coverage step), **#145** (adr-coverage-missing hard to clear: misleading hint + two since-fixed frictions that need regression pins to close the issue), and **#197** (post-flatten gate hints still print the dead `.mindspec/docs/domains/` path). The outcome is **less gate ceremony, not more**: bad domain labels are rejected EARLY (spec/plan validate) with a fix hint that mechanically works when followed; correct inputs resolve and pass; every remaining hint names a path and a remedy that actually exist. No new gate lanes, no new flags, no new config keys. Semantic ADR conformance (#181) is explicitly deferred to a reviewer lens (Non-Goals).

## Background

**The resolver family, as shipped today (verified in this worktree):**

- **The shared normalizer.** `normalizeImpactedDomains` (`internal/validate/ownership_resolve.go:73`, spec 100 R1) resolves each raw `## Impacted Domains` entry by three rules: **Rule 1** (`:114-117`) — the entry names a domain dir (OWNERSHIP.yaml exists under the layout-aware domains root) → keep verbatim; **Rule 2** (`:119-125`) — a bare token with no `/` that is NOT a domain dir → **keep verbatim, no error** ("legacy named-domain without an on-disk manifest; the coverage gate reports its state"); **Rule 3** (`:127-160`) — a path-like entry → glob-match against every domain's OWNERSHIP `paths:`; exactly one owner → replace with the owner's dir-name, zero owners → hard error (`:154-156`), multi-owner → ambiguity error.
- **Its consumers.** Spec-approve runs it via `checkImpactedDomainsResolutionParity` (`internal/validate/spec.go:71`, func at `:217`, spec 110 R5); plan-validate at `internal/validate/plan.go:143`; the bead-time divergence lane at `internal/validate/divergence.go:155` (with `ownerRef` so branch-only manifests are read), invoked from `mindspec complete` (`internal/complete/complete.go:910`) and `mindspec impl approve` (`internal/approve/impl.go:366`) through `CheckADRDivergence` (`internal/validate/adr_divergence.go:45` → `:78`).
- **The #178 hole.** The comment at `spec.go:212-214` states it outright: "a bare-name-no-manifest entry that normalizeImpactedDomains keeps verbatim (Rule 2, no error) still passes." A descriptive label like `api (orders — models)` (no `/`, not a domain-dir name) survives normalization into the divergence lane's `candidateDomains` (`divergence.go:162`), where `attributeDomainCached` matches each changed file only against that candidate slice → returns `""` → `adr-divergence-unowned` **on every changed source file of every bead** (`divergence.go:219-222`). Zero/multi-owner PATH entries already error at every consumer; only Rule-2 bare labels slip through authoring and detonate at bead time. This is exactly #178's repro, and the "surface early, not at complete-time one bead at a time" remedy #147 asked for.
- **The 6ou2 hole (the same resolver family, other side of the comparison).** Once the spec side is normalized to owner **names**, every coverage decision compares those names against the cited ADR's `Domain(s)` line by **case-folded literal string equality**: `checkADRCitations` uses `intersectFold(a.Domains, impactedDomains)` (`plan.go:485`, → `adr-cite-irrelevant` at `:491`), and `coverageOf`/`IsDomainCovered` (`plan.go:641`/`:628`) test literal domain membership for `adr-coverage-missing` (`plan.go:548/:550`) and the bead-time coverage probe (`divergence.go:230`). ADR `Domain(s)` lines in real repos are path-shaped (`api/app/lola/`) or tuple-shaped (`api (lola, tools)`) — this repo's own ADR-0032 declares `validation, adr, lifecycle, workflow`, three of which are not domain dirs. So a spec touching `api/app/lola/` that cites the very ADR declaring `api/app/lola/` still fails: the spec side resolved to a name, the ADR side stayed a path, and `name ≠ path` literally. There is no transitive resolution — `mindspec-6ou2` items 3 and 4, and the still-red tail of #147's repro (whose ADR-0001 was forced by the old gate to declare file paths as domains).
- **The #145 residue.** Of its three frictions: **(1) the misleading hint** was half-fixed by spec 100 R2 — `plan.go:548` now offers "add `%q` to the `Domain(s)` line of an existing cited Accepted ADR" — but ONLY when the plan already **cites** an Accepted ADR (`hasAcceptedCitation`, `plan.go:560-577`). In #145's actual repro the plan cited nothing yet an Accepted ADR-0001 sat in the store; the `:550` branch still hints only `run: mindspec adr create --domain <d>`, which literally followed creates a spurious ADR instead of citing the governing one. **(2) main-worktree-only ADR resolution is FIXED**: validation overlays the spec worktree's ADR dir over the main checkout (`adrStoreForSpec`, `plan.go:236`, mindspec-ew79; consumed by plan `:157`, divergence `divergence.go:137`, spec `spec.go:271`), `adr create` writes worktree-local (`cmd/mindspec/adr.go:32`, mindspec-8lzq), and `adr show`/`list` read through the same overlay (`adrReadStore`, `adr.go:89-113`, mindspec-3cfr). **(3) format-sensitive `Domain(s)` parsing is FIXED**: `internal/adr/parse.go:74-82` matches by `strings.Contains(trimmed, "**Domain(s)**:")`, so both the list-item form (`- **Domain(s)**: …`) and the plain bold form (`**Domain(s)**: …`) parse. The bonus friction (undocumented `adr_citations` key) is fixed — the plan scaffold comments it (`internal/approve/spec.go:255`). Items 2 and 3 need **regression pins** so #145 can close on evidence, not memory.
- **The #197 stale hints.** `migrate layout` moves `.mindspec/docs/domains` → `.mindspec/domains` (`internal/layout/mover.go:75`) and rewrites tracked references (`internal/layout/rewrite.go:46`), and ownership LOADING is layout-aware (flat → canonical → legacy; `workspace.DomainsDir`, `internal/workspace/workspace.go:617`; `internal/validate/ownership.go:82-83`). But three operator-facing gate hint templates hard-code the pre-flatten path: the `adr-divergence-unowned` remediation "create a new domain dir at `.mindspec/docs/domains/<name>/OWNERSHIP.yaml`" (`divergence.go:221`) and both `internal-docs` doc-sync hints "no doc updates under `.mindspec/docs/domains/<domain>/`" (`filepath.Join(".mindspec", "docs", "domains", …)` at `docsync.go:517` and `:585`). In a flattened workspace the hint names a directory that does not exist; an operator following it creates files in the wrong, resurrected layout.
- **The corrosion this produces.** #147's filer overrode with `--override-adr` on 7 of 10 beads; #178's filer on every source-touching bead. An override recorded on every bead is not an audit trail — it is a habit. The panel-vetted position on the related #181 (gates check anchoring/touch, not semantic conformance to the cited ADR's decisions) is that the gate must NOT be inflated to chase semantics; the correct move is the opposite: shrink the gate's claims to what it can mechanically verify, make those verifications true, and hand semantic conformance to a review lens. This spec does the first two; the lens is named future work.

**Design chosen (pinned here, not plan-level):** symmetry by **name resolution**, not path-overlap intersection. The spec side already resolves entries to owning-domain names; this spec makes (a) Rule-2 non-resolving labels an authoring-time ERROR when the workspace actually uses the ownership model, and (b) the ADR side resolve its path-shaped `Domain(s)` entries through the SAME owner resolution before comparison, so both sides meet as domain names. 6ou2's alternative (a), second half — "raw path overlap: A and B intersect if any of A's claimed paths overlap any of B's" — is rejected: it would make coverage depend on glob-vs-glob intersection with no single owner identity, a new resolution model that would warrant a new ADR and reopen ADR-0036's no-guessing doctrine. Name-resolution symmetry is a refinement of the model ADR-0032 (as amended by spec 100) already records, so it lands as a second amendment to ADR-0032 (see ADR Touchpoints; the new-ADR trigger is explicitly flagged there).

## Impacted Domains

- **workflow**: `internal/validate/**` (the normalizer Rule-2 hard-fail + its error text; ADR-side `Domain(s)` resolution before `intersectFold`/`coverageOf`; the truthful unowned/coverage hints; layout-aware hint paths in `divergence.go` and `docsync.go`), `internal/adr/**` (regression pins for both `Domain(s)` header forms; no parser change expected), `cmd/mindspec/**` (regression pins for the worktree-aware ADR read/write stores; no command change expected), `internal/domain/**`/`internal/ownership/**` only if the plan homes the layout-aware hint-root helper there.
- **core**: `internal/workspace/**` — CONDITIONALLY: only if the plan adds a small "domains-root for hints" accessor beside `DomainsDir` (`workspace.go:617`) instead of homing it in `internal/validate`. Declared so the bead-time blast-radius guard does not fire on that plan choice; if untouched, nothing changes here.

`execution` and `context-system` are not impacted. These attributions match `.mindspec/domains/workflow/OWNERSHIP.yaml` (`internal/validate/**`, `internal/adr/**`, `cmd/**`) and `.mindspec/domains/core/OWNERSHIP.yaml` (`internal/workspace/**`) as they stand in this worktree.

## ADR Touchpoints

- [ADR-0032](../../adr/ADR-0032-adr-semantic-gates.md) (Semantic ADR Coverage Gates) — **AMENDED by this spec** (Requirement 6), its second amendment (spec 100 added the first: path-like Impacted-Domains identifiers normalize to their dir-name owner). Two additions, both refinements of the coverage model it already records: **(a) authoring-time resolvability** — in a workspace that uses the ownership model (≥1 domain dir with a loadable OWNERSHIP.yaml), an `## Impacted Domains` entry that is neither a domain-dir name nor a path resolving to exactly one owner is an ERROR at spec-approve/plan-validate/bead-time alike (the Rule-2 verbatim-keep is retired there; it survives only in manifest-less workspaces), because a label that can never own a file makes every downstream coverage decision vacuously false; **(b) symmetric comparison** — every spec-domains-vs-ADR-`Domain(s)` intersection resolves BOTH sides to owning-domain names through the same OWNERSHIP resolution before comparing; ADR-side entries that do not resolve (legacy prose labels like ADR-0032's own `validation, adr`) compare literally as today and are never a new error class. **New-ADR trigger, flagged**: if the plan finds name-resolution symmetry insufficient and reaches for path-overlap/transitive intersection (6ou2 suggestion (a), second half) or any owner-identity model not expressible as "resolve to dir-name, then compare", that is a resolution-model CHANGE and MUST land as a new ADR superseding ADR-0032's coverage semantics rather than a third amendment — the plan must stop and say so, not bury it.
- [ADR-0036](../../adr/ADR-0036-ownership-discovery.md) (Ownership Discovery — zero framework cognition) — unchanged, APPLIED: every new resolution leg (Rule-2 rejection, ADR-side normalization, real-owner attribution in hints) reads EXPLICIT per-domain manifests through the existing loaders/caches; no synthesized fallback, no guessed ownership. The manifest-less carve-out (Requirement 1) is this ADR's doctrine expressed at the gate: where no manifests exist there is nothing to resolve against, so nothing new may fail.
- [ADR-0041](../../adr/ADR-0041-gate-before-mutate.md) (Gate-Before-Mutate) — unchanged, APPLIED: all three gates remain preflight checks that run before any mutation (`complete.go:910` and `impl.go:366` both sit in the gate phase); this spec changes what the gates EVALUATE, never their ordering, and every new refusal carries an ADR-0035 single-lever recovery line. Moving the Rule-2 failure from bead-time to authoring-time is this ADR's earliest-derivable-refusal principle applied one lifecycle stage earlier.
- [ADR-0031](../../adr/ADR-0031-doc-sync-gate.md) (Doc-Sync gate) — unchanged: the `internal-docs` lane's enforcement semantics are untouched; only its hint text becomes layout-aware (Requirement 4).
- [ADR-0039](../../adr/ADR-0039-flat-layout-v2.md) (Flat layout) — unchanged, APPLIED: hint paths follow the same flat → canonical → legacy resolution order the loaders already honor (`workspace.go:617`), so the printed path is the one that actually resolves in the operator's workspace.
- [ADR-0035](../../adr/ADR-0035-agent-error-contract.md) — unchanged; the Requirement 1 refusal and the Requirement 4 finding texts each carry the single, complete recovery action.

## Requirements

1. **Reject non-resolving Impacted-Domains labels EARLY, with a fix hint that works (#178; #147's early-surfacing ask).** In a workspace where the ownership model is IN USE — defined mechanically as: the layout-aware domain enumeration (`resolveDomains`) returns ≥1 domain dir whose OWNERSHIP.yaml loads — `normalizeImpactedDomains`'s Rule 2 (`ownership_resolve.go:119-125`) MUST return a hard error for a bare entry that names no domain dir, instead of keeping it verbatim. Because the fix lands in the SHARED helper, all three consumers surface it at once: `spec approve`/`validate spec` (via `checkImpactedDomainsResolutionParity`, `spec.go:217` — its "Rule 2 still passes" carve-out comment is retired), `validate plan` (`plan.go:143`), and the bead-time divergence lane (`divergence.go:155`), where the existing errors-then-return path (`divergence.go:156-160`) means a mid-flight legacy spec gets ONE `impacted-domains-resolve` error naming the bad label instead of N per-file `adr-divergence-unowned` errors. The error text MUST contain, in one message: the offending entry verbatim (escaped per the existing `termsafe` discipline at this site), why it failed (neither a domain dir nor a path claimed by any OWNERSHIP `paths:`), the enumerated available domain-dir names, and both working remedies — replace the label with one of the listed domain names, or declare a claimed path instead (claiming it in `<layout-aware domains root>/<name>/OWNERSHIP.yaml` if not yet claimed). **Carve-out**: in a manifest-less workspace (zero loadable OWNERSHIP.yaml), Rule 2 keeps verbatim exactly as today — no new failure for repos that never adopted the ownership model. **Falsified if**: the #178 repro (domain dir `orders` owning `src/orders/**`; spec declaring `api (orders — models)`) still passes `mindspec validate spec` or `validate plan` clean; if a bead in such a spec still reaches per-file `adr-divergence-unowned` instead of the single resolve error; if the error message omits the entry, the available-domain list, or either remedy; or if a manifest-less workspace with bare-name domains newly errors.
2. **Resolve the ADR side of every coverage comparison through the same owner resolution (`mindspec-6ou2`; the red tail of #147).** Before any spec-domains-vs-ADR-domains decision, the cited ADR's parsed `Domain(s)` entries MUST pass through the same resolution the spec side already gets: an entry that names a domain dir stays verbatim; a path-shaped entry — including the directory forms with a trailing `/` (`api/app/lola/`) and without one — that is claimed by exactly one domain's OWNERSHIP `paths:` resolves to that owner's dir-name; a non-resolving or ambiguous entry stays verbatim and compares literally (it can only fail to intersect, exactly as today — ADR-side resolution introduces NO new error class, because ADR `Domain(s)` lines are historical documents this spec must not force churn on; this repo's own ADR-0032 line `validation, adr, lifecycle, workflow` keeps working via its literal `workflow`). The resolved sets feed ALL comparison sites: `intersectFold` in `checkADRCitations` (`plan.go:485` → `adr-cite-irrelevant`), the membership test inside `coverageOf`/`IsDomainCovered` (`plan.go:641`/`:628` → `adr-coverage-missing`/`adr-coverage-proposed`), and the bead-time coverage probe (`divergence.go:230`). Resolution MUST read through the same per-run ownership cache/ref plumbing as the spec side (`newOwnershipCache`, `ownerRef`-anchored at bead time) so both sides see the same tree, and MUST be memoized per run beside the existing `newMemoStore` so per-(file × citation) loops don't re-resolve. **Falsified if**: the 6ou2 repro (spec Impacted Domains `src/orders/api.py`, domain dir `orders` claiming `src/orders/**`, plan citing an Accepted ADR whose `Domain(s)` line reads `src/orders/`) still emits `adr-cite-irrelevant` or `adr-coverage-missing` at `validate plan`; if the same shape still fails the bead-time coverage probe at `complete`; or if an ADR whose `Domain(s)` line holds only legacy prose labels newly produces any error today's build does not.
3. **The `adr-coverage-missing` hint must name the governing fix, including uncited ADRs (#145 friction 1, completed).** When `checkADRCoverage` finds domain `d` notCovered, and the ADR store (the same worktree-overlay store already in hand, `plan.go:157`) contains one or more UNCITED Accepted ADRs whose resolved `Domain(s)` (per Requirement 2) cover `d`, the error MUST name those ADR IDs and the actual remedy — add them to the plan's `adr_citations` frontmatter — as the FIRST hint, ahead of the existing spec-100 remedies (amend a cited Accepted ADR's `Domain(s)` line; `mindspec adr create --domain <d>` last, only as the create-new fallback). The hint set MUST stay truthful in the degenerate case: when the store holds no covering ADR at all, the existing `:548`/`:550` behavior stands unchanged. **Falsified if**: the #145 repro (store contains Accepted ADR-0001 whose `Domain(s)` cover `d`; plan cites nothing) still emits a hint whose only action is `mindspec adr create --domain <d>`; or if the hint names an ADR that is already cited, non-Accepted, or non-covering.
4. **Every gate hint names a path and an owner that are TRUE in the operator's workspace (#197; #178's unowned-message vagueness).** (a) The three hard-coded pre-flatten hint paths — `divergence.go:221` ("create a new domain dir at `.mindspec/docs/domains/<name>/OWNERSHIP.yaml`") and the two `internal-docs` templates (`docsync.go:517`, `:585`) — MUST render through one shared layout-aware helper that prints the domains root the loaders actually resolve in that workspace (flat `.mindspec/domains` when flattened; canonical `.mindspec/docs/domains` pre-flatten; matching `workspace.DomainsDir`'s precedence, `workspace.go:617`), and a sweep guard MUST pin that no operator-facing gate message string hard-codes a domains root again. (b) The `adr-divergence-unowned` finding MUST split into two truthful findings: when the changed file IS claimed by some domain's OWNERSHIP but that domain is not in the spec's declared (resolved) Impacted Domains, the message names the real owning domain and the remedy "add `<domain>` to `## Impacted Domains`" — the scope-drift signal #178's option 2 identified as the gate's actual intent; only a file claimed by NO domain's manifest keeps the "genuinely unowned" message with the (now layout-aware) claim-it remedy. The real-owner attribution reads the same full-enumeration path doc-sync already uses (`resolveDomains` + `attributeDomainCached`, the `docsync.go` pattern) — no new discovery mechanism. The gate's PASS/FAIL boundary is unchanged: both findings remain errors under the same `--override-adr`/`--supersede-adr` escapes; only the message truth changes. **Falsified if**: in a flattened workspace any doc-sync or adr-divergence failure output contains `.mindspec/docs/domains/`; if in a pre-flatten workspace the hint stops naming the canonical path that actually resolves there; if a file owned by an undeclared domain is still reported with the unqualified "not claimed by any OWNERSHIP.yaml" text instead of naming its owner; or if the sweep guard fails to go red when a literal domains-root string is reintroduced into a gate message.
5. **Pin the already-shipped #147/#145 fixes as regressions, so both issues close on evidence.** The following shipped behavior MUST be pinned by tests that fail if it regresses (none exist today as issue-shaped pins): (a) the #147 core — file-path `## Impacted Domains` entries resolve via OWNERSHIP `paths:` globs to their owning domain (spec 100 R1) — pinned END-TO-END through the `complete`-shaped divergence path with the full #147 fixture (impacted domains as file paths; ADR `Domain(s)` as those same file-path strings — green only through Requirement 2, red today at the coverage step; plus a standalone resolution-half pin that is green today and goes red if `divergence.go:155` is reverted); (b) #145 friction 2 — an ADR present or edited ONLY in the spec worktree is honored by `validate plan`, `adr show`, and `adr list` (the overlay stores, `plan.go:236`, `cmd/mindspec/adr.go:89-113`), and `adr create` writes worktree-local (`adr.go:32`); (c) #145 friction 3 — both `Domain(s)` header forms (`- **Domain(s)**: …` and `**Domain(s)**: …`) parse identically (`parse.go:74-82`). **Falsified if**: reverting the spec-100 normalization call at `divergence.go:155`, the overlay store at `plan.go:236`, or the `Contains`-based header match at `parse.go:74` leaves the new suite green.
6. **ADR-0032 second amendment, landing with the code.** ADR-0032 MUST be amended with Requirement 1's authoring-time-resolvability rule (including the manifest-less carve-out and the bead-time single-error demotion path) and Requirement 2's symmetric-comparison rule (including the ADR-side no-new-error doctrine and the pinned rejection of path-overlap intersection with its new-ADR trigger), in the same bead as the first code that implements them, so the ADR-divergence gate sees the declared touchpoint. **Falsified if**: the amendment text lacks either rule, lacks the carve-out, lacks the new-ADR trigger sentence, or lands in a different bead than the implementing code.
7. **Net ceremony goes DOWN — nothing new to operate.** This spec adds NO new gate lane, NO new CLI flag, NO new config key, and NO new override mechanism; every change either rejects earlier (Requirement 1, with the message carrying the complete remedy), passes a previously-false-failing correct input (Requirements 2, 3), or corrects hint text (Requirement 4). The existing escapes (`--override-adr`, `--supersede-adr`; `complete.go:229-252`, `impl.go:119-135`) are untouched. This repo's own lifecycle corpus stays green: `validate spec` and `validate plan` over the current `.mindspec/specs/*` with today's `.mindspec/domains` manifests and ADRs produce no NEW errors after this spec. **Falsified if**: `mindspec complete --help`, `impl approve --help`, or `mindspec config` output gains any new flag/key; if any spec in this repo's corpus that validates green before this change validates red after it; or if a new blocking check fires on an input class not named in Requirements 1–4.

## Scope

### In Scope

- `internal/validate/ownership_resolve.go` — Rule-2 hard-fail with the ownership-model-in-use predicate and the manifest-less carve-out; the enriched error text (Requirement 1).
- `internal/validate/plan.go` — ADR-side `Domain(s)` resolution feeding `intersectFold` (`:485`) and `coverageOf` (`:641`); the uncited-covering-ADR hint in `checkADRCoverage` (`:548-551`) (Requirements 2, 3).
- `internal/validate/divergence.go` — the bead-time coverage probe consuming resolved ADR domains (`:230`); the two-way truthful unowned finding and layout-aware hint (`:219-222`) (Requirements 2, 4).
- `internal/validate/docsync.go` — layout-aware hint paths at `:517`/`:585`; no enforcement change (Requirement 4).
- The shared layout-aware hint-root helper (home plan-level: `internal/validate` beside the loaders, or a small accessor in `internal/workspace` beside `DomainsDir`) and its sweep guard (Requirement 4).
- Regression pins for the shipped #147 core, the ADR worktree overlay stores, and both `Domain(s)` header forms (`internal/validate`, `internal/adr`, `cmd/mindspec` tests) (Requirement 5).
- The ADR-0032 amendment text (Requirement 6).
- Delivery housekeeping (not requirements): closing #147, #178, #145, #197, and `mindspec-6ou2` against the landed ACs; commenting #181 with the Non-Goal disposition and the lens follow-up reference.

### Out of Scope

- **#181 — semantic conformance of a change to a cited ADR's decisions (and doc-sync content accuracy).** The gates verify mechanical anchoring (ownership, coverage, doc-touch) only; verifying that the DIFF obeys the ADR's decisions, or that the touched doc is now accurate, requires judgment and is deferred to a REVIEWER LENS (see Non-Goals). No conformance-affirmation markers, no deviation ledger, no gate renaming in this spec.
- The #178 option-2 redesign of the divergence gate's attribution model (attribute-against-all-domains as the pass/fail basis): Requirement 4(b) uses real-owner attribution for MESSAGE truthfulness only; the candidate-set pass/fail semantics and the blast-radius guard (`divergence.go:145-155` comment) are unchanged.
- Path-overlap/transitive domain intersection (6ou2 suggestion (a), second half) and any OWNERSHIP schema change (e.g. the `:: spec` keyword of 6ou2 suggestion (b)) — rejected in Background; the new-ADR trigger in ADR Touchpoints guards the boundary.
- The ADR `Domain(s)` PARSER (`internal/adr/parse.go`) — pinned, not changed; ADR file format and `adr create` output are untouched.
- `mindspec migrate layout` itself, the loaders' resolution order, and `mindspec doctor`'s ownership checks (`internal/doctor/ownership.go`) — already layout-aware; only gate hint TEXT changes.
- Auto-rewriting authored `spec.md`/`plan.md`/ADR files to fix labels — the gates emit errors with remedies; they never mutate the author's documents.
- The `bd` binary; the panel decision matrix; any change to `--override-adr`/`--supersede-adr` semantics or their audit metadata.

## Non-Goals

- **No semantic-conformance checking in any gate (#181 — explicit, panel-vetted).** A change that is anchored (owned domain, covered ADR) but CONTRADICTS the cited ADR's decision still passes these gates, by design: that judgment belongs to review, not to a mechanical gate whose false green would over-promise worse than today. The forward path is a dedicated reviewer LENS — an `ms-panel` reviewer persona/lane that reads the cited ADRs' decision sections against the diff and surfaces candidate divergences for human adjudication — named here as future work for a separate spec/skill change, deliberately NOT begun here. Until then the gates' claims are shrunk to what they verify, which this spec makes true.
- No inflation of gate scope to compensate: no new blocking lanes, no WARN-spam, no "conformance affirmed" checkbox ceremony.
- No weakening of the gates either: every input class that correctly fails today still fails (out-of-scope files, genuinely uncovered domains, zero/multi-owner paths); the overrides remain the only escapes, and Requirement 1 moves one failure class EARLIER — it does not delete it.
- No retroactive spec-corpus migration: existing approved specs with resolvable domains are untouched; a mid-flight spec with a newly-rejected label is fixed by editing its `spec.md` (the error says exactly how), not by a migration verb.
- No new domain/ownership model: dir-name identity, explicit manifests, glob claims — all per ADR-0036 — are the fixed substrate; this spec only makes both comparison sides meet on it.
