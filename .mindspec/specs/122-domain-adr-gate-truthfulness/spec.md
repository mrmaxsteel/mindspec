# Spec 122-domain-adr-gate-truthfulness: Domain/ADR Gate Truthfulness

## Goal

Make the three domain/ADR gates — `adr-divergence` (bead/impl-approve time), `doc-sync`'s domain lane, and the plan-time `adr-coverage`/`adr-citations` pair — **truthful in both directions**. Today they lie both ways: they **false-positive on correct inputs** (a correct, present OWNERSHIP.yaml and a genuinely-governing cited ADR still fail the gate, forcing `--override-adr` on every source-touching bead — which is corrosive, because it trains operators to normalize gate bypass until the gate carries zero signal), and they **false-negative on unresolvable inputs at authoring time** (a spec whose `## Impacted Domains` labels can never resolve passes `spec approve` and `validate plan` clean, then fails `adr-divergence` on every single bead, far from the authoring mistake that caused it). Four filed issues and one bead ground the fix: **#178** (Rule-2 bare labels silently kept → per-bead `adr-divergence-unowned`), **`mindspec-6ou2`** (ADR `Domain(s)` literal-string intersection blocks specs whose paths the cited ADR itself declares), **#147** (the original file-path-vs-domain-name mismatch — its resolver core shipped in spec 100, but the full repro chain is still red at the coverage step), **#145** (adr-coverage-missing hard to clear: misleading hint + two since-fixed frictions that need regression pins to close the issue), and **#197** (post-flatten gate hints still print the dead `.mindspec/docs/domains/` path). The outcome is **less gate ceremony, not more**: bad domain labels are rejected EARLY (spec/plan validate) with a fix hint that mechanically works when followed; correct inputs resolve and pass; every remaining hint names a path and a remedy that actually exist. No new gate lanes, no new flags, no new config keys. Semantic ADR conformance (#181) is explicitly deferred to a reviewer lens (Non-Goals).

## Background

**The resolver family, as shipped today (verified in this worktree):**

- **The shared normalizer.** `normalizeImpactedDomains` (`internal/validate/ownership_resolve.go:73`, spec 100 R1) resolves each raw `## Impacted Domains` entry by three rules: **Rule 1** (`:114-117`) — the entry names a domain dir (OWNERSHIP.yaml exists under the layout-aware domains root) → keep verbatim; **Rule 2** (`:119-125`) — a bare token with no `/` that is NOT a domain dir → **keep verbatim, no error** ("legacy named-domain without an on-disk manifest; the coverage gate reports its state"); **Rule 3** (`:127-160`) — a path-like entry → glob-match against every domain's OWNERSHIP `paths:`; exactly one owner → replace with the owner's dir-name, zero owners → hard error (`:154-156`), multi-owner → ambiguity error.
- **Its consumers.** Spec-approve runs it via `checkImpactedDomainsResolutionParity` (`internal/validate/spec.go:71`, func at `:217`, spec 110 R5); plan-validate at `internal/validate/plan.go:143`; the bead-time divergence lane at `internal/validate/divergence.go:155` (with `ownerRef` so branch-only manifests are read), invoked from `mindspec complete` (`internal/complete/complete.go:910`) and `mindspec impl approve` (`internal/approve/impl.go:366`) through `CheckADRDivergence` (`internal/validate/adr_divergence.go:45` → `:78`).
- **The #178 hole.** The comment at `spec.go:212-214` states it outright: "a bare-name-no-manifest entry that normalizeImpactedDomains keeps verbatim (Rule 2, no error) still passes." A descriptive label like `api (orders — models)` (no `/`, not a domain-dir name) survives normalization into the divergence lane's `candidateDomains` (`divergence.go:162`), where `attributeDomainCached` matches each changed file only against that candidate slice → returns `""` → `adr-divergence-unowned` **on every changed source file of every bead** (`divergence.go:219-222`). Zero/multi-owner PATH entries already error at every consumer; only Rule-2 bare labels slip through authoring and detonate at bead time. This is exactly #178's repro, and the "surface early, not at complete-time one bead at a time" remedy #147 asked for.
- **The 6ou2 hole (the same resolver family, other side of the comparison).** Once the spec side is normalized to owner **names**, every coverage decision compares those names against the cited ADR's `Domain(s)` line by **case-folded literal string equality**: `checkADRCitations` uses `intersectFold(a.Domains, impactedDomains)` (`plan.go:485`, → `adr-cite-irrelevant` at `:491`), and `coverageOf`/`IsDomainCovered` (`plan.go:641`/`:628`) test literal domain membership for `adr-coverage-missing` (`plan.go:548/:550`) and the bead-time coverage probe (`divergence.go:230`). ADR `Domain(s)` lines in real repos are path-shaped (`api/app/lola/`) or tuple-shaped (`api (lola, tools)`) — this repo's own ADR-0032 declares `validation, adr, lifecycle, workflow`, three of which are not domain dirs. So a spec touching `api/app/lola/` that cites the very ADR declaring `api/app/lola/` still fails: the spec side resolved to a name, the ADR side stayed a path, and `name ≠ path` literally. There is no transitive resolution — `mindspec-6ou2` items 3 and 4, and the still-red tail of #147's repro (whose ADR-0001 was forced by the old gate to declare file paths as domains).
- **The #145 residue.** Of its three frictions: **(1) the misleading hint** was half-fixed by spec 100 R2 — `plan.go:548` now offers "add `%q` to the `Domain(s)` line of an existing cited Accepted ADR" — but ONLY when the plan already **cites** an Accepted ADR (`hasAcceptedCitation`, `plan.go:560-577`). In #145's actual repro the plan cited nothing yet an Accepted ADR-0001 sat in the store; the `:550` branch still hints only `run: mindspec adr create --domain <d>`, which literally followed creates a spurious ADR instead of citing the governing one. **(2) main-worktree-only ADR resolution is FIXED**: validation overlays the spec worktree's ADR dir over the main checkout (`adrStoreForSpec`, `plan.go:236`, mindspec-ew79; consumed by plan `:157`, divergence `divergence.go:137`, spec `spec.go:271`), `adr create` writes worktree-local (`cmd/mindspec/adr.go:32`, mindspec-8lzq), and `adr show`/`list` read through the same overlay (`adrReadStore`, `adr.go:89-113`, mindspec-3cfr). **(3) format-sensitive `Domain(s)` parsing is FIXED**: `internal/adr/parse.go:74-82` matches by `strings.Contains(trimmed, "**Domain(s)**:")`, so both the list-item form (`- **Domain(s)**: …`) and the plain bold form (`**Domain(s)**: …`) parse. The bonus friction (undocumented `adr_citations` key) is fixed — the plan scaffold comments it (`internal/approve/spec.go:255`). Items 2 and 3 need **regression pins** so #145 can close on evidence, not memory.
- **The #197 stale hints.** `migrate layout` moves `.mindspec/docs/domains` → `.mindspec/domains` (`internal/layout/mover.go:75`) and rewrites tracked references (`internal/layout/rewrite.go:46`), and ownership LOADING is layout-aware (flat → canonical → legacy; `workspace.DomainsDir`, `internal/workspace/workspace.go:617`; `internal/validate/ownership.go:82-83`). But three operator-facing gate hint templates hard-code the pre-flatten path: the `adr-divergence-unowned` remediation "create a new domain dir at `.mindspec/docs/domains/<name>/OWNERSHIP.yaml`" (`divergence.go:221`) and both `internal-docs` doc-sync hints "no doc updates under `.mindspec/docs/domains/<domain>/`" (`filepath.Join(".mindspec", "docs", "domains", …)` at `docsync.go:517` and `:585`). In a flattened workspace the hint names a directory that does not exist; an operator following it creates files in the wrong, resurrected layout.
- **The corrosion this produces.** #147's filer overrode with `--override-adr` on 7 of 10 beads; #178's filer on every source-touching bead. An override recorded on every bead is not an audit trail — it is a habit. The panel-vetted position on the related #181 (gates check anchoring/touch, not semantic conformance to the cited ADR's decisions) is that the gate must NOT be inflated to chase semantics; the correct move is the opposite: shrink the gate's claims to what it can mechanically verify, make those verifications true, and hand semantic conformance to a review lens. This spec does the first two; the lens is named future work.

**Design chosen (pinned here, not plan-level):** symmetry by **name resolution**, not path-overlap intersection. The spec side already resolves entries to owning-domain names; this spec makes (a) Rule-2 non-resolving labels a FORWARD-ONLY authoring-time ERROR (Requirement 1 — hard-fails only a spec being newly authored/approved, grandfathering the already-Approved corpus), and (b) the ADR side resolve its path-shaped `Domain(s)` entries through the SAME deterministic owner resolution before comparison, so both sides meet as domain names. 6ou2's alternative (a), second half — "raw path overlap: A and B intersect if any of A's claimed paths overlap any of B's" — is rejected: it would make coverage depend on glob-vs-glob intersection with no single owner identity, a new resolution model that would warrant a new ADR and reopen ADR-0036's no-guessing doctrine. Name-resolution symmetry is a refinement of the coverage model ADR-0032 (as amended twice already: tri-state coverage, then spec-100 Impacted-Domains normalization) records, so it lands as a NEW (third) amendment section on ADR-0032 (see ADR Touchpoints; the new-ADR trigger is explicitly flagged there).

**Evidenced supersession of bead 6ou2's 6/6 panel decision (2026-06-26).** Requirement 2 must be honest that it REVERSES a recorded 6/6 unanimous ZFC-aware panel decision on bead `mindspec-6ou2` (3 Claude + 3 Codex), which explicitly REJECTED "resolve-ADR-side-through-OWNERSHIP by name" in favor of "enforce + teach short-tags; the real fix is lola-side migration." Findings are never out-voted, so this reversal is discharged with evidence against that decision's three load-bearing objections, NOT by re-tally: **(1) the ZFC / guessed-ownership objection** ("resolving the ADR side requires heuristically parsing free-form strings like `api (lola, tools)` — the exact ADR-0036 violation") is answered by scope: R2 resolves ONLY deterministic path-shaped `Domain(s)` entries by glob-matching them against EXPLICIT per-domain OWNERSHIP `paths:` (the identical mechanism spec 100 already blessed for the spec side) — a tuple/prose token like `api (lola, tools)` is NOT parsed or guessed; it stays literal and compares exactly as today (the no-new-error policy), so no free-form heuristic is introduced. **(2) the "doesn't even fix it — `impacted-domains-resolve` fires spec-side first" objection** was true in lola's specific state (path-like tuple Impacted-Domains that fail `normalizeImpactedDomains` before any ADR comparison), but that spec-side failure is exactly what Requirement 1's forward-only early-reject + spec 100's normalization now handle FIRST; once the spec side resolves to a valid domain name, the REMAINING failures are precisely 6ou2's filed items 3 (`adr-cite-irrelevant`) and 4 (`adr-coverage-missing`), which fire purely on the ADR-side literal mismatch — the tail R2 closes. **(3) the "backward-compat is SAFE — mindspec's own ADRs already use short-tags" premise is factually FALSE and is itself the strongest evidence for R2's no-new-error policy over the panel's "reject non-short-tag ADR tokens" fix**: ADR-0032's own `Domain(s)` line reads `validation, adr, lifecycle, workflow`, three of which (`validation`, `adr`, `lifecycle`) are not domain dirs; ADR-0031 declares `validation, doc-sync, lifecycle, ownership`; the panel's proposed ADR-side reject-check would redden mindspec's OWN ADR corpus on day one — the collision R2's no-new-error policy structurally avoids. The panel's spec-side recommendation (enforce + teach) is NOT reversed — it is ADOPTED as Requirement 1. Bead 6ou2's design note is updated to record this supersession at close (Scope § housekeeping), and 6ou2's remaining items are dispositioned: **item 1** (backtick-wrapped Impacted-Domains paths) is ALREADY FIXED — `internal/contextpack/spec.go:79-81` strips `**`/backticks before normalization — and is pinned by Requirement 5; **item 2** (auto-generated OWNERSHIP stub with empty `paths:` claims nothing) is NOT-A-BUG per ADR-0036's no-synthesized-ownership doctrine (an empty manifest correctly owns nothing) and needs no change.

## Impacted Domains

- **workflow**: `internal/validate/**` (the normalizer Rule-2 hard-fail + its error text; ADR-side `Domain(s)` resolution before `intersectFold`/`coverageOf`; the truthful unowned/coverage hints; layout-aware hint paths in `divergence.go` and `docsync.go`), `internal/adr/**` (regression pins for both `Domain(s)` header forms; no parser change expected), `cmd/mindspec/**` (regression pins for the worktree-aware ADR read/write stores; no command change expected), `internal/domain/**`/`internal/ownership/**` only if the plan homes the layout-aware hint-root helper there.
- **core**: `internal/workspace/**` — CONDITIONALLY: only if the plan adds a small "domains-root for hints" accessor beside `DomainsDir` (`workspace.go:617`) instead of homing it in `internal/validate`. Declared so the bead-time blast-radius guard does not fire on that plan choice; if untouched, nothing changes here.

`execution` and `context-system` are not impacted. These attributions match `.mindspec/domains/workflow/OWNERSHIP.yaml` (`internal/validate/**`, `internal/adr/**`, `cmd/**`) and `.mindspec/domains/core/OWNERSHIP.yaml` (`internal/workspace/**`) as they stand in this worktree.

## ADR Touchpoints

- [ADR-0032](../../adr/ADR-0032-adr-semantic-gates.md) (Semantic ADR Coverage Gates) — **AMENDED by this spec** (Requirement 6): a NEW (third) `## Amendment` section, after the two it already carries (tri-state coverage, PR #126; spec-100 Impacted-Domains normalization). Two additions, both refinements of the coverage model it already records: **(a) forward-only authoring-time resolvability** — in a workspace that uses the ownership model (≥1 domain dir with a loadable OWNERSHIP.yaml), an `## Impacted Domains` entry that is neither a domain-dir name nor a path resolving to exactly one owner is an ERROR when a spec is being NEWLY authored/approved (its status is not-yet-Approved at `validate spec`/`spec approve`/`validate plan` time); an already-Approved spec is grandfathered — its Rule-2 verbatim-keep stands, so the existing corpus never newly reddens — and manifest-less workspaces keep verbatim always. A label that can never own a file makes every downstream coverage decision vacuously false, so it is caught before the spec locks; **(b) symmetric comparison** — every spec-domains-vs-ADR-`Domain(s)` intersection resolves BOTH sides to owning-domain names through the same DETERMINISTIC explicit-manifest OWNERSHIP resolution before comparing; ADR-side entries that do not resolve (legacy prose/tuple labels like ADR-0032's own `validation, adr, lifecycle`) compare literally as today and are never a new error class. **This §(b) supersedes bead `mindspec-6ou2`'s 6/6 panel decision (2026-06-26) that rejected ADR-side name-resolution** — the amendment text MUST carry the evidenced-supersession rationale (deterministic explicit-manifest resolution, not free-form heuristic parsing, answers the ZFC objection; the no-new-error-on-ADR-side policy avoids forced ADR churn AND is required because mindspec's own ADR-0032/-0031 `Domain(s)` lines carry non-short-tag tokens the panel's reject-fix would redden; Requirement 1 answers the "doesn't fix `impacted-domains-resolve`" objection). **New-ADR trigger, flagged**: if the plan finds name-resolution symmetry insufficient and reaches for path-overlap/transitive intersection (6ou2 suggestion (a), second half) or any owner-identity model not expressible as "resolve to dir-name, then compare", that is a resolution-model CHANGE and MUST land as a NEW ADR superseding ADR-0032's coverage semantics rather than a fourth amendment section — the plan must stop and say so, not bury it.
- [ADR-0036](../../adr/ADR-0036-ownership-discovery.md) (Ownership Discovery — zero framework cognition) — unchanged, APPLIED: every new resolution leg (Rule-2 rejection, ADR-side normalization, real-owner attribution in hints) reads EXPLICIT per-domain manifests through the existing loaders/caches; no synthesized fallback, no guessed ownership. The manifest-less carve-out (Requirement 1) is this ADR's doctrine expressed at the gate: where no manifests exist there is nothing to resolve against, so nothing new may fail.
- [ADR-0041](../../adr/ADR-0041-gate-before-mutate.md) (Gate-Before-Mutate) — unchanged, APPLIED: the gates continue to validate the committed bead tip AFTER artifact-materialization and BEFORE the merge/close mutation, exactly where they run today (`complete.go:910`, `impl.go:366`); this spec preserves that ordering and changes ONLY what the gates evaluate, and every new refusal carries an ADR-0035 single-lever recovery line. Moving the Rule-2 failure from bead-time to the authoring gates is this ADR's earliest-derivable-refusal principle applied one lifecycle stage earlier — the fact was always derivable at `spec approve`; this spec derives it there.
- [ADR-0031](../../adr/ADR-0031-doc-sync-gate.md) (Doc-Sync gate) — unchanged: the `internal-docs` lane's enforcement semantics are untouched; only its hint text becomes layout-aware (Requirement 4).
- [ADR-0039](../../adr/ADR-0039-flat-layout-v2.md) (Flat layout) — unchanged, APPLIED: hint paths follow the same flat → canonical → legacy resolution order the loaders already honor (`workspace.go:617`), so the printed path is the one that actually resolves in the operator's workspace.
- [ADR-0035](../../adr/ADR-0035-agent-error-contract.md) — unchanged; the Requirement 1 refusal and the Requirement 4 finding texts each carry the single, complete recovery action.

## Requirements

1. **Reject non-resolving Impacted-Domains labels EARLY and FORWARD-ONLY, with a fix hint that works (#178; #147's early-surfacing ask).** The trigger: a workspace where the ownership model is IN USE — defined mechanically as the layout-aware domain enumeration (`resolveDomains`) returning ≥1 domain dir whose OWNERSHIP.yaml loads — carrying an `## Impacted Domains` entry that is a bare token naming no domain dir (`normalizeImpactedDomains` Rule 2, `ownership_resolve.go:119-125`). **Forward-only gating (the corpus-collision resolution).** This condition MUST become a hard `impacted-domains-resolve` ERROR ONLY when the spec is being NEWLY authored/approved — mechanically, when the spec's frontmatter `status` is not-yet-`Approved` at the authoring gate that runs it: `validate spec`/`spec approve` (via `checkImpactedDomainsResolutionParity`, `spec.go:217` — its "Rule 2 still passes" carve-out comment is retired) and `validate plan` (`plan.go:143`). An already-`Approved` spec is **grandfathered**: the Rule-2 verbatim-keep stands, so mindspec's own 58-of-122 bare-label corpus specs (`tracking`, `cli`, `viz`, `validation`, …) keep passing `validate spec` — AC-14/R7 corpus-green is preserved by construction, not by exception. This makes forward-only STRICTLY cleaner than the fallback (demote-pre-existing-to-WARN + hard-fail-new): a grandfathered spec emits nothing new at all, so no WARN noise on 58 specs. Because the status signal lives at the authoring gates, the SHARED helper distinguishes the Rule-2-non-resolving finding and the CALLER applies it as an error only in forward mode; the exact plumbing (a mode flag into the helper, or a caller-side severity gate on the distinguished finding) is plan-level, fixed observably by the ACs. The error text MUST contain, in one message: the offending entry verbatim (escaped per the existing `termsafe` discipline at this site), why it failed (neither a domain dir nor a path claimed by any OWNERSHIP `paths:`), the enumerated available domain-dir names, and both working remedies — replace the label with one of the listed domain names, or declare a claimed path instead (claiming it in `<layout-aware domains root>/<name>/OWNERSHIP.yaml` if not yet claimed). **Carve-outs**: (i) an already-Approved spec (grandfathered, above); (ii) a manifest-less workspace (zero loadable OWNERSHIP.yaml) — Rule 2 keeps verbatim exactly as today, no new failure for repos that never adopted the ownership model. **Falsified if**: the #178 repro (domain dir `orders` owning `src/orders/**`; a NOT-yet-Approved spec declaring `api (orders — models)`) still passes `mindspec validate spec` or `validate plan` clean; if an already-Approved bare-label corpus spec NEWLY fails `validate spec` after this change; if the error message omits the entry, the available-domain list, or either remedy; or if a manifest-less workspace with bare-name domains newly errors.
2. **Resolve the ADR side of every coverage comparison through the same owner resolution (`mindspec-6ou2`; the red tail of #147).** Before any spec-domains-vs-ADR-domains decision, the cited ADR's parsed `Domain(s)` entries MUST pass through the same DETERMINISTIC explicit-manifest resolution the spec side already gets: an entry that names a domain dir stays verbatim; a path-shaped entry that is claimed by exactly one domain's OWNERSHIP `paths:` resolves to that owner's dir-name; a non-resolving, ambiguous, or non-path (tuple/prose) entry stays verbatim and compares literally (it can only fail to intersect, exactly as today — ADR-side resolution introduces NO new error class, because ADR `Domain(s)` lines are historical documents this spec must not force churn on; this repo's own ADR-0032 line `validation, adr, lifecycle, workflow` keeps working via its literal `workflow`, and its non-domain tokens stay literal and simply do not match). **Directory-shape completeness (point-to-glob):** the path resolution MUST claim a directory-shaped ADR label in BOTH forms — with a trailing slash (`api/app/lola/`, `src/orders/`) and without one (`api/app/lola`, `src/orders`) — against an OWNERSHIP glob like `src/orders/**`, so an ADR that writes either form resolves identically; a bare file path (`src/orders/api.py`) resolves the same way. The resolved sets feed ALL comparison sites: `intersectFold` in `checkADRCitations` (`plan.go:485` → `adr-cite-irrelevant`), the membership test inside `coverageOf`/`IsDomainCovered` (`plan.go:641`/`:628` → `adr-coverage-missing`/`adr-coverage-proposed`), and the bead-time coverage probe (`divergence.go:230`). Resolution MUST read through the same per-run ownership cache/ref plumbing as the spec side (`newOwnershipCache`, `ownerRef`-anchored at bead time) so both sides see the same tree; per-run memoization beside the existing `newMemoStore` is a plan-level performance choice (no observable contract, so not a MUST here). **Falsified if**: the 6ou2 repro (below, both slash forms) still emits `adr-cite-irrelevant` or `adr-coverage-missing` at `validate plan`; if the same shape still fails the bead-time coverage probe at `complete`; if either directory slash form fails to resolve where the other succeeds; or if an ADR whose `Domain(s)` line holds only legacy prose/tuple labels newly produces any error today's build does not.
3. **The `adr-coverage-missing` hint must name the governing fix, including uncited ADRs (#145 friction 1, completed).** WHENEVER `checkADRCoverage` finds domain `d` notCovered — regardless of whether the plan cites zero ADRs OR cites some ADRs none of which covers `d` — and the ADR store (the same worktree-overlay store already in hand, `plan.go:157`) contains one or more UNCITED Accepted ADRs whose resolved `Domain(s)` (per Requirement 2) cover `d`, the error MUST name those ADR IDs and the actual remedy — add them to the plan's `adr_citations` frontmatter — as the FIRST hint, ahead of the existing spec-100 remedies (amend a cited Accepted ADR's `Domain(s)` line; `mindspec adr create --domain <d>` last, only as the create-new fallback). The trigger is the EXISTENCE of an uncited covering ADR in the store, NOT the emptiness of the citation list — a plan that already cites some non-covering ADRs still gets the name-the-covering-ADR hint. The hint set MUST stay truthful in the degenerate case: when the store holds no covering ADR at all, the existing `:548`/`:550` behavior stands unchanged. **Falsified if**: the #145 repro (store contains Accepted ADR-0001 whose `Domain(s)` cover `d`; plan cites nothing) still emits a hint whose only action is `mindspec adr create --domain <d>`; if a plan that cites ≥1 non-covering ADR while an uncited covering ADR exists gets NO name-the-covering-ADR hint; or if the hint names an ADR that is already cited, non-Accepted, or non-covering.
4. **Every gate hint names a path and an owner that are TRUE in the operator's workspace (#197; #178's unowned-message vagueness).** (a) The three hard-coded pre-flatten hint paths — `divergence.go:221` ("create a new domain dir at `.mindspec/docs/domains/<name>/OWNERSHIP.yaml`") and the two `internal-docs` templates (`docsync.go:517`, `:585`) — MUST render through one shared layout-aware helper that prints the domains root the loaders actually resolve in that workspace (flat `.mindspec/domains` when flattened; canonical `.mindspec/docs/domains` pre-flatten; matching `workspace.DomainsDir`'s precedence, `workspace.go:617`), and a sweep guard MUST pin that no operator-facing gate message string hard-codes a domains root again. (b) The `adr-divergence-unowned` finding MUST split into two truthful findings: when the changed file IS claimed by some domain's OWNERSHIP but that domain is not in the spec's declared (resolved) Impacted Domains, the message names the real owning domain and the remedy "add `<domain>` to `## Impacted Domains`" — the scope-drift signal #178's option 2 identified as the gate's actual intent; only a file claimed by NO domain's manifest keeps the "genuinely unowned" message with the (now layout-aware) claim-it remedy. The real-owner attribution reads the same full-enumeration path doc-sync already uses (`resolveDomains` + `attributeDomainCached`, the `docsync.go` pattern) — no new discovery mechanism. The gate's PASS/FAIL boundary is unchanged: both findings remain errors under the same `--override-adr`/`--supersede-adr` escapes; only the message truth changes. **Falsified if**: in a flattened workspace any doc-sync or adr-divergence failure output contains `.mindspec/docs/domains/`; if in a pre-flatten workspace the hint stops naming the canonical path that actually resolves there; if a file owned by an undeclared domain is still reported with the unqualified "not claimed by any OWNERSHIP.yaml" text instead of naming its owner; or if the sweep guard fails to go red when a literal domains-root string is reintroduced into a gate message.
5. **Close #147 and #145 on evidence: cite the EXISTING regression pins; add only the genuinely-missing #147 end-to-end fixture.** Most of the shipped-fix pins these issues need ALREADY EXIST as named tests — the requirement is to CITE them as the issue→test evidence map at close, not re-author them: the #147 spec-100 resolution core is pinned by `TestValidateDivergenceFilePathImpactedDomainResolves` (`internal/validate/divergence_test.go`); #145 friction 2 (worktree ADR visibility) by `TestPlanSpecWorktreeADRVisible` (`internal/validate/plan_test.go`) and the user-facing `TestAdrShowWorktree_FindsWorktreeLocalADR` / `TestAdrListWorktree_ListsWorktreeLocalADR` / `TestAdrShowWorktree_StillFindsMainOnlyADR` (`cmd/mindspec/adr_test.go`), plus worktree-local `adr create` by `TestADRCreate_WritesIntoInvokingWorktree` (`cmd/mindspec/adr_test.go:92`); #145 friction 3 (both `Domain(s)` header forms) by `TestParseADR_NonListDomainLine` (`internal/adr/parse_test.go`). The GENUINELY-NEW pin this spec must ADD is the #147 END-TO-END fixture — really the red/green AC for Requirement 2 — through the `complete`-shaped divergence path with the FULL #147 shape (impacted domains as file paths; a cited Accepted ADR whose `Domain(s)` are those same file-path strings): green only through Requirement 2, RED today at the coverage step (the resolution half alone is already green via the existing test above). ## Non-Goals

- **No semantic-conformance checking in any gate (#181 — explicit, panel-vetted).** A change that is anchored (owned domain, covered ADR) but CONTRADICTS the cited ADR's decision still passes these gates, by design: that judgment belongs to review, not to a mechanical gate whose false green would over-promise worse than today. The forward path is a dedicated reviewer LENS — an `ms-panel` reviewer persona/lane that reads the cited ADRs' decision sections against the diff and surfaces candidate divergences for human adjudication — named here as future work for a separate spec/skill change, deliberately NOT begun here. Until then the gates' claims are shrunk to what they verify, which this spec makes true.
- No inflation of gate scope to compensate: no new blocking lanes, no WARN-spam, no "conformance affirmed" checkbox ceremony.
- No weakening of the gates either: every input class that correctly fails today still fails (out-of-scope files, genuinely uncovered domains, zero/multi-owner paths); the overrides remain the only escapes, and Requirement 1 moves one failure class EARLIER for new specs — it does not delete it.
- No retroactive spec-corpus migration and no new failures on existing work: already-Approved specs (the 58 bare-label corpus specs and any mid-flight Approved spec) are grandfathered by Requirement 1's forward-only gating — they never newly fail. Only a not-yet-Approved spec with a bad label is rejected, and it is fixed by editing its `spec.md` before approval (the error says exactly how), never by a migration verb.
- No new domain/ownership model: dir-name identity, explicit manifests, glob claims — all per ADR-0036 — are the fixed substrate; this spec only makes both comparison sides meet on it.
