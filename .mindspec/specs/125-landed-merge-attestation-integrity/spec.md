# Spec 125-landed-merge-attestation-integrity: Landed-merge attestation integrity

## Goal

Make the spec-121 landed-merge attestation substrate TRUE in real practice. Two confirmed, pre-existing bugs sit on the merge critical path — the path every `mindspec complete` and `impl approve` crosses — and together they make the substrate's promise false end-to-end: **(1)** `mindspec complete` never actually persists the merge-time landed-binding ADR-0041 §2(ii) mandates — `ensureLandedBinding` swallows a locate MISS as a silent nil (`internal/executor/mindspec_executor.go:1516-1519`) and writes no `mindspec_landed_merge_sha`, so `internal/lifecycle.FindLandedMerge` later fail-closes with `*LandedMergeNoEvidence` on every branch-deleted, genuinely-merged bead; and **(2)** the revert-awareness leg misclassifies EVOLVED content as REVERTED — `gitutil.ContentSubsumedOutcome` returns `SubsumptionCleanDivergence` ("backed out") for a dependency whose touched surface later honest work evolved cleanly (the real `mindspec-8nhe.2` case), so `FindLandedMerge` refuses a dep whose work is still present. Every landed-merge consumer inherits both: `complete`'s merged-unclosed reconcile, the doctor stale-OPEN cross-check, and spec 124's MF-3 readiness signal, which today false-rejects EVERY genuinely-merged dependency — a ceremony trapdoor.

The diagnosed root cause of Bug 1 (reproduced E2E, **755/757 closed beads across the fleet — including specs 121/122/123 — carry no binding**) is deeper than a bare locate miss: merge-candidate identity is keyed to an EXACT subject string `"Merge "+beadBranch`, and that string is routinely defeated by conflict-recovery. The `.beads/issues.jsonl` export is auto-committed into every bead and spec worktree, so the 2nd+ bead under a spec routinely hits a real add/add conflict on it; `CompleteBead`'s conflict path (`beadToSpecConflictFailure`, `internal/executor/mindspec_executor.go:1600`) prints the recovery `git merge --no-ff <beadBranch>` with **no `-m`** (`:1623`), so an operator/agent following it verbatim produces git's DEFAULT subject `Merge branch 'bead/x' into 'spec/…'`, which the identity scan never matches; the recovery re-run of `mindspec complete` then sees an already-ancestor branch (no-op merge), the subject scan misses, and `ensureLandedBinding` silently returns nil. So the fix is structural, not just outcome-level: identity must survive a benign subject variation.

Target outcome: after any real `mindspec complete` that performs a bead→spec merge — INCLUDING the conflict-recovery path — the landed-binding is durably on the bead (visible in `bd show`); merge-candidate identity is corroborated by second-parent identity, not defeated by a benign subject variation; a genuine bind failure is LOUD, fail-closed, and cleanup-suppressing — never a silent nil; content evolved by later honest work is identified as landed while genuinely backed-out content still refuses; already-merged history (the current fleet state: closed beads, deleted branches, no bindings) is recoverable through an audited, fail-closed, identity-corroborated re-attest operation; and every fixed class is pinned by a CI-hermetic regression test.

## Background

All citations verified in this worktree (fork of `main` at `ae53877e`, post-spec-123).

### Bug 1 — `complete` never persists the landed-binding

- Both producer legs record the binding fail-closed-before-cleanup exactly as ADR-0041 §2(ii) requires — `CompleteBead` (`internal/executor/mindspec_executor.go:448-453`) and `FinalizeEpic`'s auto-merge loop (`:672`, `:726`) call `ensureLandedBinding` before any branch/worktree cleanup, and a bind-WRITE failure suppresses cleanup via `guard.NewFailure`. But `ensureLandedBinding` (`:1514`) first calls `locateLandedMergeByIdentity(g.Root, specBranch, beadBranch)` (`:1468`), and on a locate MISS (`errNoLandedMergeIdentified`, `:1451`) it **silently returns nil** (`:1516-1519`) — no binding written, no warning, cleanup proceeds, branch deleted. The silent-nil design assumed the only miss is the legitimate zero-own-commits ancestor branch ("nothing to bind"); in real practice the locate misses for beads that DID merge with real commits, so the assumed-impossible state is the common one.
- Downstream, `FindLandedMerge` (`internal/lifecycle/landed.go:249`) then has no admissible corroboration for the branch-deleted bead: no surviving branch tip, usually no registered-panel `reviewed_head_sha` at the consuming site, no binding — subject match alone is inadmissible by design (§2(ii)) — so it returns `*LandedMergeNoEvidence` (`landed.go:337-345`) on every merged dep. **Proven**: manually writing `mindspec_landed_merge_sha`/`mindspec_landed_second_parent` onto the bead makes the consumer pass — the engine, read path (`landedBindingForBead`, `landed.go:154-160`), corroboration ladder, and write path (`bead.MergeMetadata`) all work; only `complete`'s write goes missing.
- **The exact miss condition is now PINNED** (was an open question; the parallel diagnostic reproduced it E2E with real `mindspec`+`bd`). It is NOT root-vs-worktree resolution (disproven). It is **exact-merge-subject identity defeated by conflict-recovery**: `MergeInto` (`internal/gitutil/gitops.go:154-164`), `locateLandedMergeByIdentity` (`mindspec_executor.go:1469`), and `FindLandedMerge` (`internal/lifecycle/landed.go:257`) all require the merge commit's subject to be exactly `"Merge "+beadBranch`. The auto-committed `.beads/issues.jsonl` (bd `no-auto-flush:false`) makes the 2nd+ bead under a spec routinely hit a real add/add conflict; the conflict-recovery message (`beadToSpecConflictFailure:1623`) omits `-m`, so the human/agent recovery merge lands git's default subject `Merge branch 'bead/x' into 'spec/…'`; the recovery re-run of `complete` finds the bead already an ancestor (no-op merge), the exact-subject scan misses, and the silent-nil swallow completes with no binding. This is why 755/757 fleet beads lack a binding. The structural fix (Requirement 5) is second-parent identity corroboration so a benign subject variation cannot defeat identification; Requirements 1/2 make persistence reliable and any residual miss loud; the recovery message is fixed to emit an identifiable merge.

### Bug 2 — evolved content reads as reverted

- `FindLandedMerge`'s revert-awareness leg (`landed.go:347-382`) discriminates `ContentSubsumedOutcome`'s trichotomy correctly at its own level: `SubsumptionConflict` → landed-then-evolved → identified; `SubsumptionCleanDivergence` → backed out → refused (`landed.go:379-382`).
- But the underlying primitive (`internal/gitutil/neteffect.go:217-244`) classifies as `SubsumptionCleanDivergence` ANY clean three-way whose result differs from the tip — which captures not only the true `git revert` shape (tip returned to the pre-merge state on the merge's own paths) but also evolved shapes where later honest work changed the merge's touched surface in a way that still auto-merges cleanly (content relocated, removed-and-replaced, overlapping-surface doc reconciliation). The doc comment concedes the residual ("content later removed cleanly and fully by honest work is content-indistinguishable from a revert", `landed.go:370-374`); `mindspec-8nhe.2` — later beads and doc-recon touched the dep's overlapping surface — proves the residual is the REAL evolved case, not an edge. Existing fixtures (`landed_test.go:712`, `:760`) only pin evolved shapes that CONFLICT; no fixture pins the evolved-but-cleanly-divergent shape, so the false-reject ships green.
- The refusal is a §2(i) violation in practice: no recovery converges (the content genuinely evolved; restoring the branch re-corroborates and re-refuses at the same leg), leaving only the bd-close bypass or reverting legitimate later work — the exact deadlock class ADR-0041 forbids.

### Consumers and blast radius

- `complete`'s merged-unclosed reconcile treats `*LandedMergeNoEvidence` as the attested-restore refusal (`internal/complete/complete.go:606-613`, `:1397-1405`) — a manual, human-marker recovery now demanded for every routinely-completed bead.
- The doctor stale-OPEN cross-check consumes the same predicate (`internal/lifecycle/stale_open.go:92`).
- Spec 124's MF-3 ("dependency closed AND landed-merged") consumes `FindLandedMerge` verbatim and fail-closes on any error — with Bug 1, that is a false FAIL for every dependency completed by the real lifecycle. This spec is 124's prerequisite; MF-3 itself is out of scope here.

## ADR Touchpoints

- [ADR-0041: Gate-Before-Mutate](../../adr/ADR-0041-gate-before-mutate.md) — **AMENDED by this spec (narrowly), otherwise APPLIED**. Requirements 1–3 and 5 are pure applications: §2(ii) already mandates the fail-closed, before-cleanup merge-time binding (Bug 1 is an implementation failure of it — the silent miss-swallow is exactly the "warning and continuing" the clause forbids, so making it loud needs NO amendment) AND already mandates identity by corroborating datum "NEVER subject text alone" — Requirement 5 makes the implementation FAITHFUL to §2(ii) by moving off the exact-subject gate onto second-parent corroboration, tightening toward the clause rather than diverging from it, so it needs NO amendment; and §2(i)+(ii) jointly already forbid the evolved-content permanent refusal ("a permanent 'ever-reverted ⇒ reject' rule would itself violate §2(i)'s deadlock-free rule"), so Requirement 3 needs NO amendment either. Requirement 4 DOES need one narrow §2(ii) touchpoint: §2(ii) admits "the merge-time landed-binding" as a corroborating datum, and a recovery re-attestation writes that same datum AFTER merge time. The amendment records that the binding's admissibility derives from its corroborated-identity discipline, not from when it is written: a re-attested binding is admissible iff DERIVED from an independent git scan (second-parent identity under the R5 precedence+uniqueness rule) — never from an operator-asserted SHA pair corroborating itself, and never subject-only — and carrying an integrity-preserving audit record (acting identity/authority, before/after values, timestamp, invoking operation, corroborating datum; append-only or tamper-evident). The amendment MUST enumerate the exact admissible datum set so it matches R4: (a) the git-derived second-parent identity from the scan; (b) a surviving bead-branch tip; (c) a registered panel's `reviewed_head_sha`; (d) an existing consistent binding; and (e) the ADR-0035 q9ea human attested-restore marker — admitted ONLY as an explicitly-blessed, audited exit for the genuinely-no-mechanical-corroboration case, and NEVER the sole corroboration where a scan could derive the identity. Amendment lifecycle per the 122/123 precedent: pre-drafted at plan time, landed by the same bead that lands the re-attest surface, so the ADR-divergence gate sees the declared touchpoint (pinned by AC-11).
- [ADR-0035: Agent Error Contract](../../adr/ADR-0035-agent-error-contract.md) — unchanged, APPLIED: the new loud bind-failure refusal and every re-attest refusal route through `internal/guard` with a named, copy-pastable recovery (re-run `mindspec complete <id>`; the re-attest operation for historical beads); no refusal ends without a forward exit.
- [ADR-0025: JSONL as Build Artifact](../../adr/ADR-0025-jsonl-as-build-artifact.md) — unchanged, RESPECTED: all binding and audit writes go through the existing `internal/bead` metadata helpers and the export-before-commit discipline; no direct JSONL edits, no raw `bd update --metadata` surfaces.
- [ADR-0023: Beads/Dolt as Single State Authority](../../adr/ADR-0023.md) — unchanged, RESPECTED: the binding remains advisory corroboration metadata, never lifecycle authority — re-attestation writes identity evidence, it does not open, close, or reclassify any bead.

## Requirements

Each requirement carries a hard falsifier. "Real topology" below means the shape production `complete` actually runs: merge performed in the spec worktree, locate/bind driven from the main root (`g.Root`), bead branch and worktree deleted immediately after.

1. **`complete` reliably persists the landed-binding.** After every bead→spec merge performed by `mindspec complete` (and by `FinalizeEpic`'s auto-merge leg) for a bead branch with at least one own commit, the landed-binding (`mindspec_landed_merge_sha` + `mindspec_landed_second_parent`, per the existing key contract) MUST be durably recorded on the bead — surviving into `bd show <id> --json` metadata and matching the actual first-parent merge commit on the spec branch — across the real topology, BEFORE the branch/worktree cleanup that destroys the surviving-branch datum. The fix MUST target the pinned root cause (Background; Requirement 5): the R1 regression fixture reproduces the real miss shape (conflict on `.beads/issues.jsonl` → default-subject recovery merge → recovery re-run) so it is RED on today's `main`, and the binding is persisted whenever the bead's work has actually landed as a merge on the spec branch regardless of that merge's subject line.
   *Falsified if*: a bead with own commits, completed by a real `mindspec complete` run (including via the conflict-recovery path), lacks `mindspec_landed_merge_sha` in its bd metadata afterward; or the recorded SHAs do not match the actual merge commit and its second parent; or the binding is written only after cleanup has already deleted the branch datum.

2. **A genuine locate/bind failure is LOUD (fail-closed), never silent.** `ensureLandedBinding` MUST NOT return nil on a locate miss for a bead that DID merge. The legitimate "nothing to bind" state (a trivially-ancestor branch that never produced a bead→spec merge commit — a bd_close orphan with zero own commits) MUST be POSITIVELY classified by the ONLY structural discriminator that survives the post-merge topology: whether the bead branch's tip is reachable on `git rev-list --first-parent <specBranch>`. **A count/merge-base classifier is proven insufficient** (Background/diagnostic): after the merge, a genuinely-merged-then-ancestor bead is byte-identical to a true orphan under both `rev-list beadTip ^specBranch` (both 0) and `merge-base==tip` (both yes), because `ensureLandedBinding` runs AFTER the merge — so an own-commit-count classifier reads "nothing to bind" for a real merge and RE-HIDES the bug. The discriminator MUST instead be first-parent membership: a bead whose work landed via a bead→spec MERGE has its tip on a SECOND parent (NOT on `spec`'s first-parent line), whereas a true orphan's tip is reachable only as an unmerged head; "nothing to bind" is admissible ONLY when no bead→spec merge commit exists AND the tip is not a second parent of any first-parent merge on `spec`. Any other miss — a bead that DID merge but whose merge commit cannot be located, or whose binding cannot be written — refuses via `guard.NewFailure`, suppresses branch/worktree cleanup (preserving the surviving-branch datum, so re-invocation converges per §2(ii)), and names the recovery.
   *Falsified if*: any code path swallows a bind failure or locate miss for a bead that did merge and proceeds to cleanup; or the nothing-to-bind classification is derived from own-commit-count / merge-base (which cannot distinguish a merged-then-ancestor bead from a true orphan) rather than from first-parent-membership evidence; or the refusal deletes the branch it needs for recovery.

3. **Revert-vs-evolved discrimination is correct.** For a two-parent merge M in the spec branch's first-parent history, the landed-identity decision is computed by the three-way content merge (base = M^1, ours = tip, theirs = M) at the granularity of **M's CHANGED CONTENT — the region/hunks M actually introduced — NOT per-file-path**. The decision:
   - **REVERTED (not identified)** only when re-applying M's change onto the tip yields the base (M^1) state on M's OWN touched content — M's net contribution is removed and not replaced (the `git revert M` shape). Per-file-path reasoning is explicitly WRONG here: later honest work that changed a DIFFERENT region of a file M also touched MUST NOT read as a revert of M.
   - **EVOLVED (identified)** when M's content is present at the tip, possibly relocated, rewritten, or superseded — the tip has moved BEYOND the base on M's touched content — whether the three-way replays as a CONFLICT (today's already-correct `SubsumptionConflict` leg, preserved) or as a CLEAN divergence (the `mindspec-8nhe.2` shape, today's false-reject).
   - **Stated residual (REFUSE, not over-claimed)**: content later removed CLEANLY AND FULLY by honest work — or M's own paths deleted/relocated wholesale — is content-indistinguishable from a revert, so it also refuses. This is a deliberate false-negative floor: any datum that accepted clean full removal would accept every real revert too.
   **Octopus/parent guard**: MindSpec bead→spec merges are exactly two-parent; the identity/discrimination MUST require a candidate with ≥2 parents AND the bead's second-parent among them, and MUST NOT attempt this discrimination on an octopus (>2-parent) or non-bead merge — such a candidate is excluded, never guessed. The discrimination mechanism (a sub-classification of `SubsumptionCleanDivergence`, the region-level three-way, or a sibling primitive) is plan-level; the boolean `ContentSubsumed`/`NetEffectLanded` projections consumed by the protected-main probe and the doctor merged-carrier suppression (spec 121 R4) keep their existing behavior.
   *Falsified if*: the evolved-content-reads-as-reverted fixture (later first-parent commits move the tip BEYOND base on M's touched content — removed-and-replaced, not a pure delete/relocate — such that today's `ContentSubsumedOutcome` returns `SubsumptionCleanDivergence`) is still refused; or later work on a DIFFERENT region of a path M touched is read as a revert of M; or a genuine `git revert M` with no later rework of that content is identified as landed; or the spec-121 revert-then-reapply and boolean-consumer contracts regress.

4. **A general, audited backfill/re-attest recovery.** An **explicit operator-invoked** `mindspec`-surface operation (verb naming plan-level; it MUST NOT be an implicit-writing `doctor` check — the last mass-mutation vector across ~752 beads stays gated behind a deliberate command) MUST exist to (re)attest the landed-binding of an already-merged bead — the current fleet state: closed bead, branch deleted, no binding. It is **fail-closed and DERIVES the binding from INDEPENDENT git evidence, never from the assertion it is validating**: it scans the spec branch for the bead's actual landed merge by second-parent identity under the Requirement 5 uniqueness rule (exact-second-parent-match precedence; a descendant/ancestor-only candidate is not accepted when the exact-match exists) and writes ONLY the SHAs that scan derives. An operator-asserted merge/second-parent pair is **NOT admissible as its own corroboration** (that is circular — the assertion corroborating itself). The admissible corroborating data are the git-derived ones: second-parent identity consistent with a surviving branch tip, a registered panel's `reviewed_head_sha`, or an existing consistent binding — NEVER a first-parent subject match alone, and with NO per-spec or global bypass flag. The ADR-0035 q9ea human attested-restore exit MAY be retained for the genuinely-no-mechanical-corroboration case, but only if it is explicitly ADR-blessed (enumerated in the §2(ii) amendment, see ADR Touchpoints), audited, and NEVER the SOLE corroboration for a binding a mechanical scan could instead derive. Every write is **integrity-preserving and auditable**: the record captures the acting identity/authority, the before/after binding values, the timestamp, the invoking operation, and the corroborating git datum, and is append-only (or otherwise tamper-evident), so a wrong fleet-wide backfill is DISTINGUISHABLE and RECONSTRUCTABLE after the fact. On an already-correct binding it converges as a no-op; on a contradictory existing binding it follows the G3-1 discipline (overwrite only with the corroborated, git-derived located identity, recording the prior value in the audit trail — never silently keep the contradiction, never write an uncorroborated replacement). Refusals name the state and the forward exit.
   *Falsified if*: the operation attests on a subject-only match; or accepts an operator-asserted merge/second-parent pair as its own corroboration; or writes a binding not derived from an independent git scan; or a crafted subject-matching decoy merge (or a descendant bead's merge, per R5) gets attested; or a write leaves no reconstructable actor/before-after audit trail; or a bypass flag disables corroboration; or the surface writes bindings implicitly (e.g. from a `doctor` run) rather than under explicit operator invocation.

5. **Merge-candidate identity survives a benign subject variation — with exact-match precedence and uniqueness.** Landed-merge identification MUST NOT be gated on the merge commit carrying the exact `"Merge "+beadBranch` subject; a merge whose SECOND PARENT identifies the bead's landed tip MUST be identifiable even when its subject is git's default conflict-recovery form `Merge branch 'bead/x' into 'spec/…'`. But second-parent corroboration alone is NOT sufficient, because an ANCESTOR-consistent match misattributes across beads: if bead Y branches off `spec` AFTER bead X lands, X's tip is an ANCESTOR of Y's tip, so a naive newest-first `IsAncestor(X_tip, secondParent)` scan returns Y's merge M_Y for `FindLandedMerge(X)` — and the revert-leg then reads M_Y's content, so a genuine revert of only X's content reads as landed. Identification MUST therefore apply:
   - **Exact-second-parent-match PRECEDENCE**: a candidate merge whose second parent EQUALS the bead's landed tip (or its recorded binding SHA) is preferred over any ancestor-only-consistent candidate. The scan selects the exact match when one exists.
   - **Uniqueness**: the selected candidate must UNIQUELY identify THIS bead's merge. An ancestor-consistent later/descendant merge MUST NOT be accepted as X's landed merge when the exact-match candidate for X exists; and when no exact-match exists, an ancestor-only match is admitted only if it does not collide with another bead's exact match.
   - **Subject as disambiguator only**: where the merge subject is still consulted, it disambiguates only AMONG exact/second-parent-consistent candidates, never as a hard gate and never to promote an ancestor-only match over an exact one.
   The three identity sites — `MergeInto` (`gitutil/gitops.go:154`), `locateLandedMergeByIdentity` (`mindspec_executor.go:1469`), `FindLandedMerge` (`lifecycle/landed.go:257`) — MUST corroborate by second-parent identity under this precedence+uniqueness rule. The anti-forgery posture is UNCHANGED: identity still requires a corroborating datum — a subject match ALONE remains inadmissible (§2(ii)), so relaxing the subject requirement never weakens the fail-closed floor. Additionally, the conflict-recovery message (`beadToSpecConflictFailure:1623`) MUST print a recovery that produces an IDENTIFIABLE merge — supplying `-m "Merge <beadBranch>"` on the recovery `git merge --no-ff` line (and/or relying on the second-parent identification above making the subject irrelevant) — so an operator following it verbatim does not strand the binding.
   *Falsified if*: a bead whose merge commit carries git's default `Merge branch '…' into '…'` subject but whose second parent is the bead's landed tip is not identified as landed; or, given bead X and a later descendant bead Y both merged with default subjects, `FindLandedMerge(X)` returns M_Y (the descendant's merge) instead of M_X, or a `git revert` of only X's content is not refused because the revert-leg read M_Y's content; or identity is accepted on a subject match with no second-parent corroboration; or the printed conflict-recovery recovery still yields an unidentifiable merge.

6. **Regression evidence, CI-hermetic.** Every fixed class MUST be pinned by an automated test that is RED on today's `main` (or on fix-revert) and runs in CI's bd-less `go test -short ./...` lane: (a) a real-git e2e through the production `CompleteBead` path in the real topology — INCLUDING the pinned conflict-recovery miss shape (add/add conflict on `.beads/issues.jsonl` → default-subject recovery merge → recovery re-run → binding today absent) — merge, binding persisted, branch deleted, followed by `lifecycle.FindLandedMerge` positively identifying the merge for the branch-DELETED bead (the exact contract spec 124's MF-3 consumes); (b) an explicit `*LandedMergeNoEvidence` fail-closed test preserving the no-datum refusal; (c) the evolved-vs-true-revert discrimination pair of Requirement 3 (evolved shape = tip-moved-BEYOND-base on M's touched content, NOT a pure delete/relocate); (d) the loud-miss refusal of Requirement 2, driven by a merged-WITH-own-commits bead whose locate is forced to miss (not the orphan fixture, which cannot distinguish a correct impl from a re-hiding one); (e) the default-subject/second-parent identification of Requirement 5 INCLUDING the multi-bead X-then-descendant-Y misattribution guard; (f) the `FinalizeEpic` auto-merge/already-ancestor producer legs persisting the binding. Hermeticity per the spec-119 bd-less no-skip-gating convention: bd-touching legs run through the existing injectable seams (`mergeBindingFn`/`mergeBindingReadFn`) backed hermetically, with the seam DEFAULTS pinned to the real `bead.MergeMetadata`/`bead.GetMetadata` symbols by an anti-drift test (the `netEffectLandedFn` AC-17 pattern), so the gate cannot go hollow — no new test may skip-when-bd-missing, and any test that genuinely requires a real `bd` FATALs rather than skips when its declared environment lacks it.
   *Falsified if*: any of the six classes lacks a test that fails on today's behavior; or a new gating test silently skips in the bd-less CI lane; or the seam defaults can be rewired off the real bd write path without a test going red.

## Scope

### In Scope

- `internal/executor/mindspec_executor.go`: `locateLandedMergeByIdentity`, `ensureLandedBinding`, the CompleteBead/FinalizeEpic bind legs, and `beadToSpecConflictFailure`'s recovery line — the diagnosed subject-identity fix, second-parent corroboration, positive nothing-to-bind classification, loud fail-closed refusal, identifiable-recovery-merge message (Requirements 1, 2, 5).
- `internal/gitutil/gitops.go` + `internal/lifecycle/landed.go`: the second-parent-corroborated identity at the two peer sites (`MergeInto`/`FindLandedMerge`), subject demoted from hard gate to disambiguator (Requirement 5).
- `internal/gitutil/neteffect.go` (and, if the plan chooses, a sibling primitive): the reverted-vs-evolved discrimination; boolean projections preserved (Requirement 3).
- `internal/lifecycle/landed.go`: the revert-leg consumption of the corrected discrimination; refusal message accuracy (Requirement 3).
- `cmd/mindspec/**` + `internal/bead` metadata helpers: the re-attest operation and its audit keys (Requirement 4).
- The narrow ADR-0041 §2(ii) re-attestation amendment (ADR Touchpoints; Requirement 4).
- Regression, anti-drift, and e2e tests per Requirement 5 across `internal/executor`, `internal/lifecycle`, `internal/gitutil`.

A natural decomposition is three beads: (1) subject-identity robustness + persistence + loud-miss + recovery-message (R1, R2, R5, and their tests); (2) discrimination (R3 + tests); (3) re-attest surface + ADR amendment (R4 + tests). R6's evidence lands with each. Final structure is a plan decision; note beads 1 and 3 both touch the binding write/read seams and the second-parent identity primitive, so the plan sequences 3 after 1.

### Out of Scope

- **Spec 124's MF-3 / readiness gate** — signal design, fail-posture, ready-check verb: 124's concern. This spec only makes the substrate MF-3 consumes truthful.
- **Panel/gate authority** — no change to the ADR-0037 decision ladder, panel gating, or the complete-gate order.
- **Broad executor refactor** — no restructuring of `CompleteBead`/`FinalizeEpic` beyond the bind legs; the executor/lifecycle import boundary (executor must not import `internal/lifecycle`) stands.
- **bd schema changes** — bindings and audit records stay in the existing free-form metadata map via existing helpers.
- **The attested-restore human path** (`complete.go:1397-1405`, q9ea) — unchanged in semantics; R4 adds a mechanical sibling for corroborable states, it does not relax the no-datum human-marker exit.
- **Retro-fleet backfill execution** — this spec ships the operation; actually running it across historical beads is operator work, not a migration this spec performs. (The backfill surface is explicit-operator-invoked, not an implicit `doctor` write — R4/F3-2.)
- **The routine `.beads/issues.jsonl` add/add merge CONFLICT trigger stays in place** — this spec makes attestation conflict-SOURCE-INDEPENDENT (identity survives the default-subject recovery merge) rather than eliminating the conflict itself; changing the export/commit cadence or the merge driver so 2nd+ beads stop conflicting on the tracker JSONL is a separate concern (F2-2).
- **`directMergeConflictFailure`'s `-m`-less recovery line** (`mindspec_executor.go:1634`) — the sibling spec→main conflict message also prints `git merge --no-ff` without `-m`, but the spec→main subject is NEVER identity-scanned (only bead→spec merges carry landed-binding identity), so this is harmless and left unchanged; Requirement 5's message fix targets `beadToSpecConflictFailure` only (F2-1). If the plan chooses to fix it for consistency, that is permitted, not required.
- **In Scope permits the `beadToSpecConflictFailure` conflict-recovery message change** (`mindspec_executor.go:1623`) — named here for completeness (F3-3): Requirement 5 explicitly touches this recovery line; it is the one recovery-message edit this spec makes.

## Non-Goals

- **No weakening of fail-closed identity.** Subject-only acceptance stays banned everywhere (§2(ii)); this spec makes honest evidence exist and read correctly, it does not lower the evidence bar.
- **No new lifecycle authority for the binding.** It remains corroborating metadata; no consumer gains the right to close/open beads from it.
- **No revert-detection heuristics over commit messages** — "This reverts commit …" subjects are not evidence; the discrimination is content-shape only.
- **No changes to `NetEffectLanded`'s consumers' fail-postures** (the spec-121 probe WARN-and-proceed / suppression skip-and-surface contracts stand).
