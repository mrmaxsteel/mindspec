# Spec 125-landed-merge-attestation-integrity: Landed-merge attestation integrity

## Goal

Make the spec-121 landed-merge attestation substrate TRUE in real practice. Two confirmed, pre-existing bugs sit on the merge critical path — the path every `mindspec complete` and `impl approve` crosses — and together they make the substrate's promise false end-to-end: **(1)** `mindspec complete` never actually persists the merge-time landed-binding ADR-0041 §2(ii) mandates — `ensureLandedBinding` swallows a locate MISS as a silent nil (`internal/executor/mindspec_executor.go:1516-1519`) and writes no `mindspec_landed_merge_sha`, so `internal/lifecycle.FindLandedMerge` later fail-closes with `*LandedMergeNoEvidence` on every branch-deleted, genuinely-merged bead; and **(2)** the revert-awareness leg misclassifies EVOLVED content as REVERTED — `gitutil.ContentSubsumedOutcome` returns `SubsumptionCleanDivergence` ("backed out") for a dependency whose touched surface later honest work evolved cleanly (the real `mindspec-8nhe.2` case), so `FindLandedMerge` refuses a dep whose work is still present. Every landed-merge consumer inherits both: `complete`'s merged-unclosed reconcile, the doctor stale-OPEN cross-check, and spec 124's MF-3 readiness signal, which today false-rejects EVERY genuinely-merged dependency — a ceremony trapdoor.

The diagnosed root cause of Bug 1 (reproduced E2E, **755/757 closed beads across the fleet — including specs 121/122/123 — carry no binding**) is deeper than a bare locate miss: merge-candidate identity is keyed to an EXACT subject string `"Merge "+beadBranch`, and that string is routinely defeated by conflict-recovery. The `.beads/issues.jsonl` export is auto-committed into every bead and spec worktree, so the 2nd+ bead under a spec routinely hits a real add/add conflict on it; `CompleteBead`'s conflict path (`beadToSpecConflictFailure`, `internal/executor/mindspec_executor.go:1600`) prints the recovery `git merge --no-ff <beadBranch>` with **no `-m`** (`:1623`), so an operator/agent following it verbatim produces git's DEFAULT subject `Merge branch 'bead/x' into 'spec/…'`, which the identity scan never matches; the recovery re-run of `mindspec complete` then sees an already-ancestor branch (no-op merge), the subject scan misses, and `ensureLandedBinding` silently returns nil. So the fix is structural, not just outcome-level: identity must survive a benign subject variation.

Target outcome: after any real `mindspec complete` that performs a bead→spec merge — INCLUDING the conflict-recovery path — the landed-binding is durably on the bead (visible in `bd show`); merge-candidate identity is corroborated by second-parent identity, not defeated by a benign subject variation; a genuine bind failure is LOUD, fail-closed, and cleanup-suppressing — never a silent nil; content evolved by later honest work is identified as landed while genuinely backed-out content still refuses; already-merged history (the current fleet state: closed beads, deleted branches, no bindings) is recoverable through an audited, fail-closed, identity-corroborated re-attest operation; and every fixed class is pinned by a CI-hermetic regression test.

## Background

All citations verified in this worktree (fork of `main` at `ae53877e`, post-spec-123).

### Bug 1 — `complete` never persists the landed-binding

- Both producer legs record the binding fail-closed-before-cleanup exactly as ADR-0041 §2(ii) requires — `CompleteBead` (`internal/executor/mindspec_executor.go:448-453`) and `FinalizeEpic`'s auto-merge loop (`:672`, `:726`) call `ensureLandedBinding` before any branch/worktree cleanup, and a bind-WRITE failure suppresses cleanup via `guard.NewFailure`. But `ensureLandedBinding` (`:1514`) first calls `locateLandedMergeByIdentity(g.Root, specBranch, beadBranch)` (`:1468`), and on a locate MISS (`errNoLandedMergeIdentified`, `:1451`) it **silently returns nil** (`:1516-1519`) — no binding written, no warning, cleanup proceeds, branch deleted. The silent-nil design assumed the only miss is the legitimate zero-own-commits ancestor branch ("nothing to bind"); in real practice the locate misses for beads that DID merge with real commits, so the assumed-impossible state is the common one.
- Downstream, `FindLandedMerge` (`internal/lifecycle/landed.go:249`) then has no admissible corroboration for the branch-deleted bead: no surviving branch tip, usually no registered-panel `reviewed_head_sha` at the consuming site, no binding — subject match alone is inadmissible by design (§2(ii)) — so it returns `*LandedMergeNoEvidence` (`landed.go:337-345`) on every merged dep. **Proven**: manually writing `mindspec_landed_merge_sha`/`mindspec_landed_second_parent` onto the bead makes the consumer pass — the engine, read path (`landedBindingForBead`, `landed.go:154-160`), corroboration ladder, and write path (`bead.MergeMetadata`) all work; only `complete`'s write goes missing.
- **The exact miss condition is now PINNED** (was an open question; the parallel diagnostic reproduced it E2E with real `mindspec`+`bd`). It is NOT root-vs-worktree resolution (disproven). It is **exact-merge-subject identity defeated by conflict-recovery**: `MergeInto` (`internal/gitutil/gitops.go:154-164`), `locateLandedMergeByIdentity` (`mindspec_executor.go:1469`), and `FindLandedMerge` (`internal/lifecycle/landed.go:257`) all require the merge commit's subject to be exactly `"Merge "+beadBranch`. The auto-committed `.beads/issues.jsonl` (bd `no-auto-flush:false`) makes the 2nd+ bead under a spec routinely hit a real add/add conflict; the conflict-recovery message (`beadToSpecConflictFailure:1623`) omits `-m`, so the human/agent recovery merge lands git's default subject `Merge branch 'bead/x' into 'spec/…'`; the recovery re-run of `complete` finds the bead already an ancestor (no-op merge), the exact-subject scan misses, and the silent-nil swallow completes with no binding. This is why 755/757 fleet beads lack a binding. The structural fix (Requirement 5) is second-parent identity corroboration so a benign subject variation cannot defeat identification; Requirements 1/2 make persistence reliable and any residual miss loud; the recovery message is fixed to emit an identifiable merge.

### Bug 2 — evolved content reads as reverted

- `FindLandedMerge`'s revert-awareness leg (`landed.go:347-382`) discriminates `ContentSubsumedOutcome`'s trichotomy correctly at its own level: `SubsumptionConflict` → landed-then-evolved → identified; `SubsumptionCleanDivergence` → backed out → refused (`landed.go:379-382`).
- But the underlying primitive (`internal/gitutil/neteffect.go:217-244`) classifies as `SubsumptionCleanDivergence` ANY clean three-way whose result differs from the tip — which captures not only the true `git revert` shape (tip returned to the pre-merge state on the merge's own paths) but also evolved shapes where later honest work changed the merge's touched surface in a way that still auto-merges cleanly (content relocated, removed-and-replaced, overlapping-surface doc reconciliation). The doc comment concedes the residual ("content later removed cleanly and fully by honest work is content-indistinguishable from a revert", `landed.go:370-374`); `mindspec-8nhe.2` — later beads and doc-recon touched the dep's overlapping surface — proves the residual is the REAL evolved case, not an edge. Existing fixtures (`landed_test.go:712`, `:760`) only pin evolved shapes that CONFLICT; no fixture pins the evolved-but-cleanly-divergent shape, so the false-reject ships green.
- The refusal is a §2(i) violation in practice: no recovery converges (the content genuinely evolved; restoring the branch re-corroborates and re-refuses at the same leg), leaving only the bd-close bypass or reverting legitimate later work — the exact deadlock class ADR-0041 forbids.

### Consumers and blast radius

- `complete`'s merged-unclosed reconcile treats `*LandedMergeNoEvidence` as the attested-restore refusal (`internal/complete/complete.go:606-613`, `:1397-1405`) — a manual, human-marker recovery now demanded for every routinely-completed bead.
- The doctor stale-OPEN cross-check consumes the same predicate (`internal/lifecycle/stale_open.go:92`).
- Spec 124's MF-3 ("dependency closed AND landed-merged") consumes `FindLandedMerge` verbatim and fail-closes on any error — with Bug 1, that is a false FAIL for every dependency completed by the real lifecycle. This spec is 124's prerequisite; MF-3 itself is out of scope here.

## ADR Touchpoints

- [ADR-0041: Gate-Before-Mutate](../../adr/ADR-0041-gate-before-mutate.md) — **AMENDED by this spec (narrowly), otherwise APPLIED**. Requirement 4 DOES need one narrow §2(ii) touchpoint: §2(ii) admits "the merge-time landed-binding" as a corroborating datum, and a recovery re-attestation writes that same datum AFTER merge time. The amendment records that the binding's admissibility derives from its corroborated-identity discipline, not from when it is written: a re-attested binding is admissible iff DERIVED from an independent git EXACT-second-parent match (Requirement 5: exact-match required, fail-closed on ambiguity) — never from an operator-asserted SHA pair corroborating itself, never subject-only, and never an agent-writable binding uncorroborated to a real exact merge — and carrying an INSPECTABLE audit record (acting identity/authority, before/after values, timestamp, invoking operation, corroborating datum) held in bd metadata: detectable-by-inspection, not cryptographically tamper-proof. The amendment MUST enumerate the admissible datum set so it matches R4, each STANDALONE: (a) the git-derived exact-second-parent merge from the scan (the branch-deleted happy path — no surviving tip required); (b) a surviving bead-branch tip the exact match equals; (c) a registered panel's `reviewed_head_sha` equal to an exact match's second parent; (d) an existing binding git-corroborated to an exact merge; and (e) the ADR-0035 q9ea human attested-restore marker — admitted ONLY as an explicitly-blessed, audited exit for the genuinely-no-corroboration case, and NEVER the sole corroboration where a scan could derive the identity. The amendment also records that the recovery is honestly scoped: it re-attests the git-corroborable subset and refuses the rest to the audited exit rather than claiming fleet-wide recovery. Amendment lifecycle per the 122/123 precedent: pre-drafted at plan time, landed by the same bead that lands the re-attest surface, so the ADR-divergence gate sees the declared touchpoint (pinned by AC-11).

## Requirements

Each requirement carries a hard falsifier. "Real topology" below means the shape production `complete` actually runs: merge performed in the spec worktree, locate/bind driven from the main root (`g.Root`), bead branch and worktree deleted immediately after.

**Root-of-trust principle (applies throughout R2–R5).** Two orthogonal facts are established by two different sources, and neither alone suffices:
- **OWNERSHIP — "which bead does this merge belong to?" — is the merge SUBJECT's bead-branch name.** A candidate merge is bead X's merge ONLY if its subject names `bead/X`, parsed as the bead-branch name from EITHER subject shape: `Merge bead/mindspec-X` OR git's default conflict-recovery form `Merge branch 'bead/mindspec-X' into '…'` (the branch name survives in both, so ownership is robust to the conflict-recovery variation).
- **LANDED-NESS — "did this merge actually land?" — is git TOPOLOGY**: a first-parent merge commit on `<specBranch>`, with exactly two parents, whose SECOND parent EQUALS the bead's landed tip (an EXACT second-parent match against a REAL merge), unique for that bead.

The agent-writable landed-binding metadata and a panel's `reviewed_head_sha` are a git-CORROBORATED CACHE, never authorities: they are trusted ONLY when they resolve to a merge that BOTH names this bead (ownership) AND has the exact second parent (landed-ness). A cached SHA that points at another bead's real merge fails the ownership check; one that points at no real exact merge fails the topology check; either way it is discarded, not followed. Git-corroborating a binding proves it points at a real merge — NOT that the merge is THIS bead's, which is why the subject-parsed ownership nominator is required alongside it. This stays consistent with ADR-0041 §2(ii) "never subject text ALONE": the subject only NOMINATES which bead, and topology proves landed-ness — neither is trusted alone.

Identification FAILS CLOSED — refuse — whenever ownership+landed-ness cannot be jointly established: no exact match, or the only candidates are ancestor-consistent (not exact), or competing candidates naming DIFFERENT beads. (Multiple merges that share the SAME exact second parent are NOT an ownership ambiguity — they are the same bead's re-merges; see R5's newest-exact rule.)

**Path asymmetry.** The AUTOMATIC `FindLandedMerge` path requires an EQUAL/exact corroborating datum and refuses the bare no-datum case (AC-10 pins the subject-only-spoof refusal there). The EXPLICIT, operator-invoked R4 re-attest path additionally admits subject-name nomination + topology corroboration + an audit record + operator vouching — a strictly more permissive but deliberately-gated surface; R4's nominate-then-corroborate datum MUST NOT leak into the automatic path.

Scoped to what the ACs pin: this principle closes the two concrete misattribution/spoof shapes exercised here — a cached SHA pointing at NO real exact merge (forgery, AC-2c) and one pointing at a DIFFERENT bead's real merge (ownership, AC-2d) — and the bare-fleet refusal; it does not claim to close shapes no AC exercises.

**Threat model / boundary (stated honestly).** Landed-merge identification trusts the INTEGRITY of the spec branch's git history. A party who can WRITE commits to the spec branch can forge a landed merge — author a `Merge bead/X` commit (either subject form) over an arbitrary second parent, plus a matching binding — and it will pass ownership+topology, because git history alone cannot distinguish a genuine merge from a forged one without out-of-band signatures. This is **INHERENT to any git-history-based attestation and PRE-EXISTING**: spec 121 already trusts the subject to name the bead, and that party already has full repository control AND the audited `--allow-not-ready` override (spec 124). Such an actor is therefore OUT of 125's defended threat model, and 125 attempts no git-only "fix" (there is none). What 125 DOES defend — and pins with ACs — is **accidental/mistaken misattribution and silent failure** under an honest, non-adversarial history: the wrong-cache→another-bead's-honest-merge case (AC-2d), the descendant-merge misattribution (AC-2c), the conflict-recovery subject variation (AC-1b/AC-2b), stale/non-exact bindings (AC-2c), and the silent-nil persistence failure (R1/R2). 125 does NOT worsen the pre-existing subject-trust: it ADDS topology corroboration (exact second parent + real merge) ON TOP of the subject-name ownership check spec 121 already relied on, so every path is strictly more constrained than before, never less.

1. **`complete` reliably persists the landed-binding.** After every bead→spec merge performed by `mindspec complete` (and by `FinalizeEpic`'s auto-merge leg) for a bead branch with at least one own commit, the landed-binding (`mindspec_landed_merge_sha` + `mindspec_landed_second_parent`, per the existing key contract) MUST be durably recorded on the bead — surviving into `bd show <id> --json` metadata and matching the actual first-parent merge commit on the spec branch — across the real topology, BEFORE the branch/worktree cleanup that destroys the surviving-branch datum. **WRITE vs READ (no contradiction with the R5 ownership model).** The WRITE path here (`ensureLandedBinding` during `complete`) knows the bead identity DIRECTLY — it is completing bead X — and writes the binding from that ground truth; it does NOT parse the merge subject to establish ownership, which is precisely why R1 persists **regardless of the merge's subject FORMAT** (default conflict-recovery or exact `Merge bead/X`). Subject-name parsing (the R5 ownership nominator) is a READ-path concern only: `FindLandedMerge` reading a branch-deleted bead with no surviving binding, and R4 re-attest, use it to NOMINATE which bead an unattributed historical merge belongs to. So "persist regardless of subject line" (write-time ground truth) and "subject must name bead/X" (read-time nomination) are consistent — they govern different paths. The fix MUST target the pinned root cause (Background; Requirement 5): the R1 regression fixture reproduces the real miss shape (conflict on `.beads/issues.jsonl` → default-subject recovery merge → recovery re-run) so it is RED on today's `main`, and the binding is persisted whenever the bead's work has actually landed as a merge on the spec branch. *(Residual, stated: a merge with a wholly-custom subject naming NO bead is READ-identifiable only via a binding written at complete-time — which R1's ground-truth write guarantees going forward; a historical such merge with no binding fails closed at read, which is safe.)*
   *Falsified if*: a bead with own commits, completed by a real `mindspec complete` run (including via the conflict-recovery path), lacks `mindspec_landed_merge_sha` in its bd metadata afterward; or the recorded SHAs do not match the actual merge commit and its second parent; or the binding is written only after cleanup has already deleted the branch datum.

2. **A genuine locate/bind failure is LOUD (fail-closed), never silent.** `ensureLandedBinding` MUST NOT return nil on a locate miss for a bead that DID merge. The legitimate "nothing to bind" state (a trivially-ancestor bead branch that never produced a bead→spec merge commit — a bd_close orphan that is already an ancestor of `spec` with zero own commits, so `git merge --no-ff` created no merge) MUST be POSITIVELY classified by **first-parent MEMBERSHIP**: whether the bead's tip appears as the SECOND parent of any first-parent merge on `<specBranch>` (`git log --first-parent --merges <specBranch>`, inspecting each merge's second parent — the root-of-trust exact-match test above). A bead that landed via a bead→spec MERGE has its tip as such a second parent; a true orphan does not. **A count/merge-base classifier is proven insufficient** (Background/diagnostic): after the merge, a genuinely-merged-then-ancestor bead is byte-identical to a true orphan under both `rev-list beadTip ^specBranch` (both 0) and `merge-base==tip` (both yes), because `ensureLandedBinding` runs AFTER the merge — so an own-commit-count classifier reads "nothing to bind" for a real merge and RE-HIDES the bug. "Nothing to bind" is therefore admissible ONLY when the bead's tip is the second parent of NO first-parent merge on `spec`. Any other miss — a bead whose tip IS such a second parent (it did merge) but whose merge commit cannot otherwise be located, or whose binding cannot be written — refuses via `guard.NewFailure`, suppresses branch/worktree cleanup (preserving the surviving-branch datum, so re-invocation converges per §2(ii)), and names the recovery.
   *Falsified if*: any code path swallows a bind failure or locate miss for a bead that did merge and proceeds to cleanup; or the nothing-to-bind classification is derived from own-commit-count / merge-base (which cannot distinguish a merged-then-ancestor bead from a true orphan) rather than from first-parent-membership (second-parent-of-a-merge) evidence; or the refusal deletes the branch it needs for recovery.

3. **Revert-vs-evolved discrimination is correct.** *(Round-3 note: the R3 CONTRACT and ACs are unchanged — code-verified sound in round 2; this is a PROSE-ONLY tightening to word the predicate to MATCH the `ContentSubsumedOutcome` primitive rather than backwards.)* For a two-parent merge M in the spec branch's first-parent history, the landed-identity decision is the three-way content merge (base = M^1, ours = tip, theirs = M) at the granularity of **M's CHANGED CONTENT — the region/hunks M actually introduced — NOT per-file-path**. Reading the primitive's outcomes:
   - **EVOLVED (identified)** — M's content is PRESENT at the tip. Either re-applying M's diff to the tip is a NO-OP (the merge result's tree equals the tip's tree → `SubsumptionLanded`), or it CONFLICTS because the tip advanced past base on M's own region (`SubsumptionConflict`, today's already-correct leg, preserved). "Present" includes content that later honest work moved to a DIFFERENT path, renamed, or rewrote/superseded — it still exists at the tip. This covers the `mindspec-8nhe.2` clean-divergence-but-present shape that is today's false-reject. Per-file-path reasoning is explicitly WRONG: later honest work on a DIFFERENT region of a file M also touched leaves M's own content present, so it is EVOLVED, never a revert.
   - **REVERTED (not identified)** — M's content is ABSENT at the tip: the tip sits at the base (M^1) state on M's touched content. Operationally, the three-way applies M's diff CLEANLY but the result DIFFERS from the tip (`SubsumptionCleanDivergence`) — re-applying M would RE-INTRODUCE content the tip no longer carries. This is exactly the `git revert M` shape.
   - **Stated residual (REFUSE, not over-claimed)** — M's content is ABSENT because later honest work removed it CLEANLY AND FULLY, or deleted/relocated-to-nowhere M's own paths. Absent-because-reverted and absent-because-cleanly-removed are content-INDISTINGUISHABLE, so this residual refuses too — a deliberate false-negative floor: any datum that accepted clean full removal would accept every real revert. (Contrast the EVOLVED "relocated": content moved-to-a-different-path is PRESENT and identifies; content relocated-to-NOWHERE is ABSENT and refuses.)
   **Scope — 125 fixes ONLY the CleanDivergence arm.** Spec 125's R3 change is confined to the `SubsumptionCleanDivergence`-on-evolved false-NEGATIVE (the 8nhe.2 bug: evolved-content-present wrongly refused). The `SubsumptionConflict → identified` arm (`landed.go:355-382`, the spec-121 F2-2r fix) is left UNTOUCHED. A KNOWN pre-existing residual lives on that untouched arm: an extreme revert-then-DIVERGENTLY-rewritten region can present as `SubsumptionConflict` and thus be identified — a false-POSITIVE. This is a DELIBERATE spec-121 trade-off, NOT a 125 regression: refusing the Conflict case resurrects the permanent-refusal DEADLOCK ADR-0041 §2(i) forbids (evolved work that conflicts would never converge). 125 does not worsen it — the implementation MUST change only the CleanDivergence path (grep-confirmable: the Conflict arm's behavior is byte-identical before/after 125). A precise content-presence discriminator that closes this residual WITHOUT re-introducing the deadlock is a named FOLLOW-UP, not in 125's scope (see Non-Goals).
   **Octopus/parent guard**: MindSpec bead→spec merges are exactly two-parent; the identity/discrimination MUST require a candidate with ≥2 parents AND the bead's second-parent among them, and MUST NOT attempt this discrimination on an octopus (>2-parent) or non-bead merge — such a candidate is excluded, never guessed. The discrimination mechanism (a sub-classification of `SubsumptionCleanDivergence`, the region-level three-way, or a sibling primitive) is plan-level; the boolean `ContentSubsumed`/`NetEffectLanded` projections consumed by the protected-main probe and the doctor merged-carrier suppression (spec 121 R4) keep their existing behavior.
   *Falsified if*: the evolved-content-PRESENT fixture (later first-parent commits leave M's content present but move the tip off base — removed-AND-replaced or M's region rewritten, NOT a pure delete/relocate-to-nowhere — such that today's `ContentSubsumedOutcome` returns `SubsumptionCleanDivergence`) is still refused; or later work on a DIFFERENT region of a path M touched is read as a revert of M; or a genuine `git revert M` (M's content now ABSENT) with no later rework is identified as landed; or the spec-121 revert-then-reapply and boolean-consumer contracts regress.

4. **A git-corroborated re-attest recovery — recovers the corroborable, refuses the rest.** An **explicit operator-invoked** `mindspec`-surface operation (verb naming plan-level; it MUST NOT be an implicit-writing `doctor` check — the mass-mutation vector across the merged-bead history stays gated behind a deliberate command) MUST exist to (re)attest the landed-binding of an already-merged bead. It is **fail-closed and DERIVES the binding from INDEPENDENT git topology, never from the assertion it is validating.** The bead-branch NAME (from the subject or the bead ID) may only NOMINATE a candidate; the WRITE requires a git-corroborated EXACT second-parent match under the Requirement 5 rule — ownership (the exact-matched merge's subject must name THIS bead, so a cache pointing at another bead's real merge is refused) AND landed-ness (exact second parent, real merge); fail-closed on ambiguity; for two same-second-parent exact matches, the newest names the merge while Requirement 3's content-check anchored on the OLDEST merge M₁ governs landed-vs-reverted (per Requirement 5). It writes ONLY the SHAs that scan derives. This is the deliberately more-permissive EXPLICIT path (subject-name nomination + topology + audit + operator vouching); its nominate-then-corroborate datum MUST NOT leak into the automatic `FindLandedMerge` path, which refuses the bare no-datum case (per the root-of-trust path-asymmetry note). An operator-asserted merge/second-parent pair is **NOT admissible as its own corroboration** (circular). The admissible corroborating data, each STANDALONE (matching the §2(ii) amendment enumeration): **(a)** the git-derived exact second-parent merge from the scan — this is the branch-deleted HAPPY PATH and needs NO surviving tip; **(b)** a surviving bead-branch tip, when one exists, whose SHA the exact match must equal; **(c)** a registered panel's `reviewed_head_sha`, admissible only when it equals an exact match's second parent; **(d)** an existing binding, admissible only when git-corroborated per R5. NEVER a subject match alone, NEVER an agent-asserted pair alone, and with NO per-spec or global bypass flag.
   **Honest recovery scope (not fleet-wide over-claim).** This operation RECOVERS exactly the beads whose landed merge is still git-corroborable by an exact second-parent match; it REFUSES — to the audited ADR-0035 q9ea human attested-restore exit — every bead with no corroborable exact merge (a truly-bare state: branch deleted, no binding, no exact merge, no panel-sha) rather than guessing. It therefore does NOT claim to recover the entire merged-bead history: the corroborable subset is re-attested; MF-3 (spec 124) safely fail-closes on the unrecoverable remainder; and R1 fixes attestation going FORWARD so the remainder stops growing. The q9ea exit MAY be used only for the genuinely-no-mechanical-corroboration case, is explicitly ADR-blessed (enumerated in the §2(ii) amendment, see ADR Touchpoints), is audited, and is NEVER the sole corroboration where a scan could instead derive the identity.
   **Audit — inspectable in metadata, honestly scoped.** Every write records, in bd metadata via the existing helpers (option (b) of the two honest choices — no external anchor is introduced), the acting identity/authority, the before/after binding values, the timestamp, the invoking operation, and the corroborating git datum. This record is INSPECTABLE — a wrong backfill is detectable and reconstructable BY INSPECTION of the recorded before/after values — but it lives in the same mutable bd metadata map, so it is **detectable-by-inspection, NOT cryptographically immutable/tamper-proof**; the spec claims exactly that and no more (a git-trailer/append-only external anchor is a deliberate non-goal here). On an already-correct binding it converges as a no-op; on a contradictory existing binding it follows the G3-1 discipline (overwrite only with the git-corroborated exact identity, recording the prior value in the audit — never silently keep the contradiction, never write an uncorroborated replacement). Refusals name the state and the forward exit.
   *Falsified if*: the operation attests on a subject-only match, an agent-asserted pair, or any non-exact (ancestor-only) candidate; or writes a binding not derived from an independent git exact-match scan; or a crafted subject-matching decoy merge (or a descendant bead's merge, per R5) gets attested; or it claims to recover a bead with no git-corroborable exact merge instead of refusing to the audited exit; or a write leaves no inspectable before/after/actor audit record; or the spec claims the metadata audit is tamper-proof rather than detectable-by-inspection; or a bypass flag disables corroboration; or the surface writes bindings implicitly (e.g. from a `doctor` run) rather than under explicit operator invocation.

5. **Merge-candidate identity is an EXACT second-parent match, fail-closed on ambiguity.** Landed-merge identification MUST NOT be gated on the merge commit carrying the exact `"Merge "+beadBranch` subject; a merge whose SECOND PARENT identifies the bead's landed tip MUST be identifiable even when its subject is git's default conflict-recovery form `Merge branch 'bead/x' into 'spec/…'`. But an ANCESTOR-consistent match misattributes across beads: if bead Y branches off `spec` AFTER bead X lands, X's tip is an ANCESTOR of Y's tip, so a naive newest-first `IsAncestor(X_tip, secondParent)` scan returns Y's merge M_Y for `FindLandedMerge(X)` — the revert-leg then reads M_Y's content, so a genuine revert of only X's content reads as landed. Per the root-of-trust principle above, positive identification therefore REQUIRES a git-corroborated EXACT second-parent match, and REFUSES on every ambiguity:
   - **Ownership + landed-ness are BOTH required.** A candidate is bead X's landed merge only if (ownership) its subject names `bead/X` — parsed from either `Merge bead/X` or the default `Merge branch 'bead/X' into '…'` form — AND (landed-ness) it is a two-parent first-parent merge on `spec` whose second parent EQUALS X's landed tip (an EXACT match against a real merge). An ancestor-only-consistent candidate is NEVER a positive identification on its own; a candidate that exact-matches but whose subject names a DIFFERENT bead is rejected (it is that other bead's merge). Ownership matching MUST use FULL bead-branch-name EQUALITY (the parsed token equals `bead/<full-id>` exactly), never `HasPrefix`/`Contains` — so a prefix-colliding pair like `mindspec-8nhe.1` vs `mindspec-8nhe.12` never cross-matches: a collision fails SAFE (no positive ID → refuse), never a false attribution. The parsed subject token is COMPARED DATA, never a git operand, and is `termsafe.Escape`-d before appearing in any refusal message (the spec-116/120 G2-1 discipline for agent-writable text).
   - **The agent-writable binding / panel-sha is a git-corroborated CACHE, never an authority.** A recorded `mindspec_landed_second_parent` (or `reviewed_head_sha`) is trusted ONLY when it resolves to a real merge that BOTH exact-matches the bead's tip AND whose subject names THIS bead. A forged/stale cache that points at no real exact merge (fails topology) OR at another bead's real merge (fails ownership) yields no valid identification — it is discarded, and identification refuses rather than following the cache. This closes both the no-real-match forgery (G1-3, AC-2c) and the another-bead's-real-merge misattribution (G2-R3-B1/F1-7, AC-2d).
   - **No exact-and-owned match ⟹ REFUSE (never newest-ancestor).** When no first-parent merge both names the bead and exact-matches its tip — and only ancestor-consistent or differently-owned candidates remain (reachable in the branch-deleted fleet state via an ancestor `reviewed_head_sha`, where a panel reviewed an EARLIER head, or via a cache pointing at a non-exact SHA) — identification FAILS CLOSED (`*LandedMergeNoEvidence` / refusal), NEVER a newest-ancestor pick. This "refuse on ambiguity" applies to candidates with DIFFERENT second parents or competing bead-ownership — genuine ambiguity about WHICH merge is this bead's.
   - **Two merges sharing the SAME exact second parent ⟹ same bead's re-merges: identify the merge as newest, but base the content-check on the ORIGINAL landing.** A revert-then-reapply or a re-merge lands the SAME second parent (and the SAME owning bead) on `spec` more than once. These are NOT an ownership ambiguity — they are one bead's repeated merges of the same tip — so the merge COMMIT reported (the `*LandedMerge.SHA`) is the NEWEST such same-second-parent exact match (nearest the tip on the first-parent chain). But the landed-vs-reverted CONTENT-CHECK MUST NOT be anchored on that newest merge: a later re-merge's own `M^1` can itself be the POST-REVERT state, so Requirement 3's three-way taken over the newest merge reads "no change" and MIS-attests a bead whose content is actually reverted at the tip (G2-R4-B1). The content-check REUSES Requirement 3's discrimination VERBATIM — it defines NO three-way of its own — parameterized by **M = the OLDEST same-second-parent merge M₁** (the pre-first-landing anchor): Requirement 3's three-way `(base = M₁^1, ours = current tip, theirs = M₁)` — equivalently `ContentSubsumedOutcome(base = M₁^1, ref = M₁, target = tip)`, exactly R3's argument mapping with M = M₁. The newest merge M₂ governs ONLY the reported `*LandedMerge.SHA`; it supplies NEITHER the base NOR the `theirs = M` merge of the R3 check. This yields: a revert-then-re-merge where the bead's landed content is absent at the tip → `SubsumptionCleanDivergence` → REVERTED → refuse (never `SubsumptionLanded` via a post-revert no-op anchor). The single-merge case is identical to R3: one merge ⇒ M₁ == M₂ == that merge ⇒ R3's `(base = M^1, ours = tip, theirs = M)` exactly. This is disambiguation AMONG same-owner exact matches only; it never promotes an ancestor-only or differently-owned candidate, so it does not contradict the fail-closed-on-ambiguity principle (which governs different-owner/non-exact candidates).
   The three identity sites — `MergeInto` (`gitutil/gitops.go:154`), `locateLandedMergeByIdentity` (`mindspec_executor.go:1469`), `FindLandedMerge` (`lifecycle/landed.go:257`) — MUST identify by this ownership-named, exact-match-required, fail-closed-on-ambiguity rule. The anti-forgery posture is UNCHANGED and consistent with §2(ii): the subject NOMINATES ownership but is never landed-ness evidence ALONE, and an agent-writable cache alone is likewise inadmissible unless git-corroborated to an exact, owned merge. Additionally, the conflict-recovery message (`beadToSpecConflictFailure:1623`) MUST print a recovery that produces an IDENTIFIABLE merge — supplying `-m "Merge <beadBranch>"` on the recovery `git merge --no-ff` line (and/or relying on the second-parent identification above making the subject irrelevant) — so an operator following it verbatim does not strand the binding.
   *Falsified if*: a bead whose merge commit carries git's default `Merge branch '…' into '…'` subject but whose second parent is the bead's landed tip is not identified as landed; or, given bead X and a later descendant bead Y both merged with default subjects, `FindLandedMerge(X)` returns M_Y instead of M_X, whether the exact-match for X exists (AC-2b) or NOT (AC-2c: with no binding, an ancestor `reviewed_head_sha`, and a descendant merge present, identification returns the newest ancestor-consistent merge instead of REFUSING); or a cached SHA pointing at a merge whose subject names a DIFFERENT bead is followed to a positive identification instead of being discarded (AC-2d); or a `git revert` of only X's content is not refused because the revert-leg read the wrong merge; or a revert-then-re-merge-of-the-same-second-parent whose content is ABSENT at the current tip is attested as landed because the content-check used the newest re-merge's `M^1` base instead of the original-landing base / content-at-tip (G2-R4-B1); or ownership matches by prefix/substring so a prefix-colliding bead id cross-attributes instead of failing safe; or identity is accepted on a subject/cache hint with no git-corroborated exact AND owned merge; or the printed conflict-recovery recovery still yields an unidentifiable merge.

6. **Regression evidence, CI-hermetic.** Every fixed class MUST be pinned by an automated test that is RED on today's `main` (or on fix-revert) and runs in CI's bd-less `go test -short ./...` lane: (a) a real-git e2e through the production `CompleteBead` path in the real topology — INCLUDING the pinned conflict-recovery miss shape (add/add conflict on `.beads/issues.jsonl` → default-subject recovery merge → recovery re-run → binding today absent) — merge, binding persisted, branch deleted, followed by `lifecycle.FindLandedMerge` positively identifying the merge for the branch-DELETED bead (the exact contract spec 124's MF-3 consumes); (b) an explicit `*LandedMergeNoEvidence` fail-closed test preserving the no-datum refusal; (c) the evolved-vs-true-revert discrimination pair of Requirement 3 (evolved shape = tip-moved-BEYOND-base on M's touched content, NOT a pure delete/relocate); (d) the loud-miss refusal of Requirement 2, driven by a merged-WITH-own-commits bead whose locate is forced to miss (not the orphan fixture, which cannot distinguish a correct impl from a re-hiding one); (e) the exact-second-parent identification of Requirement 5 INCLUDING the multi-bead X-then-descendant-Y misattribution guard BOTH when the exact match exists (AC-2b) AND when it does not — the fail-closed-on-ambiguity case (AC-2c), the ownership check that discards a cache pointing at another bead's real merge (AC-2d), the revert-then-re-merge content-base masked-revert guard (AC-2e), and the full-name-equality no-prefix-collision guard (AC-2f); (f) the `FinalizeEpic` auto-merge/already-ancestor producer legs persisting the binding. Hermeticity per the spec-119 bd-less no-skip-gating convention: bd-touching legs run through the existing injectable seams (`mergeBindingFn`/`mergeBindingReadFn`) backed hermetically, with the seam DEFAULTS pinned to the real `bead.MergeMetadata`/`bead.GetMetadata` symbols by an anti-drift test (the `netEffectLandedFn` AC-17 pattern), so the gate cannot go hollow — no new test may skip-when-bd-missing, and any test that genuinely requires a real `bd` FATALs rather than skips when its declared environment lacks it.
   *Falsified if*: any of the six classes lacks a test that fails on today's behavior; or a new gating test silently skips in the bd-less CI lane; or the seam defaults can be rewired off the real bd write path without a test going red.

- `cmd/mindspec/**` + `internal/bead` metadata helpers: the re-attest operation and its audit keys (Requirement 4).

## Non-Goals

- **No weakening of fail-closed identity.** Subject-only acceptance stays banned everywhere (§2(ii)); this spec makes honest evidence exist and read correctly, it does not lower the evidence bar.
- **No new lifecycle authority for the binding.** It remains corroborating metadata; no consumer gains the right to close/open beads from it.
- **No revert-detection heuristics over commit messages** — "This reverts commit …" subjects are not evidence; the discrimination is content-shape only.
- **No fix for the `SubsumptionConflict`-hides-revert false-POSITIVE residual.** The extreme revert-then-divergently-rewritten shape that reads as `SubsumptionConflict → identified` is a PRE-EXISTING spec-121 F2-2r trade-off (refusing it resurrects the §2(i) permanent-refusal deadlock). 125 leaves the Conflict arm untouched and fixes only the CleanDivergence false-negative; a content-presence discriminator that closes the residual without the deadlock is a named follow-up.
- **No defense against a forged spec-branch git history.** A party who can write commits to the spec branch can forge a landed merge + matching binding that passes ownership+topology — inherent to git-history-based attestation with no out-of-band signatures, pre-existing since spec 121, and already covered by that party's full repository control plus the audited `--allow-not-ready` override. 125 defends accidental/mistaken misattribution and silent failure, not an adversary with spec-branch write access (Threat model / boundary above). No git-only fix is attempted because none exists.
- **No changes to `NetEffectLanded`'s consumers' fail-postures** (the spec-121 probe WARN-and-proceed / suppression skip-and-surface contracts stand).
- **No cryptographically tamper-proof audit.** The re-attest audit is recorded in bd metadata (existing helpers, ADR-0025-respecting), making it detectable-and-reconstructable BY INSPECTION — not immutable. An external append-only/git-trailer anchor is deliberately out of scope; R4 claims exactly the metadata-level guarantee and no more.
- **No fleet-wide recovery guarantee.** R4 re-attests only the git-corroborable (exact-second-parent) subset of merged beads and refuses the rest to the audited exit; it does not claim to reconstruct bindings for beads whose landed merge is no longer git-corroborable. MF-3 fail-closes on that remainder; R1 stops the remainder from growing.
