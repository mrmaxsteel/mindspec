# Spec 112-per-gate-panel-config: Per-gate panel config — explicit models, first-class lenses, and model-level substitution on top of 109's `panel:` block

## Goal

Extend spec 109's `panel:` config block so the operator's standing review protocol — different reviewer mixes at different lifecycle gates, with explicit model identities, assigned lenses, and a quota-wall substitution policy — is declared, parsed, validated configuration instead of orchestrator convention held in skill prose and session memory. This is one turn of ADR-0040's ratchet: a rule moving from L4 (judgment-kernel skill prose + operator memory) down to L2 (declared config), exactly the move that ADR names as the default direction of travel.

Concretely, `panel:` gains a per-gate `gates:` map and a generalized reviewer entry, backward-compatibly: a 109-style global `reviewers` list keeps working unchanged and becomes the all-gates default; per-gate entries override it. The operator's standing protocol (rev 2026-07-07) then reads:

```yaml
panel:
  note: "fable-window 2026-07, codex-enabled"   # optional free-text; echoed by `config show`, never consumed
  reviewers:                       # 109 global list — the default for any gate not configured below
    - {family: claude, count: 3}
    - {family: codex, count: 3}
  approve_threshold: "n-1"
  substitution:
    claude_sub_on_quota: true      # legacy family-level knob; superseded while substitutes is non-empty
    substitutes:
      gpt-5.5: claude-sonnet-5     # quota wall → Sonnet, slot id kept: reviewer_id "R7 claude-sonnet-5-sub"
  gates:
    spec_approve:                  # 9 reviewers, pass >= 8 ("n-1")
      reviewers:
        - {model: claude-fable-5, count: 3}
        - {model: claude-opus-4-8, count: 3}
        - {model: gpt-5.5, count: 3}
    plan_approve:                  # same 9-reviewer mix as spec_approve
      reviewers:
        - {model: claude-fable-5, count: 3}
        - {model: claude-opus-4-8, count: 3}
        - {model: gpt-5.5, count: 3}
    bead:                          # 6 reviewers, pass >= 5; exploded form with explicit lenses
      reviewers:
        - {model: claude-opus-4-8, lens: author-of-record}
        - {model: claude-opus-4-8, lens: codebase-pin}
        - {model: claude-opus-4-8, lens: contract-stability}
        - {model: claude-sonnet-5, lens: empirical-prober}
        - {model: claude-sonnet-5, lens: adversarial}
        - {model: claude-sonnet-5, lens: integration}
    final_review:                  # 12 reviewers, pass >= 11
      reviewers:
        - {model: claude-fable-5, count: 3}
        - {model: claude-opus-4-8, count: 3}
        - {model: gpt-5.5, count: 3}
        - {model: claude-sonnet-5, count: 3}
      approve_threshold: "11"      # equals "n-1" for N=12; explicit for illustration
    # adhoc: absent → resolves to bead's mix
```

The gate decision authority does not move: the recorded `panel.json` stays the sole input to `PanelGateDecision`, `internal/panel` stays a config-free leaf, and config supplies creation-time defaults only (109 R6–R8 intact). 112 adds schema, validation, gate-scoped resolvers, a machine-consumable `config show --gate` surface, and one decision-inert recorded field — no dispatch, no spawning, no gate-logic change.

## Background

**The decision record is bead `mindspec-kkcw`** (2026-07-04): spec 109's approved `panel:` schema (109 R2) models reviewers as a single global list of `{family: claude|codex, count}` plus one `approve_threshold`. It cannot express (a) per-gate variation, (b) a mixed panel richer than the two-value `family`, or (c) explicit model identity. The gap is reviewer-side only — the per-phase *actor* model (Opus authors, Sonnet implements) is already expressible via 109 R3's `models:` map, which this spec does not touch. `kkcw` records the chosen path: a follow-up spec that ratchets the richer schema on top of 109's minimal foundation per ADR-0040's layering, rather than re-opening approved-and-planned 109 (forward-only lifecycle, ADR-0023).

**The protocol it encodes** is the operator's standing model-tiering protocol (2026-07-04, revised 2026-07-07 after Fable shipped): spec/plan review = 3 Fable + 3 Opus + 3 GPT-5.5 (9, pass ≥ 8); bead review = 3 Opus + 3 Sonnet (6, pass ≥ 5); final spec review = 3 Fable + 3 Opus + 3 GPT-5.5 + 3 Sonnet (12, pass ≥ 11); quota-wall fallback = Sonnet substitution that keeps the slot id (`reviewer_id "<slot> <model>-sub"`). Today every one of those numbers lives in a memory file and inline orchestrator prompts — precisely the "re-derived from memory or prose under time pressure" failure mode ADR-0040's Context section names from the 107/108 runs.

**Three design inputs come from the 2026-07-07 live orchestration runs:**

1. **Lens ≥ tier.** Across four 6-reviewer panels that day, defect-finding power tracked the assigned *lens* (empirical-prober / adversarial / integration) at least as much as the model tier: round-1 approvals came from the author-of-record / codebase-pin / consistency lenses (run on Opus), while every load-bearing defect came from the empirical / adversarial / integration lenses (run on Sonnet). The lens is therefore a first-class declared field on a reviewer entry, not a per-run improvisation buried in `ms-panel-run`'s "Slot lens defaults" table.
2. **Model ids must be open-vocabulary strings.** `claude-fable-5` did not exist as a model id two weeks before this spec was drafted. An enum of model names would make the config unloadable on exactly the day the operator most needs the new model. Validation checks structure and cross-field invariants, never name-membership; a curated known-model list may drive a *warning* in `config show`, never a `Load` error. (This is also 109 R3's stated rationale for free-form `models:` values — model identifiers are runner-specific — applied to the reviewer side.)
3. **Substitution is part of the protocol, not an incident response.** The codex quota wall (multiple times since June) and the 2026-07-07 Sonnet-pool outage both forced live improvisation of "who stands in, under what reviewer_id". The substitute mapping and the slot-id-preservation rule become declared config.

**Sequencing dependency (hard):** 112 layers on 109's landed schema — the `Panel` config block, the `PanelExpectedReviewers()` / `PanelApproveThresholdExpr()` resolvers, the recorded `approve_threshold` field with `ApproveThreshold()` as its single interpreter, the `ReviewerCountNote` advisory surfaces, and ADR-0040 itself, none of which exist on `main` at drafting time. **This spec must not be planned or implemented until spec 109 has merged to `main` and this branch has been rebased onto that `main`** (the same rebase-onto-main prerequisite spec 108 carries). All 109 requirement references below (R2, R5–R8, R10) are to the approved `109-orchestration-config-substrate/spec.md`.

## Impacted Domains

- **core**: `internal/config` gains the `panel.gates` map, the generalized reviewer entry (`model` / `lens` / optional `count`, with legacy `family` compatibility), the `substitution.substitutes` map, the per-gate validation refusals, the gate-scoped resolvers (`PanelGateExpectedReviewers` / `PanelGateApproveThresholdExpr` / `PanelGateReviewerSlots`), and the curated known-model advisory list. Doc-sync: `.mindspec/domains/core/interfaces.md`.
- **workflow**: `internal/panel` gains one optional decision-inert recorded field (`gate`) on the `panel.json` schema — and nothing else; it stays a config-free leaf. `internal/complete`'s panel advisory and `cmd/mindspec`'s `config show` become gate-aware in which config default they pass to the unchanged `panel.ReviewerCountNote` helper, and `config show` gains the gates/substitutes rendering, the known-model advisory, and the `--gate <name> [--json]` resolved view. Doc-sync: `.mindspec/domains/workflow/interfaces.md` (CLI surface) and `.mindspec/domains/workflow/architecture.md` (recorded-field + advisory change). The ADR-0037 amendment note touches only `.mindspec/adr/**` (doc-classified, carries itself).

## Requirements

1. **`panel.gates:` per-gate map + generalized reviewer entry (open model vocabulary).** `internal/config`'s `Panel` block (109 R2) gains an optional `gates` map whose keys are a **closed enum**: `spec_approve`, `plan_approve`, `bead`, `final_review`, `adhoc`. Each gate value carries an optional `reviewers` list and an optional `approve_threshold` expression (same grammar as 109 R2: `"n-1"` or an integer string); a configured gate must set at least one of the two. The reviewer entry — in **both** the global list and every per-gate list — is generalized from 109's `{family, count}` to `{model: string, lens: string, count: int}`: `model` is an **open-vocabulary** string (structural validation only, never name-membership — model ids are runner-specific and new ids ship faster than releases: `claude-fable-5` did not exist two weeks before this draft); `lens` is an optional open-vocabulary string; `count` is optional, defaulting to `1`, with an explicit non-positive count refused. Legacy compatibility: `family: claude|codex` remains a parseable field; an entry with `family` and no `model` is valid and resolves its model to the family string; an entry carrying both keeps both (`model` wins for slot expansion); an entry with neither is refused (R4). The `panel:` block also gains an optional free-text `note` string — inert advisory metadata so a point-in-time mix self-documents (model ids are ephemeral; e.g. `note: "fable-window 2026-07, codex-enabled"`): parsed, echoed verbatim by `config show` (R8), never validated beyond being a string, and never read by any resolver or decision. Falsified if a 109-shaped `{family, count}` entry fails to load, if an unknown `gates` key loads without error, if any model-name enum-membership check exists in `Load`, if a populated `gates` map does not round-trip through `Load` unchanged, or if the `note` value affects any validation or resolver output.

2. **Backward compatibility: absent `gates:` ⇒ behavior identical to 109; the standing protocol is the documented example, not the default.** `DefaultConfig`'s panel values are unchanged from 109 R2 — reviewers `[{family: claude, count: 3}, {family: codex, count: 3}]`, `approve_threshold "n-1"`, `claude_sub_on_quota true` — with `gates` and `substitutes` defaulting to empty. With `gates:` absent: the 109 resolvers `PanelExpectedReviewers()` / `PanelApproveThresholdExpr()` keep their exact semantics, every gate-scoped resolver (R3) returns the global-derived values, and no loaded field or gate decision differs from 109's code. A populated 109-era config file — e.g. 109 AC7's fixture (custom 2+2 `reviewers`, `approve_threshold: "3"`, full `models:`/`loop:`, `runner: claude-code-workflow`) — loads under 112 with every field equal to what 109 loads. The 2026-07-07 standing protocol (9/6/12 mixes) deliberately does **not** become `DefaultConfig`: 109 R2's falsification pins zero-config defaults at 3+3/`"n-1"`, an existing install's behavior must not change on upgrade, and the protocol is *operator policy* — exactly what L2 declared config is for (ADR-0040) — so it ships as the documented example (the Goal's YAML, reproduced in `.mindspec/domains/core/interfaces.md`) for an operator to commit to their own `.mindspec/config.yaml`. Falsified if any zero-config default differs from 109's, if a gates-absent config yields different resolver outputs than 109's resolvers, or if `DefaultConfig` contains a non-empty `gates` or `substitutes` map.

3. **Gate-scoped resolvers + deterministic slot expansion (creation-time defaults only; thresholds stay raw).** `internal/config` gains three resolvers: `(*Config).PanelGateExpectedReviewers(gate string) (int, error)`, `(*Config).PanelGateApproveThresholdExpr(gate string) (string, error)` — returning the **raw** expression, never a resolved integer: expression resolution stays single-homed in `internal/panel.Panel.ApproveThreshold()` over the recorded `panel.json` (109 R6/R7) — and `(*Config).PanelGateReviewerSlots(gate string) ([]ReviewerSlot, error)` with `ReviewerSlot{Slot, Model, Lens string}`. Resolution chain: the gate's own configured values → (for `adhoc` only) `bead`'s resolved values → the global `reviewers` list / `approve_threshold` → the built-in defaults; a gate may override only its threshold (inheriting the global reviewers) or only its reviewers. An **unknown gate name errors** — a caller typo (`final` for `final_review`) must fail loud, not silently return defaults. Slot expansion is deterministic: entries in declaration order, each `count` expanded, slot ids `"R1"…"Rn"`; an entry's explicit `lens` applies to all its expanded slots; lens-less slots are filled by a **global-cursor round-robin over the interleaved default lens ordering** `author-of-record, empirical-prober, codebase-pin, adversarial, contract-stability, integration` — structural and sharp alternating; open vocabulary per R1; declared because the 2026-07-07 panels showed defect-finding tracks the lens at least as much as the model tier. Walking the flattened slot list in declaration order, one cursor advances only over lens-less slots — an explicitly-lensed slot keeps its lens and does not consume the cursor — and each lens-less slot takes `lens[cursor mod 6]`. Because interleaved neighbors alternate class, a gate with ≥ 6 lens-less reviewers exercises all six default lenses, and every lens-less entry of `count ≥ 2` spans at least one structural (author-of-record/codebase-pin/contract-stability) and one sharp (empirical-prober/adversarial/integration) lens: the 9-reviewer spec panel resolves to Fable {author-of-record, empirical-prober, codebase-pin}, Opus {adversarial, contract-stability, integration}, GPT {author-of-record, empirical-prober, codebase-pin}. Lens coverage is thereby robust to which model fills a slot — model ids are point-in-time (the Fable window, per-operator tooling), and swapping one model for another must not silently drop or concentrate a lens class; operators wanting deliberate lens×model pinning use the explicit exploded form (the Goal's `bead` gate). Identical config yields identical slots. These resolvers are the creation-time-default surface the spec-110 writer and the `ms-panel-run` step 0 consume; nothing reads them at gate-decision time. Falsified if a threshold resolver returns a resolved integer, if an unknown gate name resolves without error, if unconfigured `adhoc` does not equal `bead`'s resolved mix, if slot expansion is nondeterministic or drops an explicit lens, if a gate with ≥ 6 lens-less reviewers omits any of the six default lenses, if a lens-less entry of `count ≥ 2` fails to include both a structural and a sharp lens, if an explicitly-lensed slot consumes the round-robin cursor, or if any resolver output becomes an input to `PanelGateDecision`.

4. **`Load` refusals extended per gate (the un-weakenable knobs, 109 R5 style).** Each refusal returns an error with an ADR-0035 recovery line: an unknown `gates` key; a configured gate entry setting neither `reviewers` nor `approve_threshold` (a silent no-op is a likely indentation mistake); a reviewer entry with neither `model` nor `family`, or an empty-string `model`; an explicit `count <= 0`; a per-gate `reviewers` list whose expanded sum is `< 2` — the per-gate replay of 109 R5's no-always-pass floor, since a sum of 1 makes `"n-1"` resolve to `0`; a per-gate `approve_threshold` that is neither `"n-1"` nor an integer in `[1, that gate's resolved reviewer sum]`; and the **cross-field inheritance check**: for every configured gate whose threshold is inherited, an inherited global *integer* expression outside `[1, the gate's resolved sum]` is refused — a global `"5"` is valid for the global 6 but unsatisfiable or degenerate for differently-sized gates, whereas `"n-1"` is valid for every gate (every resolved sum is ≥ 2, so `n-1 >= 1`). Model ids and lens values are validated structurally only; the curated known-model list (R8) must never turn membership into a `Load` error. Falsified if any listed input loads without error, or if an unknown model id alone ever causes `Load` to fail.

5. **Substitution policy generalized to model level; slot-id preservation declared.** `panel.substitution` gains `substitutes`, a map of unavailable-model → substitute-model — both sides open-vocabulary strings, validated as non-empty with key ≠ value; resolution is one-step (chains are not followed; a mutual pair `A→B, B→A` is legal and useful). If the substitute is itself unavailable at runtime, that is a runtime HALT condition surfaced to the orchestration runner (spec 111 / `loop:` territory), not a config-resolution concern — config validates only that the mapping is well-formed (non-empty sides, key ≠ value), never model availability. The map is **global, not per-gate**: unavailability is a fact about the environment, not about a gate, and per-gate substitution variation would be config surface ahead of any observed need (the `internal/harness.Agent` anti-pattern ADR-0040 §3 names). Precedence is crisp: when `substitutes` is non-empty it **is** the substitution policy and the legacy `claude_sub_on_quota` bool (109 R2) is inert; when `substitutes` is empty, `claude_sub_on_quota` keeps its 109 meaning (the family-level codex→claude special case). The slot-id-preservation rule is declared at this schema's surface and rendered by `config show`: a substituted reviewer writes its verdict as `reviewer_id "<slot> <substitute-model>-sub"`, **keeping the slot id** (R7 stays R7) so verdicts stay comparable across rounds — generalizing `ms-panel-run`'s existing `"R<slot> claude-sub"` convention, per the codex-wall and 2026-07-07 Sonnet-pool-outage precedents. Like 109's `models:`/`loop:`, this is declared-and-surfaced here and consumed by the skills and the spec-111 runner; no in-binary dispatch reads it. Falsified if a self-mapping or empty-sided entry loads, if a populated `substitutes` map fails to round-trip, if `claude_sub_on_quota` changes meaning while `substitutes` is empty, or if any in-binary gate path consumes the substitution policy.

6. **Recorded, decision-inert `gate` field on `panel.json`.** `internal/panel.Panel` gains an optional `Gate string` field (`json:"gate,omitempty"`) recording which gate mix the panel was created from — values drawn from R1's enum by convention but **parse-lenient** like `abandon_reason`: an unexpected value is surfaced by consumers, never a parse error (flagging it malformed would set `Registration.Err` and tip the gate toward fail-open, the wrong direction). The field is **decision-inert**: `PanelGateDecision` and `ApproveThreshold()` do not read it; a legacy `panel.json` without it takes byte-identical code paths. `internal/panel` stays a config-free leaf — the field is plain recorded data, and the package still imports no `internal/config`. A second amendment note appended to `.mindspec/adr/ADR-0037-panel-gate-enforced-contract.md` records the field as decision-inert metadata (§§3/6/8 untouched). The spec-110 writer stamps it; until then `ms-panel-run` step 0 may hand-write it, and its absence costs nothing (R7's fallbacks). Falsified if the field's presence, absence, or value changes any `Allow`/`Block` or any threshold, if `internal/panel` imports `internal/config`, if a gate-less legacy `panel.json` parses or decides differently than before, or if the ADR-0037 amendment note is absent.

7. **Gate-aware advisory comparison (kills the spurious-note regression per-gate config would otherwise create).** The two caller-side `ReviewerCountNote` surfaces 109 R8/Bead 4 lands — `mindspec config show`'s panel scan and `internal/complete`'s panel advisory — compare a recorded `expected_reviewers` against the **gate-appropriate** config default instead of the global one. The helper itself (`panel.ReviewerCountNote(recorded, configDefault int)`) is unchanged and stays leaf-pure; only the callers' choice of `configDefault` becomes gate-aware: `PanelGateExpectedReviewers(recorded gate)` when the panel records a known gate; the `bead` gate for a bead panel (`IsBead()`) without a recorded gate; and for a non-bead panel with no recorded gate, only when `gates:` is configured is the note **skipped** — per-gate defaults exist, the panel's gate cannot be known among spec/plan/final/adhoc, and a guessed advisory is worse than none (spurious advisories train operators to ignore advisories); when `gates:` is absent there are no per-gate defaults to mis-guess, so every panel falls back to the global-default comparison exactly as 109 ships it, preserving R2's gates-absent identity. Without this requirement, a correctly-configured 9-reviewer `spec_approve` panel or 12-reviewer `final_review` panel would permanently flag "differs from config default 6". Advisory only: no `Allow`/`Block` changes anywhere; 109 R8's precedence rule and third-defense pin hold verbatim. Falsified if a recorded panel matching its own gate's configured default still yields a note, if a gates-absent config changes the advisory behavior of any panel relative to 109, if a note ever alters a decision, or if gate-aware compare logic is added inside `internal/panel` beyond the existing pure helper.

8. **`config show` surfacing: gates + substitutes rendering, known-model advisory (warning, never error), and the `--gate` resolved view.** `mindspec config show` renders the `gates` map (per configured gate: its entries, resolved reviewer sum, and raw threshold expression), the `substitutes` map with the slot-id-preservation convention, and keeps 109 R9's "declared, not yet enforced" annotations for the still-inert blocks. Any model id absent from the **curated known-model list** — a small in-binary advisory slice seeded with the current-protocol ids `claude-fable-5`, `claude-opus-4-8`, `claude-sonnet-5`, `gpt-5.5` and the legacy family strings `claude`, `codex`; non-exhaustive by design and expected to drift — is annotated with a warning-style note ("not in the known-model list — fine if intentional") that never affects the exit code and never feeds validation — advisory-only by construction (R1/R4): a made-up id warns while `Load` and `config show` still exit `0`. A set `panel.note` is echoed verbatim (under the same control-byte escaping 109's `config show` ships). A new flag `mindspec config show --gate <name> [--json]` prints a single gate's resolved creation-time defaults — the expanded slots (`slot`/`model`/`lens`), the expected reviewer count, the raw `approve_threshold` expression, and the effective substitution policy — as text or machine-readable JSON, giving `ms-panel-run` step 0 and the spec-110 writer a consumable resolver surface without linking Go. The command stays read-only (writes nothing, exit `0` on valid config; an unknown `--gate` value exits non-zero with a recovery line) and its rendering lives in testable pure functions in `cmd/mindspec`. Falsified if the command mutates any file, if an unknown model id flips the exit code on an otherwise valid config, if the shipped known-model list omits any seeded id above, if a set `note` is not echoed by `show`, if `--gate` output disagrees with the R3 resolvers over the same config, or if the `gates`/`substitutes` keys are omitted from `show`.

## Scope

### In Scope
- `internal/config`: the `gates` map and gate-name enum, the generalized reviewer entry (`model`/`lens`/optional `count`, legacy `family` compat), the `substitution.substitutes` map, the R4 validation refusals (including the cross-field inheritance check), the three gate-scoped resolvers with deterministic slot expansion, and the curated known-model advisory list (R1–R5).
- `internal/panel`: exactly one optional decision-inert recorded field, `gate` (R6) — no other edit; the package remains import-clean of `internal/config`.
- `internal/complete` + `cmd/mindspec`: gate-aware `configDefault` selection at both `ReviewerCountNote` call sites (R7); `config show` gates/substitutes rendering, known-model advisory, and `--gate <name> [--json]` (R8).
- `.mindspec/adr/ADR-0037-panel-gate-enforced-contract.md`: a second amendment note recording the decision-inert `gate` field (R6).
- Per-domain doc-sync (gate-forward planning constraint): the `core` bead(s) touch `.mindspec/domains/core/interfaces.md` (schema, resolvers, and the standing-protocol example config); the `workflow` bead(s) touch `.mindspec/domains/workflow/interfaces.md` and/or `architecture.md`.
- Zero-config and 109-era round-trip guarantees (R2).

### Out of Scope
- The `panel.json` **writer** that stamps these creation-time defaults (including the `gate` field) — spec 110, exactly as 109 deferred it. 112 lands the resolvers and the recorded field the writer will fill.
- Runner/dispatch wiring, reviewer spawning, or any consumption of the substitution policy by in-binary code — spec 111.
- Any edit to skill files (`.claude/skills/ms-*`): they begin consuming `config show --gate` when they shrink in 110/111, not here.
- Any change to `PanelGateDecision`, the decision matrix, `ApproveThreshold()` semantics, or the `loop:` block.
- The `models:` phase-tiering map (109 R3) — the kkcw gap is reviewer-side only; the actor side is already expressible.
- Per-gate substitution policy, weighted votes, or per-slot thresholds (see Open Questions — rejected as surface ahead of need).
- A machine/user-level config **layer** (author the standing mix once, machine-wide, merged under the per-project config) — candidate spec 113, tracked as bead `mindspec-pev1`. 112's keys inherit that layer for free when it lands: layering operates on the merged config regardless of which keys exist.

## Non-Goals

- No change to the `Allow`/`Block` outcome of any existing or future `panel.json` attributable to config: the recorded panel remains the sole gate authority, and every new field this spec records (`gate`) is decision-inert. 109 R8's three-layer threshold-skip defense (config refusal + record fallback + the `threshold > 0` gate guard) is inherited untouched.
- No validation of model or lens names against a closed set, ever — the curated list drives a warning in `config show` only. Structure and cross-field invariants are validated; vocabulary is not.
- No new telemetry; `config show --gate` is a pull surface (ADR-0027 posture, as in 109).
- No default-config encoding of the operator's standing protocol — defaults stay 109's; the protocol is the shipped *example*.
- No runtime availability handling: config never probes whether a model is reachable, and a substitute that is itself unavailable at runtime is a HALT surfaced by the orchestration runner (spec 111), not a config error.

## Open Questions

_All open questions are resolved; every resolution is folded into the Requirements and Scope above._

- **RESOLVED — `adhoc` is a first-class gate key, defaulting to `bead`'s mix.** Ad-hoc/standalone-bug panels already exist as a routed surface (`.mindspec/reviews/`, commit `ddd955f1`) and review PR-sized diffs — the same shape as bead reviews — so an unconfigured `adhoc` resolving to `bead`'s mix matches practice with zero config. Making it a named key (rather than leaving ad-hoc panels outside the enum) lets an operator deliberately lighten ad-hoc review (e.g. a 2-reviewer mix for one-line bug fixes) without touching the bead gate, and gives the recorded `gate` field an honest value for such panels. The alternative — no `adhoc` key, ad-hoc panels implicitly bead-shaped — was rejected because it makes the very panels most likely to be improvised the only ones that *cannot* be declared.
- **RESOLVED — the standing protocol ships as the documented example, not as `DefaultConfig`.** Three reasons: 109 R2's falsification pins zero-config defaults at 3+3/`"n-1"`; upgrading must not change any existing install's panel sizes; and the protocol is operator policy, which is exactly the class of rule ADR-0040 assigns to L2 *declared* config — the operator declares it in their repo, the framework does not impose it.
- **RESOLVED — `panel.json` gains a recorded `gate` field (a deliberate, minimal `internal/panel` touch).** The gate-aware advisory (R7) needs record-side intent: without it, a non-bead panel's gate cannot be distinguished among spec/plan/final/adhoc and the advisory either guesses or stays globally-compared (spurious notes on every correctly-sized 9/12-reviewer panel). The rejected alternative — no field, infer from `IsBead()` only — leaves every non-bead panel permanently un-comparable. The field is parse-lenient, decision-inert, config-free, and amendment-noted on ADR-0037; it is recorded metadata in exactly the sense the existing `abandon_reason` is.
- **RESOLVED — gate names are `bead` and `final_review`, not `bead_merge` and `impl_approve`.** `loop.gate_authority` (109 R4) names *approval acts*; `panel.gates` names *review events*. They coincide for `spec_approve`/`plan_approve` (the review is the gate's evidence), but a bead panel runs per review round — possibly several times before one merge — so `bead` is truer than `bead_merge`; and the final panel reviews the whole spec branch and *precedes* the impl-approve act, so `final_review` is truer than `impl_approve`. Copying the `gate_authority` keys wholesale was rejected as a false symmetry.
- **RESOLVED — lens interacts with `count` by sharing, not by refusal; default lenses fill by interleaved global round-robin.** An entry with `count: 3` and an explicit `lens` yields three reviewers with the same lens (legitimate: e.g. two empirical probers on a risky bead); explicitly-lensed slots never consume the default-lens cursor, which fills lens-less slots per R3's interleaved round-robin. Two alternatives were rejected: round-robin over a class-grouped lens ordering concentrates the sharp lenses on whichever entry occupies the middle slots, and an entry-local count-only spread gives equal-count entries identical subsets — a 3×3 panel would repeat the same three lenses in every entry and never assign the other three. Refusing `lens`+`count>1` was rejected as needless rigidity; silently ignoring the lens was rejected as data loss. The expansion stays deterministic either way.
- **RESOLVED — substitution is global and one-step; `substitutes` supersedes `claude_sub_on_quota` when non-empty.** Per-gate substitution and chain-following were rejected as config surface ahead of any observed need (ADR-0040 §3's `harness.Agent` counter-example); every real incident (codex wall, Sonnet-pool outage) needed exactly "when X is walled, use Y, keep the slot id". The supersession rule avoids per-model precedence arithmetic between a family-level bool and a model-level map — one policy is in force at a time, and which one is decidable by inspection.
- **RESOLVED — model ids validated structurally, warned advisorily, never enum-checked.** The Fable precedent decides this: any enum would have refused the operator's protocol revision on the day the new model shipped. The curated list exists only to catch typos (`claude-opus-48`) in `config show`, where a human is looking.
