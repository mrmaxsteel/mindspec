# Spec 105-spec-grill-skill: Spec Grill Skill

## Goal

Deliver an LLM-backed "grill" SKILL — a `SKILL.md` prompt, not Go code — that interrogates a spec author one question at a time during agent-assisted `mindspec spec create`, so the resulting `spec.md` is both *structurally* valid (`mindspec validate spec <id>` passes with 0 errors) AND *semantically* strong: every requirement and acceptance criterion is falsifiable, grounded in the live repo (real domains, real ADRs, real code), free of synonym-dodges ("enable/support/improve X"), and free of cross-requirement contradictions. The concrete target outcome is agent-assisted authoring in which a vague intent ("make exports more reliable") is grilled into a spec that a downstream review panel accepts without thinness findings.

## Background

`mindspec spec create <id>` is a non-interactive scaffold: it writes a Draft `spec.md` from a template and exits. The agent-facing entry point is the existing `ms-spec-create` skill — an 11-line `SKILL.md` (`.claude/skills/ms-spec-create/SKILL.md` and its `.agents/skills/` replica) that just says "run `mindspec spec create`, then begin drafting." It offers no protocol for *how* to draft well, so authors produce thin specs that pass `mindspec validate spec` (which only checks structure: required sections present, ≥3 acceptance criteria, no unresolved open questions) yet fail human/panel review on substance.

**This spec's shape is benchmark-driven.** We built a detection benchmark (prototype at `/tmp/grill-bench`, to be tracked in-repo by this spec) with three thin-spec fixtures plus at least one held-out/blind fixture and a set of planted problems recorded in `ground_truth.tsv`, spanning categories `SEMANTIC`, `SYNONYM`, `CONTRADICTION`, `GROUNDING`, `EXACT_PHRASE`, and `STRUCTURAL`. The original prototype had 16 planted problems (SEMANTIC 8, SYNONYM 2, CONTRADICTION 1, GROUNDING 2, EXACT_PHRASE 2, STRUCTURAL 1), of which 11 fell in the LLM-only classes — but this spec EXPANDS that set (held-out fixture + CONTRADICTION raised 1→≥3; see In Scope), so the headline denominator is NOT the frozen prototype count. The load-bearing scored set is **M**, defined as *all LLM-only (SEMANTIC + SYNONYM + CONTRADICTION) problems in the final tracked `ground_truth.tsv`, including the ≥3 CONTRADICTION items and the held-out/blind fixture's LLM-only items* — these are the problems the deterministic detector cannot reach by design; the GROUNDING + EXACT_PHRASE + STRUCTURAL items are catchable by schema heuristics and are not part of M. Two detectors were scored against it:

- A **deterministic detector** (`det_detect.sh`), faithfully encoding the *ceiling* of schema-driven heuristics a Go interview engine can reach — empty/placeholder detection, exact bad-phrase lists ("support X", "works correctly"), domain-set membership, `<3` ACs — caught **7/16 (44%)**. It scored 100% on `STRUCTURAL`/`GROUNDING`/`EXACT_PHRASE`. On the LLM-only classes its honest recall on the prototype was **2/11 ≈ 0.18**, NOT zero: its `functions? properly` and `handles .* correctly` regexes *accidentally* caught two items — spec2-P4 (SYNONYM "function properly") and spec3-P4 (SEMANTIC "handles concurrency correctly") — by lucky exact-phrase overlap, **not** by any semantic understanding. So it scored a single SEMANTIC and a single SYNONYM hit via exact-phrase overlap, and 0 on truly novel-phrased semantic problems and on CONTRADICTION. The tracked eval (AC1/AC5) restores a TRUE ~0 deterministic baseline on these classes by cleaning those two fixture items — re-phrasing spec2-P4 and spec3-P4 into forms with no exact-phrase overlap the regexes can catch — so the asserted baseline (0/M) matches reality rather than papering over the leak. Worse for the Go-engine case, the detector's wins largely *duplicate* what `mindspec validate spec` already does (empty sections, AC count, bad phrases), so a deterministic Go interview engine adds little over existing validation.
- An **LLM grill** (driven by `claude -p`) caught **16/16 (100%)** and additionally surfaced 4 real, *unplanted* problems.

This is why the prior deterministic interview engine attempt (spec **104**) is treated as superseded: deterministic interrogation cannot reach the semantic/synonym/contradiction classes that constitute the majority of real spec thinness, and what it *can* reach is already covered. The value lives in the LLM grill, and the LLM grill belongs in the agent/skill layer — where the human author and Claude Code already sit — not inside the binary. Keeping the binary non-interactive preserves the contract that automation and CI can call `mindspec spec create` without a blocking stdin read (the agent reasoning *is* the grill). This boundary — interactive, judgment-bearing work lives in the skill/agent layer; the binary stays deterministic and non-interactive — extends the agent-integration principle established in ADR-0019.

## Impacted Domains

- workflow: owns the skill source and setup machinery this spec creates and edits — `plugins/mindspec/skills/**`, `.claude/skills/**`, and `internal/setup/**` are already workflow-owned per `.mindspec/domains/workflow/OWNERSHIP.yaml`. **`.agents/skills/**` is NOT yet owned by any domain** (only `.claude/skills/**` is listed today); because the grill's `.agents/skills/ms-spec-grill/SKILL.md` is git-tracked, it would hit `adr-divergence-unowned` and BLOCK `mindspec complete`. This spec therefore makes `.agents/skills/**` a **deliverable**: it adds `- .agents/skills/**` to the workflow `OWNERSHIP.yaml` (see Requirement 12) so the grill's `.agents/` replica normalizes to the workflow domain. The `bench/grill/**` eval tree is intentionally NOT added to OWNERSHIP (see Requirement 12 / Hard Constraints): `bench` is an excluded first-segment, safely skipped by the divergence gate, and adding it is a hard `LoadOwnership` schema error.

Precedent for collapsing redundant workflow surface area — informs the decision to add ONE lean `ms-spec-grill` skill that `ms-spec-create` invokes rather than duplicating grill logic, and for treating spec 104's deterministic engine as superseded rather than carried forward.

## Requirements

1. **Grill protocol — relentless one-at-a-time interrogation grounded in the live repo.** The SKILL.md MUST instruct the agent to ask exactly ONE question at a time, never batching, and to refuse to advance until the current answer is concrete. Grounding MUST be live: cross-check every declared domain against the directories under `.mindspec/domains/`; surface the real ADRs under `.mindspec/adr/` relevant to each impacted domain and suggest the next-free ADR number when a new principle appears; and reality-check factual claims ("X already does Y") against the actual code/tree before accepting them.
2. **Coaching technique — domain & glossary alignment.** The SKILL.md MUST require the agent to validate each `## Impacted Domains` entry against the real domain set and reject invented domains (e.g. "caching", "scheduling"), mapping file-path entries to owning domains per the ADR-0036 model.
3. **Coaching technique — synonym / fuzzy-language detection.** The SKILL.md MUST require the agent to flag and reject requirements and ACs that use support/enable/improve/handle-style verbs with no falsifiable behavior (e.g. "Enable resumable exports", "improve reliability") — the SYNONYM/SEMANTIC class the deterministic detector scores 0 on.
4. **Coaching technique — falsifiability coaching.** The SKILL.md MUST require the agent to refuse non-falsifiable claims ("it works", "is fast", "reasonable time", "performance is acceptable") and coach the author into a measurable assertion with a threshold and an observable.
5. **Coaching technique — contradiction detection.** The SKILL.md MUST require the agent to scan requirements pairwise for mutual contradiction (e.g. "run concurrently" vs "at most one at a time") and force resolution — the CONTRADICTION class no deterministic rule set can reach.
6. **Coaching technique — scenario-driven edge-case probing.** The SKILL.md MUST require the agent to drive concrete scenarios (failure, concurrency, empty-input, boundary) to expose unstated edge cases and convert them into requirements or explicit Non-Goals.
7. **Thinness refusal & AC floor.** The SKILL.md MUST instruct the agent to refuse thin/empty answers and to require at least 3 falsifiable acceptance criteria, each pairing an assertion with a runnable proof, and to resolve or explicitly defer every Open Question before the grill is considered complete.
8. **Composition with the non-interactive binary.** The skill MUST drive `mindspec spec create <id>` to produce the scaffold, then grill the author to fill it; the binary MUST remain the non-interactive scaffold with no new blocking stdin read, so non-agent automation/CI is unaffected (the grilling is agent reasoning, not a binary prompt).
9. **Install & registration as a PLUGIN skill.** The grill MUST ship as a NEW **plugin** skill: the single source of truth is ONE file at `plugins/mindspec/skills/ms-spec-grill/SKILL.md` with valid frontmatter (`name` + `description`). It MUST follow the PLUGIN-skill mechanism used by `ms-bead-*`/`ms-panel-*` — filesystem-discovered via `plugins/mindspec`'s `//go:embed skills/*/SKILL.md` and merged by `pluginmindspec.SkillFiles()` — NOT the lifecycle-gate inline-literal mechanism (`lifecycleSkillFiles()` in `internal/setup/claude.go`) that `ms-spec-create` uses. (`ms-spec-create` is a lifecycle-gate skill physically defined as an inline raw-string literal and present only under `.claude/skills/` + `.agents/skills/`, never under `plugins/`; grill must NOT be added there.) `mindspec setup` propagates the embedded plugin SKILL.md to `.claude/skills/ms-spec-grill/SKILL.md` and `.agents/skills/ms-spec-grill/SKILL.md`. The 3-path `test -f` install ACs therefore pass only AFTER `mindspec setup` runs.
10. **`ms-spec-create` auto-chains into the grill (trigger contract).** `ms-spec-create` MUST auto-invoke `ms-spec-grill` by default: after scaffolding via `mindspec spec create <id>`, the updated `ms-spec-create` SKILL.md (all replicas) MUST contain an explicit, grep-provable handoff that invokes `ms-spec-grill` to grill the draft — the grill fires UNLESS the author explicitly opts out. Auto-chain-by-default is chosen over opt-in precisely because the failure mode being fixed is *authors skipping rigor*; an opt-in reference the author must remember to trigger would let the grill ship dead. The invocation MUST NOT be left to author discretion (the handoff text is the contract).
11. **New skill, not an inline rewrite of ms-spec-create.** The deliverable MUST be a NEW skill named `ms-spec-grill` (rationale: a single-responsibility grill skill is independently testable by the eval, reusable from contexts beyond create, and aligns with ADR-0034's collapse-don't-duplicate posture); `ms-spec-create` is updated only to auto-invoke it (Req 10).
12. **Ownership deliverable: claim `.agents/skills/**`.** This spec MUST add `- .agents/skills/**` to the workflow `.mindspec/domains/workflow/OWNERSHIP.yaml` so the git-tracked `.agents/skills/ms-spec-grill/SKILL.md` normalizes to the workflow domain and does NOT trigger `adr-divergence-unowned` on `mindspec complete`. The eval tree MUST NOT be claimed: NO `bench/**` glob may be added (`bench` is an excluded first-segment — the divergence gate skips it; adding it is a hard `LoadOwnership` schema error).
13. **Setup count assertions stay green.** Adding the 8th plugin skill breaks hardcoded count assertions in workflow-owned `internal/setup/**`. This spec MUST update them so impl does not ship red tests: `internal/setup/skills_test.go` (`len(all)` must become 12: 4 lifecycle + 8 plugin) and `internal/setup/claude_test.go` (the "7 plugin skills" / "11 total" expectations become 8 plugin / 12 total, and any dependent created/skipped item counts adjusted accordingly).
14. **Eval scoring is deterministic, model-pinned, and `claude`-CLI gated.** The eval MUST score the LLM grill's free-text findings against `ground_truth.tsv` using a DETERMINISTIC free-text→problem-id matching rule (keyword/line-anchored matching), NOT an LLM mapping/judging step (no LLM-judging-LLM). It MUST invoke `claude -p` with a FIXED model id and `temperature 0` so the recall figure is reproducible run-to-run. `claude` is an eval PRECONDITION: it must be installed and authenticated (logged-in CLI or `ANTHROPIC_API_KEY`); `run_eval.sh` MUST SKIP-with-notice (exit 0, print a skip message) when `claude` is absent rather than hard-fail, so the eval AC is runnable on fresh machines/CI.

## Scope

### In Scope
- A new `ms-spec-grill` **plugin** SKILL.md — single source at `plugins/mindspec/skills/ms-spec-grill/SKILL.md` — encoding the protocol + all coaching techniques of Requirements 1–7. `mindspec setup` propagates it to `.claude/skills/ms-spec-grill/` and `.agents/skills/ms-spec-grill/` via the plugin `//go:embed skills/*/SKILL.md` mechanism (it is NOT inlined into `lifecycleSkillFiles()`).
- A minimal edit to `ms-spec-create` (all replicas) to **auto-invoke** `ms-spec-grill` by default (Req 10) with an explicit, grep-provable handoff.
- Adding `- .agents/skills/**` to `.mindspec/domains/workflow/OWNERSHIP.yaml` (Req 12) so the grill's git-tracked `.agents/` replica is owned and does not block `complete`. (No `bench/**` glob — excluded first-segment; see Hard Constraints.)
- Updating workflow-owned setup count assertions for the 8th plugin skill (Req 13): `internal/setup/skills_test.go` and `internal/setup/claude_test.go`.
- A repo-tracked version of the `/tmp/grill-bench` detection benchmark under `bench/grill/`: the thin-spec fixtures (with spec2-P4/spec3-P4 re-phrased to remove the deterministic-detector exact-phrase leak so the asserted 0/M baseline is true), `ground_truth.tsv`, the deterministic baseline `det_detect.sh`, and a model-pinned, deterministic-matching, `claude`-CLI-gated LLM-grill eval runner `run_eval.sh`.
- At least one HELD-OUT (blind) fixture not used while authoring or tuning the grill skill, plus raising the CONTRADICTION class to ≥3 items, to mitigate overfit (the skill author writes both the prompt and the fixtures). The held-out fixture's LLM-only problems and all ≥3 CONTRADICTION items MUST be INCLUDED in the scored denominator **M** (the headline recall metric of AC1/AC5) — they may not be excluded from M; the held-out fixture's recall MAY additionally be reported separately so a reader sees generalization.

### Out of Scope
- Changing the binary's validation rules or thresholds (`internal/validate/**`): this spec adds an agent-layer grill, not new deterministic checks.
- Replacing human judgment or the downstream review panel: the grill assists the author; it is not an approval authority.
- Building a standalone / non-agent interactive TUI or wiring a blocking stdin read into the `mindspec` binary.
- Reviving or extending the superseded spec 104 deterministic interview engine.

## Non-Goals

- This spec does not aim to make the grill deterministic or hermetic; the eval is an explicitly thresholded LLM score over a fixed fixture set, run on demand, not a CI unit test gated on byte-exact output.
- This spec does not add or modify any `internal/**` Go package *behavior*; the only `internal/**` edits are the `internal/setup/**` count-assertion updates (Req 13), which change test expectations, not runtime logic. No new CLI subcommand is introduced.
- This spec does not author ADR-0039 (the agent-layer-authoring principle); that is deferred (see Open Questions).
- **`ms-spec-grill` is intentionally NOT added to the managed CLAUDE.md skill table.** That table is generated from a HARDCODED literal (`claudeMDManagedBlock` in `internal/setup/claude.go`), not data-driven from the installed skill set, and AGENTS.md has no skill table at all. `ms-spec-grill` is an internal coaching skill auto-invoked by `ms-spec-create`, not a user-facing slash gate, so its omission from that table is a DELIBERATE choice, not a doc-sync miss. This spec does not edit `claudeMDManagedBlock`.
- **Overfit is a known limitation, not eliminated.** Because the skill author writes both the grill prompt and the benchmark fixtures, ≥90% recall on the 3 known fixtures is gameable. This spec mitigates — not eliminates — overfit via the held-out/blind fixture (In Scope) and the ≥3-item CONTRADICTION class (n=1 was unfalsifiable); it does not claim a tamper-proof eval.

## Hard Constraints

- The grill is a `SKILL.md` prompt only — no new Go code paths and no new binary subcommand. (The only Go-side edits are the setup count-assertion updates of Req 13, which add no behavior.)
- `mindspec spec create` MUST stay non-interactive: no new blocking stdin read; automation/CI behavior is unchanged. This MUST be provable: `mindspec spec create <id> </dev/null` exits 0 with no blocking read (AC).
- The eval is non-deterministic (LLM-backed). It MUST be specified as a thresholded score over a fixed, repo-tracked fixture set, run on demand (by an author or opt-in CI job), never as a hermetic pass/fail unit test. final-review/CI MUST treat the LLM recall figure as ADVISORY — a single bad sample must NOT red the panel; the exit-0 is for the on-demand runner, not a hermetic gate.
- **Reproducibility pin.** The eval's `claude -p` invocation MUST use a FIXED model id and `temperature 0`; without the pin the recall figure is non-reproducible run-to-run. AC1 MUST run the eval N≥5 times and require the MIN (or median) recall over those runs to meet the threshold, where the threshold is **≥90% of M** — M being the full LLM-only (SEMANTIC + SYNONYM + CONTRADICTION) problem set in the final tracked `ground_truth.tsv`, including the ≥3 CONTRADICTION items and the held-out/blind fixture's LLM-only items — computed as an exact count over that fixed set, not a single run, and never a frozen literal that excludes the expanded items.
- The grill is a PLUGIN skill (single source `plugins/mindspec/skills/ms-spec-grill/SKILL.md`, propagated by `mindspec setup` to `.claude/skills/` and `.agents/skills/`), with valid `name` + `description` frontmatter. It MUST NOT be added to `lifecycleSkillFiles()` (that is the lifecycle-gate mechanism for `ms-spec-create` et al.).
- This spec MUST NOT add a `bench/**` glob to any OWNERSHIP.yaml (`bench` is an excluded first-segment; the claim is a hard `LoadOwnership` schema error). It MUST add `- .agents/skills/**` to the workflow OWNERSHIP.yaml (Req 12).

## Acceptance Criteria

- [ ] **AC1 — Eval beats the deterministic baseline on the LLM-only classes (advisory).** Define **M** = all LLM-only (SEMANTIC + SYNONYM + CONTRADICTION) problems in the final tracked `ground_truth.tsv`, INCLUDING the ≥3 CONTRADICTION items and the held-out/blind fixture's LLM-only items. Over N≥5 runs of the tracked grill eval against the fixture set, the LLM grill's MIN (or median) recall on M is ≥90% of M (an exact count, e.g. ⌈0.9·M⌉ of M), measured by a DETERMINISTIC keyword/line-anchored match of the grill's free-text findings against `ground_truth.tsv` (no LLM-judging-LLM). The held-out/blind fixture's problems and all ≥3 CONTRADICTION items MUST be INCLUDED in the scored denominator M and may NOT be excluded from this headline metric. The same run reports the deterministic baseline's recall on those classes for comparison. Proof: `bench/grill/run_eval.sh` invokes `claude -p` with a FIXED model id and `temperature 0`, loops N≥5 times, deterministically diffs detected problem ids against `ground_truth.tsv`, computes M from the tracked `ground_truth.tsv` (not a hardcoded literal), prints e.g. `LLM recall (semantic+synonym+contradiction) min over 5 runs = <hits>/M` (and MAY additionally print the held-out fixture's recall separately) and `deterministic baseline (those classes) = 0/M`, and exits 0 only when the min/median count ≥ ⌈0.9·M⌉. If `claude` is not installed/authenticated the script SKIPS-with-notice and exits 0. Honesty notes: this is an ADVISORY thresholded LLM eval run on demand, not a hermetic gate; a single bad sample must not red the panel. The held-out/blind fixture (In Scope) is included in M so the figure is not pure overfit.
