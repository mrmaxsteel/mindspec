# Lifecycle Control-Plane Integrity

## Goal

Close the RESIDUAL integrity gaps in the mutating lifecycle verbs — `mindspec complete`, `mindspec impl approve`, and `mindspec plan approve` — by enforcing one invariant on the paths that still lack it: **gate-before-mutate**. Every immutable gate fact (the bead's parent epic and owning spec, branch ancestry, epic membership, plan and obligation state) is resolved and validated BEFORE any state mutation, so a verb either refuses cleanly up front or completes without corrupting the state machine. Every recovery path is forward-reconcilable — a failed or interrupted run is repaired by re-invoking the same verb, converging either to completion or to a clean, recoverable refusal with a named recovery command — never by rollback. Divergence the verbs gate on is surfaced through `doctor` using the same predicates, and — critically — the forward-reconcile paths this spec adds must NOT become governance bypasses: the reconcile skip-path applies the SAME gate decisions the normal path would — ADR-0037's decision ladder unchanged, INCLUDING decision (5)'s MissingRef → Allow + Warn, which is the decision the ladder itself renders in the deleted-ref state — and carries its anti-bypass on positive landed-merge evidence (no evidence → refuse) plus the doc-sync and ADR-divergence gates evaluated against the landed content, so a lifecycle-external merge never gets an evidence-free close.

This spec is deliberately scoped against what specs 096/097/098/115 already shipped: the orphaned-sibling predicate (`internal/lifecycle/orphans.go`, wired into `complete`, `next`, and `doctor`), the read-after-write + forced-`bd dolt commit` close verification (`internal/complete/complete.go`), and the `work_chunks[].depends_on` dependency wiring at plan approve (`internal/approve/plan.go`) are SHIPPED baseline, consumed as-is, and not re-specified here.

## Background

The lifecycle control plane has accumulated a cluster of field-reproduced integrity bugs during the 107–118 delivery runs. Each bullet below is re-grounded against the code on `main` as of this revision:

- **`complete` resolves the owning spec from cwd, not lineage.** `internal/complete/complete.go:251-270`: `resolveTargetFn(localRoot, specIDHint)` consults the session cwd (worktree context) FIRST; the bead's parent epic is only consulted as a fallback when cwd resolution errors. A run from a different spec's worktree therefore succeeds with the WRONG spec (bead `wpjv.2` field incident), and an explicit `--spec` hint wins outright (`internal/resolve/target.go:90-94`) with no cross-check against the bead's lineage (`mindspec-zty3`).
- **`complete` auto-commits tracker artifacts onto whatever branch is checked out — including protected `main`.** The artifact-dirt self-heal (`complete.go:440-475`) commits `.beads/issues.jsonl` in the resolved checkout with no branch guard, and staging is not pathspec-scoped — unrelated dirty files have been swept into lifecycle auto-commits (twice deleting the inherited `review/` tree; the in-binary half of `mindspec-adwu`).
- **A merged-but-unclosed bead cannot be reconciled — but the wall that blocks it is also a gate.** The per-bead gates anchor on `exec.MergeBase(specBranch, beadHead)` (`complete.go:492-495`); when the successful first attempt (or a lifecycle-external merge) already deleted the bead branch, the merge-base fails (exit 128) before the close and before the durable-obligation reconciliation (`reconcilePendingRefutations`, `complete.go:570`) — so a merged-unclosed bead (`mindspec-zty3`) and an already-closed branch-less bead with an unsettled obligation (`mindspec-h4n5`) both need manual branch archaeology. Crucially, that same wall is today the ONLY thing forcing a lifecycle-external merge (raw `git merge` + branch delete) through the doc-sync/ADR-divergence/ADR-0037 panel gates. A naive "skip the gates when the branch is gone" reconcile would convert the bug into a clean gate BYPASS; the fix must skip only the git plumbing, never the gate EVALUATION — applying the same decision the normal merge path would, including ADR-0037 §6's registered-panels-only fail-open scope (`ADR-0037-panel-gate-enforced-contract.md:56-59`, `internal/complete/panel_advisory.go:176-177`) for genuinely panel-free flows AND including decision (5) of the ladder itself: a GENUINELY deleted `bead/<id>` ref resolves to MissingRef → Allow + Warn, the rerun-after-merge pass-through (`internal/panel/gate.go:139`, `:208-214`) — which is exactly the state the merged-unclosed reconcile is in. So for the panel leg the ADR's own decision in that state is Allow + Warn, and the enforceable anti-bypass there is positive landed-merge evidence plus the doc-sync/ADR-divergence gates evaluated against the landed content — not a panel Block the ladder never produces.
- **Tracker/git divergence is only half-surfaced.** The shipped `doctor` orphan check (`internal/doctor/orphaned_beads.go`) covers closed-in-tracker + unmerged-in-git. The INVERSE state — a bead still OPEN/in_progress in the tracker while its `bead/<id>` branch is fully merged into the spec branch (the recurring "verify-close gotcha" residue: `4d9m` and `lczt` sat OPEN ~2 days post-merge) — has no detector anywhere (`mindspec-yqdf` residual). And nothing in CI ever runs `mindspec doctor`, so even the shipped checks catch divergence only when a human runs them (`.github/workflows/ci.yml` has no doctor step — the genuine residual of `mindspec-4gsz`).
- **Epic finalization merges other specs' work.** `FinalizeEpic`'s auto-merge leg enumerates candidates by WORKTREE listing, not epic membership: `internal/executor/mindspec_executor.go:383-386` iterates `g.WorktreeOps.List()` and merges every branch with the `bead/` prefix — finalizing spec 112 attempted to merge `bead/mindspec-fbel.4`, an in_progress bead of spec 110 (`mindspec-blp6`). The enumeration is also fail-open: a `List()` error silently skips the entire auto-merge leg. Relatedly (`mindspec-3xqm`): orphaned finalize state (unmerged `chore/finalize-*` branches, stale committed tracker state on `main`) has no doctor/instruct surface; orphan-case `CommitCount`/`DiffStat` are computed against possibly-stale local `main` (`mindspec_executor.go:433-437`); and `gitutil.RemoteHeadSHA` (`internal/gitutil/gitops.go:327-346`) returns `fields[0]` of the first `ls-remote --heads` line without checking the refname column, so a decoy ref like `refs/heads/aaa/chore/finalize-105` suffix-matches `chore/finalize-105`.
- **The plan scaffold and the dependency wiring don't meet.** Plan approve ALREADY wires bd dependency edges from the structured `work_chunks[].depends_on` frontmatter (`internal/approve/plan.go:354-373`, spec 097 R3 — the prose scrape was deliberately retired). But the scaffold emitted at spec approve (`internal/approve/spec.go:230-268`) emits a DECORATIVE prose `**Depends on**` section the wiring ignores, and no `work_chunks` block at all — so an author who fills the scaffold as given gets a silently fail-open, edge-less plan (`bd ready` offered spec 118's Bead 2 alongside its prerequisite; `mindspec-ffm4`). The same scaffold omits the per-bead `**Acceptance Criteria**` section that the plan validator hard-requires (`internal/validate/plan.go:838`, check `bead-acceptance-criteria`), so every filled-as-given scaffold fails plan approve (`mindspec-cpqc`). Per-bead scope is entirely unverified — `ValidateDivergence` attributes changed files to the SPEC's domains only (`mindspec-jli8`).
- **A vestigial refusal survives in `impl approve`.** The `CommitCount`-based refusal disjunction (`internal/approve/impl.go:440-445`) is unreachable past spec 115's Leg 3 orphan-obligation gate — but the `CommitCount` CALL itself is a documented deliberate retention (consensus-revision-9 call-order pin, `TestApproveImplCallOrder`), not dead code. Only the unreachable refusal branch should go (`mindspec-qb03`, narrowed).
- **The instruct test helper is not hermetic.** `setupRunTestProject` (`internal/instruct/run_test.go:16-25` — it exists ONLY there; there is no `internal/harness` twin) fakes a `.git` marker directory that `phase.ResolveContextFromDirWithCache` does not anchor on; resolution walks UP into the real enclosing worktree and reads live lifecycle state, failing `TestRun_IdleNoBeads` on every full-suite run inside an active-spec worktree (`mindspec-z4ps`).

These are one architectural defect seen from several angles: the verbs interleave fact-gathering with mutation, so a mid-flight surprise leaves half-mutated state whose recovery re-runs then trip over the mutation's own side effects. This spec makes the preflight → commit → reconcile protocol explicit and regression-pinned on the residual paths — without re-shipping the floors that already exist.

## ADR Touchpoints

- ADR-0041 **WILL BE CREATED**: *Gate-Before-Mutate Lifecycle Verb Contract (preflight → commit → reconcile)*. One-line decision: every mutating lifecycle verb resolves all immutable gate facts (spec/epic membership from bead lineage, ancestry, plan and obligation state) and evaluates every derivable refusal BEFORE its first state mutation (the idempotent ADR-0034 legacy migration exempted — see Requirement 1); tracker-only commits never target protected `main`; and every failure mode is recovered by forward re-invocation of the same verb, converging to completion or to a clean named refusal — never by rollback. The implementation MUST include this ADR so the ADR-divergence gate sees the declared touchpoint.
- [ADR-0037: Panel Gate as Enforced Contract](../../adr/ADR-0037-panel-gate-enforced-contract.md) — unchanged, NO amendment required: Requirement 4's merged-unclosed reconcile path applies ADR-0037's decision ladder EXACTLY as written — including decision (5), MissingRef → Allow + Warn (`internal/panel/gate.go:139`, `:208-214`): the merged-unclosed state inherently presents a genuinely deleted `bead/<id>` ref, so the panel leg resolves to Allow + Warn (the rerun-after-merge pass-through), and the reconcile path surfaces that warning rather than fabricating a Block the ladder does not produce. §6's registered-panels-only fail-open scope is likewise preserved (`ADR-0037-panel-gate-enforced-contract.md:56-59`, enforced at `internal/complete/panel_advisory.go:176-177`): a genuinely panel-free bead completes with no panel gate at all. The anti-bypass burden in the deleted-ref state is carried by Requirement 4's positive landed-merge-evidence requirement and by the doc-sync/ADR-divergence gates evaluated against the landed content — not by strengthening the panel decision, which would require an ADR-0037 amendment this spec deliberately does not make (see the named follow-up in Out of Scope). This spec reorders when git plumbing runs relative to mutation; it neither weakens nor strengthens what evidence the gates demand.
- [ADR-0040: Orchestration Layering Ratchet](../../adr/ADR-0040-orchestration-layering-ratchet.md) — unchanged; this spec is an application of its ratchet: rules that today live in operator memory ("run complete from the right worktree", "check for stray finalize branches") ratchet down into L1 in-binary gates and doctor checks generated from the gates' own predicates.
- [ADR-0030: Executor Boundary](../../adr/ADR-0030-executor-boundary.md) — unchanged, with its scope stated CORRECTLY: the boundary binds the ENFORCEMENT packages (`internal/validate`, `internal/approve`, `internal/complete`, `internal/state`, `internal/phase` — the exact list pinned by `internal/lint/boundary_test.go` `TestEnforcementHasNoGitLeaks`), which must consume git facts exclusively through `executor.Executor`. It does NOT forbid other workflow-side packages from importing `internal/gitutil` — `internal/lifecycle/orphans.go` does so today and CI passes. New code in this spec follows the actual rule: enforcement-package changes stay executor-mediated; changes in `internal/lifecycle`, `internal/doctor`, and executor-side code may use `gitutil` directly.
- [ADR-0032: ADR Semantic Gates](../../adr/ADR-0032-adr-semantic-gates.md) — unchanged. The per-bead scope check introduced here is explicitly ADVISORY (warn-only) and does not alter ADR-0032's hard ownership/divergence gate semantics.

## Requirements

1. **Preflight before mutation, modulo the idempotent migration.** Each mutating lifecycle verb (`complete`, `impl approve`, `plan approve`) MUST resolve every immutable gate fact — the bead's parent epic and owning spec, branch existence/ancestry, target-epic child set, plan parse and obligation state — and MUST evaluate every refusal condition derivable from those facts BEFORE its first state mutation (tracker write, branch creation/merge/deletion, commit, or file write). EXEMPT: the one-shot ADR-0034 legacy-to-metadata migration (`complete.go:286-293`, `impl.go:186-192`, `plan.go:70-76`), which is idempotent, already precedes all gates, and may keep running before preflight. A refusal raised in preflight MUST leave tracker and git state byte-identical to the pre-invocation state, modulo that idempotent migration.
2. **Spec resolution from bead lineage.** `complete` MUST resolve the bead's owning spec from the bead's parent epic as the AUTHORITATIVE source, for any invocation cwd inside the repository (a `main` checkout, the bead's own worktree, or a different spec's worktree). Outside-repo cwd behavior is unchanged: `findRoot()` fails before the bead is read (`cmd/mindspec/complete.go:41-44`). An explicit `--spec` hint that RESOLVES to the same spec as the lineage proceeds; a hint that mismatches the lineage MUST refuse in preflight, naming both spec IDs — the hint never silently overrides lineage. Joint from-`main` end-state (with Requirement 3): a `complete` invoked from a `main` checkout acts on the lineage-resolved spec, routes any tracker commit to that spec's branch (or refuses with a named re-invocation command), and leaves `main`'s history untouched.
3. **`complete`'s tracker commits never target protected `main`; staging is pathspec-scoped.** Artifact/tracker auto-commits made by `complete` MUST target the spec branch. When the working checkout is on `main`, `complete` MUST either route the tracker commit to the spec branch or refuse with a named forward-reconciliation command — it MUST NOT create a commit on `main`. EXPLICITLY EXCLUDED: the shipped protected-`main` finalize recovery (spec 115 / wu7t, `finalizeOrphanedSpecBranch`, `internal/executor/mindspec_executor.go:602`), which INTENTIONALLY commits the epic-close tracker export on a fresh `chore/finalize-<specID>` branch created from `origin/main` — that carrier is deliberately not-spec-branch (the spec branch is a dead PR carrier in that state) and is consumed as shipped, not re-routed by this requirement. Additionally, `complete`'s artifact auto-commit staging MUST use an explicit pathspec limited to lifecycle artifacts (e.g. `.beads/issues.jsonl`), never an `add -A` equivalent; unexpected dirty paths MUST be surfaced as a warning, not silently swept into the commit.
4. **Merged-unclosed and branch-less reconciliation — skip the plumbing, apply ADR-0037's actual decision.** `complete` MUST detect the merged-but-unclosed state (bead work already an ancestor of the spec branch, bead branch possibly deleted) in preflight and reconcile forward: positively identify the landed merge commit, skip ONLY the merge-base git plumbing that cannot run against a deleted branch (`complete.go:492-495`) and the merge/branch-cleanup legs, and evaluate the doc-sync and ADR-divergence gates against the landed merge commit's content. The ADR-0037 panel gate is evaluated with its decision ladder UNCHANGED and IN FULL: every decision applies on the reconcile path exactly as on the normal path — in particular a registered-but-malformed/unparseable panel STILL Blocks at decision (2) (`internal/panel/gate.go:173-178`) and a round-mismatched registered panel STILL Blocks at decision (4) (`gate.go:200-205`), because those decisions PRECEDE the missing-ref check; a deleted ref does not short-circuit them. Decision (5), MissingRef → Allow + Warn (`gate.go:139`, `:208-218`), applies specifically to a READABLE, NON-ABANDONED, ROUND-CONSISTENT registered panel whose `bead/<id>` ref is genuinely deleted — the state the merged-unclosed reconcile inherently presents for a well-formed registration — and there the reconcile path MUST surface the rerun-after-merge warning and MUST NOT fabricate a panel Block the ladder does not produce; the ladder's own earlier Blocks, (2) and (4), remain in force. §6's registered-panels-only fail-open scope is likewise preserved (`ADR-0037-panel-gate-enforced-contract.md:56-59`; `panel_advisory.go:176-177` returns no gate when no panel is registered): a panel-free bead completes through the reconcile path exactly as it would through the normal path, with no panel warning at all. No ADR-0037 amendment is made or required. The anti-bypass property is carried by two mandatory legs OUTSIDE the panel decision: (a) POSITIVE landed-merge evidence — the reconcile path MUST record durable evidence identifying the landed merge commit it reconciled against, and MUST refuse (not close) when no landed merge commit can be positively identified (bead open, work NOT merged, branch deleted → refusal, not close); and (b) the doc-sync and ADR-divergence gates evaluated against the landed content — an unsatisfied gate there refuses, naming the failing gate, not a close. The same skip-legs path MUST reconcile an already-closed, branch-less bead carrying an unsettled `refutation_pending_entries` obligation (reaching `reconcilePendingRefutations`) without requiring manual branch-ref restoration. The stronger fail-closed variant — refusing when a REGISTERED panel's gate facts are unsatisfied in the deleted-ref state — is a NAMED FOLLOWUP (see Out of Scope): it requires an ADR-0037 decision-5 amendment plus a reconstructed landed-bead-tip staleness target, and no criterion in this spec compares `reviewed_head_sha` to the merge commit (a non-fast-forward merge commit's SHA differs from the bead tip, so that comparison would false-stale a normally-reviewed bead).
5. **Doctor stale-OPEN cross-check — landed work, not mere ancestry.** `doctor` MUST gain a NEW check — the inverse of the shipped `checkOrphanedBeads` — flagging a bead that (a) is OPEN or in_progress in the tracker, (b) whose `bead/<id>` branch carries at least ONE commit beyond its fork point from the spec branch (the bead did work), and (c) all of that work is an ancestor of the spec branch (the work landed) — reporting `Status=Error` with the recovery command `mindspec complete <id>`, and reporting healthy when tracker and git agree. Mere ancestry is NOT sufficient to flag: bead branches fork AT the spec tip (`DispatchBead` creates the branch from the spec branch), so a freshly-claimed bead branch with ZERO commits beyond its fork point is trivially an ancestor of the spec branch and MUST NOT be flagged — otherwise every in-progress bead would fail the CI doctor run (Requirement 7). The git mechanism for computing the fork point after the work has merged (plain `merge-base` collapses to the bead tip once it is an ancestor — e.g. `merge-base --fork-point`/reflog, the recorded creation base, or first-parent-history exclusion) is a plan-level choice; the observable contract is fixed by AC-10's positive and negative fixtures.
6. **Lifecycle-scoped finalization, BOTH enumerations, fail-closed.** `FinalizeEpic` enumerates `bead/`-prefixed worktree branches in TWO independent legs: the auto-merge leg (`mindspec_executor.go:383-386`) and the lingering-worktree/branch CLEANUP leg (`mindspec_executor.go:526-534`, which removes the worktree and deletes the branch for every `bead/`-prefixed entry). The allow-set for BOTH enumerations MUST be the finalizing spec's PLAN-DECLARED LIFECYCLE BEADS — the task-type children in the plan's bead set, per the explicit lifecycle classification phase derivation already uses (`internal/phase/derive.go` `classifyChild`: only `task` and empty/unknown issue types are lifecycle; `bug` and every other explicit non-`task`, non-`epic` type is a non-lifecycle follow-up) — with `parent == finalizing epic` as a NECESSARY cross-check but NOT the sole selector. Parent-only scoping is insufficient: spec 095 deliberately permits an OPEN non-lifecycle follow-up child (e.g. a `bug`) to remain attached to the epic through impl approve (`internal/approve/impl.go:240-263`, advisory-only, never blocking), so a same-epic child can hold a live worktree and branch while sitting OUTSIDE the lifecycle. Neither leg may merge, worktree-remove, or branch-delete (i) a bead of a DIFFERENT epic or (ii) a NON-LIFECYCLE child of the finalizing epic itself — open or closed; the exclusion boundary is lifecycle identity, not open-status or worktree presence. Enumeration and classification failures MUST be fail-closed: an error listing worktrees, resolving a candidate's epic membership, or resolving its lifecycle classification aborts the finalize with a named error rather than silently skipping the leg or silently admitting the candidate.
7. **Finalize-orphan surfacing, CI wiring, and remote-ref hardening.** `doctor` (and the corresponding `instruct` guidance, generated from the same check) MUST detect and report (a) an outstanding unmerged `chore/finalize-*` branch and (b) stale committed tracker state on `main` left by an interrupted finalize. Orphan-case `CommitCount`/`DiffStat` MUST be computed against `origin/main`, not possibly-stale local `main` (`mindspec_executor.go:433-437`). `gitutil.RemoteHeadSHA` MUST match the exact refname on the `ls-remote` refname column (a decoy ref such as `refs/heads/aaa/chore/finalize-105` MUST NOT match `chore/finalize-105`), and the `PushBranchForceWithLease` doc comment MUST be corrected: the read-then-lease pattern protects only against races between the `RemoteHeadSHA` read and the push — it overwrites whatever tip was observed, so "never silently clobbered" overstates the guarantee. Finally, CI MUST run `mindspec doctor` so that its non-zero exit fails the build — wiring the shipped orphan checks and the new checks into `.github/workflows/ci.yml` (today no workflow invokes doctor).
8. **Narrow the vestigial refusal — preserve the retention.** The now-unreachable refusal DISJUNCTION inside `impl approve`'s pre-finalize check (`impl.go:440-445`: `count == 0 && (planErr != nil || len(beadIDs) == 0)`, unreachable past spec 115's Leg 3 gate) MUST be removed. The `exec.CommitCount` CALL and its consensus-revision-9 position between the phase-metadata write and `FinalizeEpic` MUST be PRESERVED, with `TestApproveImplCallOrder` continuing to pin that order — the call is a documented retention, not dead code. A test MUST continue to pin that a missing/empty plan is refused pre-mutation by the Leg 3 orphan-obligation gate.
9. **The scaffold emits structured dependencies — no prose parser; wiring gaps are loud.** The plan scaffold emitted at spec approve (`internal/approve/spec.go` `scaffoldPlan`) MUST emit a `work_chunks` frontmatter block with per-chunk `id` and `depends_on` fields (matching the shape `internal/validate/plan.go`'s `WorkChunk` parses), so that an author who fills the scaffold as given gets dependency edges wired by the SHIPPED `work_chunks[].depends_on` path (`plan.go:354-373`, spec 097 R3). The per-bead prose `**Depends on**` section is KEPT in the scaffold — the plan validator's `bead-depends` check (`internal/validate/plan.go:841-842`) warns on any bead section lacking a "Depends on" declaration, so removing it would make every filled-as-given scaffold warn — but the scaffold MUST label it as human-readable documentation that is never parsed; `work_chunks[].depends_on` is the SOLE wired source of truth. No prose dependency parsing may be introduced — it would reverse 097 R3 and create dual conflicting sources of truth. Two silent fail-opens MUST become visible warnings: (a) `plan approve` on a plan with NO `work_chunks` frontmatter MUST emit a warning naming the absence (edges cannot be wired), with approve behavior otherwise unchanged for legacy plans; and (b) the shipped edge wiring is best-effort (`internal/approve/plan.go:368-370` `continue`s on a `bd dep add` error with no output) — a failed `bd dep add` invocation MUST emit a warning naming the unwired edge (both bead IDs), with approve still proceeding (best-effort behavior otherwise preserved).
10. **Validator-complete scaffolds.** The plan scaffold MUST include every section the plan validator hard-requires — in particular a per-bead `**Acceptance Criteria**` section (`bead-acceptance-criteria`, `internal/validate/plan.go:838`) — and a "templates round-trip through their own validators" test MUST pin, for BOTH the plan scaffold and the spec scaffold, that the emitted template, minimally filled, passes its own validator. The plan-scaffold round-trip MUST additionally pin that the filled scaffold's `work_chunks` produce wired dependency edges (coupling to Requirement 9).
11. **Advisory per-bead scope.** `complete` MUST emit a non-fatal WARN (exit unchanged) when the bead's changed files fall outside the bead's declared `Domain:` while still within the spec's impacted domains. `plan approve` MUST flag (plan-lint) any single file assigned to two or more beads' step lists. Neither check is a hard gate: legitimate seams cross domain boundaries atomically. New path/ID-bearing warning output introduced by this spec MUST be routed through spec 116's `internal/termsafe` escaping. The jli8 per-path ownership-RECORDING leg is OPTIONAL and out of scope (see Scope).
12. **Hermetic test isolation.** `setupRunTestProject` (`internal/instruct/run_test.go` — the only definition; there is no `internal/harness` twin) MUST be hermetic: phase/git resolution from within the test sandbox MUST NOT escape to an enclosing real worktree — via `GIT_CEILING_DIRECTORIES`, a real `git init` in the temp dir, or an explicit root override honored by `phase.ResolveContextFromDirWithCache`. `go test ./...` (harness excluded per existing convention) MUST pass when run from inside an active-spec worktree whose enclosing repo is mid-lifecycle.
13. **Fault-injection regression coverage; doctor shares the verbs' predicates.** Every failure class fixed by this spec MUST be pinned by an automated regression test that reproduces the original trigger and goes RED if the fix is reverted. Forward-reconciliation MUST be exercised by fault injection at MULTIPLE post-preflight mutation points (not only the first), and re-invocation MUST converge either to the completed end state or to a clean, recoverable REFUSAL with a named recovery command — both are valid forward-reconcilable outcomes (the known acceptable case: `complete --commit-msg`'s own auto-commit advances the bead tip past `reviewed_head_sha`, so a killed-then-re-invoked run converges to a panel-staleness refusal). Doctor's new divergence checks MUST invoke the same exported predicates the verbs gate on, so check and gate cannot drift.

## Scope

### In Scope

- Restructure `internal/complete` preflight: lineage-authoritative spec resolution with `--spec` mismatch refusal, pre-mutation gate evaluation (modulo the idempotent migration), merged-unclosed/branch-less reconciliation that requires positive landed-merge evidence and evaluates the doc-sync/ADR gates against the landed merge commit while applying ADR-0037's decision ladder unchanged (decision-5 MissingRef → Allow + Warn in the deleted-ref state; §6 fail-open scope preserved), pathspec-scoped spec-branch-targeted tracker commits (the `chore/finalize-*` recovery carrier explicitly excluded).
- New `doctor` checks: stale-OPEN (open-in-tracker + merged-in-git) cross-check; unmerged `chore/finalize-*` and stale-committed-tracker-on-`main` surfacing with generated `instruct` guidance; wire `mindspec doctor` non-zero exit into `.github/workflows/ci.yml`.
- Scope BOTH of `FinalizeEpic`'s bead-branch enumerations (the auto-merge leg AND the cleanup leg) to the finalizing spec's plan-declared lifecycle beads (`parent == finalizing epic` as a necessary cross-check, never the sole selector; same-epic non-lifecycle children excluded whether open or closed), fail-closed on enumeration/classification errors; compute orphan stats vs `origin/main`; harden `RemoteHeadSHA` refname matching; correct the `PushBranchForceWithLease` doc comment; remove the unreachable refusal disjunction while preserving the `CommitCount` retention.
- Scaffold work: emit a `work_chunks` frontmatter block (with `depends_on`) and a per-bead `**Acceptance Criteria**` section, keeping the prose `**Depends on**` section labeled non-authoritative (the validator's `bead-depends` check expects it); warn on missing `work_chunks` and on failed `bd dep add` at plan approve; scaffold/validator round-trip tests including edge wiring.
- The advisory bead-scope WARN at `complete` and the double-assignment plan-lint at `plan approve` (termsafe-escaped output).
- Make `setupRunTestProject` hermetic so the full suite is green inside an active-spec worktree.
- Author ADR-0041 recording the gate-before-mutate contract.
- Fault-injection regression tests for every fixed class, using the existing `MockExecutor`/test seams.

### Out of Scope

## Non-Goals

- No rollback machinery. A partially-completed verb run is never "undone"; recovery is always forward re-invocation converging on completion or a clean named refusal.
- No in-binary lint/full-test quality gate. The shipped binary gains no `golangci-lint` or whole-suite hard gate at `complete`; gate-truthfulness is the descoped adwu follow-up doctrine (see Out of Scope).
- No prose dependency parser. Dependency truth stays single-sourced in `work_chunks` frontmatter (spec 097 R3); this spec fixes the scaffold to feed it, not the parser to read prose.
- No hard per-bead domain gate. The bead-scope check stays advisory (WARN + plan-lint) because legitimate seams cross domain boundaries atomically.
- No new configuration surface. This spec adds no `.mindspec/config.yaml` knobs; the gate-before-mutate contract is an L1 invariant, not per-run policy.
- No redesign of the finalize/PR flow for protected `main` (spec 115's legs are consumed as shipped), and no relaxation of `next`'s existing blocking behavior.

## Open Questions

None. The mechanism by which the reconcile path anchors the doc-sync/ADR/panel gates on the landed merge commit (Requirement 4) is a plan-level choice, constrained only by the requirement that the gates evaluate the landed content and by ADR-0037. The hermetic-isolation mechanism (`GIT_CEILING_DIRECTORIES` vs real `git init` vs explicit root override) is likewise a plan-level choice; the core domain is declared only for the override variant. The `.github/workflows/ci.yml` ownership handling (claim into workflow vs recorded unowned attribution) is resolved by the plan per the Impacted Domains note. Squash-tolerant orphan detection and the adwu gate-truthfulness doctrine are explicitly deferred as named follow-ups rather than open questions. Likewise the stronger fail-closed-on-registered-unsatisfied-panel-in-the-deleted-ref-state reconcile variant: it is a NAMED FOLLOWUP (see Out of Scope), not an open question, because it is blocked on an ADR-0037 decision-5 amendment (today MissingRef → Allow + Warn) and on a reconstructed landed-bead-tip staleness target — a non-fast-forward merge commit's SHA differs from the bead tip, so anchoring staleness on the merge commit would false-stale normally-reviewed beads; this spec deliberately ships the decision-5-faithful variant instead.
