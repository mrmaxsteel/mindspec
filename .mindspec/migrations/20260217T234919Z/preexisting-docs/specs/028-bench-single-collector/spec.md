# Spec 028-bench-single-collector: Single-Collector Bench Architecture

## Goal

Replace bench's 3-collector-per-session architecture with a single shared OTLP collector. Sessions are differentiated using `OTEL_RESOURCE_ATTRIBUTES` (`bench.label`) injected at the source, eliminating port conflicts, simplifying orchestration, and enabling bench to share a collector with the recording system.

## Background

`bench run` currently starts 3 separate OTLP collectors ‚Äî one per session (A, B, C) on ports 4318‚Äì4320. This causes two problems:

1. **Port conflict with recording**: The per-spec recording collector (spec 027) runs on port 4319. When bench claims 4318‚Äì4320, it collides with an active recording and/or with AgentMind's live viz port (4318).

2. **No shared analysis**: When bench was configured to point at an external OTLP endpoint (e.g., AgentMind's collector), no local JSONL was captured and the quantitative report was blank ‚Äî the data flowed to the external collector but bench couldn't read it back.

The root issue is that bench conflates **collection** (receiving OTLP events) with **session identity** (which events belong to which session). These can be cleanly separated:

- **Collection**: A single collector on one port receives all events into one file.
- **Session identity**: Each session sets `OTEL_RESOURCE_ATTRIBUTES=bench.label=a|b|c`, which Claude Code propagates into every OTLP event's resource attributes. The collector already extracts resource attributes into `CollectedEvent.Resource` ‚Äî no collector changes needed.

ADR-0009 explicitly confirms that Claude Code passes `OTEL_RESOURCE_ATTRIBUTES` through to its OTLP exporter (see "Custom resource attributes" section). Each session also gets a unique `session.id` UUID in `CollectedEvent.Data`, which serves as a secondary discriminator for per-invocation drill-down (useful when retries produce multiple invocations per label).

## Impacted Domains

- **bench**: Core change ‚Äî session orchestration, event collection, report parsing
- **observability**: Port allocation strategy (bench no longer claims 3 ports)

## ADR Touchpoints

- [ADR-0009](../../adr/ADR-0009.md): Confirms `OTEL_RESOURCE_ATTRIBUTES` flows through Claude Code's OTLP exporter; defines OTLP as the ingestion protocol
- [ADR-0010](../../adr/ADR-0010.md): Recording collector uses port 4319 ‚Äî bench must not collide with it

## Requirements

### 1. Single Collector

1. `bench run` starts **one** OTLP collector on a single port before any session begins. All three sessions send telemetry to this same endpoint.

2. All events are written to a single `bench-events.jsonl` file in the benchmark output directory.

3. The collector is shut down after all sessions complete and the report is generated.

### 2. Session Tagging via Resource Attributes

4. Each session's `runClaude()` invocation sets `OTEL_RESOURCE_ATTRIBUTES=bench.label=<label>` (where label is `a`, `b`, or `c`) in the process environment.

5. The `bench.label` value appears in `CollectedEvent.Resource["bench.label"]` for every event emitted by that session, including sub-agent events.

6. Retry invocations for the same session label inherit the same `bench.label` value. Multiple `session.id` values may exist per label (one per invocation), but all share the same `bench.label`.

### 3. Filtered Report Parsing

7. A new function `ParseSessionByLabel(path, label string) (*Session, error)` reads the single JSONL file and aggregates only events where `Resource["bench.label"]` matches the given label.

8. The existing `ParseSession(path, label string)` function continues to work unchanged for standalone JSONL files (e.g., `bench report` on legacy per-session files). It does **not** filter by label ‚Äî it aggregates all events in the file.

9. The quantitative comparison report is generated by calling `ParseSessionByLabel` once per session label against the shared `bench-events.jsonl`.

### 4. Port Simplification

10. Remove per-session `Port` field from `SessionDef`.

11. Use a single bench collector port (not 4319, to avoid conflict with the recording collector).

12. Remove the separate port constants (`portA`, `portB`, `portC`) from the runner.

### 5. Session ID Tracking (Secondary)

13. `SessionResult` gains a `SessionIDs []string` field populated by scanning the JSONL for unique `data["session.id"]` values matching the session's `bench.label` after each invocation completes.

14. Session IDs are included in the report metadata for drill-down and debugging.

## Scope

### In Scope

- `internal/bench/session.go` ‚Äî Remove `Port` from `SessionDef`, inject `bench.label` via `OTEL_RESOURCE_ATTRIBUTES`, extract session IDs after each run
- `internal/bench/runner.go` ‚Äî Single collector lifecycle, remove per-session port constants, use `ParseSessionByLabel` for report
- `internal/bench/report.go` ‚Äî Add `ParseSessionByLabel()`, keep `ParseSession()` for backward compat
- `internal/bench/markdown.go` ‚Äî Single `bench-events.jsonl` output, update report template
- `cmd/mindspec/bench.go` ‚Äî Remove per-session port flags if any remain
- `internal/bench/runner_test.go` ‚Äî Update test expectations, add `ParseSessionByLabel` tests

### Out of Scope

- Modifying the recording collector (spec 027) or its port
- Fan-out / proxying to multiple OTLP endpoints
- Changing `CollectedEvent` schema or collector parsing logic (resource attributes are already extracted)
- AgentMind live viz integration (bench operates independently)

## Non-Goals

- Not replacing or merging with the per-spec recording collector ‚Äî bench and recording are separate concerns that happen to share collector code
- Not supporting parallel session execution in this spec (sessions run sequentially; `bench.label` tagging would work for parallel too, but orchestration changes are out of scope)
- Not adding `bench.label` filtering to AgentMind replay ‚Äî bench reports are the consumer

## Acceptance Criteria

- [ ] `bench run` starts exactly one OTLP collector (verified by port allocation: single port, not three)
- [ ] All three sessions emit events to the same `bench-events.jsonl` file
- [ ] Events from session A have `Resource["bench.label"] == "a"`, B has `"b"`, C has `"c"`
- [ ] Quantitative report shows per-session breakdowns (tokens, cost, duration, model usage) parsed from the single JSONL file
- [ ] `bench run` does not bind port 4319 (recording collector port)
- [ ] `ParseSessionByLabel("bench-events.jsonl", "a")` returns only session A's metrics
- [ ] `ParseSession("legacy-session.jsonl", "a")` continues to work on standalone files without `bench.label` filtering
- [ ] `SessionResult.SessionIDs` contains the correct `session.id` UUIDs for each session
- [ ] Retry invocations (multiple `runClaude()` calls for the same label) all aggregate under the same label
- [ ] `make build` succeeds
- [ ] `go test ./internal/bench/...` passes

## Validation Proofs

- `mindspec bench run --spec-id <id> --prompt "..." --skip-qualitative --skip-commit --skip-cleanup`: produces single `bench-events.jsonl` with events from all sessions
- `grep bench.label <benchmark-dir>/bench-events.jsonl | python3 -c "import json,sys; labels=set(); [labels.add(json.loads(l).get('resource',{}).get('bench.label','?')) for l in sys.stdin]; print(labels)"`: outputs `{'a', 'b', 'c'}`
- Quantitative report shows non-zero token/cost/duration for each session
- `make test`: all tests pass

## Open Questions

*None ‚Äî all design decisions are resolved.*

- `bench.label` via `OTEL_RESOURCE_ATTRIBUTES` confirmed by ADR-0009 and collector code review
- Collector already extracts resource attributes into `CollectedEvent.Resource` ‚Äî no collector changes needed
- Port choice: avoid 4319 (recording) and 4318 (AgentMind live) ‚Äî use a distinct port (e.g., 4320) or let the OS assign one
