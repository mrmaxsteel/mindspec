# ADR-0007: Parallel Workstreams via Per-Worktree State

- **Date**: 2026-02-12
- **Status**: Proposed
- **Domain(s)**: workflow, git
- **Deciders**: MindSpec maintainers
- **Supersedes**: n/a (amends ADR-0005)
- **Superseded-by**: n/a

---

## Context

ADR-0005 established `.mindspec/state.json` as the single source of truth for current mode and active work. The schema supports exactly one active workstream:

```json
{
  "mode": "implement",
  "activeSpec": "008-worktree-lifecycle",
  "activeBead": "beads-xxx",
  "lastUpdated": "..."
}
```

This works for strictly serial workflows: one spec → one plan → one set of implementation beads, completed before starting the next spec.

However, real workflows involve parallelism at multiple levels:

1. **Parallel implementation beads** — already supported via worktrees (one worktree per bead, each isolated). But the state file can only track one `activeBead`.
2. **Parallel specs** — a user may want to draft spec 008 while spec 007 is still being implemented. The state file cannot track two `activeSpec` values.
3. **Parallel spec + plan** — speccing one feature while planning another. Same limitation.

The root problem: **state is project-level (one file on main) when it should be workstream-level (one state per active branch of work).**

Git worktrees already solve filesystem isolation for parallel work. The missing piece is making state follow the worktree — each worktree knows what it's working on, independently.

### Why not store state in Beads?

Beads provides `bd kv` (persistent key-value storage) and shared state across worktrees via redirect files. It was considered as an alternative to `.mindspec/state.json`. However:

- **Beads state is shared across worktrees by design** — all worktrees see the same `.beads/` database via redirects. This is correct for issue tracking (all worktrees should see the same issues), but wrong for mode tracking. If worktree A is implementing bead X and worktree B is implementing bead Y, a shared `mindspec.mode` key can't represent both. Per-worktree keys (e.g., `mindspec.worktree-beads-xxx.activeBead`) would fight Beads' shared-state model rather than work with it.
- **ADR-0003 draws a clear boundary** — MindSpec owns mode semantics and orchestration; Beads owns execution tracking. Mode is orchestration state, not tracking state. The orchestrator should own its own state.
- **Hard dependency** — storing mode in Beads means MindSpec can't function without Beads. Currently `state.json` works even during bootstrap or if Beads breaks.

The right separation: MindSpec's `state.json` tracks "what am I doing" (orchestration). Beads tracks "what work exists" (issues, dependencies, status). These are complementary, not redundant.

## Decision

Each git worktree maintains its own `.mindspec/state.json` on its own branch. State travels with the worktree, not with `main`. MindSpec's orchestration state remains in `.mindspec/state.json` (not in Beads).

**How it works:**

- **Main worktree**: `state.json` reflects idle or the last completed work. This is the "project-level bookmark" described in ADR-0005.
- **Spec/plan worktree** (e.g., `worktree-spec-009`): `state.json` = `{mode: "spec", activeSpec: "009-..."}`. Lives on branch `spec/009-...`.
- **Implementation worktree** (e.g., `worktree-beads-xxx`): `state.json` = `{mode: "implement", activeSpec: "008-...", activeBead: "beads-xxx"}`. Lives on branch `bead/beads-xxx`.
- **On merge to main**: The branch's state.json merges in. Since only one branch completes at a time, the merge updates main's state to reflect the latest completed phase.

**Serial workflows are unchanged**: If you only ever work on one thing at a time, state.json on main tracks it exactly as ADR-0005 describes. Worktrees are optional.

**Parallel workflows are enabled**: Multiple worktrees, each with their own state, working independently. `mindspec instruct` in each worktree emits guidance for that worktree's active work.

### Relationship to Beads worktree support

Beads has first-class worktree support (`bd worktree create/list/remove/info`). Beads worktrees set up a `.beads/redirect` file so all worktrees share the same issue database. MindSpec's worktree tooling should use `bd worktree create` (not raw `git worktree add`) to ensure Beads infrastructure is set up correctly, then layer MindSpec-specific concerns (state file, naming conventions, bead status validation) on top.

This preserves the intended layering:
- **Beads**: shared issue database across worktrees (via redirect)
- **MindSpec**: per-worktree orchestration state (via branch-local `state.json`)

## Decision Details

### A) State File Remains Committed

Per ADR-0005, state.json is committed to git. In the per-worktree model, each branch has its own committed state. This preserves session continuity: checking out any branch immediately tells `mindspec instruct` what mode and work are active.

### B) Merge Conflict Resolution

When merging a branch to main, state.json may conflict if main's state was updated by a different branch. Resolution is trivial: the merging branch's state wins (it represents the latest completed work). This can be automated via a `.gitattributes` merge driver or documented as a convention.

### C) Branch Naming Conventions

Extending the existing `bead/<bead-id>` convention to all phases:

- Spec phase: `spec/<spec-id>`
- Plan phase: `plan/<spec-id>`
- Implementation phase: `bead/<bead-id>` (already established)

These conventions are used by worktree tooling to identify what each worktree is for.

### D) Worktree Naming Conventions

Extending the existing `worktree-<bead-id>` convention:

- Spec worktree: `worktree-spec-<spec-id>`
- Plan worktree: `worktree-plan-<spec-id>`
- Implementation worktree: `worktree-<bead-id>` (already established)

### E) Workflow Command Implications

- `mindspec next` creates implementation worktrees (via `bd worktree create` + MindSpec state setup). Sets state in the worktree, not on main.
- A future `mindspec spec-start` or extended `/spec-init` could create a spec worktree for parallel spec work.
- `mindspec instruct` reads state from the **current worktree's** `.mindspec/state.json`. This already works since it reads from the project root.
- `mindspec complete` cleans up the worktree and resets state on main.

### F) Beads Daemon Considerations

Beads' daemon does not work correctly with git worktrees (it may commit to the wrong branch). Worktrees should use `--no-daemon` mode or the sync-branch feature. MindSpec tooling should either set `BEADS_NO_DAEMON=true` in worktrees or document this requirement.

### G) Backward Compatibility

Serial workflows (no worktrees for spec/plan) continue to work identically to ADR-0005. Per-worktree state is an extension, not a replacement.

## Consequences

### Positive

- **Parallel spec/plan work becomes possible** — each workstream is isolated with its own state
- **State follows the work** — checking out a branch or entering a worktree automatically provides the right context
- **No schema change** — same `state.json` format, just one per worktree/branch
- **Backward compatible** — serial workflows unchanged
- **Aligns with git model** — each branch is an independent line of work with its own state
- **Clean layering with Beads** — Beads owns shared issue tracking; MindSpec owns per-worktree orchestration state

### Negative / Tradeoffs

- **Merge conflicts on state.json** — trivial to resolve but still friction. Could be automated.
- **State drift** — a long-lived branch's state may not reflect changes made on main. Mitigated by `mindspec instruct`'s cross-validation.
- **Complexity** — developers must understand that state is per-worktree, not global. `mindspec state show` in different worktrees shows different state.
- **Depends on worktree adoption** — parallel spec/plan work requires creating worktrees (or at least branches), which is more ceremony than working directly on main.
- **Beads daemon limitation** — worktrees require `--no-daemon` or sync-branch configuration.

## Alternatives Considered

### 1. Store state in Beads (`bd kv`)

Use Beads' key-value store for MindSpec mode/spec/bead tracking.

- Pros: Single system; no merge conflicts; leverages existing Beads infrastructure
- Cons: Shared across worktrees by design (wrong model for per-worktree mode tracking); blurs orchestrator/substrate boundary (ADR-0003); hard dependency on Beads availability; per-worktree keys would fight Beads' shared-state design

### 2. Multi-item state file

Extend `state.json` to track multiple active workstreams:

```json
{
  "workstreams": [
    {"mode": "spec", "activeSpec": "009-...", "activeBead": ""},
    {"mode": "implement", "activeSpec": "008-...", "activeBead": "beads-xxx"}
  ]
}
```

- Pros: Single file, no worktree requirement for parallelism
- Cons: Complex schema; `mindspec instruct` must know which workstream is "current"; conflicts when two developers edit simultaneously; doesn't provide filesystem isolation

### 3. Keep state serial, only parallelize at implementation level

Accept that only one `activeSpec` exists at a time. Parallel implementation beads share the same `activeSpec` but have different worktrees. Spec/plan work is strictly serial.

- Pros: Simplest model; no state changes needed
- Cons: Cannot spec two features simultaneously; `activeBead` still only tracks one bead even when multiple are in-progress

### 4. Gitignore state file (per-developer state)

Remove state.json from git. Each developer's state is local.

- Pros: No merge conflicts; each developer tracks their own work
- Cons: Loses session continuity (fresh clones start blank); per ADR-0005, state is project-level, not personal

## Validation / Rollout

1. Update ADR-0005 to note the per-worktree extension
2. Extend worktree tooling (Spec 008) to use `bd worktree create` and set per-worktree state
3. Update `mindspec instruct` to handle per-worktree state correctly (likely already works since it reads from project root)
4. Document branch naming conventions in CONVENTIONS.md
5. Document Beads daemon workaround (`--no-daemon` in worktrees)
6. Test parallel spec work: two worktrees, two specs, independent `mindspec instruct` output
