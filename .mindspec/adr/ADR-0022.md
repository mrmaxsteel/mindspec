# ADR-0022: Worktree-Aware Spec Resolution and Context Switching

- **Date**: 2026-03-01
- **Status**: Proposed
- **Domain(s)**: workflow, git, workspace
- **Deciders**: Max
- **Supersedes**: n/a
- **Superseded-by**: n/a
- **Related**: ADR-0006 (branch topology), ADR-0019 (enforcement layers)

---

## Context

ADR-0006 establishes that all work happens on branches in worktrees, with spec artifacts (lifecycle.yaml, spec.md, plan.md) living in the spec worktree. ADR-0019 enforces that agents cannot write to main.

However, neither ADR addresses how code **finds** spec artifacts at runtime. Currently, `workspace.FindRoot()` always resolves to the main repo root (by following git worktree links back to the common dir). Functions like `ActiveSpecs()`, `ResolveActiveBead()`, and `advanceState()` then look for lifecycle.yaml under the main repo's docs directory ã where it doesn't exist during implementation, because it was written to the spec worktree.

This causes `mindspec complete` and other commands to fail with "no active specs found" when the agent's CWD is anywhere other than the bead worktree. The bug (mindspec-ofp0) was patched with per-callsite workarounds, but the root issue is systemic: every function that reads spec files needs to independently know about worktrees.

Additionally, there are no rules for **context switching** ã what happens when you encounter unrelated work (a bug, a different feature) while in the middle of a spec lifecycle. The branching point (main vs spec branch) depends on where the issue lives.

## Decision

### 1. Spec artifact resolution is worktree-aware by default

**`workspace.SpecDir(root, specID)` checks the spec worktree before the main repo.**

Resolution order:
1. `root/.worktrees/worktree-spec-<specID>/.mindspec/specs/<specID>/` ã if it exists
2. `root/.mindspec/specs/<specID>/` ã fallback (main repo)
3. `root/docs/specs/<specID>/` ã legacy fallback

This means every caller of `SpecDir` ã including `ActiveSpecs`, `ResolveActiveBead`, `advanceState`, `ReadLifecycle`, context-pack builder ã automatically finds spec files regardless of which worktree the process is running from, without per-callsite workaround code.

`FindRoot()` continues to resolve to the main repo root. This is correct ã the main root is the stable anchor for configuration, worktree paths, and git operations. Only spec *artifact* paths are worktree-aware.

### 2. Focus file as authoritative fallback for active state

When filesystem scanning (lifecycle.yaml) finds nothing, the focus file (`.mindspec/focus`) serves as an authoritative fallback for:
- **Active spec**: `focus.activeSpec`
- **Active bead**: `focus.activeBead`
- **Active worktree**: `focus.activeWorktree`

This is not a new data source ã focus already exists and is written by `mindspec next`, `mindspec complete`, and mode transitions. The change is that `ResolveTarget` and bead resolution functions use it as a fallback instead of failing.

### 3. Context switching: branch from the right base

All work happens in worktrees. When you need to switch context, the branching point depends on where the issue lives:

| Situation | Branch from | Worktree name | Example |
|-----------|-------------|---------------|---------|
| New spec | `main` | `worktree-spec-NNN-slug` | Normal spec lifecycle |
| Bead within a spec | `spec/NNN-slug` | `worktree-<bead-id>` | Implementation bead |
| Bug on main (found while on spec branch) | `main` | `worktree-bugfix-<slug>` | Production bug unrelated to current spec |
| Bug in spec implementation (not on main yet) | `spec/NNN-slug` | `worktree-bugfix-<slug>` | Bug in current spec's code |
| Infrastructure/tooling fix | `main` | `worktree-bugfix-<slug>` | CI, build, tooling changes |

**Rule**: if the bug/issue exists on main (or will need to be on main independently of the current spec), branch from main. If it only exists on the spec branch, branch from the spec branch.

**Returning to previous work**: after the detour, switch back to the original worktree. The original worktree's state is preserved ã nothing was modified because the detour happened in a separate worktree.

### 4. The worktree invariants

1. **Main is read-only** ã no direct commits, no file writes, no CWD for editing. Only receives merges.
2. **One worktree per unit of work** ã specs, beads, and bugfixes each get their own worktree.
3. **Worktrees are disposable** ã created by `mindspec next`/`spec-init`/`git worktree add`, removed by `mindspec complete`/`cleanup`. No precious state lives only in a worktree.
4. **Context switch = worktree switch** ã not branch switch within the same directory. Each worktree has its own filesystem state.
5. **Focus tracks the current context** ã `.mindspec/focus` on main records which worktree/spec/bead is active. Updated on every transition.

## Decision Details

### A) SpecDir resolution implementation

```go
func SpecDir(root, specID string) string {
    // Check spec worktree first
    wtPath := filepath.Join(root, ".worktrees", "worktree-spec-"+specID)
    wtSpecDir := filepath.Join(wtPath, ".mindspec", "docs", "specs", specID)
    if exists(wtSpecDir) {
        return wtSpecDir
    }
    // Canonical path in main repo
    canonical := filepath.Join(root, ".mindspec", "docs", "specs", specID)
    if exists(canonical) {
        return canonical
    }
    // Legacy path
    return filepath.Join(root, "docs", "specs", specID)
}
```

This replaces `EffectiveSpecRoot` + manual path construction. Callers just use `SpecDir(root, specID)` and get the right path.

### B) ActiveSpecs becomes worktree-aware

`ActiveSpecs(root)` scans both:
1. `root/.mindspec/specs/*/lifecycle.yaml` (main repo)
2. `root/.worktrees/worktree-spec-*/` directories, reading lifecycle.yaml from each

This handles the case where specs are active in worktrees but haven't been merged to main yet. Deduplication by specID ensures no double-counting.

### C) DocsDir stays root-relative

`DocsDir(root)` continues to return the main repo's docs dir. It doesn't take a specID, so it can't be worktree-aware. Functions that need worktree awareness use `SpecDir(root, specID)` instead.

### D) Write paths are explicit

Worktree-aware resolution only applies to **reads**. Functions that write spec files (like `specinit.Run()`) already have the correct worktree path as their `root` argument. The resolution logic doesn't affect writes.

## Consequences

### Positive

- **Eliminates a class of bugs** ã any function reading spec artifacts works regardless of CWD
- **Removes per-callsite workarounds** ã no more `EffectiveSpecRoot` sprinkled through `ResolveActiveBead`, `advanceState`, etc.
- **Context switching is principled** ã clear rules for when to branch from main vs spec branch
- **Focus as fallback is robust** ã focus is already maintained; using it as fallback adds resilience without new data

### Negative / Tradeoffs

- **SpecDir does filesystem I/O** ã checking worktree existence adds stat calls. Negligible for CLI commands.
- **ActiveSpecs scans worktree dirs** ã slightly more I/O than scanning one directory. Still fast.
- **Implicit resolution can surprise** ã a function might read from a worktree without the caller realizing. Mitigated by: this is the desired behavior (find the truth wherever it lives).

## Alternatives Considered

### 1. Keep per-callsite EffectiveSpecRoot (status quo after bugfix)

Each function that reads spec files calls `EffectiveSpecRoot` independently.

- Pros: Explicit; each callsite controls resolution
- Cons: Every new callsite must remember to use it; already caused a P1 bug when callers didn't; duplicated path construction logic
- Rejected: the pattern is a maintenance burden and bug magnet.

### 2. Make FindRoot return the spec worktree root

When CWD is in a spec worktree, `FindRoot` would return the worktree path instead of resolving to main.

- Pros: All path functions automatically use the worktree
- Cons: Breaks callers that need the main root (config, git operations, worktree management); `root` becomes ambiguous
- Rejected: `root` must be the stable main repo anchor.

### 3. Pass both mainRoot and specRoot through the call chain

Functions receive two root paths: one for configuration/git, one for spec artifacts.

- Pros: Explicit; no implicit resolution
- Cons: Doubles the root parameter everywhere; major refactor; callers must decide which root to use
- Rejected: too much ceremony for the benefit.

## Future Work

### `mindspec switch` command

Git does not auto-switch CWD when a branch is already checked out in a worktree ã it simply refuses the checkout. There is no built-in mechanism to resolve a branch name to its worktree path and `cd` there.

A `mindspec switch <spec-id|bead-id|branch>` command would:

1. Look up which worktree has that branch (via `git worktree list`)
2. Print `cd /path/to/worktree` (or create the worktree if it doesn't exist yet)
3. Update focus to track the new active context

This would formalize the context-switching rules from this ADR into a single command, replacing manual `cd` + focus management. Shell aliases or IDE integrations could wrap this for an even smoother experience.
