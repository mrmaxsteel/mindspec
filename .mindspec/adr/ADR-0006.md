# ADR-0006: Protected Main Branch with PR-Based Merging

- **Date**: 2026-02-12
- **Status**: Accepted
- **Domain(s)**: workflow, git
- **Deciders**: Max
- **Supersedes**: n/a
- **Superseded-by**: n/a

## Context

> **Note**: This section describes the state of the world *before* this decision was accepted. The workflow described below is what this ADR replaces.

MindSpec's original workflow assumed direct commits to `main` at every mode transition:

- **Spec exit**: `spec(<bead-id>): ...` committed directly to main
- **Plan exit**: `plan(<bead-id>): ...` committed directly to main
- **Implementation exit**: `impl(<bead-id>): ...` committed directly to main

This is documented in AGENTS.md (Milestone Commits), MODES.md (Exit Gates), and the instruction templates emitted by `mindspec instruct`.

Direct-to-main works well for single-developer, local-first workflows. However, it has limitations:

- **No review gate**: Milestone commits land without review. For teams or when working with AI agents, a review step before merging to main provides an additional safety net.
- **No rollback unit**: Individual commits on main are harder to revert cleanly than merged PRs, which revert as a single unit.
- **No CI gate**: Branch protection enables required status checks (tests, linting, validation) before code reaches main.
- **Spec/plan phases unprotected**: Even markdown-only spec and plan changes land on main without review, which matters when specs define scope and acceptance criteria that govern subsequent implementation.

A decision is needed on whether MindSpec should adopt a protected-main model where all phases use branches merged via pull request.

## Decision

**Zero changes on main. All work happens on branches in worktrees. One PR per spec lifecycle.**

Every mindspec-managed change — spec, plan, and implementation — happens on a branch in a dedicated worktree. Nothing is committed directly to main. Changes reach main only through a merge (PR when a remote is configured, local merge otherwise).

### Branch topology

The spec/plan branch serves as the integration branch for the entire spec lifecycle:

```
main ──────────────────────────────────────────────────●──────
                                                      ↑ single PR
spec/NNN-slug ────●────────────────●──────●──────●────●
  (spec, plan)    ↑               ↑      ↑      ↑
                  │    bead/xxx ──●      │      │
                  │                      │      │
                  │    bead/yyy ─────────●      │
                  │                             │
                  │    bead/zzz ────────────────●
```

1. **`mindspec spec-init`** creates a `spec/NNN-slug` branch and worktree. All spec and plan work happens here.
2. **`/plan-approve`** approves the plan. The spec/plan worktree stays open — it becomes the integration point for implementation beads.
3. **`mindspec next`** creates `bead/<bead-id>` branches from `spec/NNN-slug` (not from main). Each impl bead gets its own worktree.
4. **`mindspec complete`** merges each impl bead back to `spec/NNN-slug` (not to main).
5. **When all beads are done**, a single PR merges `spec/NNN-slug` to main. This PR contains the complete body of work: spec, plan, and all implementation.

### Merge strategy (configurable)

| Strategy | Behavior | Use case |
|----------|----------|----------|
| `pr` | Push branch, create PR, wait for CI/review | Teams, production repos |
| `direct` | Local branch merge to main | Solo dev, local-only, no remote |
| `auto` | PR if remote exists, direct merge if local-only | Default |

### Enforcement

Enforcement is deterministic, not prompt-based. See ADR-0019 for the three-layer enforcement mechanism (git hooks, CLI guards, agent hooks).

## Decision Details

### A) Spec/plan share a worktree

Spec and plan phases are sequential (never parallel within a spec) and edit the same directory (`docs/specs/NNN/`). They share a single `spec/NNN-slug` branch and worktree. This avoids branch churn between spec and plan phases.

### B) Branch naming conventions

- Spec+plan phase: `spec/<spec-id>` (e.g., `spec/046-worktree-enforcement`)
- Implementation phase: `bead/<bead-id>` (e.g., `bead/mindspec-xxx`)

### C) Impl beads branch from spec, not main

Implementation beads branch from `spec/NNN-slug` because:

- **Zero wait time** — impl beads start immediately after plan approval, no waiting for a spec/plan PR to merge
- **One PR per spec** — the single `spec/NNN-slug → main` PR contains the full body of work (spec, plan, all impl). Easier to review as a coherent unit.
- **Atomic rollback** — revert one PR to undo an entire spec's worth of work
- **Natural scoping** — the PR maps 1:1 to the spec

### D) Per-bead PRs (optional)

Teams that want per-bead CI/review can configure per-bead PRs against the spec branch (`bead/xxx → spec/NNN-slug`). This is opt-in — the default is direct merge of beads back to the spec branch.

### E) Worktree lifecycle

| Event | Worktree | Branch |
|-------|----------|--------|
| `spec-init` | Create `worktree-spec-NNN-slug` | `spec/NNN-slug` from main |
| `/spec-approve` | Keep worktree | Same branch |
| Plan work | Keep worktree | Same branch |
| `/plan-approve` | Keep worktree open | Same branch |
| `mindspec next` (per bead) | Create `worktree-bead-xxx` | `bead/xxx` from `spec/NNN-slug` |
| `mindspec complete` (per bead) | Remove bead worktree | Merge `bead/xxx` → `spec/NNN-slug` |
| `/impl-approve` (all beads done) | Remove spec worktree | PR or merge `spec/NNN-slug` → main |

### F) State file management

`.mindspec/state.json` lives on the branch, not on main. Each worktree has its own state reflecting its active work. State on main only updates when a spec lifecycle merges.

## Consequences

### Positive

- **Zero changes on main** — main only changes via controlled merges
- **One PR per spec** — coherent unit of review containing spec, plan, and all implementation
- **Atomic rollback** — revert one PR to undo an entire spec's worth of work
- **No dead time** — impl beads start immediately after plan approval (branch from spec, not main)
- **CI integration** — branch protection enables required status checks before anything reaches main
- **Team-ready** — scales to multi-developer workflows
- **Agent safety net** — AI-generated specs, plans, and code are all reviewed before landing

### Negative / Tradeoffs

- **Long-lived spec branches** — `spec/NNN-slug` lives from spec-init through final impl-approve. May diverge from main. Mitigated by periodic rebases.
- **Large PRs** — the final PR contains everything. For large specs this could be a big diff. Mitigated by per-bead PR option for incremental review.
- **Merge conflicts** — impl beads merging back to spec branch may conflict with each other. Mitigated by: beads are scoped to avoid overlap (that's the point of bead scoping in plans).
- **Ceremony** — even solo developers must create a worktree before making any change. Mitigated by: `mindspec spec-init` and `mindspec next` handle this automatically.
- **Local-only limitation** — PR strategy requires a remote. `direct` and `auto` strategies handle local-only gracefully.

## Alternatives Considered

### 1. Keep direct-to-main (status quo)

Simple, fast, works well for single-developer local-first workflows. No ceremony. The tradeoff is no review gate and no CI enforcement on main.

### 2. Protect main for implementation only

Spec/plan phases stay direct-to-main (markdown-only, low risk). Only implementation creates PRs.

- Pros: Less overhead for spec/plan; code changes get review
- Cons: Inconsistent model; spec mistakes still land on main without review; agents still make changes on main during spec/plan phases
- Rejected: "zero changes on main" is a cleaner invariant than "some changes on main are OK"

### 3. Per-phase PRs (spec PR + plan PR + N impl PRs)

Each phase creates its own PR to main.

- Pros: Incremental review; smaller PRs
- Cons: 4+ PRs per spec; dead time waiting for spec/plan PR to merge before starting impl; ceremony overhead
- Rejected: one-PR-per-spec is simpler and the spec branch as integration point eliminates dead time

### 4. Protect main with squash merging

Same as the full proposal but squash-merge each phase into a single commit on main.

- Pros: Clean main history
- Cons: Loses per-commit granularity within the spec lifecycle

### 5. Git hooks only (no PR workflow)

Use pre-push hooks to run validation before allowing pushes to main.

- Pros: Simpler; no remote required
- Cons: No human review step; hooks can be bypassed

## Validation / Rollout

1. Update MODES.md exit gates to reflect branch-based workflow
2. Update AGENTS.md commit conventions with branch naming
3. Update instruction templates (`spec.md`, `plan.md`, `implement.md`) for worktree-based workflow
4. Extend `mindspec spec-init` to create spec branch + worktree
5. Extend `mindspec next` to branch from spec branch (not main)
6. Extend `mindspec complete` to merge bead → spec branch (not main)
7. Add `/impl-approve` step to create final PR (`spec/NNN-slug → main`)
8. Add `.mindspec/config.yaml` with `merge_strategy` setting
9. Dogfood on the next spec cycle
