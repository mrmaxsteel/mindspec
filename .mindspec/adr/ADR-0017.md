# ADR-0017: Agent Onboarding Architecture — Prompt Emission, Init/Setup Separation, and AGENTS.md Layering

- **Date**: 2026-02-21
- **Status**: Accepted
- **Domain(s)**: workflow, agent-integration
- **Deciders**: Max
- **Supersedes**: n/a
- **Superseded-by**: n/a

---

## Context

MindSpec needs to onboard new and existing repositories into its spec-driven workflow. This involves two distinct concerns:

1. **Documentation migration** — reorganizing existing markdown files into the canonical `.mindspec/docs/` structure.
2. **Agent bootstrapping** — configuring coding agents (Claude Code, Copilot CLI, Codex) with hooks, slash commands, and instruction files so they understand and enforce the MindSpec workflow.

The original `mindspec migrate` was ~700 lines across `internal/brownfield/` (6 files), implementing deterministic path classification, LLM subprocess calls for ambiguous docs, transactional staging, archiving, and lineage tracking. This was over-engineered: the coding agent running `mindspec migrate` can already read, move, and merge files itself.

For agent bootstrapping, the question is how to handle the diversity of agent tooling. Claude Code uses `.claude/settings.json` for hooks, `.claude/commands/*.md` for slash commands, and `CLAUDE.md` for instruction. Copilot CLI has its own hook format. Codex reads `AGENTS.md` at startup. Each has different extensibility models and lifecycles.

## Decision

### 1. Migrate emits a prompt, not code

`mindspec migrate` scans the repository for markdown files and emits a structured prompt to stdout instructing the agent to reorganize them. It does not move, copy, merge, or archive any files. The prompt includes the canonical directory structure, a category rubric, discovered source files, existing canonical files, and step-by-step instructions.

This aligns with Goal #8 (CLI-first, minimal IDE glue): delegate work the agent can do natively rather than reimplementing it in Go. The agent is better at content-aware classification than deterministic path matching, and the prompt approach requires no transactional rollback logic.

### 2. Init and setup are separate commands with distinct lifecycles

`mindspec init` creates the project structure (`.mindspec/` directories, glossary, context map, policies, state) and the universal `AGENTS.md` instruction file. It runs once per repository.

`mindspec setup <agent>` creates agent-specific configuration. `mindspec setup claude` creates `.claude/settings.json` (hooks), `.claude/commands/*.md` (slash commands), and `CLAUDE.md` (with pointer to AGENTS.md). It can be re-run after MindSpec upgrades to pick up new hooks or commands, and supports `--check` for dry-run inspection.

The separation exists because:
- **Different lifecycle**: init runs once at project creation; setup may run multiple times as MindSpec or agent tooling evolves.
- **Different scope**: a team may use multiple agents (Claude + Copilot) and run setup for each independently.
- **Re-runnability**: setup is idempotent by design — it merges new hooks into existing settings, skips existing command files, and uses markers for instruction file appends.

### 3. AGENTS.md is the universal instruction layer

`AGENTS.md` is the cross-agent instruction file, read by Codex, Copilot, Cursor, Amp, and Jules at startup. `mindspec init` creates it with the full MindSpec workflow guidance (modes, conventions, build/test commands). If `AGENTS.md` already exists, init appends a MindSpec block using a `<!-- mindspec:managed -->` marker for idempotency.

Agent-specific files (e.g., `CLAUDE.md`) contain a pointer to `AGENTS.md` for shared conventions, plus agent-specific content (slash command table, hook documentation). This avoids duplicating workflow guidance across multiple instruction files.

### 4. Idempotent append via managed marker

Both `mindspec init` and `mindspec setup` use the `<!-- mindspec:managed -->` HTML comment as an idempotency marker when appending to existing files (`AGENTS.md`, `CLAUDE.md`). If the marker is present, the append is skipped. If absent, the MindSpec block (prefixed with the marker) is appended to the end of the file.

This pattern ensures:
- First run adds the block.
- Subsequent runs are no-ops.
- Users can edit the file freely above the marker without interference.
- The marker is invisible in rendered markdown.

## Consequences

- **Simpler codebase** — ~2,900 lines of brownfield code deleted, replaced by ~200 lines of prompt emission + ~300 lines of setup logic.
- **Better classification** — the agent uses content-aware judgment rather than path-based heuristics, with no LLM subprocess overhead.
- **Multi-agent support** — adding `mindspec setup copilot` or `mindspec setup codex` in the future is straightforward: implement a new setup function with that agent's configuration format.
- **No migration state** — there is no staging directory, transaction log, or lineage tracking. The agent handles file operations directly, and git history serves as the audit trail.
- **Settings merge complexity** — `mindspec setup claude` must merge hooks into an existing `settings.json` without clobbering user-added hooks. This is handled by matcher-based deduplication (if a hook with the same matcher exists, it's skipped).
- **Marker fragility** — if a user manually deletes the `<!-- mindspec:managed -->` marker but keeps the MindSpec content, a subsequent run will append a duplicate block. This is an acceptable edge case given the simplicity of the approach.
