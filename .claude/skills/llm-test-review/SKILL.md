# LLM Test Scenario Review

You are reviewing an LLM test scenario for design quality. Read the scenario definition in `internal/harness/scenario.go` and answer each section below. Present your findings to the user as a structured report.

## 1. Spirit of the Test

What is this test intended to validate? State the core behavior being tested in one sentence. Is it testing:
- A lifecycle transition (e.g., plan → implement)?
- Error recovery (e.g., stale worktree, crash resume)?
- Edge case handling (e.g., multi-spec disambiguation)?
- Guidance adherence (e.g., branch policy, no raw git)?

## 2. Setup Completeness Audit

Read the `Setup` function and check every layer of sandbox state. For each item, state whether it is present, missing, or N/A for this scenario:

- **Beads state**: Is there a spec epic? Bead issues? Are they in the right status (open/in_progress/closed)? Are dependencies set if needed?
- **MindSpec focus**: Is `.mindspec/focus` written? Does it have the correct mode, activeSpec, specBranch, activeWorktree, activeBead fields for the scenario's starting state?
- **Git branches**: Are spec/ and bead/ branches created? Do they branch from the right parents (main → spec → bead)?
- **Worktrees**: Are worktrees created when the scenario expects them? Are they deliberately absent when testing stale/missing worktree recovery?
- **Spec artifacts**: Is there a valid `spec.md` with correct YAML frontmatter (title, status)? Is it in the right directory (`.mindspec/specs/<id>/`)?
- **Plan artifacts**: Is there a valid `plan.md` with correct YAML frontmatter (status, spec_id, version)? Does it pass `ValidatePlan` if the scenario starts post-plan-approval?
- **File contents**: Are any source files pre-created that the agent is expected to find or modify?
- **Git state**: Are setup files committed? Is the working tree clean when the agent starts? Are commits on the right branches?
- **CWD**: Where does the agent start — main repo root, spec worktree, or bead worktree? Is this correct for the scenario's starting mode?
- **Claude hooks**: SessionStart and PreToolUse hooks are installed by `setup.RunClaude()` — is this happening? Does the scenario need `agent_hooks: true/false`?
- **CLAUDE.md**: Is it generated by `setup.RunClaude()`? Does it contain the right instruct references?

Flag any inconsistencies where the setup state doesn't match what `mindspec instruct` would emit for the scenario's starting mode.

## 3. Prompt Quality

Read the `Prompt` string and evaluate:

- **Alignment with spirit**: Does the prompt test what the scenario claims to test, or does it test something else?
- **Tautology check**: Does the prompt tell the agent exactly which commands to run? If so, the test validates prompt-following, not product quality. The prompt should describe the *task* and *exit criteria*, not the *commands*.
- **Initial state leakage**: Does the prompt over-describe the starting state (e.g., "you are in implement mode with bead X claimed")? The agent should infer this from `mindspec instruct` output, not from the prompt. Some context is OK (e.g., "you are resuming work"), but specific state details should come from the tool, not the prompt.
- **Exit criteria**: Does the prompt tell the agent when to stop? It should specify the desired end-state (e.g., "finish the bead through the MindSpec lifecycle") without naming specific commands.
- **Imperative tone**: Haiku needs direct instructions. Is the prompt imperative enough to prevent conversational responses, without being tautologically prescriptive?
- **Scope**: Is the prompt asking for too much (agent runs out of turns) or too little (trivial to pass without exercising the workflow)?

## 4. Expected Happy Path

Describe the ideal sequence of agent actions, step by step:
1. What should the agent do first? (e.g., read CLAUDE.md, run `mindspec instruct`)
2. What lifecycle commands should it discover and run?
3. What files should it create/modify?
4. How should it finish? (e.g., `mindspec complete`, `mindspec approve impl`)

Note any alternative valid paths (e.g., `bd close` as acceptable recovery).

## 5. Assertions Audit

Read the `Assertions` function and evaluate:

- **Coverage**: Do the assertions check the things that matter for the spirit of the test? Are there important outcomes that aren't asserted?
- **Strictness**: Are assertions too strict (rejecting valid alternative paths) or too lenient (passing when the agent didn't actually exercise the workflow)?
- **Tiered assertions**: If there are fallback/recovery assertion paths, are they clearly ordered from preferred to acceptable?
- **File assertions**: If the agent should create files, is `FileExists` or content checked?
- **State assertions**: Is the final mode/focus state checked? Are branches/worktrees cleaned up if expected?
- **Negative assertions**: Are there checks that the agent did NOT do something wrong (e.g., didn't commit to main, didn't skip lifecycle)?

## 6. Recent Results & Failure Modes

Read the scenario's improvement history table in `internal/harness/HISTORY.md`. Analyze the last 3-5 runs:

- **Pass rate**: How often does this test pass? Is it reliable, flaky, or reliably failing?
- **Failure pattern**: When it fails, is it always the same root cause or different each time? Common patterns:
  - Agent never calls the expected lifecycle command (e.g., uses raw git or `bd close` instead of `mindspec complete`)
  - Agent gets stuck retrying a failing command
  - Agent creates the artifact but doesn't finish the lifecycle
  - Agent takes a valid-but-unasserted alternative path
  - Setup issue prevents the agent from starting properly
- **Trends**: Are metrics (turns, events, forward ratio) improving, stable, or regressing across runs?
- **Model sensitivity**: Has this test ever been run with a non-Haiku model? If so, did it behave differently? (Check history for "Sonnet" mentions)
- **Root cause classification**: For recent failures, is the problem in:
  - The **product** (CLI error messages, missing guidance in instruct templates)?
  - The **test** (incomplete setup, wrong assertions, tautological prompt)?
  - The **model** (nondeterministic Haiku behavior — agent understands the task but picks a non-lifecycle path)?

If the test reliably fails with the same root cause, note whether a product fix or test fix is more appropriate (per the fix surface rule: product guidance fixes go in instruct templates/CLI messages, never in test prompts).

## 7. Verdict

Summarize:
- **Setup**: Complete / Has gaps / Needs rework
- **Prompt**: Good / Tautological / Too vague / Needs tuning
- **Assertions**: Strong / Adequate / Too lenient / Too strict
- **Overall**: Ready / Needs minor fixes / Needs redesign

If there are actionable fixes, list them in priority order.
